# 真實 C++ 專案實測：四大 Claude Code 工作流哪個最好用？

網路上介紹各種 Claude Code 工作流的文章很多，但幾乎都是單獨介紹某一個工具，缺少在同一個真實專案上的橫向比較。這篇文章用一個 4000+ 檔案的複雜 C++ 專案，實際跑了 openSpec、gsd、superpower、feature-dev 四套工作流，把結果整理給你看。

## 測試背景與方法

**專案背景**：一個大型 C++ 專案，4000+ 個檔案，涉及靜態圖、動態圖、動態 shape、靜態子圖等複雜邏輯。

**測試方式**：
- 使用同一個真實需求（51 個檔案、+1196 行程式碼）
- 將程式碼回退到需求合入前，複製多份目錄，分別只安裝對應工具
- 所有工具使用**完全相同的輸入**：一個 `spec_input.txt` 需求分析文件

**比較維度**：程式碼品質、Token 消耗、時間消耗、互動次數

> [!note] 關於 gstack
> gstack 是一個 Headless browser 工具，主要用於 Web 應用的 QA 測試，**不適合 C++ 開發場景**，因此本次未納入實際程式碼比較。

## 結果總覽

先上結論表格，後面再細說每個工具的表現：

| 工作流 | Token 消耗 | 時間消耗 | 互動次數 | 程式碼品質 | 特色 |
|--------|-----------|---------|---------|-----------|------|
| **openSpec** | ~10M | ~1h | 很少 | 中（有細節遺漏） | 進入快，但會多次暫停等確認 |
| **gsd** | ~30M | ~3h | 非常多 | 中（同上） | 引導式、有 phase 規劃，但互動負擔重 |
| **superpower** | ~7M | 中等 | 中等 | 較好 | 生成設計文件後再規劃，使用者體驗友善 |
| **feature-dev** | 未記錄 | ~1h | 較少 | 好 | 7 個步驟清晰，自動化程度高 |
| **null（直接 AI）** | ~14M | ~1h | 很少 | 中（漏掉記憶體釋放邏輯） | 會進入 plan 模式，但計畫沒有落盤 |

**一句話結論**：如果追求省心，用 **feature-dev**；如果要最詳細的設計文件，用 **superpower**；如果不在意互動次數、想要最有結構的開發流程，用 **gsd**。

## 各工具詳細表現

### gsd（get-shit-done）⭐ 結構最完整，但互動負擔最重

gsd 建議用 `claude --dangerously-skip-permissions` 啟動，整個流程高度結構化。

**運作方式**：
- 第一次使用會啟動 4 個 agent 探索程式碼庫
- 需求會被拆成多個 **phase（階段）**，本次需求被分成 4 個 phase
- 每個 phase 分為三個子步驟：

```
/gsd:discuss-phase 1
/gsd:plan-phase 1
/gsd:execute-phase 1
```

**優點**：
- 高度引導式，每個動作完成後會告訴你下一步是什麼
- 每個子任務都會自動 commit，本次需求產生了 30+ 個 commit
- plan 階段會生成詳細的 `.planning/phases/XX-CONTEXT.md` 設計文件，細化到程式碼層面

**缺點**：
- 互動次數爆炸：4 個 phase × 3 個步驟 = 至少要手動輸入 12 次指令，每個 discuss 階段還要額外確認
- Token 消耗最多（~30M），耗時最長（~3h）
- Plan 文件包含大量程式碼細節，不容易快速掃讀

> [!warning] 使用體驗注意
> gsd 的 plan-phase 研究階段可能持續 20 分鐘以上，中途如果你去做別的事，它會等你回應。如果遇到 agent 卡住，可以用 `/btw` 詢問狀況，或 `Ctrl+C` 中斷。

---

### superpower ⭐ Token 效率最佳，設計文件品質高

superpower 在所有工具中 **Token 消耗最少（~7M）**，且使用者互動體驗最友善。

**運作方式**：
- 在生成 plan 前，會先產出一份**設計文件**（架構設計、整合點說明）
- 然後再根據設計文件生成執行計畫
- 支援子代理並行執行

**程式碼品質**：
- 正確抓取了 FileConstant 的檔案名稱（其他工具在這裡出錯）
- 部分函式實作（如 `LoadExecutorFromModelData`）略遜於 gsd

**適合場景**：需要留下設計文件、又不想花太多時間在工具互動上的團隊。

---

### feature-dev ⭐ 自動化最高，適合想少說話的人

feature-dev 是 Claude Code 內建的特性開發流程，分為 **7 個固定階段**：

1. 理解需求
2. 探索程式碼庫
3. 釐清問題
4. 架構設計
5. 實作
6. 品質審查
7. 完成

**最大特色**：只有在「釐清問題」階段會詢問你關鍵細節，之後幾乎全自動跑完。

**程式碼品質**：整體品質好，但測試案例的豐富度不如 superpower。

**適合場景**：需求清楚、希望最少人工干預的場景。

---

### openSpec 需求文件驅動，但細節確認不足

openSpec 是一個開源的規格驅動開發（SDD）工具（GitHub 35.4k stars），透過 `proposal.md`、`design.md`、`tasks.md` 等文件驅動開發。

**表現**：
- 進入工作流程快
- Token 消耗中等（~10M）
- **問題**：在取得 FileConstant 檔案名稱的方式上出錯；且對於使用者只提供部分 fileconstant 地址的情況，應該自行申請補齊，但工具選擇直接報錯

這個工具更適合需求文件已經很完整的場景，對需要深度探索程式碼庫的複雜需求支援略顯不足。

---

## 選哪個？一個簡單的判斷流程

```
你的需求清楚嗎？
├── 是 → 想要最少互動？
│        ├── 是 → feature-dev
│        └── 否，想要設計文件 → superpower
└── 否，需要先釐清需求 → gsd（有 discuss phase）
    或 openSpec（透過文件驅動釐清）
```

---

## 下一步

1. **先試 feature-dev**：如果你還沒用過任何工作流，這是入門門檻最低的選擇，輸入一個需求看看它怎麼跑。
2. **Token 消耗是真實成本**：gsd 的 30M token 在實際使用中代表更高的 API 費用和更長的等待時間，選工具時要納入考量。
3. **不同需求配不同工具**：小需求直接 null（無工作流）其實也夠用，只是要記得計畫沒有落盤，context 滿了就麻煩了。
