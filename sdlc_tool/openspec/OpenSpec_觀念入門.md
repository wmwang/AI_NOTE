## 為什麼需要 OpenSpec

### AI 編碼的挑戰

AI 編碼助手功能強大，但當需求只存在於聊天歷史中時，結果會變得難以預測：

```
問題：
┌─────────────────────────────────────────────────────────────┐
│  你說：「幫我加一個深色模式」                       │
│  AI：建立了一些代碼...                               │
│  你：「喔，我想要也支援系統偏好設定」                 │
│  AI：修改了代碼...                                  │
│  你：「那個切換按鈕的位置不對」                     │
│  ...                                               │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  結果：                                             │
│  • 聊天歷史變得很長，很難回溯需求                   │
│  • 沒有正式的規格文件可以參考                     │
│  • 團隊成員無法理解變更的全貌                       │
│  • 代碼和需求可能脫節                               │
└─────────────────────────────────────────────────────────────┘
```

### OpenSpec 的解決方案

OpenSpec 添加了一個輕量級的規格層，讓人類和 AI 在撰寫任何代碼**之前**先達成共識：

```
解決方案：
┌─────────────────────────────────────────────────────────────┐
│  1. 建立變更：明確定義要做什麼                      │
│     └─ proposal.md（為什麼、做什麼）                  │
│                                                      │
│  2. 規格化：描述行為需求                             │
│     └─ specs/（需求 + 情境）                         │
│                                                      │
│  3. 設計：決定技術方案                                │
│     └─ design.md（如何實作）                          │
│                                                      │
│  4. 任務：分解執行步驟                               │
│     └─ tasks.md（檢查清單）                         │
│                                                      │
│  5. 實作：AI 執行具體任務                           │
│     └─ 實際撰寫代碼                                 │
│                                                      │
│  6. 驗證：檢查是否符合規格                           │
│     └─ /opsx:verify                                  │
│                                                      │
│  7. 封存：合併到主要規格                             │
│     └─ /opsx:archive（specs 成為來源真實值）         │
└─────────────────────────────────────────────────────────────┘
```

## 核心哲學

OpenSpec 建立在四個原則之上：

### 1. 流動而非僵化

傳統規格系統將你鎖定在階段中：先規劃、再實作、然後完成。但真實的工作並非如此運作。

```
傳統方式（階段鎖定）：

  規劃 ────────► 實作 ────────► 完成
      │                    │
      │   「無法回退」     │
      └────────────────────┘

OpenSpec 方式（流動動作）：

  proposal ──► specs ──► design ──► tasks ──► implement
      │           ▲          ▲                    │
      └───────────┴──────────┴────────────────────┘
            隨時可以更新任何文件
```

**核心觀念：** 沒有「階段」，只有你可以執行的動作。隨時可以更新任何文件。

### 2. 迭代而非瀑布

需求會變化。理解會加深。一開始看起來不錯的方法，看到代碼後可能行不通。OpenSpec 接納這個現實。

```
瀑布式問題：
┌─────────────────────────────────────────────────────────────┐
│  規劃 → 規格 → 設計 → 實作                        │
│         ▲                                             │
│         └─────「哦，我發現設計有問題」                 │
│              但規劃階段已經結束了！                     │
└─────────────────────────────────────────────────────────────┘

迭代式優勢：
┌─────────────────────────────────────────────────────────────┐
│  proposal ──► specs ──► design ──► tasks ──►       │
│      │                                              │
│      └───「實作時發現設計有問題」                   │
│            無妨，直接更新 design.md 繼續！            │
└─────────────────────────────────────────────────────────────┘
```

### 3. 簡單而非複雜

有些規格框架需要大量設定、嚴格格式或繁重的流程。OpenSpec 不妨礙你工作：

```
複雜的框架：
├─ 需要 Python 環境
├─ 大量的 Markdown 文件
├─ 嚴格的格式驗證
├─ 複雜的設定過程
└─ 學習曲線陡峭

OpenSpec：
├─ npm install -g @fission-ai/openspec
├─ openspec init
├─ /opsx:propose "你的想法"
└─ 完成！
```

### 4. 適合既有專案而非只有全新專案

大多數軟體工作不是從頭開始 — 而是修改現有系統。OpenSpec 的增量方法讓規格變更成為一等公民：

```
新專案（綠地開發）：
描述整個系統 ──► 實作

既有專案（棕地開發）：
描述這次變更什麼 ──► 實作
         ↑
   增量規格的優勢！
```

## 核心概念

### Specs（規格）與 Changes（變更）

```
┌─────────────────────────────────────────────────────────────────┐
│                        openspec/                         │
│                                                          │
│   ┌─────────────────────┐      ┌──────────────────────────┐  │
│   │       specs/        │      │         changes/          │  │
│   │                     │      │                        │  │
│   │  來源真實值        │◄────│  提議的修改            │  │
│   │  系統目前如何運作   │ 合併│  每個變更 = 一個資料夾  │  │
│   │                     │      │  包含文件 + 增量規格     │  │
│   └─────────────────────┘      └──────────────────────────┘  │
│                                                          │
└─────────────────────────────────────────────────────────────────┘
```

- **Specs** 是來源真實值 — 描述系統目前如何運作
- **Changes** 是提議的修改 — 在準備好之前獨立存在

這個分離是關鍵：
- 可以同時處理多個變更而不會衝突
- 可以在變更影響主要規格之前審查
- 封存變更時，增量規格會乾淨地合併

### 文件（Artifacts）

每個變更包含指導工作的文件：

```
┌─────────────────────────────────────────────────────────────────┐
│                   變更資料夾結構                        │
├─────────────────────────────────────────────────────────────────┤
│                                                          │
│  openspec/changes/add-feature/                             │
│  ├── proposal.md      ──► 為什麼、做什麼                 │
│  ├── design.md        ──► 如何、技術方案                   │
│  ├── tasks.md         ──► 步驟、檢查清單                 │
│  └── specs/           ──► 變更內容（增量規格）             │
│                                                          │
└─────────────────────────────────────────────────────────────────┘
```

#### 文件流程

```
proposal ──────► specs ──────► design ──────► tasks ──────► implement
   │               │             │              │
   為什麼          什麼變更      如何實作       執行步驟
 + 範圍          相對規格      技術方案       到寫代碼
```

文件相互構建，每個文件都為下一個提供上下文。

#### Proposal（提案）

```markdown
# 提案：加入深色模式

## 意圖
使用者請求提供深色模式選項，以減少夜間使用時的眼睛疲勞。

## 範圍
**範圍內：**
- 設定頁面中的主題切換
- 系統偏好設定偵測
- localStorage 持久化偏好設定

**範圍外：**
- 自訂顏色主題（未來工作）
- 各頁面獨立主題覆蓋

## 方法
使用 CSS 自訂屬性進行主題化，配合 React context 進行狀態管理。
```

**什麼時候更新提案：**
- 範圍變更（縮小或擴大）
- 意圖更清楚
- 方法根本改變

#### Specs（規格）

規格描述需求，每個需求都有情境：

```markdown
# 驗證規格

## 目的
應用程式的驗證和會話管理。

## 需求

### 需求：使用者驗證
系統必須在成功登入時發行 JWT token。

#### 情境：有效憑證
- 假設擁有有效憑證的使用者
- 當使用者提交登入表單
- 那麼會回傳 JWT token
- 並且使用者被重新導向至儀表板

#### 情境：無效憑證
- 假設無效憑證
- 當使用者提交登入表單
- 那麼會顯示錯誤訊息
- 並且不發行 token
```

**關鍵元素：**

| 元素 | 目的 |
|------|------|
| `## 目的` | 此規格領域的高層次描述 |
| `### 需求：` | 系統必須擁有的特定行為 |
| `#### 情境：` | 需求的具體範例 |
| SHALL/MUST/SHOULD | RFC 2119 關鍵字，表示需求強度 |

#### Design（設計）

```markdown
# 設計：加入深色模式

## 技術方法
主題狀態透過 React Context 管理，避免 prop drilling。
CSS 自訂屬性啟用執行時切換而無需切換 class。

## 架構決策

### 決策：使用 Context 而非 Redux
使用 React Context 管理主題狀態，因為：
- 簡單二元狀態（亮/暗）
- 沒有複雜狀態轉換
- 避免新增 Redux 依賴

### 決策：CSS 自訂屬性
使用 CSS 變數而非 CSS-in-JS，因為：
- 適用於現有樣式表
- 無執行時開銷
- 瀏覽器原生解決方案

## 資料流程
```
ThemeProvider (context)
       │
       ▼
ThemeToggle ◄──► localStorage
       │
       ▼
CSS Variables (applied to :root)
```

## 檔案變更
- `src/contexts/ThemeContext.tsx` (新檔案)
- `src/components/ThemeToggle.tsx` (新檔案)
- `src/styles/globals.css` (修改)
```

#### Tasks（任務）

```markdown
# 任務

## 1. 主題基礎設施
- [ ] 1.1 建立帶有亮/暗狀態的 ThemeContext
- [ ] 1.2 新增顏色的 CSS 自訂屬性
- [ ] 1.3 實作 localStorage 持久化
- [ ] 1.4 新增系統偏好設定偵測

## 2. UI 組件
- [ ] 2.1 建立 ThemeToggle 組件
- [ ] 2.2 將切換器新增至設定頁面
- [ ] 2.3 更新 Header 以包含快速切換器

## 3. 樣式
- [ ] 3.1 定義深色主題調色盤
- [ ] 3.2 更新組件以使用 CSS 變數
- [ ] 3.3 測試對比度是否符合無障礙標準
```

## 增量規格

增量規格是讓 OpenSpec 適合既有專案開發的核心概念。它們描述**什麼正在變更**，而非重述整個規格。

### 格式

```markdown
# Auth 的增量

## 新增的需求

### 需求：兩步驗證
系統必須支援基於 TOTP 的兩步驗證。

#### 情境：2FA 註冊
- 假設未啟用 2FA 的使用者
- 當使用者在設定中啟用 2FA
- 那麼會顯示供驗證器應用程式設定的 QR code
- 並且使用者在啟用前必須用程式碼驗證

#### 情境：2FA 登入
- 假設已啟用 2FA 的使用者
- 當使用者提交有效憑證
- 那麼會呈現 OTP 挑戰
- 並且只有在有效 OTP 時登入完成

## 修改的需求

### 需求：會話過期
系統必須在 15 分鐘不活動後過期會話。
（先前：30 分鐘）

#### 情境：閒置逾時
- 假設已驗證的會話
- 當 15 分鐘過去且無活動
- 那麼會話失效

## 移除的需求

### 需求：記住我
（已被 2FA 取代。使用者應每次會話重新驗證。）
```

### 增量區段

| 區段 | 意義 | 封存時發生什麼 |
|------|------|----------------|
| `## 新增的需求` | 新的行為 | 附加到主要規格 |
| `## 修改的需求` | 改變的行為 | 取代現有需求 |
| `## 移除的需求` | 淘汰的行為 | 從主要規格刪除 |

### 為什麼用增量而非完整規格

**清晰度：** 增量精確顯示什麼在變更。讀完整規格時，你必須心理上與目前版本做 diff。

**避免衝突：** 兩個變更可以觸及同一個規格檔案而不衝突，只要它們修改不同的需求。

**審查效率：** 審查者看到的是變更，而非未變的上下文。專注於重要的部分。

**適合既有專案：** 大多數工作是修改現有行為。增量讓修改成為一等公民，而非事後補充。

## 工作流程模式

### 動作而非階段

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         OPENSPEC 流程                                │
│                                                                      │
│   ┌────────────────┐                                                  │
│   │  1. 開始      │  /opsx:propose (預設) 或 /opsx:new (擴展)        │
│   │     變更      │                                                   │
│   └───────┬────────┘                                                  │
│           │                                                           │
│           ▼                                                           │
│   ┌────────────────┐                                                  │
│   │  2. 建立      │  /opsx:ff 或 /opsx:continue (擴展工作流程)          │
│   │     文件      │  建立 proposal → specs → design → tasks                │
│   │                │  (基於 schema 依賴)                               │
│   └───────┬────────┘                                                  │
│           │                                                           │
│           ▼                                                           │
│   ┌────────────────┐                                                  │
│   │  3. 實作      │  /opsx:apply                                       │
│   │     任務      │  執行任務，勾選完成項目                             │
│   │                │◄──── 隨學習更新文件                               │
│   └───────┬────────┘                                                  │
│           │                                                           │
│           ▼                                                           │
│   ┌────────────────┐                                                  │
│   │  4. 驗證      │  /opsx:verify (可選)                              │
│   │     工作      │  檢查實作是否符合文件                               │
│   └───────┬────────┘                                                  │
│           │                                                           │
│           ▼                                                           │
│   ┌────────────────┐     ┌──────────────────────────────────────────────┐ │
│   │  5. 封存      │────►│  增量規格合併到主要規格                   │ │
│   │     變更      │     │  變更資料夾移至 archive/                    │ │
│   └────────────────┘     │  規格成為更新的來源真實值                   │ │
│                          └──────────────────────────────────────────────┘ │
│                                                                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

**良性循環：**

1. 規格描述目前行為
2. 變更提議修改（作為增量）
3. 實作讓變更成為現實
4. 封存合併增量到規格
5. 規格現在描述新行為
6. 下次變更基於更新後的規格

### 更新還是新建

什麼時候應該更新現有變更，什麼時候應該開始新的？

#### 更新現有變更當：

- **相同意圖，精緻化執行**
  - 你發現沒考慮到的邊緣情況
  - 方法需要微調但目標不變
  - 實作揭示設計稍微偏離

- **範圍縮小**
  - 你意識到完整範圍太大，想先出 MVP
  - 「加入深色模式」→「先加入深色模式切換器（系統偏好設定 v2）」

- **學習驅動的修正**
  - 代碼庫結構不是你想像的那樣
  - 依賴運作不如預期
  - 「使用 CSS 變數」→「改用 Tailwind 的 dark: 前綴」

#### 開始新變更當：

- **意圖根本改變**
  - 問題本身現在變不同了
  - 「加入深色模式」→「加入完整主題系統，含自訂顏色、字型、間距」

- **範圍爆炸**
  - 變更成長太多，本質上是不同的工作
  - 原始提案在更新後會變得無法辨認
  - 「修正登入 bug」→「重寫驗證系統」

- **原始變更可完成**
  - 原始變更可以被標記為「完成」
  - 新工作獨立存在，不是精緻化
  - 完成「加入深色模式 MVP」→封存→新變更「增強深色模式」

### 決策流程

```
                    ┌─────────────────────────────────────┐
                    │     這是相同的工作嗎？            │
                    └──────────────┬──────────────────────┘
                                   │
                       ┌───────────┼───────────┐
                       │           │           │
                       ▼           ▼           ▼
                相同意圖？    >50% 重疊？  原始可以
                相同問題？    相同範圍？  不用這些
                       │           │       變更完成？
                       │           │           │
             ┌────────┴────────┐ ┌───┴───┐  ┌────┴────┐
             │               │ │       │  │         │
            是              否 是       否  否        是
             │               │ │       │  │         │
             ▼               ▼ ▼       ▼  ▼         ▼
          更新            新建 更新    新建 更新     新建
```

| 測試 | 更新 | 新變更 |
|------|------|--------|
| **身分** | 「相同事物，精緻化」 | 「不同工作」 |
| **範圍重疊** | >50% 重疊 | <50% 重疊 |
| **完成** | 沒有這些變更無法「完成」 | 可以完成原始工作，新工作獨立存在 |
| **故事** | 更新鏈告訴連貫故事 | 補丁會比澄清更令人困惑 |

## 最佳實踐

### 保持變更專注

每個變更一個邏輯單位。如果你在「加入功能 X 且也重構 Y」，考慮兩個獨立變更。

**為什麼重要：**
- 更容易審查和理解
- 更乾淨的封存歷史
- 可以獨立出貨
- 需要時更簡單的回滾

### 使用 Lite Spec 預設

OpenSpec 旨在避免官僚主義。使用最輕量級別，但仍讓變更可驗證。

**Lite spec（預設）：**
- 行為優先的簡短需求
- 清楚的範圍和非目標
- 一些具體的驗收檢查

**Full spec（用於較高風險）：**
- 跨團隊或跨 repo 變更
- API/合約變更、遷移、安全/隱私考量
- 模糊性可能導致昂貴重做的變更

大多數變更應保持在 Lite 模式。

### 人類 + Agent 協作

許多團隊中，人類探索，agent 起草文件。預期的循環：

1. 人類提供意圖、上下文和約束。
2. Agent 將其轉換為行為優先的需求和情境。
3. Agent 將實作細節保留在 `design.md` 和 `tasks.md`，而非 `spec.md`。
4. 驗證在實作前確認結構和清晰度。

這讓規格對人類可讀，對 agent 一致。

### 逐步驗證

不要等到最後才驗證整個變更：

```text
好的做法：
├─ 完成任務 1.1
├─ 驗證運作
├─ 完成任務 1.2
├─ 驗證運作
├─ ...
└─ 最後 /opsx:verify

較差的做法：
├─ 一次完成所有任務
├─ 發現大問題
└─ 需要回溯大量工作
```

### 給變更好名字

好名字讓 `openspec list` 有用：

```
好的：                               避免：
add-dark-mode                      feature-1
fix-login-redirect                  update
optimize-product-query              changes
implement-2fa                      wip
```

### 理解 Spec 是什麼（以及不是什麼）

規格是**行為合約**，而非實作計畫。

**好的規格內容：**
- 使用者或下游系統依賴的可觀察行為
- 輸入、輸出和錯誤條件
- 外部約束（安全、隱私、可靠性、相容性）
- 可測試或明確驗證的情境

**規格中避免：**
- 內部類別/函式名稱
- 函式庫或框架選擇
- 逐步實作細節
- 詳細執行計畫（那些屬於 `design.md` 或 `tasks.md`）

快速測試：
- 如果實作可以在不改變外部可見行為的情況下變更，它很可能不屬於規格。
