OpenSpec

AI 原生的規格驅動開發工具

讓 AI 和人類在寫代碼「之前」先達成共識

問題：AI 編碼的挑戰

當需求只存在於聊天歷史中...

❌ 沒有規格的痛苦
  • 聊天歷史變得很長，很難回溯
  • 沒有正式文件可以參考
  • 團隊成員無法理解變更全貌
  • 代碼和需求可能脫節
✓ OpenSpec 的優勢
  • 共識先行 — 先同意再寫代碼
  • 結構化規格 — 清楚的需求和情境
  • 追蹤進度 — 任務檢查清單
  • 持續演進 — 規格隨專案成長

核心哲學

四個原則，改變開發方式

🌊 流動而非僵化
沒有階段閘門，按意義執行。隨時可以更新任何文件，而非被鎖定在「規劃階段」。
🔄 迭代而非瀑布
需求會變化。理解會加深。隨學習精緻化規格，而非遵循固定計畫。
⚡ 簡單而非複雜
輕量級設定，最少儀式。數秒鐘初始化，立即開始工作，按需自訂。
🏗️ 棕地優先
適合既有專案，不只是新專案。增量方式讓規格修改成為一等公民。

5 分鐘快速開始

# 1. 安裝 OpenSpec
$ npm install -g @fission-ai/openspec@latest

# 2. 初始化專案
$ cd your-project
$ openspec init

# 3. 告訴 AI 要做什麼
$ /opsx:propose 加入深色模式

# 4. AI 自動建立...
$ ✓ proposal.md — 為什麼要做
$ ✓ specs/       — 需求和情境
$ ✓ design.md    — 技術方案
$ ✓ tasks.md     — 實作清單

# 5. 開始實作
$ /opsx:apply

核心指令

預設快速路徑

/opsx:propose
→
/opsx:apply
→
/opsx:archive
1
propose — 提出新變更
一次性建立所有規劃文件（proposal、specs、design、tasks）
2
apply — 執行實作
逐一執行 tasks.md 中的任務，寫入代碼並勾選完成
3
archive — 封存變更
合併增量規格到主要規格，歸檔變更以供審計
4
explore — 探索想法
自由調查、比較方案，不需要結構

專案結構

openspec/
├── specs/              # 來源真實值 — 系統目前如何運作
│   └── <領域>/
│       └── spec.md
│
├── changes/            # 提議的修改 — 每個變更一個資料夾
│   └── <變更名稱>/
│       ├── proposal.md   # 提案：為什麼、做什麼
│       ├── design.md     # 設計：技術方案
│       ├── tasks.md      # 任務：實作清單
│       └── specs/       # 增量規格 — 變更內容
│           └── <領域>/
│               └── spec.md
│
└── config.yaml         # 專案配置（可選）
specs/ = 來源真實值，changes/ = 提議的修改

四大核心文件

proposal
→
specs
→
design
→
tasks
📄 Proposal
意圖：解決什麼問題？
範圍：做什麼、不做什麼？
方法：如何解決？
📋 Specs
需求：系統必須有的行為
情境：具體可測試的範例
格式：Given/When/Then
🔧 Design
技術方案：架構和方法
決策記錄：為什麼這樣做
資料流程：組件如何互動
✅ Tasks
實作步驟：具體可執行的任務
追蹤進度：勾選框標記完成
分層結構：1.1, 1.2, 2.1...

增量規格（Delta Specs）

只描述「什麼在變更」，而非重述整個規格

# Auth 的增量規格

## 新增的需求

### 需求：兩步驗證
系統必須支援基於 TOTP 的兩步驗證。

#### 情境：2FA 註冊
- 假設未啟用 2FA 的使用者
- 當使用者在設定中啟用 2FA
- 那麼會顯示供驗證器應用程式設定的 QR code

## 修改的需求

### 需求：會話過期
系統必須在 15 分鐘不活動後過期會話。
（先前：30 分鐘）

## 移除的需求

### 需求：記住我
（已被 2FA 取代）
ADDED 附加到規格
MODIFIED 取代現有
REMOVED 從規格刪除

工作流程比較

🔒 傳統階段鎖定
規劃
→
實作
→
完成
  • 階段強制線性進度
  • 實作中無法回退修改設計
  • 一開始就要建立所有文件
  • 僵硬不靈活
🌊 OpenSpec 流動動作
new
↔
continue
↔
apply
↔
archive
  • 動作是你可以做的事，非你處於的階段
  • 隨時可以更新任何文件
  • 可以一次建立或逐步建立
  • 依賴是啟用器，非閘門

實際範例：加入深色模式

# 對話流程

你: /opsx:propose 加入深色模式

AI: ✓ 已建立 openspec/changes/加入深色模式/
    ✓ proposal.md — 使用者請求減少眼疲勞
    ✓ specs/ui/spec.md — 主題切換 + 系統偏好
    ✓ design.md — React Context + CSS 變數
    ✓ tasks.md — 8 個實作任務
    準備好開始實作！

你: /opsx:apply

AI: ✓ 1.1 建立 ThemeContext
   ✓ 1.2 新增 CSS 自訂屬性
   ✓ 1.3 實作 localStorage 持久化
   ✓ 2.1 建立 ThemeToggle 組件
   ✓ 2.2 將切換器新增至設定頁面
   ... 所有任務完成！

你: /opsx:archive

AI: ✓ 已同步規格至 openspec/specs/ui/spec.md
   ✓ 已移至 openspec/changes/archive/2025-01-24-加入深色模式/
   變更已成功封存！

最佳實踐

✓
保持變更專注
一個邏輯單位 = 一個變更。不要混雜不同功能
✓
給變更好名字
使用 `add-dark-mode` 而非 `feature-1` 或 `wip`
✓
人類 + Agent 協作
人類提供意圖，AI 起草文件，保持規格人類可讀
✓
逐步驗證
不要等到最後。完成任務後立即驗證運作
✓
使用 Lite Spec
預設使用輕量規格，只在風險高時使用完整規格
✓
更新 vs 新建
相同意圖精緻化→更新；不同工作→新建變更
