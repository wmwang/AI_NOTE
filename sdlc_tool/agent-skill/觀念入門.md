# Agent Skills 觀念入門

理解 Agent Skills 的核心概念與設計哲學。

---

## 什麼是 Agent Skills？

Agent Skills 是一個生產級工程技能庫，專為 AI 編程代理設計。每個技能都編碼了資深工程師的開發流程。

### 核心問題

**AI 編程代理的預設行為：**
- 走捷徑
- 跳過測試
- 忽略審查
- 產生「看起來沒問題」但不穩定的程式碼

**Agent Skills 的解決方案：**
- 強制結構化工作流程
- 每個技能都有驗證步驟
- 對抗「我會補測試」的藉口
- 將最佳實踐嵌入 AI 行為

---

## 核心概念

### 1. 技能是工作流程，不是參考文件

```
參考文件：你讀它來了解「如何做 X」
技能：AI 遵循它來「一步一步執行 X」
```

每個技能都有：
- 清晰的步驟順序
- 驗證檢查點
- 退出標準

### 2. 反合理化表格

AI 常見的藉口：

| 藉口 | 反駁 |
|------|------|
| 「這很簡單，不需要規格」 | 簡單任務不需要長規格，但仍需驗收標準 |
| 「我會後來寫測試」 | 後來永不來。現在寫 |
| 「規格會拖慢我們」 | 15 分鐘規格防止數小時的重新設計 |

### 3. 驗證不可協商

```
任務完成？不。
「看起來沒問題」？不。
通過測試、建置、驗證？是。
```

證據要求：
- 測試通過
- 建置成功
- 執行時數據
- 手動檢查結果

### 4. 漸進式揭露

不要一次載入所有技能。載入相關的：

```
處理 UI？載入 frontend-ui-engineering
偵錯？載入 debugging-and-error-recovery
```

節省 tokens，提升效果。

## 開發生命週期

### 六個階段

```
DEFINE → PLAN → BUILD → VERIFY → REVIEW → SHIP
```

### 完整流程

```
1. Idea Refine
   模糊想法 → 具體提案

2. Spec-Driven Development
   什麼 + 為什麼 + 如何驗證

3. Planning & Task Breakdown
   規格 → 可執行的小任務

4. Incremental Implementation
   實作 → 測試 → 驗證 → 提交 → 下一個切片

5. Code Review & Quality
   五軸審查（正確性/可讀性/架構/安全/效能）

6. Shipping & Launch
   發布檢查清單 → 監控 → 還原程序
```

## 設計哲學

### 1. 簡單優先

```
1000 行程式碼？失敗。
100 行行得通？成功。
```

問自己：
- 能用更少行數嗎？
- 抽象是否值得其複雜度？
- 資深工程師會說「為什麼不直接...」嗎？

### 2. 範圍紀律

```
不要「順便清理」相鄰的程式碼
不要移除你不理解的東西
不要增加規格中沒有的功能
```

外科手術般的精確，不是主動翻修。

### 3. 程式碼是負債

每行程式碼都是未來的維護成本。

```python
# 糟糕：重複的 if-else
if type == 'user':
    process_user()
elif type == 'admin':
    process_admin()
elif type == 'guest':
    process_guest()

# 良好：策略模式
handlers = {'user': process_user, 'admin': process_admin, 'guest': process_guest}
handlers.get(type, default)()
```

### 4. 上下文工程

在正確的時間提供正確的資訊：

```
錯誤：載入整個專案
正確：載入相關模組 + 當前任務規格
```

### 5. 源頭驅動

所有框架決策基於官方文件：

```
錯誤：「我記得 React Hook Form 這樣用...」
正確：「根據官方文件 v7.x...[引用]」
```

## 適用原則（適用於所有技能）

### 1. 明確假設

```
我正在做的假設：
1. 使用 PostgreSQL
2. 認證用 session cookies
→ 現在更正我，否則繼續
```

### 2. 主動管理困惑

遇到不一致時：
1. 停止
2. 命名具體困惑
3. 提出權衡或問題
4. 等待解決

### 3. 必要時推回

不是說「是」的機器。

```
「當然！」→ 錯誤
「這個方法有問題，因為...建議替代方案」→ 正確
```

### 4. 強制簡單性

自然傾向是過度複雜。主動抵抗。

### 5. 驗證，不假設

每個技能以驗證結束。證據要求是非可協商的。

## 為什麼有效

### 谷歌工程文化

來自 Google 的最佳實踐：

- **Hyrum's Law**（API 設計）
- **Beyonce Rule**（測試）
- **測試金字塔**（測試）
- **變更大小**（程式碼審查）
- **Chesterton's Fence**（簡化）
- **主幹開發**（Git）
- **Shift Left**（CI/CD）

這些不是抽象原則 — 它們嵌入在 AI 遵循的逐步工作流程中。

## 總結

**Agent Skills 本質：**

1. 結構化工作流程 > 自由輸出
2. 驗證 > 「看起來沒問題」
3. 小變更 > 大爆炸
4. 簡單 > 聰明
5. 對抗合理化 > 接受藉口

**記住：**

- AI 預設走捷徑
- 技能強制紀律
- 驗證不可協商
- 簡單優先

開始使用三個核心技能，體驗差異！
