## 簡介

Agent Skills 是一個包含 22 個生產級工程技能的集合，專為 AI 編程代理設計。每個技能都編碼了資深工程師在開發軟體時遵循的具體工作流程、品質閘門和最佳實踐。

### 開發生命週期


## 快速開始

### 最小安裝

對於大多數使用者，三個核心技能就足夠了：

1. **spec-driven-development** — 定義要建構什麼
2. **test-driven-development** — 證明它有效
3. **code-review-and-quality** — 合併前驗證品質

### 五分鐘入門

```bash
# 1. 複製專案
git clone https://github.com/addyosmani/agent-skills.git

# 2. 將核心技能加入你的規則文件
# 複製 skills/spec-driven-development/SKILL.md
# 複製 skills/test-driven-development/SKILL.md
# 複製 skills/code-review-and-quality/SKILL.md

# 3. 開始使用
```

## 核心概念

### 技能的結構

每個技能都遵循一致的結構：

- **Frontmatter** — 名稱和描述
- **Overview** — 這個技能做什麼
- **When to Use** — 觸發條件
- **Process** — 逐步工作流程
- **Rationalizations** — 借口 + 反駁
- **Red Flags** — 問題的跡象
- **Verification** — 證據要求

### 關鍵原則

1. **流程，不是散文** — 技能是代理遵循的工作流程
2. **反合理化** — 對抗 AI 跳過步驟的傾向
3. **驗證不可協商** — 每個技能都以證據結束
4. **漸進式揭露** — 需要時才載入相關內容

## 技能詳解

### Meta — using-agent-skills
發現並叫用適當的技能。這是元技能，管理所有其他技能的使用。

### Define — 定義階段

| 技能 | 用途 | 何時使用 |
|------|------|---------|
| idea-refine | 結構化發散/收斂思考 | 模糊想法需要探索 |
| spec-driven-development | 寫 PRD 規格 | 新專案/功能/重大變更 |

### Plan — 計畫階段

| 技能 | 用途 | 何時使用 |
|------|------|---------|
| planning-and-task-breakdown | 分解為可驗證任務 | 有規格需要實作單元 |

### Build — 建構階段

| 技能 | 用途 | 何時使用 |
|------|------|---------|
| incremental-implementation | 薄垂直切片 | 多檔案變更 |
| test-driven-development | Red-Green-Refactor | 實作邏輯/修復錯誤 |
| context-engineering | 正確的上下文 | 會話開始/品質下降 |
| source-driven-development | 基於官方文件 | 需要權威程式碼 |
| doubt-driven-development | 對抗性決策審查 | 高風險/不熟悉程式碼 |
| frontend-ui-engineering | 生產級 UI | UI 工作 |
| api-and-interface-design | 合約優先設計 | API/模組邊界 |

### Verify — 驗證階段

| 技能 | 用途 | 何時使用 |
|------|------|---------|
| browser-testing-with-devtools | DevTools MCP | 瀏覽器工作 |
| debugging-and-error-recovery | 五步驟分流 | 測試失敗/壞了 |

### Review — 審查階段

| 技能 | 用途 | 何時使用 |
|------|------|---------|
| code-review-and-quality | 五軸審查 | 合併前 |
| code-simplification | 降低複雜度 | 程式碼難讀/維護 |
| security-and-hardening | OWASP 預防 | 使用者輸入/認證 |
| performance-optimization | 測量優先 | 效能需求/懷疑迴歸 |

### Ship — 部署階段

| 技能 | 用途 | 何時使用 |
|------|------|---------|
| git-workflow-and-versioning | 主幹開發/原子提交 | 任何程式碼變更 |
| ci-cd-and-automation | Shift Left/品質閘門 | 設定/修改 pipeline |
| deprecation-and-migration | 程式碼作為負債 | 移除舊系統 |
| documentation-and-adrs | ADR/文件 | 架構決策/API 變更 |
| shipping-and-launch | 發布檢查清單 | 部署到生產 |

## 命令使用

### Claude Code 斜杠命令

| 命令 | 觸發技能 | 核心原則 |
|------|---------|---------|
| `/spec` | spec-driven-development | 程式碼前的規格 |
| `/plan` | planning-and-task-breakdown | 小的、原子任務 |
| `/build` | incremental-implementation + test-driven-development | 一次一片 |
| `/test` | test-driven-development | 測試是證據 |
| `/review` | code-review-and-quality | 改善程式碼健康 |
| `/code-simplify` | code-simplification | 清晰優於聰明 |
| `/ship` | shipping-and-launch | 更快更安全 |

### 使用範例

```
# 定義新功能
/spec 建立一個任務管理系統

# 製作實作計畫
/plan 根據 SPEC.md

# 實作並測試
/build 實作任務 1-3

# 審查程式碼
/review 審查當前分支

# 部署到生產
/ship 準備發布 v1.0.0
```

## 代理角色

### code-reviewer
資深工程師視角的五軸程式碼審查。用於嚴格審查時。

### test-engineer
QA 專家的測試策略和覆蓋率分析。用於設計/評估測試時。

### security-auditor
安全工程師的弱點偵測和 OWASP 評估。用於安全審查時。

## 最佳實踐

1. **從規格開始** — 非瑣碎工作先寫規格
2. **漸進式實作** — 每次一個切片，測試後移動
3. **測試優先** — 先寫失敗測試，再讓它通過
4. **審查每個變更** — 合併前進行審查
5. **保持變更小** — ~100 行為目標
6. **明確假設** — 不要默默填補模糊需求
7. **驗證優先** — 「看起來沒問題」不夠，需要證據

## 故障排除

**技能沒有觸發？** 明確參考技能名稱。

**輸出品質不佳？** 使用 context-engineering 提供更好的上下文。

**變更太大？** 使用 incremental-implementation 分解它。
