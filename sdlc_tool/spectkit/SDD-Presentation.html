GitHub Spec Kit

什么是 Spec-Driven Development？

SDD 是一种将规格说明作为核心驱动力的软件开发方法论。

规格说明不再只是指导实施——它们直接生成工作实现。

↓
规格说明
→
实施计划
→
代码实现

传统开发的问题

📜 传统模式

  • 代码是真相源
  • 文档很快过时
  • 需求变更代价高昂
  • 文档与代码存在鸿沟
  • 技术债务累积

✨ SDD 模式

  • 规格是真相源
  • 文档保持同步
  • 需求变更成本可控
  • 无鸿沟，只有转换
  • 架构持续演进

为什么现在需要 SDD？

🤖 AI 能力成熟

AI 可以可靠地从自然语言规格说明生成工作代码

📊 复杂性增长

现代系统集成数十个服务，手动对齐越来越困难

⚡ 变革加速

需求变化速度极快，快速迭代成为常态

SDD 的核心原则

  • 📌 规格说明作为通用语言——维护软件就是演进规格说明
  • ⚙️ 可执行规格说明——精确、完整、无歧义
  • 🔄 持续细化——一致性验证持续进行
  • 🔬 研究驱动上下文——自动收集技术上下文
  • ↔️ 双向反馈——生产现实通知规格演进
  • 🌳 分支探索——从同一规格生成多种实现

Spec Kit 工具包

GitHub Spec Kit 是实现 SDD 的开源工具包

核心命令

/speckit.constitution — 建立项目原则

/speckit.specify — 创建功能规格

/speckit.plan — 制定实施计划

/speckit.tasks — 生成任务列表

/speckit.implement — 执行实施

SDD 工作流程

想法
↓
规格说明 (specify)
↓
实施计划 (plan)
↓
任务分解 (tasks)
↓
代码实现 (implement)

每个阶段都有 AI 辅助和结构化验证

Constitution 治理架构

九大开发条款确保每个生成的实现保持一致性、简单性和质量

📦 库优先

每个功能首先作为独立库

💻 CLI 接口

所有功能通过命令行可访问

✅ 测试优先

不可协商：先写测试再写代码

🎯 简单性

最多 3 个项目，不进行未来证明

🚫 反抽象

直接使用框架，单一模型表示

🔗 集成优先

使用真实环境而非 mock

快速开始

5 分钟上手

# 安装

uv tool install specify-cli --from git+https://github.com/github/spec-kit.git

# 初始化

specify init my-project --integration claude

# 建立原则

/speckit.constitution 创建关注代码质量和测试的原则

# 创建规格

/speckit.specify 构建一个任务管理应用

# 制定计划

/speckit.plan 使用 React 和 Node.js，PostgreSQL 数据库

# 执行实施

/speckit.implement

SDD 的价值

🎯 保持对齐

规格说明作为单一真相源，消除与实现的差距

🚀 快速迭代

需求变更变成系统性重生成，而非手动重写

✅ 保证质量

通过 Constitution 强制执行架构原则

⚡ 提高效率

自动化机械翻译，专注于创造性工作
