# Spec-Driven Development 概念入门

> 规格驱动开发（SDD）是一种将规格说明作为核心驱动力的软件开发方法论，通过让规格说明成为可执行的源头来生成实现，而不是仅仅指导实现。

## 核心理念

### 传统开发：代码为王

几十年来，代码一直占据着软件开发的统治地位。规格说明只是我们搭建的脚手架，一旦开始"真正"的编码工作就被丢弃。我们编写 PRD 来指导开发，创建设计文档来通知实施，绘制图表来可视化架构。但这些始终从属于代码本身。代码是真相，而其他一切都只是美好的意图。

**传统模式的问题**：
- 规格说明与代码之间存在永远无法弥合的鸿沟
- 需求变更需要手动传播到代码中
- 文档很快过时，与实现脱节
- 重构困难，因为没有可信赖的规格基础

### SDD：规格为王

Spec-Driven Development（SDD）彻底颠覆了这个权力结构。规格说明不再服务于代码——代码服务于规格说明。PRD 不是实施的指南；它是生成实施的源头。技术计划不是通知编码的文档；它们是产生代码的精确定义。

**SDD 模式**：
```
规格说明（意图）
    ↓
实施计划（技术定义）
    ↓
代码实现（自动生成）
```

**这种转变带来的好处**：
- 消除了意图与实现之间的鸿沟
- 规格说明成为可执行的源头
- 代码是规格在特定语言和框架中的表达
- 维护软件意味着演进规格说明

### 开发语言的提升

在 SDD 世界中，开发团队的意图用自然语言表达（**意图驱动开发**）、设计资产、核心原则和其他准则。开发的通用语言提升到了更高层次，代码成为最后一英里的实现方式。

- **调试**：修复产生错误代码的规格说明和实施计划
- **重构**：为清晰度而重组结构
- **开发工作流**：围绕规格说明作为核心真相源重新组织

## 为什么现在需要 SDD

三个趋势使 SDD 不仅成为可能，而且变得必要：

### 1. AI 能力达到临界点

AI 现在能够可靠地从自然语言规格说明生成工作代码。这并不是要取代开发者——而是通过自动化从规格说明到实施的机械翻译来提升他们的效率。

**AI 带来的改变**：
- 放大探索和创造力
- 支持"重新开始"而不需要重写所有代码
- 支持增删改和批判性思考

### 2. 软件复杂性指数级增长

现代系统集成数十个服务、框架和依赖项。通过手动流程保持所有这些组件与原始意图对齐变得越来越困难。SDD 通过规格驱动生成提供系统性对齐。

**复杂性的挑战**：
- 组件间的协调变得极其困难
- 技术栈的快速演进
- 系统边界的模糊化

### 3. 变革速度加速

需求变化的速度比以往任何时候都快。转向不再是例外——而是被期望的。现代产品开发要求基于用户反馈、市场条件和竞争压力进行快速迭代。

**SDD 如何应对变化**：
- 需求变更变成正常工作流的一部分
- 改变 PRD 中的核心需求，受影响的实施计划自动更新
- 修改用户故事，相应的 API 端点重新生成
- 这不仅关于初始开发——更关乎通过不可避免的变革保持工程速度

## SDD 的关键原则

### 1. 规格说明作为通用语言

规格说明成为主要工件。代码是其在特定语言和框架中的表达。维护软件意味着演进规格说明。

### 2. 可执行规格说明

规格说明必须精确、完整且无歧义，足以生成工作系统。这消除了意图与实现之间的差距。

**可执行规格的特点**：
- 清晰的用户故事和验收标准
- 明确的功能需求
- 可衡量的成功标准
- 无歧义的边界条件

### 3. 持续细化

一致性验证持续进行，而不是作为一次性门控。AI 持续分析规格说明中的歧义、矛盾和缺口，作为持续改进的过程。

### 4. 研究驱动上下文

在整个规格说明过程中，研究代理收集关键上下文，调查技术选项、性能影响和组织约束。

**研究上下文包括**：
- 库兼容性分析
- 性能基准测试
- 安全影响评估
- 组织约束的自动应用

### 5. 双向反馈

生产现实通知规格说明的演进。指标、事件和运营学习成为规格说明细化的输入。

**反馈循环**：
```
规格 → 实施 → 生产
   ↑              ↓
   └── 反馈 ──────┘
```

### 6. 分支探索

从同一规格说明生成多种实施方法，以探索不同的优化目标——性能、可维护性、用户体验、成本。

## SDD 与传统开发对比

| 维度 | 传统开发 | Spec-Driven Development |
|------|---------|------------------------|
| **核心工件** | 代码 | 规格说明 |
| **文档角色** | 指导实施，很快过时 | 可执行源头，保持同步 |
| **需求变更** | 手动传播，容易出错 | 系统性重生成 |
| **测试策略** | 代码编写后添加 | 作为规格的一部分 |
| **架构演进** | 技术债务累积 | 通过规格演进驱动 |
| **重用性** | 代码片段难以重用 | 规格可跨技术栈重用 |

### 传统开发流程

```
想法 → PRD → 设计文档 → 代码实现 → 测试 → 部署
       ↑_______需求变更需手动回溯______________|
```

**痛点**：
- 文档与代码脱节
- 需求变更代价高昂
- 技术债务累积
- 重构困难

### SDD 开发流程

```
想法 → 规格说明 → 实施计划 → 代码生成 → 测试 → 部署
       ↑←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←┘
                   持续反馈循环
```

**优势**：
- 规格作为单一真相源
- 需求变更成本可控
- 技术决策可追溯
- 可并行探索多种实现

## Constitution 治理架构

SDD 的核心是一个 Constitution——一组不可变的原则，支配规格如何变成代码。Constitution（`.specify/memory/constitution.md`）作为系统的架构 DNA，确保每个生成的实现保持一致性、简单性和质量。

### 开发的九大条款

#### 条款 I：库优先原则

每个功能必须首先作为独立库存在——没有例外。这从一开始就强制模块化设计。

```
Spec Kit 中的每个功能必须首先作为独立库存在。
不得直接在应用程序代码中实现任何功能，而不先将其抽象为可重用的库组件。
```

#### 条款 II：CLI 接口强制要求

每个库必须通过命令行接口公开其功能。

```
所有 CLI 接口必须：
- 接受文本作为输入（通过 stdin、参数或文件）
- 产生文本作为输出（通过 stdout）
- 支持用于结构化数据交换的 JSON 格式
```

#### 条款 III：测试优先强制性

最具变革性的条款——没有测试就没有代码：

```
这是不可协商的：所有实施必须遵循严格的测试驱动开发。
在编写以下内容之前不得编写实施代码：
1. 单元测试已编写
2. 测试已由用户验证和批准
3. 测试已确认失败（红色阶段）
```

#### 条款 IV：简单性原则

- 初始实施最多使用 3 个项目
- 不进行未来证明设计

#### 条款 V：反抽象原则

- 直接使用框架功能而不是包装它们
- 单一模型表示

#### 条款 VI：集成优先测试

```
测试必须使用现实环境：
- 优先使用真实数据库而不是 mock
- 使用实际服务实例而不是 stub
- 实施前强制执行契约测试
```

### Constitution 的执行

实施计划模板通过具体检查点将这些条款操作化：

```markdown
### 阶段 -1：前实施门控

#### 简单性门控（条款 VII）

- [ ] 使用 ≤3 个项目？
- [ ] 没有未来证明设计？

#### 反抽象门控（条款 VIII）

- [ ] 直接使用框架？
- [ ] 单一模型表示？

#### 集成优先门控（条款 IX）

- [ ] 契约已定义？
- [ ] 契约测试已编写？
```

这些门控充当架构原则的编译时检查。AI 无法在通过门控或在"复杂性跟踪"部分记录合理的例外的情况下继续。

## SDD 工作流程

SDD 工作流程从想法开始——通常模糊而不完整。通过与 AI 的迭代对话，这个想法变成全面的 PRD。

### 迭代细化

1. **AI 提出澄清问题**
   - 识别边缘情况
   - 定义精确的验收标准

2. **研究代理收集上下文**
   - 调查库兼容性
   - 性能基准测试
   - 安全影响评估

3. **持续一致性验证**
   - AI 分析规格说明中的歧义、矛盾和缺口
   - 不是作为一次性门控，而是作为持续细化过程

### 从 PRD 到实施计划

AI 生成将需求映射到技术决策的实施计划：

- 每个技术选择都有文档化的理由
- 每个架构决策都追溯到特定需求

### 代码生成

当规格说明和实施计划足够稳定时，代码生成就开始了：

- 领域概念变成数据模型
- 用户故事变成 API 端点
- 验收场景变成测试

### 持续反馈循环

生产指标和事件不仅仅触发热修复——它们更新下一次生成的规格说明：

- 性能瓶颈成为新的非功能性需求
- 安全漏洞成为影响所有未来生成的约束

## 常见误解

### 误解 1：SDD 取代开发者

**真相**：SDD 不是要取代开发者或自动化创造力。它是关于通过自动化机械翻译来提升人类能力，让开发者专注于：

- 创造力
- 实验
- 批判性思考

### 误解 2：规格说明必须一开始就完美

**真相**：规格说明可以早期生成，不必"完全"：

- 早期生成可能是探索性的——测试规格在实践中是否有意义
- 通过实施反馈持续改进
- 生产问题回馈到规格说明中

### 误解 3：SDD 只适用于新项目

**真相**：SDD 也适用于遗留系统：

- 增量式引入功能
- 现代化遗留系统
- 采用增量流程

### 误解 4：规格说明太复杂难写

**真相**：Spec Kit 提供的结构化命令和模板使规格说明变得简单：

- `/speckit.specify` 命令引导您完成过程
- 模板强制执行必要部分
- AI 帮助细化和澄清需求

### 误解 5：SDD 限制技术选择

**真相**：SDD 实际上增加了灵活性：

- 规格说明是技术中立的
- 可以从同一规格说明生成多种实施
- 支持平行探索不同优化目标

---

## 总结

Spec-Driven Development 代表了软件开发范式的根本转变。它将规格说明从"必要的恶"转变为开发的驱动力，使团队能够：

1. **保持对齐**：规格说明作为单一真相源
2. **快速迭代**：需求变更变成系统性重生成
3. **保证质量**：通过 Constitution 强制执行架构原则
4. **提高效率**：自动化机械翻译，专注于创造性工作

这不是要替代开发者，而是要赋能他们做更重要的事情：解决问题、创造价值、驱动创新。
