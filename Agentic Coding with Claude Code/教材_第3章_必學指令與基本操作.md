# Claude Code 核心指令全覽——從認證到 Skills 的實戰地圖

第一章讓你理解 Context Engineering 的哲學，第二章帶你認識 Claude Code 的整體輪廓。這一章正式進入「每天都會用到」的操作層面：費用怎麼算、session 怎麼管理、Hook 怎麼掛、記憶怎麼運作、Rewind 怎麼用，以及如何打造自己的 Skills（自訂 slash 指令）。

讀完這章，你會知道 Claude Code 的每個設計決策背後在解決什麼問題，而不只是記住指令怎麼打。

---

## 費用與認證：先搞清楚錢從哪裡燒

Claude Code 有三種付費方式，選哪種直接決定你怎麼監控用量。

### 選項一：Anthropic API Key（按用量計費）

用 API Key 登入的話，是 pay-as-you-go，每個 token 都收費。

適合對象：
- 想在訂閱前先評估 Claude Code 是否值得
- 用量不穩定、有時密集有時沒在用

**要注意的事：**

用 `/cost` 指令隨時看當前 session 燒了多少錢：

```
Total cost:         $0.2873
Total duration (API): 3.9s
Usage by model:
  claude-3-5-haiku: 87 input, 27 output, 0 cache read, 0 cache write
  claude-opus:       3 input, 13 output, 0 cache read, 15.3k cache write
```

**真正的風險：** 後面章節會用到 agents、sub-agents、orchestration 等模式，一個設定錯誤可能同時觸發很多 agents，費用會急速疊加。建議在 Anthropic Console 設定用量上限，超過就停止服務。

### 選項二：Claude 訂閱制

訂閱制用 `/usage` 而不是 `/cost` 來看用量：

```
Current session         0% used    Resets 2pm
Current week (all models) 0% used
Current week (Sonnet only) 0% used
```

### 選項三：第三方雲端平台（企業用）

Amazon Bedrock、Microsoft Azure AI、Google Cloud Vertex AI 都可以接，billing 走雲端帳戶。適合需要集中計費、合規控管的企業環境。

### 三種模型的定位

用 `/models` 看可用模型：

| 模型 | 定位 | 適合的任務 |
|------|------|---------|
| **Claude Opus** | 最強推理，200K context | 複雜規劃、深度研究、架構設計 |
| **Claude Sonnet** | 速度與智慧平衡，200K context | 日常程式碼生成（日常主力） |
| **Claude Haiku** | 最快最輕量 | 簡單任務、低延遲需求 |

## Slash 指令：你和 Claude Code 之間的控制台

在 Claude Code 輸入 `/` 就會看到完整指令清單。這裡講你每天都會碰到的幾個。

### 管理 session 長度：`/clear` 與 `/compact`

對話越長，有兩個問題：越慢、越容易讓 Claude 搞混（Context Rot）。

| 指令 | 做什麼 | 什麼時候用 |
|------|--------|---------|
| `/clear` | 清掉所有對話歷史，回到空白 | 任務結束、對話跑歪了、要開新任務 |
| `/compact` | 壓縮對話歷史成摘要，保留關鍵決策 | 同一個任務還要繼續，但 context 快滿 |

這兩個指令是 Context Engineering 在實際操作層面的體現——你在主動管理 Claude 的工作記憶。

### 管理設定：`/config`

打開 Claude Code 的設定面板：

```
Settings — Configure Claude Code preferences

Auto-compact        true
Use todo list       true
Checkpointing       true
Verbose output      false
Auto-updates        true
Theme               Dark mode
Notifications       Auto
Editor mode         normal
Model               Default (recommended)
```

值得注意的是 **Auto-compact**：這是讓 Claude Code 在快達到 token 上限時自動執行 `/compact`，不需要你手動觸發。

### 設定的三個層級

| 層級 | 位置 | 範圍 |
|------|------|------|
| 使用者設定 | `~/.claude/settings` | 套用到所有 Claude Code instance |
| 專案設定 | `.claude/settings.json` | 只套用到這個專案，可 commit 進 git |
| 本地設定 | local settings | 個人調整，不影響共享的專案設定 |

## Hooks：讓 Claude Code 在對的時機自動做對的事

Hooks 是 Claude Code 裡面容易被忽視但極其強大的功能。**Hook 是在 Claude Code 工作流程的特定時間點自動執行的 shell 指令。**

### Hook 的五個觸發點

| Hook 事件 | 觸發時機 |
|-----------|---------|
| `PreToolUse` | 工具執行**前** |
| `PostToolUse` | 工具執行**後** |
| `Notification` | 送出通知時 |
| `UserPromptSubmit` | 使用者送出 prompt 時 |
| `Stop` | Claude 完成回應、即將結束前 |

Hook 可以做的事：
- 格式化程式碼
- 跑測試
- 阻擋不安全的操作
- 觸發 sub-agents
- 自動切換 context

> **安全警告**：Hooks 以你的完整使用者權限執行任意 shell 指令，不需要確認。來源不明的 hook 絕對不要隨便用。

### 實作：做一個「完成通知音效」Hook

這是最簡單的 Hook 示範，每次 Claude 回應完就播放音效，讓你不用一直盯著螢幕。

**Step 1：建立音效播放腳本**

建立 `hooks/play_sound.sh`：

```bash
#!/bin/bash
# macOS
afplay /path/to/sound.wav

# Linux（需要 aplay 或其他音效工具）
# aplay /path/to/sound.wav
```

**Step 2：設定 Hook**

打開 `/hooks`，選擇 hook 事件 **Stop**，設定指令：

```bash
bash /path/to/play_sound.sh
```

Claude Code 會把設定存進 `.claude/settings.json`：

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash /path/to/play_sound.sh"
          }
        ]
      }
    ]
  }
}
```

`matcher: ""` 代表這個 hook 套用到所有 Stop 事件，不做額外篩選。

**Step 3：重啟 Claude Code，測試 hook 有沒有觸發**

輸入任何 prompt，Claude 回應完畢後應該聽到音效。

**進階提示：** 對於高頻觸發的 hook，盡量用輕量指令（`afplay` 比啟動整個 Python runtime 快很多）。重的腳本留給真正需要複雜邏輯的 hook。

## 記憶系統：讓 Claude 跨 session 記住重要的事

Claude Code 的記憶系統讓 AI 能在對話結束後保留設定、規範、偏好，下次開新 session 還是知道你在做什麼。記憶主要靠 **`CLAUDE.md`** 檔案實現。

### 記憶的五個層級

| 層級 | 位置 | 範圍 | 備註 |
|------|------|------|------|
| **Level 1 — Managed Policy** | 系統管理路徑 | 整個組織 | 不能被覆蓋 |
| **Level 2 — 使用者記憶** | `~/.claude/CLAUDE.md` | 個人，跨所有專案 | |
| **Level 3 — 專案記憶** | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 與團隊共享，commit 進 git | |
| **Level 4 — 專案規則** | `./.claude/rules/*.md` | 與團隊共享，commit 進 git | 可拆成多個主題檔案 |
| **Level 5 — 本地個人設定** | `./CLAUDE.local.md` | 個人，自動被 git ignore | |

層級越高優先級越高，高層無法被低層覆蓋。

### Claude Code 怎麼讀記憶？

啟動時，Claude Code 從當前工作目錄**往上遞迴搜尋**所有 `CLAUDE.md`，全部載入並合成一個工作 context。

舉例，目錄結構如下：

```
/Users/yourname/
├── .claude/
│   └── CLAUDE.md          ← 使用者記憶
└── Desktop/
    └── my-company/
        ├── CLAUDE.md       ← 公司/專案記憶
        └── frontend/
            └── CLAUDE.md   ← 前端子專案記憶
```

如果從 `/Users/yourname/Desktop/my-company/frontend/` 啟動，三個 `CLAUDE.md` 都會被載入。

**Lazy loading：** 子目錄的記憶檔不會立即載入——只有當 Claude Code 實際存取那個子目錄的檔案時才會載入。

### 怎麼新增記憶？

**快速新增：** 用 `#` 開頭的訊息，Claude Code 會自動把資訊存進適當的記憶檔。

```
# 這個專案的 API 回應格式必須符合 RFC 7807 標準
```

Claude Code 會問你要存進哪一層記憶，選了之後下次對話就記住了。

**直接編輯：** 用 `/memory` 指令在程式碼編輯器裡開啟記憶檔案直接修改。

## Rewind：後悔藥

Rewind 讓你可以安全還原 Claude Code 對檔案做的修改。

### Rewind 的運作原理

每次 Claude Code 修改檔案前，都會自動建立一個 checkpoint。Rewind 就是回到這些 checkpoint 的機制。

**觸發方式：**
- 按兩下 `Esc`
- 輸入 `/rewind`

### 為什麼 Rewind 重要？

Rewind 提供安全網，讓你敢嘗試更大膽的修改——「試試看這個方向」的實驗成本趨近於零。快速從 AI 生成的錯誤中恢復，是評估 AI 工具實用性的重要能力。

### 實際示範

假設你依序做了三件事：

1. 建立 landing page
2. 新增 FAQ section
3. 新增 footer

發現 footer 太重，想拿掉：

1. 輸入 `/rewind`，出現這個 session 的時間軸，每個 prompt 旁邊標示有沒有修改檔案
2. 選擇「新增 footer」這個 prompt
3. 系統提供三個選項：
   - **Restore code and conversation**：程式碼和聊天記錄都還原（最常用）
   - **Restore conversation**：只還原聊天記錄，保留程式碼修改
   - **Restore code**：只還原程式碼，保留聊天記錄
4. 選 **Restore code**，footer 立刻消失，但你還有完整的對話脈絡

### Rewind 的限制

| 限制 | 說明 |
|------|------|
| Bash 指令不追蹤 | `/rewind` 只追蹤檔案修改，shell 指令的副作用沒辦法還原 |
| Claude Code 外部的手動修改沒有 checkpoint | 你自己改的檔案不在 rewind 範圍內 |
| 不取代 Git | Checkpoints 是短期安全網，不是版本控制系統 |

## Skills：把重複工作包成自訂 Slash 指令

Skills 是讓你把常用的工作流程包成一個指令，之後用 `/skill-name` 就能執行的機制。

### 建立一個 Skill

Skill 放在 `.claude/skills/` 目錄裡，每個 skill 是一個子目錄，裡面有一個 `SKILL.md`：

```
.claude/
└── skills/
    └── commit-code/
        └── SKILL.md
```

`SKILL.md` 用 YAML frontmatter 定義 metadata，主體是 prompt：

```markdown
---
allowed-tools: Bash(git add:*), Bash(git status:*), Bash(git commit:*)
description: Create a git commit
---

Review the files that have changed, and create a commit with a commit message summarizing the business logic of the changes.
```

### Skill 的兩個關鍵設計

**1. 工具限制（allowed-tools）**

這是「最小權限原則」的體現。這個 commit skill 只能用 git 相關指令，沒辦法讀你的其他檔案或做其他事。

如果沒有限制工具，當 context 被污染（比如惡意 input），agent 可能用所有可用工具做出意外的操作。

**2. $ARGUMENTS 佔位符**

讓 Skill 接受動態輸入：

```markdown
---
allowed-tools: Bash(git add:*), Bash(git status:*), Bash(git commit:*)
description: Create a git commit
---

Review the files that have changed, and create a commit with a commit message.

Use user hints as the message main subject: $ARGUMENTS
```

使用方式：

```
/commit-code fix auth bug
```

`fix auth bug` 這串文字會取代 `$ARGUMENTS`。

### 一個更完整的 Skill 設計

加入 shell 指令讓 Claude 先取得即時資訊再產生輸出：

```markdown
---
allowed-tools: Bash(git add:*), Bash(git status:*), Bash(git commit:*)
description: Create a git commit
---

## Context

- Current git status: `git status`
- Current git diff: `git diff HEAD`
- Current branch: `git branch --show-current`
- Recent commits: `git log --oneline -10`

## Your task

Based on the above changes, create a single git commit.
```

Skill 裡可以執行 shell 指令（用 backtick 包起來），Claude 執行 Skill 時會先跑這些指令、把輸出結果帶進 context，再根據真實的 git 狀態生成 commit message。

## 本章重點整理

| 功能 | 核心作用 | 常用指令/操作 |
|------|---------|-------------|
| **費用與認證** | 決定 billing 方式和用量監控方法 | `/cost`（API key）、`/usage`（訂閱制）|
| **Slash 指令** | 管理 session、設定、整合 IDE | `/clear`、`/compact`、`/config` |
| **Hooks** | 在 Claude 工作流特定時間點自動執行邏輯 | `/hooks`，選 PreToolUse/PostToolUse/Stop 等事件 |
| **記憶系統** | 跨 session 保留專案設定和個人偏好 | `CLAUDE.md`、`/init`、`/memory`、`#` 快速記憶 |
| **Rewind** | 安全還原 Claude Code 對檔案的修改 | `Esc Esc` 或 `/rewind` |
| **Skills** | 把重複工作流程打包成可重用的自訂指令 | `.claude/skills/你的skill名稱/SKILL.md` |

**三個立即可以實作的事：**

1. 在你的專案根目錄跑 `/init`，讓 Claude 自動生成一份 `CLAUDE.md`，從此每個 session 都有專案 context
2. 找一個你最常重複對 Claude 下的指令（比如「幫我寫 commit message」），把它包成一個 Skill
3. 設定一個 Stop hook，在 Claude 回應完後自動通知你（音效或系統通知），這樣就不用一直盯著螢幕等
