Mental Model // 心理模型

Planning Mode vs. Acting Mode

PLANNING MODE
  • Research-only (Read-only)
  • Propose changes in text
  • Map out the "Why" and "How"
  • Zero risk to the codebase
ACTING MODE
  • Write-enabled
  • Follows the blueprint
  • Executes shell commands
  • Runs tests and fixes errors

Command // 指令詳解

Entering /plan Mode

Use /plan [task] to start a non-destructive session.

使用 /plan 啟動非破壞性工作階段。

➜ /plan refactor auth logic

Claude Code will:
1. Grep for existing auth patterns.
2. Read relevant service files.
3. Propose a step-by-step list.
Key Benefit

Claude won't start typing code until you say "Go". This saves context and prevents hallucination spirals.

Infrastructure // 基礎架構

Isolating Agents with Git Worktrees

Avoid "Conflict Hell" by giving each agent its own folder and branch.

$ git worktree add ../agent-feature-x feature-x
$ cd ../agent-feature-x
$ claude
# This agent is now isolated and safe.

This allows multiple agents to run tests and build processes without interfering with each other.

State // 狀態溝通

Passing Context Between Agents

Agents don't talk to each other directly. Files are the medium.

Agent A

Writes API in api.ts

➡️
Agent B

Reads api.ts to write UI

Always ensure one agent is the "Producer" and the other is the "Consumer" for a specific file.

Case Study // 實戰案例

Project: Database Migration

Task: Move from SQLite to PostgreSQL across 5 services.

  1. Plan Agent: Analyzes SQL queries and writes a migration guide.
  2. Dev Agent 1: Updates Service A & B.
  3. Dev Agent 2: Updates Service C & D.
  4. Test Agent: Runs integration tests in a Docker container.
Result:

Migration completed in 2 hours instead of 2 days. 100% test coverage ensured before merge.

Warnings // 避坑指南

Why Multi-agent Workflows Fail

1. Under-planning

Starting execution before the dependencies are clear.

2. Context Overlap

Two agents trying to refactor the same shared utility function.

3. Siloed Errors

Agent A ignores a breaking change that Agent B introduced.

4. Analysis Paralysis

Spending too much time in /plan mode for a simple bug fix.

Review // 審查流程

The Review Gate

You are the Quality Gate. Never skip the review of an Agent's plan.

➜ "Review the plan in SPEC.md. Are you sure about the data types?"
➜ "Wait, don't use library X, we are phasing it out."
➜ "OK, this looks solid. Proceed to step 1."

Proactive redirection is 10x cheaper than fixing buggy code.
