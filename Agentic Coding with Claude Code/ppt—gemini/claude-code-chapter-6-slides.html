Why Planning Mode?

Ad-hoc prompting works for small tasks, but fails on Complex Systems.

隨機的提示詞對小任務有效，但在處理複雜系統時會失效。

  • Reduces Unintended Side Effects
    減少非預期的副作用
  • Ensures Architectural Consistency
    確保架構的一致性
  • Spec-driven vs. Trial-and-error
    規格驅動 vs. 試錯驅動

"Planning is the bridge between a vague idea and a successful implementation."

Commands // 指令集

Tool: The /plan Mode

When you enter /plan, Claude Code switches to a Read-Only state.

當進入計畫模式，Claude Code 會切換至「唯讀」狀態。

Allowed

grep_search, ls, read_file, MCP calls

Blocked

write_file, replace, git commit, shell execution

➜ /plan
# System: Entered planning mode.

➜ "Design a new API for users"
# Claude: I will research current auth and DB schemas before proposing a plan.

Process // 核心流程

The 5-Step Planning Workflow

01
Research
Map dependencies and understand constraints. (調查依賴關係與理解約束)
02
Draft Spec
Create a detailed implementation plan. (起草詳細的實作計畫)
03
Refine
Iterate on the plan with human feedback. (根據反饋優化計畫)
04
Approve
Exit planning mode and commit to action. (退出計畫模式並開始行動)

Safety // 安全機制

Managing Parallel Conflicts

Multiple agents on the Same Branch is fragile. (在同一個分支運行多代理是脆弱的。)

The Risk
  • Overwrite each other's changes.
    互相覆蓋對方的改動
  • Incompatible API assumptions.
    不兼容的 API 假設
The Solution
  • Git Branches per Agent.
    每個代理獨立分支
  • Use Git Worktrees.
    使用 Git Worktrees 隔離

Wrap-up // 總結

Chapter 06 Key Takeaways

  • Use /plan to separate Strategy from Execution.
  • Spec-driven development prevents "hallucination loops".
  • Parallelize independent tasks using multiple terminal windows.
  • Isolate agents using branches to avoid merge conflicts.
