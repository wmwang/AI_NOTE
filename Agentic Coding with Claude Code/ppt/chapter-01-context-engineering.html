什麼是環境工程？

環境工程（Context Engineering）是指圍繞 LLM 的系統，負責管理工具調用、控制代理迴圈、處理錯誤、執行防護措施，以及決定每一步要發送什麼環境給模型的編排層（Agent Harness）。

提示工程 vs. 環境工程

  • 提示工程：靜態的，一次性提示詞設計
  • 環境工程：動態的，持續的環境管理系統
  • 環境來自多個來源且不斷增長
  • 需要動態系統來構建正確的環境

核心概念

"Garbage in, garbage out" —— 這是導致代理系統無法正常運作的最常見原因之一。

為什麼需要環境工程？

  • 環境來自多個來源：開發者、使用者、先前互動、工具調用結果
  • 提示詞是靜態的，環境是高度動態的
  • 構建正確的環境需要動態系統，而不只是靜態提示詞
  • 這是提示工程的自然演進，但是一個更深入的概念

代理編排器（Agent Harness）

什麼是 Agent Harness？

Agent Harness 是圍繞 LLM 的編排層，它不僅是對 LLM 的「包裝器」，而是一個需要深度知識和大量工程工作的複雜系統。

Harness 的職責

  • 管理工具調用（Tool Calls）
  • 控制代理迴圈（Agent Loop）
  • 處理錯誤與重試
  • 執行防護措施（Guardrails）
  • 決定每一步要發送什麼環境給模型

關鍵洞察

大多數實際工程不在 LLM 調用本身，而在其周圍的系統。決定代理是否能可靠運作的關鍵在於周圍系統如何管理狀態、工具、記憶和環境。

長時間任務的挑戰

環境視窗持續增長

在長時間和複雜任務中，我們會累積來自工具調用的回饋，導致環境視窗持續增長，充滿工具調用結果和中間輸出。

Claude Code 的四種環境工程策略

1. Write Context

持久記憶系統 —— 專案記憶、使用者記憶、動態記憶匯入

2. Select Context

智慧環境檢索與注入 —— 根據情境動態選擇相關環境

3. Compress Context

高效環境表示 —— 壓縮和摘要以保持環境可管理

4. Isolate Context

多代理和範圍環境管理 —— 使用專門的子代理處理不同任務

策略 1：撰寫環境 —— 持久記憶

三層記憶階層

專案記憶（./CLAUDE.md）

團隊共享的專案環境，包含架構、標準或任何與特定專案相關的資訊。版本控制，所有成員皆可存取。

使用者記憶（~/.claude/CLAUDE.md）

儲存在使用者家目錄，包含跨越所有專案的使用者偏好和捷徑。不提交到 GitHub，每個使用者有不同值。

動態記憶匯入（@ 語法）

允許使用 @ 符號從其他記憶檔案匯入專門資訊，類似於載入一般環境，但可以有專門的記憶檔案。

@path/to/memory/file.md
@./relative/path/context.md
@~/global/user/context.md

策略 2：選擇環境 —— 智慧檢索

動態環境發現

  • Claude 自動掃描資料夾尋找有用的環境檔案
  • 在子資料夾中時，也會從父資料夾提取環境
  • 優先使用最近使用和頻繁存取的資訊
  • 當有更詳細的資訊可用時，使用更詳細的資訊

環境切換

# 根據 Git 分支動態更新環境

branch=$(git branch --show-current)

case $branch in
"feature/auth-"*)
add_context "@./context/auth-system.md"
;;
"feature/payment-"*)
add_context "@./context/payment-flow.md"
;;
esac

策略 3：壓縮環境 —— 高效表示

為什麼需要壓縮？

環境視窗有限，且環境持續增長。壓縮環境可以保持其可管理性，確保最相關的資訊保持在視窗內。

壓縮技術

  • 摘要：將長文件濃縮為關鍵點
  • 分層：先提供高層次概述，再提供細節
  • 優先級：優先傳送最相關的環境
  • 清理：移除不再相關的舊環境

原則

使用最少的文字傳達最多的資訊。避免重複，使用結構化的格式（標題、項目符號）讓資訊易於掃描。

策略 4：隔離環境 —— 多代理管理

為什麼隔離？

複雜任務需要不同類型的專業知識。使用專門的子代理可以隔離環境，防止干擾並提高效率。

隔離的好處

  • 專業化：每個代理專注於特定任務類型
  • 效率：避免單一代理處理所有事情
  • 穩定性：一個代理的錯誤不會影響其他代理
  • 可維護性：更容易除錯和更新特定功能

子代理範例

  • 代碼審查代理
  • 測試代理
  • 文件代理
  • 部署代理

系統提示詞 —— Goldilocks 區域

太具體

硬編碼邏輯
缺乏彈性

剛剛好

清晰且具體
足夠的引導

太模糊

缺乏具體指引
行為不一致

太具體的問題

  • 將 LLM 視為確定性狀態機
  • 硬編碼特定邏輯（如：必須問三個後續問題）
  • 變成維護噩夢，每個新邊緣案例都需要修改
  • 可能不適合實際輸入的預定義路徑

太模糊的問題

  • 缺乏可操作的指引
  • 假設模型知道公司、品牌和規範（實際上不知道）
  • 未定義的邊界（如：「如有需要則升級到人工」）
  • 導致不一致的行為，不同執行產生不同方法

系統提示詞最佳實踐

給予指示就像指路

如果只說「去那邊」，他們會困惑。但如果給他們 50 頁的手冊說明每個可能的轉彎和街道，又會讓他們不知所措。我們要清晰、具體、給予剛好足夠的資訊。

良好系統提示詞特徵

  • 清晰的角色定義：明確說明代理是什麼
  • 可用的工具：列出代理可以使用的工具
  • 行為準則：提供如何處理問題的框架
  • 明確的邊界：定義何時升級到人工
  • 迭代演進：系統提示詞隨時間演進
