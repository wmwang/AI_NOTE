# 第 11 章：Understanding Deep Agents — 讓 AI 能做真正的長規劃

之前的章節教過 Subagents、Skills、MCP Server，這些都是讓 Claude Code「單次任務」做得更好的工具。但如果你想讓 AI 處理那種「需要分階段思考、每個階段都依賴前一階段結果」的複雜任務，就需要更深層的概念——**Deep Agents**。

---

## Shallow Agent vs Deep Agent：差別在哪？

### Shallow Agent（淺層 Agent）

你平常跟 Claude Code 對話的方式，本質上就是 Shallow Agent：

```
你：幫我寫一個登入功能
Claude：直接寫出程式碼
```

特點：
- **單次 reasoning**：收到 prompt 之後，思考一次，輸出結果
- **沒有內部規劃階段**：不會先思考「我該分幾步做」，而是直接開始做
- **適合明確任務**：需求清楚、步驟明確的任務做得好

| 任務類型 | Shallow Agent 適用度 |
|---------|-------------------|
| 寫一個函式、修改一個 bug | ✅ 非常適合 |
| 轉換檔案格式、重命名變數 | ✅ 非常適合 |
| 需要多階段設計的系統架構 | ❌ 可能會亂改 |
| 需要研究再實作的新功能 | ❌ 容易踩到沒想到的坑 |

### Deep Agent（深層 Agent）

Deep Agent 的核心是：**在真正動手之前，先做內部規劃和思考**。

```
你：幫我建立一個完整的電商訂單系統
Deep Agent：
  ├─ 內部思考：這個需求需要分析什麼？
  ├─ 研究階段：現有程式碼長怎樣？資料庫 schema？
  ├─ 設計階段：API 端點、資料模型、錯誤處理
  ├─ 實作階段：依照設計寫程式碼
  └─ 驗證階段：測試、確認符合需求
```

特點：
- **多階段 reasoning**：先分析、再設計、再實作、再驗證
- **內部狀態**：每個階段的結果會記下來，傳給下一階段用
- **適合複雜任務**：需求模糊、影響範圍廣、需要多次迭代的工作

---

## 為什麼需要 Deep Agent？

### 問題：Shallow Agent 的「一次到位」思維

讓 Claude Code 直接實作複雜功能，常遇到這幾個問題：

| 問題 | 現象 | 原因 |
|------|--------|------|
| 想到哪寫到哪 | 寫到一半發現忘了考慮某個情況，要重寫 | 沒有先做完整設計 |
| 改錯檔案 | 本來只想改 A，結果連 B、C 都動了 | 沒有先定好 scope |
| 誤解需求 | 做出來的功能跟你想要的差很多 | 沒有先確認需求理解 |
| Context 混亂 | 對話越來越長，Claude 開始胡言亂語 | 沒有規劃 context 分段管理 |

### 解決：Deep Agent 的「分階段」思維

Deep Agent 會把任務拆成多個階段，每個階段專注做一件事：

```
階段 1：理解與分析
  ├─ 複述需求，確認理解正確
  ├─ 列出不清楚的地方，問問題
  └─ 確定要做什麼、不做什麼

階段 2：研究與資訊蒐集
  ├─ 讀取現有程式碼
  ├─ 查相關文件
  └─ 了解技術限制

階段 3：設計與規劃
  ├─ 設計 API 端點
  ├─ 設計資料結構
  └─ 規劃實作順序

階段 4：實作
  ├─ 依照設計寫程式碼
  ├─ 每完成一個元件就測試
  └─ 處理邊緣情況

階段 5：驗證與改進
  ├─ 測試完整流程
  ├─ 確認符合需求
  └─ 做必要的調整
```

這種方式讓每個階段都是「輸入明確、目標清楚」，大大降低錯誤率。

## Deep Agent 的實現方式

### 方式一：用 Planning Mode + 手動串接

這是最簡單的方式，用 `/plan` 進入規劃模式，讓 Claude Code 只能讀不能寫。

```
/plan

幫我設計一個會員積分系統
```

Claude Code 會先研究、設計，產出一份 spec 文件。你 review 之後核准，它再開始實作。

優點：
- 完全手動控制，每個階段你都要同意
- 安全，不會亂改檔案
- 適合需要 review 的團隊環境

缺點：
- 比較慢，需要你多次確認
- 如果你不熟 spec-driven development，可能覺得麻煩

### 方式二：用 MCP Server 串接外部規劃工具

有些 MCP Server 能幫你自動做分階段處理。例如：

- **OpenSpec MCP**：把需求轉成結構化的 change document，包含 design、spec、tasks
- **專門的 deep agent server**：有些第三方 server 實作了完整的 deep agent 邏輯

```
你用 OpenSpec MCP：

1. 創建 change：opsx:new "會員積分系統"
2. OpenSpec 自動生成：
   - Design document（架構設計）
   - Spec（API endpoint、資料模型）
   - Tasks list（實作清單）
3. 你 review 這些文件
4. 執行實作：opsx:apply
5. 驗證：opsx:verify
```

優點：
- 自動化程度高，不用手動切換模式
- 每個階段都有 artifact 留下來，方便追溯
- 適合複雜專案，多階段文件管理很重要

缺點：
- 需要額外安裝 MCP server
- 初期學習成本較高

### 方式三：自訂 Deep Agent Workflow

如果你對 deep agent 的概念有理解，可以用 Claude Code 的設定檔自訂一個「預設就會做分階段」的工作流程。

在 `CLAUDE.md` 加入：

```markdown
## Deep Agent Workflow

當收到複雜任務時，遵循以下階段：

### 階段 1：需求確認
- 複述使用者的需求
- 列出需要確認的問題
- 確定實作範圍（做什麼、不做什麼）

### 階段 2：研究與設計
- 讀取現有程式碼
- 設計 API 端點和資料結構
- 產出設計文件

### 階段 3：實作
- 依照設計寫程式碼
- 每個元件寫完後確認可以跑
- 處理邊緣情況

### 階段 4：驗證
- 測試完整流程
- 確認符合需求
- 回報任何需要調整的地方
```

這樣每次你給複雜任務，Claude Code 就會自動照著這個 workflow 走。

## 什麼時候該用 Deep Agent？

判斷標準很簡單：**問自己「這個任務我能不能一句話說清楚要做什麼？」**

| 能一句說清楚 | 不能一句說清楚 |
|------------|---------------|
| Shallow Agent ✅ | Deep Agent ✅ |
| 幫我寫一個 login 函式 | 幫我建立一個完整的訂單系統 |
| 這個 bug 要怎麼修 | 幫我設計一個會員積分機制 |
| 把這段 code 重構成 cleaner | 幫我做一個能處理各種付款方式的 checkout |

更具體的判斷條件：

**用 Shallow Agent：**
- 需求非常明確（改特定函式、修特定 bug）
- 影響範圍小（只改一兩個檔案）
- 不需要研究（你清楚程式碼現狀）
- 快速迭代、試不同方案

**用 Deep Agent：**
- 需求模糊，需要先確認理解
- 影響範圍廣，涉及多個模組
- 需要研究現有架構和技術限制
- 需要多階段實作和驗證
- 需要產出設計文件給團隊 review

## Deep Agent 的 Context 管理挑戰

Deep Agent 雖然強大，但也帶來一個問題：**Context window 更容易被撐爆**。

### 問題：每個階段都累積 context

```
階段 1 的對話 (20K tokens)
  +
階段 2 的對話 (30K tokens)
  +
階段 3 的對話 (50K tokens)
  +
  ...
  = Context window 爆炸
```

### 解決：分階段 compact / clear

Deep Agent 的工作方式本身就提供了解決方案：**每個階段結束後，主動處理 context**。

```bash
# 階段 1 結束（需求確認完成）
/compact

# 階段 2 結束（設計完成）
/compact

# 階段 3 結束（實作完成）
/clear   # 或者開新 session
```

更進階的做法：用 `/rewind` 回到上一個階段的 checkpoint，但保留重要的決策在記憶裡。

## 實際範例：建立會員積分系統

讓我們用 Deep Agent 的方式走一個完整流程。

### 階段 1：需求確認

```
/plan

幫我設計一個會員積分系統。需求：
- 會員下單獲得訂單金額 1% 的積分
- 積分可以折抵下次訂單（100 積分 = 1 元）
- 積分有效期 1 年
- 需要記錄積分變動歷史
```

Claude Code 會先研究現有程式碼，然後提問：

```
問題：
1. 目前有沒有現成的會員系統可以接？
2. 積分歷史需要保留多久？永久嗎？
3. 積分折抵時，要不要限制最低消費金額？
```

你回答問題，確認需求理解正確。

### 階段 2：設計

Claude Code 產出設計文件：

```markdown
# 會員積分系統設計

## 資料庫 Schema

- points（積分餘額表）
- point_transactions（積分變動歷史表）

## API 端點

- GET /api/points/:userId - 查詢積分
- POST /api/points/earn - 獲得積分
- POST /api/points/redeem - 折抵積分
- GET /api/points/history - 變動歷史

## 業務邏輯

- 訂單完成時自動計算積分
- 積分有效期檢查（建立日期 + 1 年）
- 折抵時檢查餘額足夠
```

你 review 之後，如果滿意就進下一階段。

### 階段 3：實作

```bash
/plan  # 離開規劃模式

根據上一階段的設計，開始實作會員積分系統
```

Claude Code 會：
1. 建立資料庫 migration
2. 建立 API endpoint
3. 實作積分計算邏輯
4. 寫測試

每完成一個部分，它會問你繼續下一個嗎，你有機會中途調整方向。

### 階段 4：驗證

```
測試以下場景：
1. 下單 $1000，確認獲得 10 積分
2. 積分折抵 $5，確認扣除 500 積分
3. 查看積分歷史，確認有紀錄
```

Claude Code 會跑測試，確認功能正常，然後回報結果。

## 本篇重點整理

| 概念 | Shallow Agent | Deep Agent |
|------|-------------|------------|
| **思維方式** | 收到需求後直接開始做 | 先分析、設計、再做 |
| **適合任務** | 明確、簡單、單次推理 | 模糊、複雜、多階段 |
| **Context 管理** | 對話越來越長容易混亂 | 每階段結束後 compact/clear |
| **實現方式** | 直接對話 | Planning Mode / MCP Server / 自訂 workflow |
| **速度** | 快 | 較慢，但品質更高 |

**判斷標準：**
- 一句話能說清楚要做什麼 → Shallow Agent
- 需要多階段研究、設計、實作 → Deep Agent

**三個立即可以做的事：**

1. **試一次 Planning Mode**：下一個比較複雜的功能，先輸入 `/plan`，讓 Claude Code 幫你寫設計文件，你再核准實作。體驗一下「先想清楚再動手」的好處。

2. **學會 `/compact`**：對話開始變長、Claude 的回答感覺跑掉時，試試 `/compact`，看看壓縮後 context 清不乾淨。

3. **把複雜任務拆階段**：不是每次都交給 Claude 自己判斷。你可以主動說「這個任務分三個階段做：先確認需求，再設計，最後實作」，讓整個流程更可控。
