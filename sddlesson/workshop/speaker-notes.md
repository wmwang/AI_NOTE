## M1：規格先行（0:15 - 0:45）

### 投影片：規格的問題

OK，M1 開始。

我先問大家：你們公司的需求文件放在哪裡？

*（讓大家說：Notion、Jira、Confluence、Word...）*

這些都是很好的工具，但它們有一個共同的問題：**靜態文件和動態程式碼天然漂移**。

需求文件寫在 Notion，程式碼在 GitHub。改了需求，Notion 更新了，但程式碼不知道。改了程式碼，Notion 不會自動更新。

更重要的是：**AI 讀不懂 Notion**。即使你把 Notion 頁面貼給 AI，它也不知道哪個是最新版本、哪個才是真正的驗收條件。

真正的問題不是「規格寫得不夠詳細」，而是「規格沒有跟程式碼綁在一起」。

---

### 投影片：可執行規格

那什麼叫「可執行規格」？

看這個對比：

*（念出傳統規格）* 「使用者可以用 Google 帳號登入系統」

這句話給 AI，AI 不知道：Google 登入後要跳轉哪裡？新用戶和舊用戶的行為一樣嗎？如果 Google 授權失敗怎麼辦？AI 會自己猜，每次猜的可能不同。

*（念出 Gherkin）*

```
Given 使用者尚未登入
When  使用者點擊「用 Google 登入」並完成授權
Then  使用者應被導向首頁
And   頂部導航列應顯示使用者名稱
```

這個格式，你可以直接把它轉換成測試程式碼。「完成」的定義是清楚的——Then 就是你的驗收條件。

---

### 投影片：Discovery → Formulation → Automation

這是我們今天工作坊的核心流程。

**Discovery**：和 PO、QA 一起做的探索工作。問三個問題：誰在用、在什麼情況下用、用了之後系統怎麼改變。這一步是人做的，AI 幫不了你——因為業務知識在人的腦子裡。

**Formulation**：把探索結果寫成 Gherkin。這一步可以 AI 協助，但主導的還是人。

**Automation**：AI 根據 Gherkin 生成測試和實作。這一步 AI 可以做 80% 的工作。

重點是：**人做策略，AI 做戰術**。Gherkin 是策略和戰術之間的橋樑。

---

### 投影片：Gherkin 語法入門

我帶大家看 Gherkin 語法。不需要背，講義裡有速查表。

`Feature`：功能模組的名稱。

`Background`：所有 Scenario 共用的前置條件，不用每個 Scenario 都重複寫。

`Scenario`：一個具體的使用情境。一個 Feature 通常有 3-10 個 Scenario。

`Given`：前置狀態——系統現在是什麼樣子？

`When`：觸發動作——使用者做了什麼？**只有一個動作**，不要塞太多。

`Then`：預期結果——系統應該有什麼反應？這是你的 assertion。

`And`：連接多個 Given、When、或 Then。

這個格式設計的初衷是：讓業務人員（PO、QA）和工程師都能讀懂同一份文件。

---

### 投影片：DSL vs. ISA Level

這是今天最重要的概念之一。Gherkin 有兩個層次：

**DSL Level**：業務語言。抽象，讓 PO 和 QA 可以讀懂、參與討論。不含技術細節。

**ISA Level**：整合規格層級。具體，含有真實參數、HTTP endpoint、DB 欄位名稱。這是 AI 生成測試所需的格式。

*（指著投影片的兩段對比）*

看這個「折扣計算」的例子：
DSL Level 說「套用折扣碼 SAVE10（打九折）」——業務人員理解。
ISA Level 說「discount_code: 'SAVE10'（discount_rate: 0.9），呼叫 calculateOrderTotal(items, discountCode)，回傳值應為 360」——AI 可以直接生成測試。

你需要維護兩個 Level，但流程是：先寫 DSL，再展開 ISA。

*（帶大家一起做 Discovery 三問）*

## M2：AI × TDD（0:45 - 1:30）

### 投影片：TDD 核心

M2 開始，這是今天最核心的技術模組。

TDD 大家聽過，但可能沒有持續在做。原因通常是：寫測試太麻煩了。

AI 的出現改變了這個等式。

Red-Green-Refactor 這個循環：先寫一個會失敗的測試，再寫最少的程式碼讓測試通過，最後在有測試保護的情況下重構。

關鍵心態：測試是規格的可執行形式，不是事後補的東西。你有了 Gherkin，測試就有了材料。

---

### 投影片：AI 在 TDD 的位置

我帶大家看這個流程圖。

*（逐步說明）*

你負責的：Discovery、寫 DSL Gherkin、展開 ISA Gherkin、執行測試。

AI 幫你做的：把 ISA Gherkin 轉成測試程式碼、生成讓測試通過的實作、提供重構建議。

注意一件事：「執行測試」還是你做的。因為你要確認 RED（測試確實失敗），確認 GREEN（測試通過），確認 REFACTOR 後還是 GREEN。這是你的責任。

這樣的分工，開發速度可以提升 3-5 倍，但品質不會下降，因為你還在掌控方向。

---

### 投影片：Prompt 1

我們來看四個核心 Prompt。

Prompt 1 是從 ISA Gherkin 生成測試。

*（逐行解說 Prompt 的每個部分）*

「你是一位資深 TDD 工程師」——給角色，影響 AI 的思維框架。

「只生成測試，不實作被測函式」——這是 RED Phase，我們要先看到失敗的測試，不要讓 AI 跳過去直接給你成功的實作。

「Given → Arrange，When → Act，Then → Assert」——這個對應告訴 AI 怎麼把 Gherkin 翻譯成程式碼結構。

### 投影片：Demo - Prompt 1 輸出

我在工作坊前，用一個 Gherkin 生成了這個測試。

*（帶大家看程式碼）*

注意幾件事：
AI 自動推導出 UserService 這個介面——因為 Gherkin 的語意暗示了「有一個服務」。

---

### 投影片：Prompt 2 - Green Phase

現在我們要讓測試通過。

Prompt 2 的關鍵詞：「最小實作」。

不要做超過測試要求的任何事。如果測試只要求 `updateProfile()` 回傳更新後的 nickname，就只做這件事，不要順手加分頁、不要加 audit log、不要加 cache。

這對很多工程師來說很反直覺——我們天生想要設計「完美的架構」。但 TDD 說：先讓它能用，再讓它更好（那是 REFACTOR 的事）。

AI 在這個 Phase 非常強大，它可以快速生成符合測試的最小實作。

---

### 投影片：Prompt 3 - Edge Case 挖掘

這個 Prompt 是品質保障的利器。

我想問大家：你通常怎麼想到 edge case？

*（讓大家說：靠經驗、靠 QA 回報、靠線上 bug）*

線上 bug 是最貴的 edge case 發現方式。

AI 可以幫你系統性地挖掘 edge case。你給它 Happy Path，它從四個維度分析：輸入邊界、狀態邊界、業務規則邊界、並發情況。

這不是說 AI 比你更有業務知識——它不是。但它可以系統性地問問題，讓你確認「這個情況業務規則是什麼」。

### 投影片：Exercise 2 說明

主要練習。25 分鐘。

情境是「折扣計算引擎」。你要：
1. 展開一個 ISA Gherkin
2. 用 Prompt 1 生成測試
3. 用 Prompt 2 生成實作

重點不是測試能不能在你的電腦上跑起來（環境問題很常見）。重點是你走完了這個思維流程：Gherkin → 測試 → 實作。

## M3：Context Engineering（1:40 - 2:05）

### 投影片：為什麼 Prompt 不穩定

大家回來了，M3 開始。

休息時有沒有人在想：「如果我的 Prompt 已經很好，為什麼 AI 的輸出還是會變？」

原因是：每次對話，AI 面對的是不同的 context。你沒有說清楚的部分，AI 會自己填補。每次填補的方式不同，輸出就不同。

看這個對比：

*（念出兩個 Prompt）*

「幫我寫一個購物車的測試」——AI 要猜：什麼語言？什麼框架？測試風格？結果每次不同。

「根據以下 ISA-Level Gherkin，使用 TypeScript + Jest，生成 Step Definitions...」——AI 不需要猜，結果穩定。

---

### 投影片：Context Engineering 三要素

Context Engineering 是一個系統性地給 AI 提供背景資訊的方法。

三個要素：角色、格式、範例。

**角色**：就像雇用一個顧問，先告訴他他的背景。「你是一位資深 TDD 工程師」和「幫我寫測試」給 AI 的訊號完全不同。角色影響 AI 的思維框架。

**格式**：AI 默認會加很多解釋文字。如果你只需要程式碼，要明說。「只輸出程式碼，不要任何解釋文字」——讓輸出直接可用。

**範例**（Few-shot）：這是最強的工具。給 AI 一個輸入/輸出對，AI 的模仿能力非常強。這讓你把隱性標準變成顯性規格。

---

### 投影片：Prompt 模板化

好的 Prompt 是資產，不是一次性的提問。

*（展示模板格式）*

把你調整好的 Prompt 存進一個 `prompt-library.md`，把它放在 Git repo 裡，和程式碼一起版本控制。

好處：
你的團隊所有人都能用同一套 Prompt。
Prompt 也可以做 code review——有更好的寫法就提 PR。
Prompt 的效果可以被追蹤（版本 A vs. 版本 B，哪個更穩定）。

今天的講義裡有一個 Prompt 模板庫的起點，你可以直接帶回去用。

---

### 投影片：Source-Driven Development

我要介紹 agent-skills 裡的一個重要概念：Source-Driven Development。

「每一個框架決策都應該基於官方文件，而不是 AI 的記憶。」

AI 的知識有截止日期。你用一個較新的 npm 套件，AI 可能記得舊版的 API。

解法：把官方文件貼進 Prompt。不是讓 AI 靠記憶告訴你怎麼用，而是「根據以下文件，生成符合這個 API 的程式碼」。

特別適合：新版框架、你不熟悉的套件、有破壞性更新的版本。

## M4：完整工作流（2:05 - 2:35）

### 投影片：完整工作流

OK，M4 是把前面所有東西串起來。

*（逐步解說流程圖）*

Discovery → Event Storming → DSL Gherkin：確認業務理解。

API-First 後端三巨頭：在寫任何程式碼之前，定義好 API Contract、DB Schema、Domain Events。

ISA Gherkin：把 DSL 展開成 AI 可以生成測試的具體規格。

並行開發：前端不用等後端。前端用 Mock Server 模擬後端，後端用 AI x BDD 開發。

整合：有了 Contract，整合時的差異立刻浮現，不用靠猜測。

---

### 投影片：API-First 哲學

API-First 解決了前後端協作最大的痛點：等待。

傳統模式：後端開發完，前端才能接。整合時發現不符合，返工。

API-First：前後端共同定義 API Contract。前端用 Mock Server 開發，後端用 TDD 實作。整合時，Contract 是唯一的驗收標準。

這個做法的前提：**Contract 要先於程式碼存在**。這需要工作坊結束後，你跟你的前後端同事一起在 Sprint 開始前，先花一個小時對齊 API。

---

### 投影片：後端三巨頭

每個後端功能在開工前，都要確認這三份文件：

API Contract：這個 endpoint 接受什麼、回傳什麼、有哪些錯誤碼。
DB Schema：需要哪些新的 Table 或欄位。
Domain Events：這個功能完成後，會發出什麼事件給其他服務。

*（展示「建立訂單」的三巨頭範例）*

有了這三份文件，前端知道要 Mock 什麼、後端知道要實作什麼、QA 知道要驗什麼。AI 有了足夠的 Context 可以生成正確的程式碼。

### 投影片：Live Demo

好，我們現在走一遍完整流程。功能是「使用者可以訂閱電子報」。

Context：在首頁或文章底部看到訂閱入口。
Outcomes：成功訂閱（得到確認信）、Email 格式錯誤、已訂閱過。

*（DSL Gherkin）*
*（快速寫出三個 Scenario）*

*（後端三巨頭）*
API: POST /api/subscriptions

## M5：企業導入（2:35 - 2:50）

### 投影片：J Curve

導入新流程一定有 J Curve。開始的時候會比之前慢，然後慢慢超過。

預期：第 2-4 週最痛苦，第 6-8 週開始回本，第 3 個月後成為標準流程。

這不是鼓勵你忍耐，而是讓你對自己誠實：如果第一週感覺很慢，不要放棄。

---

### 投影片：三段式導入路徑

第一階段：個人 MVP。下週找一個要開發的功能，用這套方法做一次。記錄時間。

第二階段：雙人協作。找一個同事，一起建立 Prompt 模板庫，在 Sprint 裡加入 Gherkin Review 環節。

第三階段：團隊標準化。把 Gherkin 放進 User Story 模板，PR 必須附帶對應的 Gherkin，CI 自動跑 BDD 測試。

不要跳過第一階段就想做第三階段。先讓自己成功一次，再分享給別人。

---

### 投影片：量表評估

花兩分鐘，誠實評估你的團隊現狀。1 到 5 分。

*（讓大家安靜評估，然後問）*

最低分的那個維度是什麼？那就是你的第一優先。

一次只改一件事。

---

### 投影片：常見阻力

*（逐條說，如果有人有共鳴，讓他們分享）*

「寫 Gherkin 太花時間」：先只在複雜功能用，簡單 CRUD 不強制。

「PO 不願意參與 Discovery」：先讓 Dev + QA 寫，事後給 PO 確認。

「舊系統沒有測試」：針對新功能導入，舊的不動。這叫「Boy Scout Rule」——每次進一個地方，讓它比進去前更乾淨一點。

「Senior 工程師不信任 AI」：這是最有意思的一個。解法不是說服他信任 AI，而是讓他做最難的那部分——規格設計、Prompt 工程——AI 做最繁瑣的那部分——程式碼生成。Senior 的角色不是被取代，而是被放大。
