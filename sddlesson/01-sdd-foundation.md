# Ch1：規格驅動開發的前提

## 為什麼「把規格寫清楚」是假議題？

大多數團隊以為問題是「需求文件寫得不夠好」，但真正的問題是：

- 需求文件是靜態的，程式碼是動態的，兩者天然漂移
- 規格寫在 Notion/Confluence，但 AI（和測試）讀不懂
- 沒有可執行的規格，就沒有可驗證的「完成」定義

**核心命題：規格必須是可執行的（Executable Specification）**

---

## Discovery → Formulation 流程

```
Discovery（探索）
  誰在用？用在哪個情境？會發生什麼事？
       ↓
Formulation（形式化）
  把探索結果寫成 Gherkin 情境
       ↓
Automation（自動化）
  AI 根據 Gherkin 生成測試程式碼 + 實作程式碼
```

### 實戰：三個關鍵問題（Discovery 用）

1. **Actor**：誰在做這件事？（用戶角色）
2. **Context**：在什麼前提/狀態下？（Given 的來源）
3. **Outcome**：做了之後，系統狀態如何改變？（Then 的來源）

## Gherkin 語言入門

Gherkin 是 BDD 的「規格語言」，結構固定，可被工具解析。

```gherkin
Feature: 使用者登入

  Background:
    Given 系統中存在帳號 "alice@example.com"

  Scenario: 正確密碼登入成功
    Given 使用者在登入頁面
    When  使用者輸入正確的帳號與密碼
    Then  使用者應被導向首頁
    And   畫面應顯示 "歡迎回來，Alice"

  Scenario: 密碼錯誤登入失敗
    Given 使用者在登入頁面
    When  使用者輸入錯誤的密碼（連續 3 次）
    Then  帳號應被鎖定 30 分鐘
    And   畫面應顯示錯誤訊息 "帳號已鎖定"
```

### Gherkin 關鍵字對照

| 關鍵字 | 用途 | 對應問題 |
|--------|------|---------|
| `Feature` | 功能名稱 | 這個功能是什麼？ |
| `Background` | 共用前置條件 | 所有情境的共同前提 |
| `Scenario` | 單一使用情境 | 一個具體的「例子」 |
| `Given` | 前置狀態 | 系統現在是什麼狀態？ |
| `When` | 觸發動作 | 使用者做了什麼？ |
| `Then` | 預期結果 | 系統應有何反應？ |
| `And` / `But` | 連接多個條件 | 補充說明 |

## 用「模型思維」驅動規格

不釐清模型就寫規格，等於在沙地上蓋房子。

### 三問模型（適用 Discovery 階段）

```
1. 這個功能的「主角」是誰？（Actor）
2. 這個功能的「核心業務事件」是什麼？（Domain Event）
3. 這個功能的「邊界條件」有哪些？（Boundary / Edge Cases）
```

### 練習：電商「建立訂單」

| 問題 | 答案 |
|------|------|
| Actor | 已登入的顧客 |
| Happy Path Event | 顧客結帳 → 訂單建立 → 庫存扣減 |
| Edge Cases | 庫存不足、付款失敗、優惠券已過期 |

## 沒有規格時，AI 開發的問題

| 情況 | 問題 |
|------|------|
| 需求口頭描述 | AI 每次理解不同，輸出不穩定 |
| 需求模糊 | AI 會「腦補」，填入開發者偏好的預設值 |
| 沒有驗收條件 | AI 無法自我驗證，Debug 靠人工肉眼 |
| 規格與測試分離 | 改了規格，測試沒跟著改，靜默失敗 |

**結論：Gherkin 是 AI 的「輸入格式」，讓 AI 有東西可以對齊。**
