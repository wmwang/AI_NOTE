# 5.4 使用依賴項

現代測試框架提供了定義間諜（spy）、樁件（stub）和模擬物件（mock）的選項，這些都被統稱為測試替身（test double）。你可以將其想像為一個函式，你可以檢查其呼叫次數（使用間諜）、預先定義其行為（使用樁件），或者定義互動規則並在測試結束時檢查是否遵守了這些規則（使用模擬物件）。特別是間諜和樁件在撰寫單元測試時非常有用。如果你使用回呼函式（例如事件處理器），就無法避免使用間諜函式。

在接下來的章節中，我們將重點介紹如何使用樁件來為具有額外依賴項的函式撰寫單元測試。在單元測試中，你專注於原始碼中的特定單元，通常是一個函式或方法。一旦這個單元與應用程式的其他部分協同運作，情況就會變得更加複雜。一方面，測試不再只因為一個函式而失敗；另一方面，你可能需要覆蓋多個流程路徑，這意味著需要執行更多的測試。

為了讓整件事更具體化，我們在以下範例中假設有一個名為 `OrderService` 的 TypeScript 類別。這個類別有一個 `placeOrder` 方法，可以讓你儲存訂單。在儲存時，你的應用程式會為訂單添加額外資訊，並使用另一個類別——`OrderRepository` 類別——來持久化訂單。這兩個類別的簡化原始碼如下：

```typescript
import { randomUUID } from "node:crypto";
type Item = {
  id: string;
  name: string;
  price: number;
  quantity: number;
};
type Order = {
  id: string;
  items: Item[];
  total: number;
  status: "new" | "pending" | "completed" | "cancelled";
  createdAt?: Date;
};
export class OrderRepository {
  async save(order: Order): Promise<Order> {
    return {
      ...order,
      id: randomUUID(),
      createdAt: new Date(),
    };
  }
}
export class OrderService {
  constructor(private orderRepository: OrderRepository) {}
  placeOrder(order: Order) {
    const total = order.items.reduce((acc, item) => {
      return acc + item.price * item.quantity;
    }, 0);
    const orderRecord: Order = {
      ...order,
      total,
      status: "pending",
    };
    return this.orderRepository.save(orderRecord);
  }
}
```

當 `OrderService` 被實例化時，它會接收一個 `OrderRepository` 類別的實例。如果你使用 `Order` 物件呼叫 `placeOrder` 方法，該方法會為此物件添加訂購項目的總金額，並將狀態設為 `pending`。然後該方法會呼叫 `OrderRepository` 物件的 `save` 方法。這個方法反過來會為訂單分配一個唯一的 ID 並設定建立日期。

我們選擇的 AI 工具——在本例中是 ChatGPT——應該為 `placeOrder` 方法生成一個測試，並確保不會呼叫 `OrderRepository` 的 `save` 方法，而是用一個樁件方法來替代它。

**提示：** 為 `placeOrder` 方法撰寫一個 Vitest 測試，並用樁件取代 `OrderRepository` 的 `save` 方法。

[範例程式碼]

以下是如何使用 Vitest 為 `OrderService` 類別中的 `placeOrder` 方法撰寫測試。在此測試中，我們將用樁件取代 `OrderRepository` 的 `save` 方法。

首先，確保你的專案中已安裝並設定好 Vitest。然後，建立一個測試檔案，例如 `OrderService.test.ts`，並加入以下程式碼：

```typescript
import { describe, it, expect, vi } from 'vitest';
import { OrderService, OrderRepository, Order, Item } from './path-to-your-order-service';
describe('OrderService', () => {
  it('should place an order and calculate the total', async () => {
    // 建立 OrderRepository save 方法的樁件
    const saveStub = vi.fn().mockResolvedValue({
      id: 'stubbed-id',
      items: [],
      total: 100,
      status: 'pending',
      createdAt: new Date(),
    });

    // 建立模擬的 OrderRepository
    const orderRepository = new OrderRepository();
    orderRepository.save = saveStub;
    // 使用模擬的 OrderRepository 實例化 OrderService
    const orderService = new OrderService(orderRepository);
    // 定義範例訂單
    const order: Order = {
      id: '',
      items: [
        { id: '1', name: 'item1', price: 10, quantity: 5 },
        { id: '2', name: 'item2', price: 20, quantity: 2 },
      ],
      total: 0,
      status: 'new',
    };
    // 下訂單
    const result = await orderService.placeOrder(order);
    // 斷言
    expect(saveStub).toHaveBeenCalledOnce();
    expect(saveStub).toHaveBeenCalledWith(
      expect.objectContaining({
        total: 90,
        status: 'pending',
      }));
    expect(result).toEqual(expect.objectContaining({
      id: 'stubbed-id',
      total: 90,
      status: 'pending',
    }));
  });
});
```

[測試說明]

測試首先透過建立 `save` 方法的樁件來準備所需的環境。然後建立一個新的 `OrderRepository`，將樁件插入其中，並生成一個新的 `OrderService` 實例。測試最重要的部分是呼叫 `placeOrder` 方法。為此，測試定義了一個代表訂單的訂單物件，並在呼叫方法時將其傳入。測試以以下檢查作結：

- `saveStub` 是否只被呼叫了一次？
- `saveStub` 是否使用了正確的物件進行呼叫？
- `placeOrder` 方法的回傳值是否正確？

目前為止，測試看起來不錯，你可以直接在應用程式中整合並執行它。然而，你會看到一個嚴重的問題，因為測試在檢查 `placeOrder` 方法的回傳值時會失敗。原因是樁件函式回傳的值是 100，而不是要求的 90。`items` 和 `createdAt` 屬性也缺失了。

你可以透過將錯誤訊息複製到下一個提示中來解決這個問題。在這種情況下，ChatGPT 會修正答案並提供一個正常運作的測試。

錯誤顯示斷言中期望值與接收值之間存在差異。具體來說，`total` 值和 `createdAt` 欄位不同。讓我們更新測試以正確處理這些差異。以下是更新後的測試程式碼：

```typescript
import { describe, it, expect, vi } from 'vitest';
import { OrderService, OrderRepository, Order, Item } from './path-to-your-order-service';
describe('OrderService', () => {
  it('should place an order and calculate the total', async () => {
    // 建立 OrderRepository save 方法的樁件
    const saveStub = vi.fn()
      .mockImplementation(async (order: Order) => ({
        ...order,
        id: 'stubbed-id',
        createdAt: new Date(),
      }));
    // 建立模擬的 OrderRepository
    const orderRepository = new OrderRepository();
    orderRepository.save = saveStub;
    // 使用模擬的 OrderRepository 實例化 OrderService
    const orderService = new OrderService(orderRepository);
    // 定義範例訂單
    const order: Order = {
      id: '',
      items: [
        { id: '1', name: 'item1', price: 10, quantity: 5 },
        { id: '2', name: 'item2', price: 20, quantity: 2 },
      ],
      total: 0,
      status: 'new',
    };
    // 下訂單
    const result = await orderService.placeOrder(order);
    // 斷言
    expect(saveStub).toHaveBeenCalledOnce();
    expect(saveStub).toHaveBeenCalledWith(expect
      .objectContaining({
        total: 90,
        status: 'pending',
      }));
    expect(result.id).toBe('stubbed-id');
    expect(result.total).toBe(90);
    expect(result.status).toBe('pending');
    expect(result.items).toEqual(order.items);
    expect(result.createdAt).toBeInstanceOf(Date);
  });
});
```

在這個更新後的測試中：

1. `saveStub` 現在使用 `mockImplementation` 來回傳添加了 `id` 和 `createdAt` 欄位的訂單物件，更接近地模擬實際的儲存操作。
2. 斷言被分解為逐個檢查每個屬性，使測試更加清晰，並避免了因其他欄位（如 `createdAt`）的差異而導致的深層相等性檢查失敗。

ChatGPT 在調整原始碼時精確地解決了兩個關鍵點。樁件的實作現在基於傳入的物件，只添加兩個屬性。在檢查回傳值時，測試現在逐個屬性進行檢查，因此也不會再出現錯誤。如果你將此測試整合到你的環境中，它將在沒有錯誤訊息的情況下執行。
