# 6.2 行內文件

## 何謂行內文件？

行內文件（Inline Documentation）指的是直接嵌入在程式碼中的註解與說明文字，用於解釋程式碼的意圖、邏輯與行為。良好的行內文件能幫助開發者（包括未來的自己）快速理解程式碼的運作方式，而不需要逐行推敲。

行內文件通常包含：

- **行內註解（Inline Comments）**：放在程式碼行內或行尾的簡短說明
- **區塊註解（Block Comments）**：用於解釋一段邏輯或演算法的多行註解
- **TODO / FIXME 標記**：標記待辦事項或需要修正的問題

## 為什麼行內文件很重要？

1. **提高可讀性**：好的註解能讓程式碼更容易理解
2. **降低維護成本**：新成員能更快上手，減少溝通成本
3. **記錄設計決策**：解釋「為什麼」這樣做，而不只是「做什麼」
4. **協助除錯**：適當的註解能幫助追蹤問題的根源

## 行內文件的最佳實踐

### 1. 解釋「為什麼」，而非「是什麼」

程式碼本身已經說明了「做什麼」，註解應該解釋「為什麼」要這樣做。

**❌ 不好的例子：**
```python
# 將價格乘以 1.08
total = price * 1.08
```

**✅ 好的例子：**
```python
# 加上 8% 營業稅（台灣標準稅率）
total = price * 1.08
```

### 2. 避免冗餘的註解

不要只是用文字重複程式碼的內容。

**❌ 冗餘的註解：**
```javascript
// 宣告變數 i 並設為 0
let i = 0;

// 當 i 小於陣列長度時執行迴圈
while (i < array.length) {
    // 將 i 加 1
    i++;
}
```

**✅ 有意義的註解：**
```javascript
// 使用 index 遍歷陣列以避免 for...in 的原型鏈問題
let i = 0;
while (i < array.length) {
    i++;
}
```

### 3. 在複雜邏輯前加上區塊註解

當程式碼邏輯較為複雜時，在上方加上簡短的區塊註解說明整體邏輯。

```python
# 使用二分搜尋法在已排序的陣列中尋找目標值
# 時間複雜度：O(log n)，適合大規模資料查詢
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1
```

### 4. 使用 TODO 和 FIXME 標記

這些標記能幫助團隊快速定位需要關注的程式碼區域。

```python
# TODO: 這個快取策略需要加入過期時間機制
# FIXME: 當輸入為空字串時會拋出例外
```

### 5. 在非直觀的行為上加註解

如果程式碼的行為不是直觀的，一定要加上註解說明。

```javascript
// setTimeout 使用 0ms 延遲是為了將回調推入事件佇列，
// 讓瀏覽器有機會先處理 UI 更新
setTimeout(() => {
    processHeavyTask();
}, 0);
```

## 使用 AI 輔助撰寫行內文件

AI 工具可以幫助你快速為程式碼添加高品質的行內文件。以下是有效的使用方式。

### **提示：** 為現有程式碼添加行內註解

```
請為以下程式碼添加適當的行內註解，解釋每個重要步驟的目的和邏輯：

[貼上程式碼]

要求：
1. 註解應解釋「為什麼」而非「是什麼」
2. 在複雜邏輯處加上區塊註解
3. 標記任何潛在的問題或 TODO 項目
```

### **提示：** 檢視並改善現有註解

```
請檢視以下程式碼中的註解品質，並提出改善建議：

[貼上程式碼與註解]

請從以下方面評估：
1. 註解是否解釋了「為什麼」？
2. 是否有冗餘或無用的註解？
3. 複雜邏輯是否有充分的說明？
4. 註解是否與程式碼保持一致？
```

### **提示：** 為複雜演算法加上詳細說明

```
請為以下演算法添加詳細的行內文件，包括：
1. 演算法的整體策略說明
2. 每個步驟的解釋
3. 時間與空間複雜度分析
4. 邊界條件的處理說明

[貼上演算法程式碼]
```

## 常見的行內文件陷阱

| 陷阱 | 說明 | 解決方案 |
|------|------|----------|
| 過時的註解 | 註解與程式碼不一致 | 修改程式碼時同步更新註解 |
| 冗餘的註解 | 只重複程式碼內容 | 聚焦於解釋意圖和原因 |
| 過度註解 | 連簡單的程式碼都加註解 | 只在必要處加註解 |
| 註解掉程式碼 | 保留被註解的舊程式碼 | 使用版本控制，刪除舊程式碼 |
| 模糊的註解 | 註解意義不明確 | 使用清晰、具體的描述 |

## 總結

良好的行內文件是程式碼品質的重要指標。記住以下原則：

1. **解釋「為什麼」**，而非「是什麼」
2. **避免冗餘**，讓每一行註解都有價值
3. **保持更新**，確保註解與程式碼同步
4. **善用 AI**，利用 AI 工具快速產生高品質的行內文件
5. **持續改善**，定期檢視和更新註解品質
