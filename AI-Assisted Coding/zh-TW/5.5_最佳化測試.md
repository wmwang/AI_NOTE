# 5.5 最佳化測試

AI 工具不僅可以幫助你實作測試，還可以幫助你改進測試。你已經在為一系列值實作大量測試時看到了一個這樣的範例。但可能性不止於此。在接下來的章節中，我們將探討兩個額外的使用案例：改進現有測試和實作缺失的測試。

## 5.5.1 最佳化測試程式碼

在實作測試時，有時會出現測試正確檢查了所需功能並因此達到了目的，但實作方式並非最佳的情況。一般來說，測試與所有其他程式碼適用相同的規則：它們應該符合專案的品質標準。AI 工具可以透過改善測試程式碼來簡化你的工作。然而，這並不意味著你不再需要關注原始碼的品質。相反地，在這種情況下，AI 是一個指示器，告訴你哪裡還有改進的空間。

以 `UserService` TypeScript 類別為例。它管理一個使用者物件的陣列，可以新增、讀取和刪除它們。其實作如下：

```typescript
export type User = {
  id: number;
  name: string;
};
export class UserService {
  private users: User[];
  constructor() {
    this.users = [];
  }

  addUser(user: User): User {
    this.users.push(user);
    return user;
  }
  getUserById(id: number): User | undefined {
    return this.users.find(user => user.id === id);
  }
  deleteUserById(id: number): User | null {
    const index = this.users.findIndex(user => user.id === id);
    if (index !== -1) {
      return this.users.splice(index, 1)[0] || null;
    }
    return null;
  }
}
```

對於這個類別，你可以撰寫如下的一系列測試：

```typescript
import { expect, it } from 'vitest';
import { UserService } from './UserService';
import type { User } from './UserService';
it('addUser should add a user to the list', () => {
  const userService = new UserService();
  const user: User = { id: 1, name: 'Alice' };
  userService.addUser(user);
  expect(userService['users']).toContain(user);
});
it('getUserById should return the correct user', () => {
  const userService = new UserService();
  const user1: User = { id: 1, name: 'Alice' };
  const user2: User = { id: 2, name: 'Bob' };
  userService.addUser(user1);
  userService.addUser(user2);
  expect(userService.getUserById(1)).toBe(user1);
  expect(userService.getUserById(2)).toBe(user2);
});
it('getUserById should return undefined for non-existing user', () => {
  const userService = new UserService();
  expect(userService.getUserById(99)).toBeUndefined();
});
it('deleteUserById should delete the correct user', () => {
  const userService = new UserService();
  const user1: User = { id: 1, name: 'Alice' };
  const user2: User = { id: 2, name: 'Bob' };
  userService.addUser(user1);
  userService.addUser(user2);
  userService.deleteUserById(1);
  expect(userService['users']).not.toContain(user1);
  expect(userService['users']).toContain(user2);
});
it('deleteUserById should return null for non-existing user', () => {
  const userService = new UserService();
  expect(userService.deleteUserById(99)).toBeNull();
});
```

這些測試涵蓋了所有三種情況，即新增、讀取和刪除。儘管如此，這些測試仍有改進的空間：

**物件重複**
`UserService` 類別和使用者物件有多個實例。

**未使用設定和清除常式**
可以在測試前初始化環境，並在測試後進行清理。

**結構不夠清晰**
測試程式碼沒有組織結構，導致難以理解、維護和擴展。

你可以透過適當的提示輕鬆解決這個問題。在這裡，同樣的原則適用：你對期望的描述越精確，結果就會越好。例如，你可以指出你期望一個清晰的結構、應盡可能消除程式碼中的重複，並且通常應遵循簡潔程式碼（clean code）的最佳實踐。

**提示：** 請最佳化以下用 TypeScript 撰寫的 Vitest 測試。為測試引入清晰的結構，消除重複，並使用簡潔程式碼原則。

[測試程式碼]

為了最佳化給定的測試，我們可以引入一個通用的設定函式，使用一致的命名慣例，並移除重複的程式碼。此外，我們可以確保每個測試都專注於單一行為。以下是重構後的程式碼：

```typescript
import { expect, it, describe, beforeEach } from 'vitest';
import { UserService } from './userService';
import type { User } from './userService';
describe('UserService', () => {
  let userService: UserService;
  let user1: User;
  let user2: User;
  beforeEach(() => {
    userService = new UserService();
    user1 = { id: 1, name: 'Alice' };
    user2 = { id: 2, name: 'Bob' };
  });
  it('should add a user to the list', () => {
    userService.addUser(user1);
    expect(userService['users']).toContain(user1);
  });
  it('should return the correct user by ID', () => {
    userService.addUser(user1);
    userService.addUser(user2);
    expect(userService.getUserById(1)).toBe(user1);
    expect(userService.getUserById(2)).toBe(user2);
  });
  it('should return undefined for non-existing user by ID', () => {
    expect(userService.getUserById(99)).toBeUndefined();
  });
  it('should delete the correct user by ID', () => {
    userService.addUser(user1);
    userService.addUser(user2);
    userService.deleteUserById(1);
    expect(userService['users']).not.toContain(user1);
    expect(userService['users']).toContain(user2);
  });
  it('should return null when deleting a non-existing user by ID', () => {
    expect(userService.deleteUserById(99)).toBeNull();
  });
});
```

[程式碼說明]

修訂後的測試品質顯著提升。它們包含在一個 `describe` 區塊中，為測試定義了一個上下文。`userService`、`user1` 和 `user2` 變數被測試重複使用，而 `beforeEach` 函式中的設定常式確保服務和使用者物件被正確實例化。這意味著每個測試都有一個乾淨的環境，測試之間的依賴關係被消除了。

儘管如此，還有一個小的批評點。在 `getUserById` 和 `deleteUserById` 的測試中，使用了 `addUser` 方法將資料寫入服務。如果這個方法有誤，其他測試也會失敗，而測試通常應該只因一個原因而失敗。更優雅的選擇是直接將資訊寫入服務或在初始化期間傳入資料。

## 5.5.2 撰寫缺失的測試

在開發應用程式時，你應該嘗試使用自動化測試來驗證整個程式碼。然而，在某些情況下，這是不可能的或太耗時。那麼，你至少應該檢查關鍵點。一個在這裡非常有用的工具是覆蓋率報告（coverage report）。在測試執行期間，它會測量原始碼中哪些行、語句和分支被測試所檢查。一般來說，80% 的覆蓋率是目標值。如果你採用測試驅動開發，這個值可以達到 100%。

大多數現代測試框架都內建支援覆蓋率報告或作為擴充功能提供。對於我們在本章中使用的 Vitest 框架，你可以在 V8 和 Istanbul 報告器之間選擇，前者是預設的。要生成報告，你需要使用 `npm install -D @vitest/coverage-v8` 命令安裝擴充功能，然後透過輸入 `vitest run --coverage` 命令來執行測試。Vitest 接著會在一系列 HTML 檔案中生成覆蓋率報告，你可以在瀏覽器中查看。

作為一個具體的範例，我們再次使用前一個範例中的 `UserService` 類別，但功能略有縮減。該類別現在只有 `addUser` 和 `getUserById` 方法。在這兩個方法中，只有 `addUser` 方法被測試覆蓋。圖 5.1 顯示了該檔案的覆蓋率報告。

以下是 `UserService` 類別的擴展測試套件，涵蓋了目前尚未測試的 `getUserById` 方法。

```typescript
import { expect, it, describe, beforeEach } from "vitest";
import { UserService } from "./UserService";
import type { User } from "./UserService";
describe("UserService", () => {
  let userService: UserService;
  let user1: User;
  let user2: User;
  beforeEach(() => {
    userService = new UserService();
    user1 = { id: 1, name: "Alice" };
    user2 = { id: 2, name: "Bob" };
  });
  it("should add a user to the list", () => {
    userService.addUser(user1);
    expect(userService["users"]).toContain(user1);
  });
  it("should return a user by ID", () => {
    userService.addUser(user1);
    const result = userService.getUserById(1);
    expect(result).toEqual(user1);
  });
  it("should return undefined if user ID does not exist", () => {
    userService.addUser(user1);
    const result = userService.getUserById(2);
    expect(result).toBeUndefined();
  });
  it("should handle multiple users correctly", () => {
    userService.addUser(user1);
    userService.addUser(user2);
    const result1 = userService.getUserById(1);
    const result2 = userService.getUserById(2);
    expect(result1).toEqual(user1);
    expect(result2).toEqual(user2);
  });
});
```

AI 工具正確地從覆蓋率報告中辨識出 `getUserById` 方法未被充分測試，並為該方法總共生成了三個額外的測試。生成測試的基礎是現有的測試檔案，新的測試無縫地嵌入其中。新測試不僅呼叫 `getUserById` 方法，還檢查是否可以讀取使用者、當請求的物件不存在時會發生什麼，以及是否可以讀取多個使用者。

在這種情況下，AI 工具大大簡化了我們生成測試的工作。如果你像這個範例一樣以現有測試的形式提供額外的上下文，結果會更好，你可以將生成的程式碼直接傳入你的應用程式而無需進行任何更改。
