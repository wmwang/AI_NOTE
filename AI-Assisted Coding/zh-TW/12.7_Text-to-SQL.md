# 12.7 Text-to-SQL

網路上充斥著使用 LLM 從一個表格五個欄位的資料庫中篩選出最大值的範例（「哪個部門創造了最多的銷售額？」）。LLM 在接收到作為上下文傳遞的表格結構後能夠制定對應的 SQL 查詢，這並不令人驚訝。但面對真實世界的資料，情況就有所不同了。

在以下各節中，我們將探討 viel-falter 計畫的資料集。在這個我們已在 本章第一部分中使用過內容的研究計畫中，蝴蝶在指定區域按物種進行識別。

## 12.7.1 從 MongoDB 匯出資料庫到 SQLite

## 12.7.2 使用 LlamaIndex 進行 Text-to-SQL

我們可以毫無顧慮地將匯出的 SQLite 資料庫提供給 LlamaIndex 函式庫，因為生產資料仍然在 MongoDB 中，腳本無法存取它。當然，你也可以在這裡連接一個生產用的 SQL 資料庫而不是匯出的副本。我們使用的 SQLAlchemy 函式庫支援常見的關聯式資料庫，如 MySQL/MariaDB、PostgreSQL、Microsoft SQL Server 和 Oracle。問題在於你無法預先知道 AI 工具會建立哪些 SQL 語句。即使 AI 工具僅對表格具有唯讀存取權限，資料庫管理員可能也會對不受控制的 SQL 查詢感到不以為然。

LlamaIndex 函式庫讓 Text-to-SQL 查詢變得輕而易舉。NLSQLTableQueryEngine 模組生成我們用於自然語言查詢的 QueryEngine。事先建立一個具有資料庫檔案參考的 SQLAlchemy 引擎。你還必須使用 pip install SQLAlchemy 命令安裝此 Python 模組：

```python
from llama_index.core import (
    Settings,
    SQLDatabase,
)
from llama_index.core.query_engine import NLSQLTableQueryEngine
from llama_index.llms.anthropic import Anthropic
from sqlalchemy import create_engine
tables = ["surveys", "speciesList", "species", "sites"]
engine = create_engine("sqlite:data/vielfalter.db")
sql_database = SQLDatabase(
    engine,
    include_tables=tables
)
query_engine = NLSQLTableQueryEngine(
    sql_database=sql_database,
    tables=tables
)
Settings.llm = Anthropic(model="claude-3-5-sonnet-20240620")

response = query_engine.query(
  "How many species were found in 2024?"
)
print(response)  
```

為了回答關於 2024 年發現了多少不同物種的最終問題，LlamaIndex 函式庫首先讀取所列表格的結構。可以想像，表格和欄位的命名非常重要，這樣 LLM 才能在之後生成正確的 SQL 命令。

```
INFO:llama_index.core.indices.struct_store.sql_retriever:>
  Table desc str: Table 'surveys' has columns: _id (TEXT),
    date (TEXT), start (TEXT), end (TEXT), notes (TEXT),
    site_id (TEXT),
    and foreign keys: ['site_id'] -> sites.['_id'].
Table 'speciesList' has columns: _id (TEXT), survey_id (TEXT),
  species_id (TEXT), count (INTEGER),
    and foreign keys: ['survey_id'] -> surveys.['_id'],
    ['species_id'] -> species.['_id'].
...  
```

這些資訊連同建立 SQL 查詢的詳細說明（我們稍後會談到）一起被發送給 LLM，LLM 反過來發回初步的 SQL 建議：

```
Predicted SQL query:
SELECT COUNT(DISTINCT sl.species_id) AS species_count
  FROM surveys s
  JOIN speciesList sl ON s.`_id` = sl.survey_id
  WHERE strftime('%Y', s.date) = '2024'  
```

然後 LlamaIndex 函式庫連接到 SQLite 資料庫並執行查詢。結果被包含在一個新的提示中，並作為 SQL Response: [(79,)] 重新發送給 LLM。LLM 組織了簡短而正確的答案（在撰寫本文時）：

> 根據 SQL 查詢結果，2024 年進行的調查中共發現了 79 個不同的物種。

即使這聽起來像是一個非常簡單的問題，Claude 也必須展示一些 SQL 專業知識。speciesList 表格必須與 surveys 表格連接，並且必須正確分析 date 欄位。我們用英語表述提示這一點有助於 LLM 建立 SQL 查詢。在我們的測試中，大型 LLM 也成功地將德語提示轉換為具有英文表格結構的正確 SQL，但複雜查詢的成功率明顯較低。

但這只是開始。在下一節中，我們將提出更困難的問題，並讓不同的語言模型來回答。

## 12.7.3 範例查詢

我們的程式啟動得相當快（在 Python 中，「相當快」總是留有解釋空間），但我們不想為每個查詢重新啟動它。使用程式中的 while 迴圈，我們不斷請求新的提示：

```python
from prompt_toolkit import PromptSession
session = PromptSession()
try:
    while True:
        q = session.prompt("Query our viel-falter database: ")
        if q == "exit":
            break
        answer(q)
except (KeyboardInterrupt, EOFError):
    print("bye")  
```

來自 prompt_toolkit（pip install prompt_toolkit）的 PromptSession 函式會儲存已提交查詢的列表。你可以使用 (Ctrl)+(P) 快速鍵捲動瀏覽較舊的條目，甚至可以透過 (Ctrl)+(R) 在這些條目中搜尋。程式可以透過輸入 exit 或使用 (Ctrl)+(C)（KeyboardInterrupt）或 (Ctrl)+(D)（EOFError）來終止。現在我們想讓 AI 工具回答一些更困難的問題。

**提示：** How many surveys were conducted in 2024 and on how many different sites?

> 根據 SQL 查詢結果，2024 年：
> 1. 總共進行了 586 次調查。
> 2. 這些調查在 184 個不同的站點進行。

答案聽起來不錯，但我們能相信嗎？看看導致這個結果的 SQL 查詢會很有趣。幸運的是，我們不需要做太多額外的程式設計，只需啟用 LlamaIndex 函式庫的日誌功能。我們使用與前面範例（第 12.6 節）相同的命令列參數來啟用日誌。如果日誌級別設為 DEBUG，則會顯示 SQL 命令。為此，我們使用 --debug 開關啟動應用程式。

日誌訊息包含與 LLM（我們選擇了 Claude 3.5 Sonnet）連線的詳細日誌，以及 LlamaIndex 生成並發送給 LLM 的提示。用於生成 SQL 查詢的相當詳細的提示很有趣：

> **LlamaIndex 提示：** Given an input question, first create a syntactically correct SQLite query to run, then look at the results of the query and return the answer. You can order the results by a relevant column to return the most interesting examples in the database. Never query for all the columns from a specific table, only ask for a few relevant columns given the question. Pay attention to using only the column names that you can see in the schema description. Be careful not to query for columns that don't exist. Pay attention to which column is in which table. Also, qualify column names with the table name when needed. You are required to use the following format, each taking one line:
>
> Question: Question here
> SQLQuery: SQL Query to run
> SQLResult: Result of the SQLQuery Answer

然後提示會被我們在程式中指定的所有表格的結構（tables 變數）以及當然的問題所擴展。在後續的通訊中，我們看到了建議的 SQL 查詢：

```
Predicted SQL query:
SELECT COUNT(*) AS total_surveys,
       COUNT(DISTINCT site_id) AS distinct_sites
  FROM surveys WHERE date LIKE '2024%'  
```

對於 SQL 純粹主義者來說，使用 LIKE 進行日期比較可能看起來有點不傳統，但在 SQLite 中，查詢按預期工作。我們在日誌輸出中找到答案為 SQL response: [(586, 184)]。然後 LLM 再次生成措辭優美的答案，將數字和上下文轉化為英文句子。因此 Claude Sonnet 做得相當不錯。

現在我們想了解 Ollama 伺服器中的本地模型在這方面表現如何。Meta 目前的開源模型 llama3.1 最初讓我們不太抱希望：

> 根據查詢回應，似乎 2024 年沒有進行任何調查。
> 因此，綜合回應將是：
> 「2024 年沒有進行任何調查。此外，由於沒有調查，因此也沒有與這些調查相關的不同站點。」

對 SQL 查詢的分析立即揭示了問題：

```sql
SELECT COUNT(DISTINCT site_id) AS total_sites,
       COUNT(*) AS total_surveys
  FROM surveys WHERE date = '2024';  
```

查詢的大部分實際上是正確的，只是 date 欄位沒有被正確分析，這令人失望。我們立即再次嘗試查詢，對於 llama3.1 突然有了一個解決方案感到有些驚訝，但結果與之前來自 Anthropic 的 AI（Claude）不同。（Llama 3.1 在結果中進行了強調。）

> 根據我們的記錄，2024 年在 76 個不同的站點進行了調查，總共在這些地點進行了 500 次調查。

Debug 日誌也幫助我們解開了這個謎團。AI 工具在這裡嘗試做了太多事情，將發現的物種列表（speciesList）與 surveys 表格連接了起來。但也有些調查中沒有發現蝴蝶，這也是一個結果（差異如此之大也是因為並非所有調查都在物種層面進行，但這裡就不深入討論了）。

```sql
Predicted SQL query:
SELECT COUNT(DISTINCT T1.site_id) AS num_sites,
    COUNT(T1._id) AS num_surveys
  FROM surveys AS T1
    INNER JOIN speciesList AS T2 ON T1._id = T2.survey_id
  WHERE STRFTIME('%Y', T1.date) = '2024'  
```

當我們再次執行查詢並得到 Claude AI 工具立即提供的正確結果時，事情變得有點可怕。三次測試在一分鐘內完成，Ollama 伺服器沒有其他計算任務。也許 llama3.1 模型對這個任務來說還不夠好。我們還在 Ollama 中測試了 deepseek-coder-v2 模型。然而，我們最初的信心在更複雜的查詢中再次受挫，因為我們面臨著每次查詢答案都不正確且不同的同樣問題。

我們的印象是，這些較小的本地模型無法正確處理複雜的 Text-to-SQL 任務。到你手中拿著這本書的時候，這些模型肯定已經更加先進，可能能夠在本地很好地執行 Text-to-SQL。

另一方面，Claude AI 和 OpenAI 的 gpt-4o 不斷讓我們驚喜。在以下查詢中——某一年在某個地區發現了某個物種的多少個體——我們 SQLite 資料庫的所有四個表格都必須在 SQL 查詢中進行連接，並且其中一個表格中的數值 count 欄位必須進行加總。nuts2 資料庫欄位源自歐洲領土單位分類（https://ec.europa.eu/eurostat/web/nuts）。

**提示：** How many individuals of species "Admiral" were found in surveys in 2024 in the Nuts2 region of Vorarlberg?

> 根據 SQL 查詢結果，2024 年在福拉爾貝格 NUTS2 地區的調查中發現了 4 個「Admiral」物種的個體。

我們自己也不可能寫出比這更好的 SQL 查詢：

```sql
SELECT SUM(sl.count) as total_count
  FROM speciesList sl
  JOIN surveys s ON sl.survey_id = s._id
  JOIN species sp ON sl.species_id = sp._id
  JOIN sites st ON s.site_id = st._id
 WHERE sp.german = 'Admiral'
   AND strftime('%Y', s.date) = '2024'
   AND st.nuts2 = 'Vorarlberg'  
```

透過對資料庫的探索，我們還偶然發現了不正確的條目。搜尋所有現有的 Nuts2 地區導致了一個令人驚訝的結果。

**提示：** How many different Nuts2 regions are in the database and list them?

> 根據 SQL 查詢結果，資料庫中有 11 個不同的 Nuts2 地區。這些地區是：
> 1. Tirol
> 2. Vorarlberg
> 3. Südtirol
> 4. Niederösterreich
> 5. Salzburg
> 6. Oberösterreich
> 7. Kärnten
> 8. Steiermark
> 9. Burgenland
> 10. Wien
>
> 值得注意的是，列表中似乎有一個空條目（由 Burgenland 和 Wien 之間的雙逗號表示），這可能代表資料庫中的空值或空字串。這個空條目被計算為 11 個不同地區之一，儘管它沒有名稱。

然而，AI 工具立即解釋了未出現在列表中的第 11 個條目來自哪裡：有一個條目的地區顯然是空的。如果對資料庫結構沒有一些了解，要找到不正確的條目將會很困難。導致結果的 SQL 查詢也在這裡幫助了我們：

```sql
SELECT COUNT(DISTINCT nuts2) AS count,
       GROUP_CONCAT(DISTINCT nuts2) AS regions 
FROM sites  
```

這是 sites 表中的一個條目。現在我們可以問 AI 工具這個表格中的哪個條目：

**提示：** Show me sites not assigned to a Nuts2 region. Include database IDs for these sites.

> 根據查詢結果，有一個站點未被分配到 NUTS2 地區。詳細資訊如下：
> 資料庫 ID：65f6a6f616634be52c735711
> 名稱：[未提供名稱]
>
> 值得注意的是，雖然資料庫 ID 存在，但該站點的名稱欄位似乎是空的。這可能意味著該站點在資料庫中沒有記錄名稱，或者這個特定站點的資料輸入可能有問題。

使用資料庫 ID 很快就修正了不正確的條目。

## 12.7.4 結論

Text-to-SQL 概念對我們來說具有巨大的潛力。這個概念可以為資料庫管理員節省工作，特別是在非技術背景的人員想要不定期地從中央資料庫查詢不同指標的環境中。這會讓資料庫管理員變得多餘嗎？目前肯定不會。

即使在 Text-to-SQL 中也無法依賴 AI 工具答案的問題，使得該系統不適合用於業務關鍵領域。但這同樣適用於在其他任務中產生幻覺的模型。

另一個限制因素目前是 LLM 在上下文中可以處理的 token 數量。如果 SQL 查詢包含太多資料，LLM 就無法再處理回應。然而，我們也期望這個領域會有快速發展。看看這項技術未來能在哪些領域站穩腳跟，一定會令人興奮。
