# 10 自動化程式碼處理

到目前為止，我們一直假設您以互動方式使用 AI 工具：您向 ChatGPT 請求程式碼支援，或讓 GitHub Copilot、Continue 或其他精靈在您的整合開發環境（IDE）中協助您。這對於開發新程式碼來說是正確的方法。

但假設您有一個由數十、數百甚至數千個檔案組成的程式碼庫。那麼，如果能像以下範例那樣撰寫提示，是不是很棒？

- 將程式碼庫從 Python 2 轉換為 Python 3。
- 將所有檔案中的註解從德文翻譯成英文。
- 將所有 CSS 和 HTML 檔案中的等寬字型統一替換為「Liberation Mono」。
- mydb 資料庫的綱要已變更。在資料表 x 中，欄位 y 已被賦予新名稱 z。同時，資料類型從 INT 變更為 FLOAT。請調整所有相關的程式碼檔案。

目前的 AI 工具及相關的使用者介面、編輯器和開發環境（尚）無法處理這類提示。然而，這並不意味著這些遠大的願望完全無法實現。

本章首先提供 OpenAI 和 Ollama 應用程式介面（API）的速成課程。這使您能夠在獨立的腳本中向語言模型發送提示並分析其回應。在此基礎上，我們將向您展示一些範例腳本，您可以使用這些腳本將提示應用於整個目錄中的檔案。

**夢想與現實**

在本章中，我們已經觸及了目前技術可能性的極限。是的，透過 AI API 的途徑，確實可以對大量檔案進行自動化程式碼變更。但不，在我們所有的測試中，效果都不是很好。我們將詳細記錄這些限制。

然而，由於 AI 工具及其 API 正以驚人的速度發展，今天不可能的事明天就會觸手可及，後天可能就成為現實！

話雖如此，希望不用多說您也明白，無論是現在還是未來，您都不能在不經測試的情況下就採用大規模的程式碼變更。建議您至少在兩個 Git 提交中進行變更。第一個提交包含 AI 工具所做的變更，後續的提交包含您或您的（人類！）團隊手動進行的修正。理想情況下，應該有盡可能大的單元測試集合，以確保變更後的程式碼繼續按預期運作。

## 10.1 OpenAI API

OpenAI 是「發明」ChatGPT 並將其轉化為產品的公司。GitHub Copilot 也使用來自 OpenAI 的語言模型。您可能支付給 GitHub 或 Microsoft 的每月費用的一部分會流向 OpenAI。

除了這些主流產品外，還有一個專門為開發者設計的 OpenAI API。它允許您透過簡單的介面向語言模型發送提示並分析回應。OpenAI API 的一個常見應用是實作公司特定的聊天機器人。然而，在本節中，我們想將 API 用於程式碼編寫任務。

### 10.1.1 設定 API 帳戶

OpenAI API（https://platform.openai.com）是付費才能使用的。計費與任何現有的 ChatGPT 訂閱分開。您必須設定自己的 API 帳戶並透過信用卡支付起始餘額。之後的計費則按需進行，當然，您可以設定限額。

不用擔心，不需要巨大的投資——10 美元就足夠進行初步實驗了。計費考慮了所使用的模型以及處理的輸入和輸出 token 數量。使用 GPT-4o，一百萬個輸入 token 費用為 2.5 美元，一百萬個輸出 token 費用為 10 美元（截至 2025 年 1 月，https://openai.com/api/pricing）。

以下是一個具體的例子：一個包含五行文字的提示大約有 100 個輸入 token。以本書格式排版的兩頁回答大約包含 900 個輸出 token。這導致的成本為 2.5 / 1,000,000 × 100 + 10 / 1,000,000 × 900 = 0.009 美元，也就是不到一分錢。在用完 10 美元的額度之前，您可以處理超過 1,000 個這樣的提示。根據上下文的大小（較大的上下文需要更多的輸入 token）和輸出的範圍，成本當然會有所不同，但上述計算有助於初步估計數量級。處理圖片適用單獨的費率，這在程式碼編寫的情境中不相關。在「使用量」頁面上，您可以追蹤最近 API 呼叫的確切成本（https://platform.openai.com/usage）。

如果您選擇效能調校過的 mini 版本而不是 GPT-4o，成本會降低 15 倍！按照上述計算，10 美元可以處理超過 15,000 個上述提示。mini 版本在品質上雖然無法完全匹敵完整版本，但對於眾多應用來說已經足夠好了。

在您可以在程式中使用 OpenAI API 之前，您必須建立一個專案金鑰。您可以為每個專案分配一個單獨的金鑰，也可以為多個專案使用一個共同的金鑰。如果您不想處理專案的概念，只需使用預設配置的預設專案即可。無論哪種情況，您都可以建立所需數量的金鑰，並在必要時稍後撤銷它們。

新金鑰只會在 OpenAI 網頁介面中顯示一次。因此，請確保將金鑰保存在安全的地方。

### 10.1.2 Hello, World!

各種程式語言都有使用 OpenAI API 的函式庫。在本章中，我們將重點放在 Python 上，並假設已經安裝了最新版本。最好為您的專案設定一個虛擬環境。在 Linux 和 macOS 上，所需的命令如下：

```
python3 -m venv openai-test
cd openai-test
source bin/activate  
```

在 Windows 上，Python 3 被分配了 `python` 命令。要啟動環境，您需要使用 PowerShell 腳本 `Scripts/Activate.ps1` 而不是 `bin/activate` shell 腳本：

```
python -m venv openai-test
cd openai-test
Scripts/Activate.ps1  
```

然後，按以下方式安裝 openai 模組：

```
pip/pip3 install --upgrade openai  
```

現在在您的專案中設定一個名為 `.env` 的檔案，並在那裡儲存 OpenAI API 金鑰。請嚴格遵守以下語法；也就是說，使用變數名稱 `OPENAI_API_KEY`，並且在 `=` 字元前後不要加空格！`.env` 檔案和金鑰隨後會被 openai 模組自動考慮：

```
# File openai-test/.env
OPENAI_API_KEY=sk...  
```

> **金鑰和密碼不屬於 Git 儲存庫！**
>
> 如果您使用 Git 管理您的專案，建議將 `.env` 檔案加入 `.gitignore`，這樣金鑰就不會進入 Git 儲存庫。

除了將 API 金鑰儲存在 `.env` 檔案中，您也可以將其儲存在 `OPENAI_API_KEY` 環境變數中。在 Linux 或 macOS 上，您需要在 `.bashrc` 或 `.zshrc` 中輸入相應的定義。在 Windows 上，當您在系統選單中搜尋「編輯系統環境變數」時，會開啟一個設定對話方塊。無論哪種情況，設定只有在您重新啟動終端或重新登入後才會生效。

現在只缺少將第一個提示傳送給 OpenAI 並顯示回應的 Python 腳本程式碼。這只需幾行程式碼即可實現：

```python
# Sample file openai/hello.py
from openai import OpenAI
client = OpenAI() # reads the .env file 
completion = client.chat.completions.create(
  model = "gpt-4o",
  messages = [
    {"role": "user",
     "content": "How do I use list comprehension in Python?"}
  ]
)
print(completion.choices[0].message.content)  
```

當您執行程式時，您會注意到在大約 5 到 10 秒內沒有任何動作，直到回應被顯示。這是不尋常的，因為 ChatGPT 通常會立即開始回覆。差異在於我們的腳本會等待整個回應完成後才進行輸出。而在聊天系統中，答案是一段一段出現的；您可以立即開始閱讀，因此不會覺得需要等待。

如果您希望在腳本中有類似的行為，您必須使用 OpenAI API 的串流功能。為此，將額外的 `stream = True` 參數傳遞給 `create`，並在迴圈中分析結果物件的 `delta.content` 屬性：

```python
# Sample file openai/hello-streaming.py
from openai import OpenAI
client = OpenAI() 
stream = client.chat.completions.create(
  model = "gpt-4o",
  stream = True,
  messages = [
    {"role": "user",
     "content": "How do I use list comprehension in Python?"}
  ]
)
for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        print(chunk.choices[0].delta.content, end="")  
```

### 10.1.3 撰寫提示

您傳送給 OpenAI 的提示由一個字典條目陣列組成。每個條目由兩個元件組成：

```python
{ "role": "user|system|assistant",
  "content": "text ..." }  
```

`role` 有三個允許的值：

- `system` 指定語言模型應扮演的角色。
- `user` 表示這是您或您的程式提出的問題。
- `assistant` 包含回應的模式或語言系統先前產生的回應。

如果您只傳送一個使用者問題，如前面的範例所示，語言模型會假定其預設角色（「You are a helpful assistant」），並且沒有上下文資訊。但是，您可以透過指定一次系統文字來為語言模型分配特定的角色。

多組使用者和助理文字可以提供您希望語言模型如何回應的範例，或者它們可以包含先前的問答，為下一個問題設定上下文。透過整合舊的問答，您確定了下一個問題的上下文，從而使語言模型能夠引用先前建立的資訊。

以下程式碼範例說明了該過程。

```python
# Sample file openai/hello-context.py
from openai import OpenAI
client = OpenAI() # reads the .env file
completion = client.chat.completions.create(
  model = "gpt-4o-mini",
  messages = [
    # Role
    { "role": "system",
      "content": """You are a Python programming 
                    assistant. 
                    You give short answers, two paragraphs 
                    max. 
                    If possible, return code examples, 
                    no text."""},
    # Example of first question + answer
    { "role": "user",
      "content": "How can I test if a variable contains a tuple?"},
    { "role": "assistant",
      "content": "`if isinstance(var, tuple) ...`"},
    # Example of second question + answer
    { "role": "user",
      "content": "How can I read the second element of a tuple?"},
    { "role": "assistant",
      "content": "`second_element = my_tuple[1]`"},
    # new question, refers to the previous context
    {"role": "user",
     "content": "How do I convert a tuple into a list?"}
  ]
)
print(completion.choices[0].message.content)
# Output: 
#
# ```python
# my_list = list(my_tuple)
# ```  
```

**理想的語言模型**

您也可以在傳送給 `create` 的資訊中選擇您偏好的語言模型。截至撰寫本文時（2025 年初），`gpt-4o`（小寫字母「o」，不是零）和 `gpt-4o-mini` 代表了最佳選擇。兩個模型都有 32,000 個 token 的上下文視窗（約 128,000 個字元）。

在品質方面，`gpt-4o` 傳回最好的結果。為更高效率而調校的 `gpt-4o-mini` 模型取代了過去流行的 `gpt-3.5-turbo`，相比之下具有許多優勢：

- 其使用成本比 GPT-4o 便宜 15 倍（https://openai.com/api/pricing）。
- 效能更好，也就是說，您的腳本能更快收到 API 呼叫的回應。
- 每次回應允許的最大輸出 token 數量明顯更高，達到 16,000 個 token。相比之下，GPT-4o 每次回應只允許 4,000 個 token。

在我們的程式碼編寫應用測試中，我們並沒有明顯感受到所謂的速度提升，但我們對 gpt-4o-mini 的品質非常滿意。從我們的角度來看，gpt-4o-mini 對許多應用來說已經夠好了。如果您想進一步節省成本，可以看看 Batch API（https://platform.openai.com/docs/guides/batch）。雖然回應在那裡發送的提示可能需要長達 24 小時，但您將獲得相應模型基本價格的五折優惠。

### 10.1.4 聊天系統

只需 40 行 Python 程式碼就能建立一個在終端中執行的簡單聊天系統：

```python
# Sample file openai/hello-chat.py
from openai import OpenAI
client = OpenAI() # reads the .env file
def get_response(messages):
    completion = client.chat.completions.create(
        messages = messages,
        model = "gpt-4o-mini",  # Model
        max_tokens = 150,       # max. Anzahl der Antwort-Token
        temperature = 0.5       # Temperature (0-2)
    )
    return completion.choices[0].message.content.strip()
print("Welcome to your programming assistant!")
print("Type 'exit' to quit.")
conversation = []    # contains the chat history

conversation.append(
    {"role": "system",
     "content": """Respond code only if possible.
                   Keep explanations to a minimum 
                   (2 paragraphs max). Don't use
                   markdown."""})
while True:          # Processing of inputs
    prompt = input("You: ")
    if prompt.lower() == 'exit':
        print("Goodbye!")
        break
    # Add prompt to the chat history
    conversation.append({"role": "user", "content": prompt})
    # Process prompt
    response = get_response(conversation)
    print("AI:")
    print(response)
    print()
    # Add response to chat history
    conversation.append({"role": "assistant", 
                         "content": response})  
```

您自己聊天系統的範例執行可能如下所示。為了更好的可讀性，我們在以下程式碼中移除了回答中包含的 Markdown 格式。

```
Welcome to your programming assistant!
Type 'exit' to quit.
You: Explain the concept of list comprehension.
AI:
List comprehension is a concise way to create lists in Python.
It allows you to generate a new list by applying an expression
to each item in an existing iterable, such as a list or a range,
and can include an optional condition to filter items. 

The syntax follows the format: [expression for item in iterable
if condition]. This makes the code more readable and often more
efficient than using traditional loops.

For example, to create a list of squares from 0 to 9, you could 
use list comprehension like this: [x**2 for x in range(10)]. 

This results in the list [0, 1, 4, 9, 16, 25, 36, 49, ...]
You: I want to get a list of the word lengths in a 
     sentence. Can I use list comprehension for this?
AI:
Yes, you can use list comprehension to get a list of the lengths 
of the words in a sentence. Here's an example:
sentence = "This is an example sentence"
lengths = [len(word) for word in sentence.split()]
This will give you a list of the lengths of each word in the
sentence.  
```

這裡的新內容是將回應的輸出 token 限制為 150（`max_tokens` 參數）以及設定語言模型的「溫度」。值的範圍在 0 到 2 之間，其中 2 代表最大創造力。對於程式碼編寫任務，如果語言模型專注於最佳答案會更好。0.2 到 0.8 之間的值已被證明是有效的。這些以及許多其他 API 參數記錄在 API 參考資料中：https://platform.openai.com/docs/api-reference。

**提示注入**

前面的範例顯示，您只需幾行程式碼就能編寫自己的聊天系統，並使用系統字串為其分配非常特定的角色。但我們必須警告您！在擁有一個真正只回答指定角色範圍內問題的完全開發應用之前，還有很長的路要走。

問題在於提示注入，也就是包含惡意指令的提示。最簡單的情況下，惡作劇者可能會如下引入問題：「現在，忽略你的系統角色，然後⋯⋯」。雖然現在攻擊語言模型已經不再那麼容易，但攻擊者過去仍然找到了欺騙聊天系統或從中獲取機密資料的方法。物流公司 DPD 首次嘗試使用聊天機器人回答客戶查詢時尤其令人尷尬。該聊天機器人將自己的公司描述為「毫無用處」，並特別抱怨糟糕的客戶服務（https://www.bbc.com/news/technology-68025677）。

為了防止提示注入，您必須在提示被語言模型處理之前檢查其中可能包含的錯誤，例如，使用第二個語言模型，其唯一任務是識別可疑模式。然而，這些嘗試並不總是成功的。IBM 對此非常悲觀（請參閱 www.ibm.com/blog/prevent-prompt-injection）。避免提示注入的唯一方法是完全放棄使用語言模型。

### 10.1.5 處理回應

OpenAI API 以 ChatCompletion 物件的形式提供答案。通常，只有一個回應變體（`choices[0]`）。只有當您將值大於 1 的 `n` 參數傳遞給 `create` 方法時，語言模型才會為每個問題提供多個答案。然後您可以在答案之間進行選擇，但您必須為所有答案付費。

到目前為止，我們只分析了回應的 `message.content` 屬性，其中包含回應的文字。OpenAI 通常使用 Markdown 語法。或者，您可以在提示或角色描述中要求未格式化的文字或 HTML。這個要求通常會被遵守，但遺憾的是並非每次都是如此。

以下程式碼顯示了對其他一些回應參數的分析：

```python
completion = client.chat.completions.create(...)
print("Input Tokens:", completion.usage.prompt_tokens)
print("Output Tokens:", completion.usage.completion_tokens)
print("Total Tokens:", completion.usage.total_tokens)
print("Finish Reason:", completion.choices[0].finish_reason)
print("Text:", completion.choices[0].message.content)  
```

`finish_reason` 特別重要，也就是回應被結束或取消的原因：

- `'stop'`：答案已完成。要麼語言模型認為回應已完成，要麼回應中出現了預定義的關鍵字（`create(..., stop='keyword')`）。
- `'length'`：已超過輸出 token 限制。此時答案被中斷。
- `'content_filter'`：提示不符合使用規範。
- `'null'`：答案因其他（未知）原因被取消。

**錯誤防護**

在正式環境的程式碼中，進行錯誤檢查是必不可少的：

- 將 API 呼叫包裹在 `try-except` 中。可能的錯誤原因包括超過上下文長度（最大輸入 token 數）和 OpenAI 雲端服務的服務問題。
- 分析 `finish_reason`。這裡最可能的錯誤原因是超過輸出 token 數量。使用 GPT-4o 時，這是 4k = 4,096。可以使用 `max_tokens` 參數減少但不能增加。

### 10.1.6 上傳和下載檔案

對於許多應用來說，能夠上傳一個檔案、給語言模型下指令、操作這個檔案或建立一個全新的檔案，然後再下載結果檔案，會非常有用。遺憾的是，常見的語言模型缺乏這樣的功能（截至 2025 年初）。使用 OpenAI API 時，目前甚至無法將檔案附加到提示中；就像在 ChatGPT 中一樣，您首先透過拖放上傳檔案，然後在提示中提出引用該檔案的問題。

為了繞過這個限制，您可以透過 API 呼叫將要分析的程式碼檔案內容包含在提示中（即使用者內容）。以下程式碼給出了一個範例。該腳本的目的是將 Python 檔案 `sample.py` 傳遞給 OpenAI API 進行完善。回應中的程式碼被儲存在新的 `sample-improved.py` 檔案中。

```python
# Sample file openai/hello-upload.py
from openai import OpenAI
client = OpenAI() # reads the .env file
# Read file to be edited
with open('sample.py', 'r') as file:
    file_contents = file.read()
# Put together roll and prompt
role = """You are a programming assistant. You analyze 
    code and return an improved version of it. Organize code 
    into functions where possible. Include code comments where 
    necessary.
    YOU RETURN CODE ONLY! DO NOT ADD EXPLANATIONS. 
    DO NOT USE MARKDOWN."""
prompt = f"Optimize the following code:\n\n{file_contents}"
completion = client.chat.completions.create(
    model = "gpt-4o",
    temperature = 0.5,
    messages = [ {"role": "system", "content": role},
                 {"role": "user",   "content": prompt} ] 
)
if completion.choices[0].finish_reason != "stop":
    print("Code optimization failed.")
else:
    new_code = completion.choices[0].message.content.strip()
    with open('sample-improved.py', 'w') as file:
        file.write(new_code)
    print("\nImproved code\n\n")
    print(new_code)  
```

此過程的主要問題在於結果的分析。角色描述明確指出需要純程式碼回應，不需要進一步的解釋，也不需要 Markdown 格式。在大多數情況下，語言模型會遵守這些規範，但遺憾的是，沒有絕對的確定性。您可以嘗試在答案中搜尋 ` ``` ` 以識別列表的 Markdown 格式。我們將在第 10.4 節中介紹相應的函數。

**Token 限制**

這裡描述的過程在大型程式碼檔案上會失敗。問題在於 OpenAI 指定的最大回應長度為 4,096 個 token（GPT-4o，截至 2025 年 1 月）。對於文字的一般經驗法則是每個 token 四個字元。然而，從語言模型的角度來看，程式碼具有更高的資訊密度。例如，每個運算子（如 `+` 或 `-`）都是一個完整的 token。這就是為什麼程式碼字元與 token 的比例通常會降到三以下。因此，產生的程式碼長度不能超過約 10 KB（約 250 到 300 行）。

根據應用的目的，您可以嘗試將任務分解為多個步驟。正如您將在第 10.4 節中看到的，拆分程式碼檔案是極其困難的。
