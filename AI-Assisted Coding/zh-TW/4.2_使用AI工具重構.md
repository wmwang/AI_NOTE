# 4.2 使用 AI 工具進行重構

現在我們簡要討論了重構的理論，是時候處理原始程式碼了。首先，您將學習如何評估是否應該修改某段原始程式碼、做出此決定的標準，以及 AI 工具如何幫助您。您還將學習特定重構流程的一些策略和最佳實踐。

## 4.2.1 識別重構需求

您作為開發者的時間非常寶貴，所以不應該浪費在不必要的工作上。因此，首先了解重構是否合適以及程式碼中的弱點在哪裡是很重要的。

有許多工具可用於此類分析，例如 SonarQube，用於執行靜態程式碼分析。這些工具可以提供整體系統的良好概覽。這也是許多 AI 工具的弱點，因為它們很少能看到應用程式的整體上下文，只能看到較小的部分。因此，不同工具的組合是理想的：經典靜態分析提供概覽，AI 工具負責細節和實際重構。您也可以將 AI 工具用作一種程式碼審查器，揭示重構潛力。

### 程式碼壞味道（Code Smells）

您可以幾乎從字面上理解「程式碼壞味道」這個詞。具有程式碼壞味道的程式碼潛藏著問題，您可以透過重構來修復。程式碼壞味道不一定是錯誤，但如果忽視太久，可能會導致嚴重問題。典型的程式碼壞味道包括：

- **重複程式碼**：程式碼在不同地方有相同或相似的區塊。如果您修正一處的錯誤，也必須在所有其他地方修正。可維護性因此降低，錯誤潛力很高。
- **過長的方法**：函式和方法應該盡可能緊湊，只處理一個方面。函式行數越多，潛在的複雜性越大。
- **過大的類別**：類別與函式方法同理。如果類別太大（以行數、屬性或方法數衡量），它很可能做了太多事情。
- **特性嫉妒（Feature envy）**：這種壞味道指的是一個方法存取另一個類別的屬性多於自己的屬性。這表示該方法不屬於當前位置。
- **資料泥團（Data clumps）**：這是在應用程式不同位置以相同方式使用的變數群組。它們應該被組合到一個單獨的類別或資料結構中。

### 使用 AI 分析程式碼

例如，我們使用一個名為 `OrderProcessor` 的 TypeScript 類別。您幾乎可以使用任何 AI 工具進行分析。在 VS Code 中使用 GitHub Copilot：

**提示：** @workspace /explain Does the OrderProcessor class contain code smells?

Copilot 找到了以下問題並提供了修正版本：

- **使用 any 型別**：失去了靜態型別系統的優勢。Copilot 建立了 Order 介面。
- **魔術數字**：如訂單限制等數字不應直接寫在程式碼中。Copilot 將其儲存為 MAX_ORDERS 屬性。
- **程式碼重複**：檢查城市是否為 Berlin 的程式碼出現兩次。
- **過長方法**：processOrders 和 generateReport 方法過長。Copilot 使用 forEach 簡化了迭代。
- **缺少錯誤處理**：存取屬性鏈時沒有檢查是否存在。
- **主控台輸出**：使用 console.log 不是最佳選擇，應該使用專門的日誌記錄器。

然而，Copilot 未發現的一些問題：類別承擔了太多責任（管理訂單和生成報告應分離），以及對訂單地址屬性的直接存取是特性嫉妒的例子。

### 具體重構請求

您也可以更具體地提出請求。例如，以下函式包含不必要的重複：

**提示：** Could you please reduce the code duplicates in the following function: [function code]

AI 工具將多個訂單狀態檢查合併為一個，使用 `continue` 跳過不符合條件的訂單，使程式碼更緊湊、更易讀。

### 反模式（Anti-Patterns）

設計模式是軟體開發中常見問題的既定可重複解決方案。反模式在某種程度上是相反的——它們是原始程式碼中長期造成問題的模式或實踐。典型的反模式包括：

- **上帝物件（God object）**：一個類別承擔了太多責任，使程式碼難以維護。
- **Singleton 模式的不當使用**：可能被用來管理應用程式的全域狀態。
- **義大利麵程式碼（Spaghetti code）**：結構不良的程式碼，拉一髮而動全身。
- **金錘子（Golden hammer）**：將一個既定解決方案用於所有問題，即使不合適。
- **過早最佳化**：程式碼過度工程化，解決尚未出現的問題。

使用 `UserService` 類別進行反模式分析：

**提示：** @workspace /explain Does the code of the UserService class contain anti-patterns?

Copilot 發現了以下問題：
- **Singleton 模式**：提醒使用時應謹慎
- **型別安全**：從 any 型別改為 User 介面
- **單一職責原則**：processUserData 方法執行過多任務，已拆分為獨立方法
- **減少巢狀**：透過拆分方法解決了深層巢狀問題
- **魔術數字**：毫秒計算儲存在適當命名的變數中
- **日誌記錄**：建議使用獨立日誌記錄器替代 console.log
- **錯誤處理**：指出缺少錯誤處理

然而，原始程式碼還有其他未解決的問題：類別承擔過多責任、calculateDiscounts 中的快取可能是過早最佳化、notifyUsers 方法僅是 processUserData 的別名。
