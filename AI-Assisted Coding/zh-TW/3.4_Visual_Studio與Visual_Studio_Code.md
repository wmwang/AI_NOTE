# 3.4 Visual Studio 與 Visual Studio Code

接下來的兩節討論 Microsoft 的兩個重要開發環境（整合開發環境 [IDE]）：Visual Studio 和 VS Code。前者可以自信地被描述為 IDE 歷史上的基石。Windows 的第一個版本早在 1997 年就推出了。直到 2014 年，這個專有軟體都只能付費使用；此後，推出了在 Microsoft 註冊後即可免費使用的 Community Edition。該 IDE 非常受歡迎，因為它為許多支援的程式語言提供了上下文敏感的說明和語法檢查。該程式需要電腦的大量資源，這就是為什麼許多開發者尋求更精簡的替代方案。

自 2015 年以來，Microsoft 一直在 Electron 平台上開發 VS Code 編輯器，該平台最初是作為免費 Atom 編輯器的基礎建立的。Electron 為 Web 應用程式提供了執行環境，能夠實現不受瀏覽器沙盒限制的桌面應用程式。VS Code 的核心是精簡的，可以透過無數的擴充功能來擴展。在 Stack Overflow 的調查中，VS Code 多年來一直位居開發者最喜愛的 IDE 之首。

## 3.4.1 VS Code 中缺少的 Python 模組

以下章節中的 Python 應用程式已經有幾年的歷史，我們不得不再次處理它。在呼叫除錯器時發生了例外，因為找不到一個整合模組。遺憾的是，該應用程式是在沒有 Python 虛擬環境以及沒有 requirements.txt 檔案的情況下建立的。

然而，VS Code 中基於 GitHub Copilot 和相應擴充功能的 AI 支援在這裡也能為我們節省一些工作。例外停止了執行並將游標定位在程式碼中的相應位置。VS Code 中 Python 的語言支援（同樣是一個擴充功能）已經指出了不存在的匯入。使用 Copilot，我們現在可以按一下按鈕（/fix）就能獲得建議的解決方案，並將相應的命令直接插入到控制台視窗中。然後，只需按下 Enter 鍵即可安裝模組。

遺憾的是，VS Code 中的 Copilot 尚無法處理在執行時發生的例外的文字。（我們將在下一節中展示這在 Visual Studio 中是如何可能的。）然而，我們懷疑這個功能離我們不遠了。

## 3.4.2 Visual Studio 例外除錯

對於使用 Microsoft Visual Studio 開發環境（不是免費的 VS Code）且同時是 GitHub Copilot 客戶的使用者來說，現在有一個非常方便的 AI 擴充功能：如果您在除錯模式下啟動應用程式並發生意外錯誤（例外），Visual Studio 提供了一鍵讓 Copilot 調查問題的選項。

我們對此功能的實驗留下了非常正面的印象。提示不僅發送例外的文字，還發送周圍的程式碼。這是我們能理解為什麼（故意造成的）System.ArgumentException 能被如此精確解釋的唯一原因。C# 呼叫 `todoItems.GetRange(0, 3)` 從 todoItems 列表中提取前三個元素。然而，我們呼叫時列表是空的，這就是為什麼發生了例外。

GitHub Copilot 能夠簡短地並使用正確的變數描述問題：

我們還得到了一個現成的修復——一個無需任何更改即可運作的程式碼建議，更是錦上添花。Microsoft 正在展示 AI 支援開發的便利性。正如我所說，我們已經期待 VS Code 的實作了。
