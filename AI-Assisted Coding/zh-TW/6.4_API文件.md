# 6.4 API 文件

## 何謂 API 文件？

API 文件是對外提供的介面說明文件，描述了 API 的端點（Endpoints）、請求方式、參數、回傳格式、錯誤碼等資訊。良好的 API 文件能讓開發者快速理解並正確使用 API，是前後端協作和第三方整合的關鍵基礎。

API 文件通常包含：

- **端點說明**：每個 API 端點的 URL、HTTP 方法和用途
- **請求參數**：路徑參數、查詢參數、請求體的格式和驗證規則
- **回傳格式**：成功的回傳結構和資料型態
- **錯誤處理**：可能的錯誤碼和對應的錯誤訊息
- **認證授權**：API 的認證方式和權限要求
- **使用範例**：實際的請求和回應範例

## API 文件的標準格式

### RESTful API 文件

#### 基本端點文件格式

```markdown
### 建立新使用者

建立一個新的使用者帳號。

**端點：** `POST /api/v1/users`

**認證：** 需要 Bearer Token（管理員權限）

#### 請求標頭

| 標頭 | 值 | 必要 |
|------|------|------|
| Content-Type | application/json | 是 |
| Authorization | Bearer {token} | 是 |

#### 請求體

| 欄位 | 型態 | 必要 | 說明 |
|------|------|------|------|
| name | string | 是 | 使用者姓名，長度 1-100 字元 |
| email | string | 是 | 有效的電子郵件格式，全域唯一 |
| role | string | 否 | 使用者角色，預設為 "member" |
| department | string | 否 | 所屬部門 |

#### 請求範例

\```json
{
    "name": "王小明",
    "email": "wang@example.com",
    "role": "developer",
    "department": "工程部"
}
\```

#### 成功回應（201 Created）

\```json
{
    "id": "usr_abc123",
    "name": "王小明",
    "email": "wang@example.com",
    "role": "developer",
    "department": "工程部",
    "created_at": "2024-01-15T10:30:00Z"
}
\```

#### 錯誤回應

| 狀態碼 | 錯誤代碼 | 說明 |
|--------|----------|------|
| 400 | INVALID_EMAIL | 電子郵件格式不正確 |
| 401 | UNAUTHORIZED | 缺少有效的認證憑證 |
| 403 | FORBIDDEN | 權限不足，需要管理員權限 |
| 409 | EMAIL_EXISTS | 該電子郵件已被註冊 |
| 422 | VALIDATION_ERROR | 請求體驗證失敗 |
```

### OpenAPI (Swagger) 規範

OpenAPI 是描述 RESTful API 的業界標準格式：

```yaml
openapi: 3.0.3
info:
  title: 使用者管理 API
  description: 提供使用者帳號的 CRUD 操作
  version: 1.0.0

paths:
  /api/v1/users:
    get:
      summary: 取得使用者列表
      description: |
        取得所有使用者的分頁列表。
        支援篩選和排序功能。
      operationId: listUsers
      tags:
        - 使用者
      parameters:
        - name: page
          in: query
          description: 頁碼（從 1 開始）
          required: false
          schema:
            type: integer
            minimum: 1
            default: 1
        - name: per_page
          in: query
          description: 每頁筆數
          required: false
          schema:
            type: integer
            minimum: 1
            maximum: 100
            default: 20
        - name: role
          in: query
          description: 依角色篩選
          required: false
          schema:
            type: string
            enum: [admin, developer, member]
      responses:
        '200':
          description: 成功取得使用者列表
          content:
            application/json:
              schema:
                type: object
                properties:
                  data:
                    type: array
                    items:
                      $ref: '#/components/schemas/User'
                  pagination:
                    $ref: '#/components/schemas/Pagination'

components:
  schemas:
    User:
      type: object
      required:
        - id
        - name
        - email
      properties:
        id:
          type: string
          description: 使用者的唯一識別碼
          example: "usr_abc123"
        name:
          type: string
          description: 使用者姓名
          example: "王小明"
        email:
          type: string
          format: email
          description: 使用者電子郵件
          example: "wang@example.com"

    Pagination:
      type: object
      properties:
        page:
          type: integer
          example: 1
        per_page:
          type: integer
          example: 20
        total:
          type: integer
          example: 150
        total_pages:
          type: integer
          example: 8
```

### GraphQL API 文件

GraphQL 使用型態系統來描述 API：

```graphql
"""
使用者型態，包含帳號基本資訊。
"""
type User {
    """使用者的唯一識別碼"""
    id: ID!

    """使用者姓名"""
    name: String!

    """使用者電子郵件（全域唯一）"""
    email: String!

    """使用者角色"""
    role: UserRole!

    """帳號建立時間"""
    createdAt: DateTime!

    """使用者的文章列表"""
    posts(
        """回傳的最大筆數"""
        limit: Int = 10

        """游標，用於分頁"""
        after: String
    ): PostConnection!
}

"""使用者角色列舉"""
enum UserRole {
    """管理員：擁有所有權限"""
    ADMIN

    """開發者：可以管理自己負責的專案"""
    DEVELOPER

    """一般成員：基本權限"""
    MEMBER
}

type Query {
    """根據 ID 查詢單一使用者"""
    user(
        """使用者的唯一識別碼"""
        id: ID!
    ): User

    """查詢使用者列表，支援篩選和分頁"""
    users(
        """篩選條件"""
        filter: UserFilter

        """分頁參數"""
        pagination: PaginationInput
    ): UserConnection!
}
```

## API 文件的最佳實踐

### 1. 結構化且一致的格式

使用統一的文件結構，讓開發者能快速找到需要的資訊。

### 2. 提供完整的範例

範例是最好的文件。提供包含請求和回應的完整範例。

```json
// 請求範例：更新使用者資料
PUT /api/v1/users/usr_abc123
Headers: {
    "Authorization": "Bearer eyJhbGciOiJIUzI1NiIs...",
    "Content-Type": "application/json"
}
Body: {
    "name": "王小明",
    "department": "技術長室"
}

// 成功回應
Status: 200 OK
Body: {
    "id": "usr_abc123",
    "name": "王小明",
    "email": "wang@example.com",
    "department": "技術長室",
    "updated_at": "2024-01-15T14:30:00Z"
}

// 錯誤回應
Status: 404 Not Found
Body: {
    "error": {
        "code": "USER_NOT_FOUND",
        "message": "找不到指定的使用者"
    }
}
```

### 3. 說明錯誤處理

清楚列出所有可能的錯誤碼和對應的處理方式：

```markdown
### 錯誤回應格式

所有錯誤回應都使用統一的格式：

\```json
{
    "error": {
        "code": "ERROR_CODE",
        "message": "人類可讀的錯誤描述",
        "details": [
            {
                "field": "email",
                "message": "電子郵件格式不正確"
            }
        ]
    }
}
\```

### 常見錯誤碼

| 錯誤代碼 | HTTP 狀態碼 | 說明 | 建議處理方式 |
|----------|-------------|------|-------------|
| INVALID_REQUEST | 400 | 請求格式不正確 | 檢查請求參數格式 |
| UNAUTHORIZED | 401 | 未提供認證憑證 | 重新取得 access token |
| TOKEN_EXPIRED | 401 | Token 已過期 | 使用 refresh token 更新 |
| FORBIDDEN | 403 | 權限不足 | 確認使用者權限 |
| NOT_FOUND | 404 | 資源不存在 | 確認資源 ID 是否正確 |
| RATE_LIMITED | 429 | 請求頻率超過限制 | 實施指數退避重試策略 |
| SERVER_ERROR | 500 | 伺服器內部錯誤 | 稍後重試或聯繫管理員 |
```

### 5. 認證與授權說明

清楚說明 API 的認證機制：

```markdown
## 認證

本 API 使用 OAuth 2.0 Bearer Token 進行認證。

### 取得 Token

\```
POST /auth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&client_id={your_client_id}&client_secret={your_client_secret}
\```

### 使用 Token

在每個請求的標頭中包含 Token：

\```
Authorization: Bearer {your_access_token}
\```

### Token 有效期

- Access Token：1 小時
- Refresh Token：30 天
```

## API 文件品質檢查清單

- [ ] **端點完整性**：所有 API 端點是否都有文件？
- [ ] **參數說明**：所有參數是否都有清楚的說明和驗證規則？
- [ ] **回應格式**：成功和錯誤的回應格式是否都有說明？
- [ ] **範例**：每個端點是否都有請求/回應範例？
- [ ] **認證說明**：認證機制是否清楚說明？
- [ ] **版本資訊**：版本和變更記錄是否完整？
- [ ] **錯誤碼**：所有錯誤碼是否都有列出和說明？
- [ ] **使用流程**：是否提供了常見使用場景的流程說明？
- [ ] **準確性**：文件是否與實際 API 行為一致？
- [ ] **可讀性**：文件是否結構清晰、易於閱讀？

## 總結

高品質的 API 文件是成功 API 的關鍵。透過 AI 輔助，可以更有效率地產生和維護 API 文件。核心原則：

1. **一致性**：使用統一的文件格式和術語
2. **完整性**：涵蓋所有端點、參數、回應和錯誤
3. **實用性**：提供豐富的範例和使用指南
4. **準確性**：確保文件與 API 實際行為一致
5. **可維護性**：建立文件更新的流程和機制
