# 10.4 範例：自動化程式碼註解

許多程式設計師偏好專注於實際的程式碼，而不是乏味地為其添加註解。但如果之後有其他人必須進一步開發或維護程式碼，就很難找到好的參考了。因此，我們有了以下想法：AI 工具能不能為現有程式碼添加註解？

在 ChatGPT 中的初步測試一直很成功（另見第 6 章）。然後我們嘗試自動化該過程，並使用了以下角色描述：

> 角色：您是一個程式設計助理。您的目標是為現有程式碼添加缺少的註解。至少要為檔案添加一個標頭以說明其用途。嘗試解釋每個函式/方法的目的。
>
> 如有必要，添加註解來解釋程式碼區塊。不要變更/移除現有註解。
>
> 除了註解之外，保持程式碼不變。程式碼的功能不得被變更。
>
> 不要在您的回應中添加解釋。
>
> 不要使用 Markdown 來格式化您的回應。

當您瀏覽以下程式碼時，基本流程應該立刻就能清楚。

```python
# Sample file openai/comment-files.py
#!/usr/bin/env python3
import os
from openai import OpenAI
client = OpenAI() # reads the .env file

# searches recursively for all files with specified identifiers
def find_files_with_extensions(directory, extensions):
    matching_files = []
    for root, _, files in os.walk(directory):
        for file in files:
            if any(file.endswith(ext) for ext in extensions):
                matching_files.append(os.path.join(root, file))
    return matching_files

# reads a code file, adds comments and saves them
def comment(filename):
    # Read file, create backup
    with open(filename, "r") as file:
        content = file.read()
    with open(filename + ".bak", "w") as backup:
        backup.write(content)
    # Put together roll and prompt
    role = """You are a coding assistant. Your goal ..."""
    prompt = f"""Please add comments to the 
                 following code:\n\n{content}"""
    try:
        completion = client.chat.completions.create(
            model = "gpt-4o",
            temperature = 0.3,
            messages = [ {"role": "system", "content": role},
                         {"role": "user",   "content": prompt} ] )
    except Exception as e:
        print("  An error occurred:", str(e))
        return
        
    # Analyze result, save modified code file
    if completion.choices[0].finish_reason != "stop":
        print("  Commenting failed.")
        print("  Input Tokens:", completion.usage.prompt_tokens)
        print("  Output Tokens:",
              completion.usage.completion_tokens)
    else:
        new_code = completion.choices[0].message.content.strip()
        with open(filename, 'w') as file:
            file.write(new_code)
        print("  Commenting done.")
        print("  Input Tokens:", completion.usage.prompt_tokens)
        print("  Output Tokens:",
              completion.usage.completion_tokens)

# Main
dir = "/Users/kofler/my-sample-files"
extensions = [".js", ".php"]
files = find_files_with_extensions(dir, extensions)
for filename in files:
    print(filename)
    comment(filename)  
```

角色的定義對此腳本的功能至關重要。在那裡您可以指定語言模型應該多加還是少加註解、註解時是否應遵循特定語法（例如 JavaDoc）等等。

> **提示：**
>
> 如果您想用自己的檔案嘗試此腳本，請將您的程式碼複製到一個單獨的目錄中，並在那裡移除現有的註解。（我相信您是一位善於寫註解的開發者，對吧？）
>
> 一個用於移除註解的有用工具是 cloc（https://github.com/AlDanial/cloc）。使用 `cloc --strip-comments=nc myfile.py`，您將建立一個不包含任何註解的新檔案 `myfile.py.nc`。除了 `.nc`（無註解）之外，您可以使用任何字母組合。

## 10.4.1 實際經驗

我們不僅使用 OpenAI API 測試了註解腳本，還使用了 Ollama 和 Groq。本書的範例檔案中包含了相應的腳本。我們得到的結果好壞參半：

**OpenAI 搭配 GPT-4o**

對於大約 10 KB 以下的小型程式碼檔案，自動註解功能運作得相當好。當然，即使語言模型也無法預知一切，但在許多情況下，註解在內容上是恰當的，至少可以作為理解程式碼的初步幫助。

當然，某些註解的有用性是可以討論的。如果一個自定義的 JavaScript 函式叫做 `updateTopicsWithChapters`，而語言模型在它上方的那行添加了註解「Function to update topics with chapters」，那就沒有太大的附加價值。但也有更正面的例子。在我們的 PHP 測試檔案中，AI 工具為每個方法添加了所有參數、傳回值和任何例外的描述。

OpenAI API 的主要問題被證明是其 4,096 個輸出 token 的限制。我們嘗試了各種方法來繞過這個限制（第 10.4.4 節），但都失敗了。

**OpenAI 搭配 GPT-4o-Mini**

令人驚訝的是，在我們的測試中，gpt-4o-mini 幾乎不比 gpt-4o 快。註解的品質是可比的。幸運的是，gpt-4o-mini 可以處理更大的檔案，因為最大輸出 token 數量要高得多。然而，gpt-4o-mini 在註解方面非常吝嗇，特別是對於長檔案，僅限於函式的單行描述。

**Ollama**

在我們的測試電腦（配備 M3 Pro CPU 的 MacBook）上，Ollama 只能以合理的速度執行相對較小的 LLM（最多約 100 億個參數），例如 llama3:8b。我們也實驗了 gemma2、codellama 和 codestral，但測試的 LLM 無法勝任此任務，特別是對於較大的檔案。答案一再地不包含帶有註解的程式碼，而是對程式碼檔案的相對簡短的文字描述。這就偏離了重點。

**Groq**

使用託管在 Groq 伺服器上的語言模型為較小的程式碼檔案添加註解運作完美。這令人驚訝，因為使用的部分語言模型與 Ollama 相同，只是具有更大的上下文視窗和更多參數（例如 llama3-70b-8192 或 mixtral-8x7b-32768）。

Groq 對於大型程式碼檔案也失敗了。在這種情況下，原因是開發者存取的速率限制。僅僅對一個大型程式碼檔案進行註解就已經超過了每分鐘 token 的限制，因此被拒絕。

我們本來希望在不受限流的情況下重複我們的測試，並願意為相應的測試帳戶付費，但我們在這方面的詢問直到本書的文稿已經完成時仍未得到回覆。Groq 似乎更針對大型企業客戶，而不是小型開發者。

**微調**

為了獲得更好的結果，您可以準備一組範例檔案——一個未加註解的版本和一個您自己認為是最佳解決方案的加註版本。從這些範例檔案中，您建立提示-回應對，並將其嵌入到 messages 陣列中。這為語言模型提供了一些可以作為參考的模式（也包括關於所需回應格式的方面）。

然而，這種方法在我們的測試中並未帶來令人信服的結果。相反，我們得到的印象是，上下文視窗中的額外範例往往會使語言模型感到困惑，特別是當範例的內容與之後要處理的程式碼檔案無關，甚至是以不同的程式語言編寫的時候。這種方法也會增加成本，因為範例必須隨每次 API 呼算一起發送，這會大幅增加輸入 token 的數量。

如果您使用範例集合來微調語言模型，可以更有效地實現同一個想法。有關微調 OpenAI 語言模型和本機語言模型（Ollama）的背景資訊和技巧，請參閱以下頁面：

- https://openai.com/index/gpt-4o-fine-tuning
- https://github.com/ollama/ollama/issues/2488
- https://huggingface.co/docs/transformers/training

## 10.4.2 從回應中提取程式碼

只有 GPT-4o 和 GPT-4o-mini 能持續傳回不含 Markdown 格式、不含引言文字（例如「Here is the code with added comments」）或後續解釋的程式碼，正如我們所要求的那樣。在我們測試的所有其他語言模型中，答案一再被以不同方式潤飾。我們嘗試使用 Python 函數 `extract_code` 從回應中提取程式碼。雖然這在我們的測試中可靠地運作，但如果語言模型不夠「聰明」來理解任務，這當然不是一個好跡象。

```python
# in the sample file groq/comment-files.py
def extract_code(md):
    md = md.strip()
    lines = md.split("\n")
    first, last = 0, len(lines)
    for i in range(len(lines)):
        if lines[i].startswith("```"):
            first = i + 1
            break
    
    if first == 0:
        return md
    else:
        print("  Markdown found")
    for i in range(first + 1, len(lines)):
        if lines[i].startswith("```"):
            last = i
            break
    return "\n".join(lines[first:last])  
```

## 10.4.3 真實的錯誤

在我們的測試過程中，我們只遇到過一個嚴重的錯誤：GPT-4o 嘗試為嵌入在 PHP 檔案中的 SQL 程式碼添加註解。如果它按照 SQL 標準使用 `--` 來引入註解，這本來是沒問題的。但它使用了 `//`，這是 GPT-4o 特有的做法，並導致 SQL 命令觸發了錯誤。

我們已經大幅簡化了以下列表以說明問題。實際上，那是一個超過 20 行的相當複雜的 SELECT 命令。

```php
// before
$sql = sprintf("
  SELECT n.*, CONCAT(p.firstname, ' ', p.lastname) AS fullname
  FROM notes n
  JOIN person p ON p.id = n.author
  WHERE n.topic = %u
  ORDER BY n.ts", topic_id);

// after
$sql = sprintf("
  SELECT n.*, CONCAT(p.firstname, ' ', p.lastname) AS fullname
  FROM notes n
  JOIN person p ON p.id = n.author
  WHERE n.topic = %u
  // sort by timestamp     <-- incorrect comment!
  ORDER BY n.ts", topic_id);  
```

即使這個錯誤是個案（通常 SQL 程式碼會被正確處理，即使它嵌入在使用其他程式語言的程式碼檔案中），這個例子證明您必須始終檢查自動插入的註解。最好的方法是使用像 Visual Studio Code（VS Code）這樣的編輯器，將加註後的檔案與上次提交時的狀態進行比較。

## 10.4.4 大型程式碼檔案的麻煩

這個範例的主要問題是，所有 API 都因各種原因無法處理大型程式碼檔案。我們建立了前面印出的腳本的幾個變體，嘗試繞過這個限制。（我們僅使用 OpenAI API 進行了這些實驗，沒有使用任何其他 API。）

遺憾的是，所有嘗試都失敗了：

1. **將程式碼檔案分割成片段**，即第 1 到 100 行，然後第 101 到 200 行，以此類推。每次呼叫時，語言模型接收 100 行並應為這些行添加註解。

   但這不是一個好主意。首先，上下文丟失了。語言模型無法再整體分析程式碼；例如，僅根據前 100 行，它無法估計整個檔案程式碼的總體目的。其次，程式碼經常在不利的位置被拆分。不匹配的括號層級、未封閉的字串等使程式碼片段在語法上不正確。如果您想將人類特質歸因於語言模型，您可以說這些錯誤讓模型失去平衡。它會違反所有指令，嘗試修正程式碼。

2. **每次都將整個程式碼檔案傳遞給語言模型**會更聰明一些。這需要一個具有足夠大上下文視窗的語言模型。對大量上下文的需求很普遍，因此越來越多的新模型在宣傳其高效處理大量上下文的能力。例如，GPT-4o 允許 128,000 個 token 的上下文大小。這即使對於巨大的檔案也足夠了。

   然後在提示中，您要求模型第一次被呼叫時註解並傳回第 1 到 100 行，下次傳回第 101 到 200 行，以此類推。與前一種策略的主要區別在於，語言模型始終可以存取整個程式碼檔案，即使它每次只處理和傳回一個行範圍。

   我們的嘗試主要因為 GPT-4o 沒有嚴格遵守行號而失敗。例如，模型本應處理第 1 到 100 行。在第 90 行，一個方法的程式碼延伸到了第 110 行。模型考慮並傳回了整個方法。為了確保程式碼在語法上正確，它在方法後面添加了一個括號來關閉類別。然而，這個在原始檔案中此位置不出現的括號，如果稍後要重新組裝各個程式碼片段，就顯得多餘了。

3. **我們下一個想法是要求語言模型只傳回 DIFF 補丁**，而不是新的完整程式碼。這有兩個優點。首先，補丁只包含變更，因此更小，使用的 token 更少。因此，可以一次處理更多的程式碼而不會超過輸出 token 限制。

   其次，我們認為應用補丁會比合併變體 2 中的程式碼片段更穩健，但我們錯了：補丁的品質很差，許多變更無法正確應用到原始程式碼檔案。部分錯誤是微不足道的；通常行號就是錯的。然後我們嘗試了上下文補丁或統一補丁（diff 選項 `-c` 或 `-u`）。這也沒有好轉。語言模型顯然不比人類更擅長用 DIFF 補丁思考。

在這個時候，我們放棄了實驗。最終，沒有辦法繞過完整地編輯每個程式碼檔案。唯一的解決方案是等待具有大上下文視窗並允許大量輸出 token 的語言模型或 API。當您讀到本書時，此類服務很有可能已經很普遍了。

## 10.4.5 翻譯現有註解

原始任務的一個變體是註解的翻譯。許多專案從小規模開始。註解是以各自的在地語言撰寫的（如果有的話）。如果專案在 10 年後出乎意料地成功並即將被出售，程式碼中的德文（或法文或義大利文）註解突然就不再好看 了。它們阻礙了國際團隊進一步開發程式碼，並降低了軟體的價值。

基本上，前面介紹的腳本結構沒有任何變化。只需調整角色描述和提示的撰寫方式：

```python
# Sample file openai/translate-comments.py
...
role = """You are a coding assistant.
    Your goal is to translate non-English comments into English.
    Apart from comments, leave the code as it is. 
    The functionality of the code MUST NOT be changed.
    DO NOT ADD EXPLANATIONS in your response. 
    DO NOT USE MARKDOWN to format your response."""
prompt = f"""In the following code, please replace the German 
    comments with English comments:\n\n{content}"""
...  
```

我們使用 OpenAI API（GPT-4o）進行的測試絕對令人滿意，儘管是在已描述的限制範圍內。因此，該腳本在處理大小超過上下文視窗或輸出 token 數量的程式碼檔案時會失敗。

在現實中，通常不僅註解不是英文的，變數和函式名稱、資料表和欄位名稱等也不是英文的。鑑於先前的經驗，我們希望在這裡不需要解釋，自動化將專案程式碼庫英文化距離目前 AI 工具的能力還很遙遠。

主要問題是，在一個類別的檔案中變更方法名稱時，也必須在使用該類別的所有其他檔案中反映出来。因此不可能單獨處理每個檔案。而是必須將程式碼庫作為一個整體來考慮。好的編輯器和 IDE 能夠做到這一點。如果能在編輯器和 AI API 之間建立一個介面，結合兩個工具的能力，那就太好了。目前，這仍然是一個對未來的夢想。

## 10.4.6 結論

我們開始這一節的工作時，期望能夠提供一個簡單但有用的範例。我們完全弄錯了，原因如下：

- 許多 LLM 甚至無法一致地完成在系統描述（角色）和提示中精確定義的任務。
- 小型檔案的註解效果相當不錯。對於大型檔案，只有當上下文視窗足夠大以處理整個檔案且回應大小不受限制時，註解才能成功。
- 在極少數情況下，語言模型變更了程式碼，使其不再正常運作。對變更進行徹底檢查是絕對必要的。
- 從 ChatGPT 等工具中，我們已習慣語言模型幾乎毫無延遲地運作。一旦您將這裡介紹的腳本應用於 10 個、100 個甚至更多檔案，您就必須告別這個想法。根據 API 或您自己的硬體，執行可能需要數分鐘，或者在最壞的情況下需要數小時。這使得測試和最佳化過程非常繁瑣。
