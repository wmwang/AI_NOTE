# 5.3 AI 輔助的測試驅動開發

測試驅動開發（TDD）是一個循環過程，通常被描述為 TDD 週期。它由三個階段組成：紅色、綠色和重構。第一步是一個失敗的測試，因為它出現在實作之前。紅色代表測試的失敗。然後你實作確保測試能成功執行的程式邏輯。這使得測試變為綠色。在最後一步，你有機會優化你的程式碼。你進行重構，其中對外部的介面保持不變，功能方面也沒有任何改變。第 4 章探討了 AI 如何在這方面幫助你，以及有哪些可用的策略。

在 TDD 中，你不會直接開始開發。相反，你首先思考你的程式碼單元（通常是一個函式）應該滿足的需求，並將這些需求以一個或多個測試的形式表述出來。然後，你實作這個函式。你以小步驟進行，即所謂的嬰兒步，逐步或更確切地說逐個測試地實作函式。

在本章中，我們使用兩種不同的 AI 工具進行 TDD，但在實際工作中，你很少會這樣做。這個例子更多的是為了展示你如何在開發過程的各個部分獲得幫助。

作為 TDD 的具體範例，我們正在實作一個用於檢查信用卡號的函式。這樣的號碼由不同的部分組成：前六位數字標識發卡金融機構，隨後的數字構成持卡人的帳號。與我們範例相關的最後幾位數字構成檢查碼。這是使用 Luhn 演算法計算的。

因為我們希望 AI 工具為我們做這項工作，而不是讓我們不必要地忙碌，所以我們讓兩個工具在此協作：ChatGPT 撰寫測試，GitHub Copilot 負責實作。

## 5.3.1 實作第一個測試

像 ChatGPT 這樣的工具非常有用，通常會嘗試給出全面的回答。然而，這對於我們當前的任務來說更像是一種阻礙。為了防止工具向你展示如何設定測試環境、提供所有測試和實作以及執行測試，你需要讓提示非常明確：

**提示：** 假設環境已經設定好，所有東西都已安裝。使用 TypeScript 在 vitest 中建立一個測試，用於驗證給定信用卡號的函式 validateCreditCardNumber。使用 TDD 和嬰兒步作為策略。只撰寫第一個測試；不需要如何執行測試的資訊。

當然，讓我們從測試驅動開發（TDD）的第一步開始：為一個尚未存在的函式 validateCreditCardNumber 撰寫測試。

此外，GitHub 推出了 Copilot Workspace，這是一個原生的 Copilot 開發環境，旨在協助開發人員完成從腦力激盪到編碼和測試的各種任務。Copilot Workspace 能夠在儲存庫中進行 AI 輔助開發，允許在更大的上下文中進行程式設計任務。

## 5.3.2 第二個測試

如果第一個測試仍然在 ChatGPT 工作階段的上下文視窗中，你可以從之前的提示開始，要求 ChatGPT 實作下一個測試：

**提示：** 建立下一個測試。

步驟 2：撰寫下一個測試。我們現在將撰寫一個測試來檢查 validateCreditCardNumber 函式是否正確驗證有效的信用卡號。以下是 validateCreditCardNumber.test.ts 的更新版本，包含下一個測試：

## 5.3.3 第三個測試

現在，又輪到 ChatGPT 來進行下一個測試了。由於已有的上下文，這個提示也非常簡單：

**提示：** 建立下一個測試。

當然，讓我們繼續為 validateCreditCardNumber 函式撰寫下一個測試。這個測試將檢查函式是否正確識別無效的信用卡號。以下是 validateCreditCardNumber.test.ts 的更新版本，包含下一個測試：

```typescript
import { describe, it, expect } from 'vitest';
import { 
  validateCreditCardNumber
} from './validateCreditCardNumber';
describe('validateCreditCardNumber', () => {
  (existing code)
  it(
    'should return false for an invalid credit card number', 
    () => {
    // Example of an invalid credit card number
    const invalidCreditCardNumber = 
'1234567890123456'; 
    
    expect(validateCreditCardNumber(invalidCreditCardNumber))
      .toBe(false);
  });
}); 
```

[程式碼說明]

這第三個測試是一個經典的負面測試。它讓你能夠測試函式是否也正確處理錯誤的輸入。這也迫使你調整函式的實作，使演算法正確。在提示中，你將三個測試傳給 GitHub Copilot，並指示它相應地更改程式碼。

**提示：** 更改實作，使以下三個測試通過：
[測試的程式碼]

```typescript
// Implementation of validateCreditCardNumber function
export function validateCreditCardNumber(
  cardNumber: string
): boolean {
  // Remove any non-digit characters from the card number
  const cleanedCardNumber = cardNumber.replace(/\D/g, "");
  // Check if the cleaned card number is empty or not
  if (cleanedCardNumber.length === 0) {
    return false;
  }
  // Use Luhn algorithm to validate the card number
  let sum = 0;
  let double = false;
  for (let i = cleanedCardNumber.length - 1; i >= 0; 
i--) {
    let digit = parseInt(cleanedCardNumber.charAt(i), 10);
    if (double) {
      digit *= 2;
      if (digit > 9) {
        digit -= 9;
      }
    }
    sum += digit;
    double = !double;
  }

  return sum % 10 === 0;
} 
```

Copilot 出色地完成了任務，撰寫的程式碼使所有三個測試都能成功通過。函式中的各個步驟都附有描述性註解，讓你可以輕鬆追蹤程式碼。

從空函式到最終實作的這一步並不符合 TDD 的理念。這裡應該有更多的中間步驟。在這種情況下，你在測試設計以及對 Copilot 的提示方面應該更加嚴格，更好地引導 AI 工具。同樣，以下原則也適用：你的提示表述得越精確，你就越能控制結果。

## 5.3.4 產生測試資料

如果你仍然對實作是否真的有效感到懷疑，你可以讓另一個測試產生，檢查一系列的信用卡號。對於這種大量測試，你可以使用 `it.each` 方法來測試一系列值，同時保持程式碼緊湊。相應的提示可以如下：

**提示：** 建立 10 個有效和 10 個無效的信用卡號，並使用 it.each 來驗證 validateCreditCardNumber 函式是否正確運作。

當然，讓我們使用 it.each 來增強我們的測試，一次驗證多個信用卡號。我們將建立 10 個有效和 10 個無效的信用卡號，並使用 it.each 來驗證 validateCreditCardNumber 函式是否正確運作。

以下是更新的 validateCreditCardNumber.test.ts：

```typescript
import { describe, it, expect } from 'vitest';
import { 
  validateCreditCardNumber 
} from './validateCreditCardNumber';

const validCreditCardNumbers = 
  [... validCreditCardNumbers ... ];
const invalidCreditCardNumbers = 
  [ ... invalidCreditCardNumbers ... ];
describe('validateCreditCardNumber', () => {
  it('should be a function', () => {
    expect(typeof 
validateCreditCardNumber).toBe('function');
  });
  it.each(validCreditCardNumbers)(
    'should return true for a valid credit card number: %s', 
    (cardNumber) => {
    
    expect(validateCreditCardNumber(cardNumber)).toBe(true);
  });
  it.each(invalidCreditCardNumbers)(
    'should return false for an invalid credit card number: %s', 
    (cardNumber) => {
    
    expect(validateCreditCardNumber(cardNumber)).toBe(false);
  });
}); 
```

[程式碼說明]

由於已有的上下文，ChatGPT 將這個提示理解為對現有測試的改進，即重構。validateCreditCardNumber 的實作保持不變，但在這種情況下測試的程式碼會有所更改。一般來說，測試的重構也是允許的。在這個例子中，你定義兩個陣列，一個包含有效的信用卡號，另一個包含無效的信用卡號。

validateCreditCardNumber 是否為函式的第一個測試保持不變。對於有效和無效的信用卡號，你使用 `it.each` 函式為每個值產生一個單獨的測試。`it.each` 函式在描述中使用佔位符，以便在輸出中插入各自的值。
