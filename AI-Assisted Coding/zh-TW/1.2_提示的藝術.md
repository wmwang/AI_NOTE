# 1.2 提示的藝術

在 AI 聊天系統中，輸入或問題通常被稱為提示（prompt）。提出正確的問題對結果的好壞或有效性有重大影響。最佳化構建提示的藝術通常被稱為提示工程（prompt engineering）。本節總結了有關此主題的一些提示：

**附加資訊**

在問題中加入有關您使用的程式語言、函式庫、工具或環境的上下文資訊。

如果您一般性地詢問正規表示式模式，ChatGPT 必須猜測指的是哪種語法變體（Portable Operating System Interface [POSIX]、POSIX Extended Regular Expressions [ERE]、Perl Compatible Regular Expressions [PCRE] 等）。然而，如果您指定要在 C# 語言中使用正規表示式模式，ChatGPT 可以考慮正規表示式的 Microsoft 特定功能。

在會話的第一個提示前簡要加上程式語言或其他規格就足夠了，例如「bash/Linux」、「nodejs 20」、「PHP 8」、「Regex/PCRE」、「SQL Server」或「zsh/macOS」。

**附加條件**

指出對您來說重要的事項：
- 您想要簡短、精簡的程式碼嗎？
- 是否應該使用盡可能少的匯入/模組/函式庫？
- 程式碼是否為安全關鍵的？
- 執行時的最大效率是否至關重要？
- 程式碼是否應盡可能遞迴地表述？或者相反，應該避免遞迴？

您可以在第一次嘗試中省略這些附加規則，看看結果如何。之後，您仍然可以要求 AI 工具根據新條件重新編寫程式碼：
- 程式碼可以運作，但太慢了。能讓它更快嗎？

**正式規則**

您可以定義正式規則。這些可以涉及最大行長度、變數和函式名稱的所需慣例（camelCase、snake_case 等）、註解的位置和範圍等等。

**語言**

即使英語不是您的母語，也請嘗試用英語表述您的問題。

**後續問題**

提出後續問題。ChatGPT 會自動考慮問題/答案歷史記錄。

**第二次嘗試絕不會有壞處**

如果您對答案不滿意，就再試一次。明確指出從您的角度來看答案的哪個細節不合適。

在 ChatGPT 網頁介面中，您可以點擊「重新產生」按鈕。新答案碰巧會略有不同，可能更符合您的期望。但只要 ChatGPT 不知道您為什麼不滿意，第二次嘗試得到更好結果的機會就相對較低。最好在後續問題中說明您對前一個答案中哪裡不滿意。

如果沒有「重新產生」按鈕，而您又沒有時間或意願來構建具體的改進要求，可以嘗試以下通用指令：**Do better!** 不用擔心此提示的簡單性；它需要的努力很少，偶爾甚至會成功。

**多個選項**

通常，您提出一個問題並得到一個答案，但從您的特定角度來看，這可能不是最好的答案。

如果您正在熟悉一個新領域或解決方案不明確，您也應該在提示中表達這一點。要求語言模型提供多個選項供您選擇，例如：

> 在 Python 中實作 REST API 有哪些選項？

AI 工具將提供各種函式庫和框架的概述。您可以在網際網路或 AI 聊天系統中研究提案的細節。然後，選擇其中一個變體並提出後續問題。

**思路鏈（推理）**

以下提示想法朝著類似的方向發展。與其要求一個特定的解決方案，您可以要求 AI 聊天機器人扮演多位專家討論一個問題，如下例所示：

> 想像三位 IT 專家正在處理一個問題。他們討論各種提案，放棄不合適的想法，最終達成一個解決方案。任務如下：如何為網頁伺服器設計一個冗餘備份系統？

**逐步進行**

對於複雜的任務，您應避免嘗試一次性將所有內容放入一個提示中。如果您逐步進行，先要求 AI 聊天機器人提供大綱，然後在後續提示中處理細節，效果會好得多。此過程也被稱為提示鏈（prompt chaining）。第一個提示可能如下所示：

> 我需要設計一個資料庫來分析我們的網頁流量。首先，請給我資料庫的大綱。（我需要哪些資料表？）我們稍後再討論每個資料表的細節。

請注意，ChatGPT 和類似服務對最大輸入和輸出長度（上下文視窗的大小和最大輸出 token 數）有嚴格的限制。因此，AI 聊天機器人永遠不會提供由數千行程式碼組成的答案。也不建議將包含許多行程式碼的檔案傳遞給提示並要求完整修訂（例如重構）。相反，您需要將任務分解為部分，並分別執行每個步驟。

**主題變更時開啟新會話**

當您變更主題時，開始一個新會話。ChatGPT 會考慮在會話中收集的資訊來回答後續問題。因此，所有先前的問題和答案構成了所有進一步問題的上下文。

然而，如果新問題涉及不同的程式語言、不同的問題或不同的函式庫，建議在單獨的會話中重新開始。

如果您對目前的結果不滿意，或者您認為您或 AI 工具已經陷入死胡同，重新開始也可能很有用。先前問題和答案的上下文可能阻止 ChatGPT 探索其他想法。

**理解並測試程式碼**

AI 工具不對您的程式碼負責——您才是！不要只是希望產生的程式碼能運作；相反，要徹底測試它，考慮所有可能的特殊情況。如果您無法理解 AI 產生的程式碼，您需要熟悉其背後的概念，即使目前不方便。

**特別注意新的或冷門的主題**

語言模型基本上在處理新技術或周邊 IT 領域時有困難。一方面，可用的訓練材料很少；另一方面，問題因當前語言模型的訓練通常至少在 6 個月前、往往 12 個月或更早之前就開始了而更加嚴重。

> **口語化提示也有效，但並非總是完美**
> 在本書中，我們努力用整潔、完整的句子來構建提示。本書的全部文字都經過了校對，英語不佳的提示根本不合適。
>
> 但不用擔心：即使您非常簡短地構建提示，AI 工具通常也能理解您。像前面提到的 Do better!、Optimize this code!、How does this work? 或 Why doesn't this work? 這樣的提示也可以。
>
> 然而，重要的是您的問題意圖必須明確。精確的表述有助於您和 AI 工具更好地理解問題。在這方面，花在「正確」構建提示上的時間通常是值得的。

## 1.2.1 提示框架

遵守各種提示框架的規則可以幫助您進行提示。在此上下文中，「框架」是一種指南，說明提示可以由哪些組件組成。

最受歡迎的框架是 **RTF 框架**。在提示中，您表述 AI 工具的角色、特定任務和所需的回應格式。在以下範例中，我們標記了提示的組件。在實際使用中，這些標記當然會被省略。

**提示：**（角色）您是一位經驗豐富的 Python 開發者。（任務）使用 Django 繪製待辦事項應用的 REST API 框架。（格式）以逐步方式呈現答案。只顯示程式碼大綱；先不要深入細節。

RTF 的替代方案包括 **RODES 框架**（角色、目標、細節、範例、意義檢查）和 **RISEN 框架**（角色、指示、步驟、最終目標、縮小範圍）。然而，表述如此多的細節需要相當大的努力。在這方面，使用此類框架更適合您透過 API 自行程式設計 AI 聊天機器人或其他應用程式時（另見第 10 章）。

使用這裡概述的提示結構框架是可選的。即使您只指定任務（即 RTF 框架意義上的任務），您也會取得有用的結果。但特別是在棘手的情況下，遵守框架規則可以帶來更好的結果。

## 1.2.2 上下文的問題

我們已經指出，AI 工具需要附加資訊，即上下文，才能提供相關的答案。上下文決定了在實際語言模型之外主動處理哪些資料。在 ChatGPT 會話中，上下文來自會話中迄今為止提出的所有問題（提示）和由此產生的答案。

語言模型的上下文大小是有限的。它通常以 token 為單位測量（見表 1.1）。有關上下文和 token 的更多技術細節將在第 1.5 節中介紹。一些語言模型有進一步的限制。ChatGPT-4o 的輸出限制為 4,000 個 token。這意味著 ChatGPT-4o 盡管上下文記憶體相對較大，但一次只能交付大約 12 KB 的程式碼。4,000 個 token 的輸出限制也適用於 Anthropic 的 Claude 3 Opus 語言模型。對於 Claude 3.5 Sonnet，它已增加到 8,000 個 token。

| 語言模型 | 最大上下文大小 | 最大文字量 | 最大程式碼量 |
|---------|--------------|-----------|------------|
| GPT-3.5 Turbo | 4,096 tokens | ~16,000 字元 | ~12 KB |
| GPT-4 | 8,000 tokens | ~32,000 字元 | ~24 KB |
| GPT-4o (omni) | 32,000 tokens | ~128,000 字元 | ~96 KB |
| Llama 3.3 | 128,000 tokens | ~500,000 字元 | ~380 KB |
| Claude 3 Opus | 200,000 tokens | ~800,000 字元 | ~600 KB |
| Claude 3.5 Sonnet | 200,000 tokens | ~800,000 字元 | ~600 KB |

> 表 1.1 來自 OpenAI、Meta 和 Anthropic 的各種語言模型的最大上下文大小（截至 2025 年 1 月）

在未來，即使是 Anthropic 模型相當大的上下文大小也可能看起來很 modest。2024 年 8 月，Magic AI 公司展示了一個能夠處理 1 億 token 上下文視窗的語言模型。這相當於大約 1,000 萬行程式碼。作為比較，整個 Linux 核心——這是最大的公共軟體專案之一——包含約 3,000 萬行程式碼。

最大上下文大小（通常也稱為上下文視窗）和最大輸出與提示有什麼關係？我們在使用 AI 工具時有最好的經驗，當我們將自己限制在小的、可管理的功能時。不要一次要求所有東西（「像 booking.com 一樣編寫飯店預訂軟體的程式碼。」）；相反，先設計程式的結構，然後將 AI 工具應用於其各個功能。

> **GPT 是無狀態的**
> 您已經習慣了 ChatGPT 和其他工具允許您在後續問題中引用先前聊天歷史記錄中的問題和答案。這給您一種 ChatGPT 正在記住對話的印象。
>
> 然而，語言模型實際上是無狀態運作的；也就是說，它們不記住任何東西，並將每個提示視為全新的。記憶的印象僅僅是透過聊天介面的程式設計產生的。從第二個提示開始，網站不僅將新提示傳輸給語言系統，還傳輸所有先前的問題和答案。這些構成了新問題的上下文，因此也構成了聊天歷史記錄的「記憶體」。
>
> 在第 10 章中，我們描述了最小聊天系統的程式設計在幕後是如何運作的。

## 1.2.3 自訂 GPT

如果您重複指定某些規則，可能值得設置一個自訂的生成式預訓練 Transformer（GPT）。（我們將在第 1.5 節中完整解釋 GPT 這個術語。）使用 ChatGPT 定義自己的 GPT 需要 ChatGPT 的付費版本。在 ChatGPT 網頁介面中，首先點擊「Explore GPTs」，然後點擊「Create」。在 GPT 編輯器中，進入「Configure」對話框。在那裡，您給自己的 GPT 取一個名稱，簡要描述預期用途，然後指定 GPT 的指令。

> 圖 1.2 左側為新 GPT 的指令，右側為應用範例

這些指令可能如下所示：

**指令：** 您的唯一目的是幫助 Python 程式設計。不要回答其他問題。請遵循以下規則：
- 使用英文名稱和註解，即使提示使用其他語言。
- 使用 camelCase 命名法。
- 盡量避免超過 65 個字元的程式碼行。
- 盡量產生簡潔易懂的程式碼。效能不是主要關注點。
- 盡可能避免需要外部安裝（pip）的模組。

根據應用，您可以使用「Upload files」按鈕上傳包含附加資訊的檔案（例如，語言模型訓練時尚不可用的新軟體函式庫或版本的文件），我們在此範例中沒有這樣做。允許的文件格式為純文字、Markdown、HTML 和 PDF。您可以在「Preview」區域立即試用您的個人化 GPT。

在圖 1.2 中，您可以看到新的程式碼精靈按預期運作：
- 提示不包含 ChatGPT 應使用哪種程式語言的任何資訊。但由於個人化 GPT 指令，ChatGPT 知道它應該使用 Python。
- 函式和變數名稱使用 camelCase 命名法。

「Code Interpreter」核取方塊允許 GPT 直接執行程式碼，例如執行測試執行。然而，這僅適用於少數程式語言，並且也與各種限制相關，例如執行時間或存取檔案或網際網路的條款。

**提示：** 你能展示一個計算 Fibonacci 數的腳本嗎？請執行腳本並計算第 25 個 Fibonacci 數。

```python
def fibonacci(n):
    """Calculate the n-th Fibonacci number."""
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a

# Calculate the 25th Fibonacci number
fibonacci(25)
```

第 25 個 Fibonacci 數是 75,025。如果您需要計算另一個 Fibonacci 數，可以透過更改輸入值來使用腳本中提供的 fibonacci 函式。

## 1.2.4 比較自訂 GPT、微調與檢索增強生成

自訂 GPT 主要改變語言模型的角色，即其基本任務。通常，角色描述是「您是一個有幫助的精靈」或類似的內容。我們將在第 10 章中更詳細地討論角色、提示和回應的概念。改變的角色影響語言模型的答案，但不改變其核心。

透過**微調**，您對給定語言模型的功能有更大的影響力。在最簡單的情況下，您給語言模型一個問題（提示）列表和您認為最好的答案。語言模型可以以此為基礎。然而，ChatGPT 目前不支援自訂 GPT 的微調。

在語言模型訓練時尚不可用的附加知識的「真正」整合，由**檢索增強生成（RAG）**提供（另見第 12 章）。如果您上傳附加知識檔案，此過程也用於自訂 GPT。

遺憾的是，不可能簡單地為特定應用重新訓練語言模型或考慮附加資訊。這種方法消耗極大的計算時間和金錢，因此只有大型（AI）公司才能實行。

> **自訂 GPT 的限制**
> 整合幾個包含附加資訊的 PDF 檔案並更改語言模型的系統指令（角色）不足以在幾分鐘內建立一個運作良好的公司特定或應用特定的機器人。此類嘗試在過去總是慘敗。
