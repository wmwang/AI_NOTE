# 5 軟體測試

測試工具和框架已經在軟體開發中站穩腳跟，成為品質保證的標準。測試涵蓋的範圍遠不止傳統的單元測試。它從為開發和手動測試建立有意義的測試資料開始，貫穿各個程式碼區塊的驗證，一直到端對端測試。

每種測試類型都有其獨特的挑戰，需要特定的方法和工具。你可以在每個領域中使用各種 AI 工具，讓它們為你完成大量工作。

然而，軟體測試有一個共同點：它們經常被重複執行，因為一組基本的測試模式會被反覆使用。這就是為什麼撰寫測試不是軟體開發人員最喜歡的工作，而且常常被認為佔用了不必要的時間，並佔用了原本可用於功能開發的資源。這正是 AI 工具可以發揮決定性作用的地方——有效地接管重複性任務，大幅減少測試工作量。

AI 工具的使用不限於特定的測試層級，而是可以在任何地方作為輔助工具。透過使用正確表述的提示和適當的上下文，AI 工具可以自動涵蓋許多測試案例。這種支援不僅減少了所需時間，還能識別出開發人員可能沒有想到的測試案例。此外，AI 可以提供「第二意見」或其他視角，使測試整體上變得更好、更穩健。

在測試驅動和伴隨測試的開發中，AI 可以對測試自動化做出決定性的貢獻，並確保全面的測試覆蓋率。然而，儘管有許多優點，也存在一些挑戰。AI 工具並不總是正確的，有時會透過幻覺產生錯誤資訊。因此，始終檢查 AI 產生的測試，至少對它們進行合理性檢查，這一點很重要。畢竟，這關係到軟體的安全，你不能在這裡冒任何風險。

未來，AI 將在軟體測試中扮演更重要的角色。它可以為你分擔大量工作，讓你能夠專注於實際的開發工作。儘管如此，AI 仍然是一種工具——就像任何其他工具一樣——必須正確使用和監控。人類專業知識與 AI 效率的結合，有望將軟體測試提升到一個新的水平，並持續改善軟體產品的品質。

## 5.1 產生測試資料

在開發過程中，你總是需要處理資料，無論是註冊新使用者的表單、客戶資料的列表顯示，還是國際銀行帳號（IBAN）的驗證函式。理想情況下，你應該在開發過程中就使用真實的資料。這正是問題開始的地方。真實的資料應該是什麼樣子，資料又從哪裡來？這種情況的一個具體例子是客戶列表的實作，這在電子商務領域經常使用。客戶列表將被實作為一個使用 TypeScript 建立的簡單 HTML 表格。在使用 AI 時，型別定義總是很有用的，用來指定 AI 應該為其產生資料的結構。你可以透過指定大致的結構並讓 AI 處理細節來獲得產生此類型別的幫助。

**提示：** 請建立一個 TypeScript 別名 Customer，包含以下欄位：id、first name、last name、email、registration date、status 和 last order date。

當然！以下是如何建立一個名為 Customer 的 TypeScript 型別別名，包含指定的欄位：

```typescript
type Customer = {
  id: string;
  firstName: string;
  lastName: string;
  email: string;
  registrationDate: Date;

  // 假設 status 是這些值之一
  status: 'active' | 'inactive' | 'suspended'; 
  // 假設 lastOrderDate 如果沒有訂單則為 null
  lastOrderDate: Date | null; 
}; 
```

以下是每個欄位的簡要說明：

[欄位說明]

產生的程式碼完全遵循提示的指示。對於不明確的欄位，例如 status 和 lastOrderDate，已插入帶有相應假設的註解。status 是一個包含三個值的聯合型別，而 lastOrderDate 可以使用日期值或 null。你可以進一步限制提示並指定特定型別，但在這種情況下，你會失去 AI 工具的優勢，因為你幾乎是自己實作結構。這種提示的替代方案可以是使用 GitHub Copilot 等工具的自動完成功能。當你開始實作客戶型別時，Copilot 會建議欄位和資料型別供你採用。

實作由兩部分組成：一個負責顯示的 HTML 檔案，以及一個動態建立資料的 TypeScript 檔案。HTML 檔案的程式碼如下所示：

```html
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <title>Document</title>
    <script type="module" src="list.js"></script>
  </head>
  <body>
    <table>
      <thead>
        <tr>
          <th>Id</th>
          <th>First name</th>
          <th>Last name</th>

          <th>Email</th>
          <th>Registration date</th>
          <th>status</th>
          <th>Last order date</th>
        </tr>
      </thead>
      <tbody></tbody>
    </table>
  </body>
</html>  
```

HTML 結構由文件的一般框架、載入 JavaScript 檔案以建立表格的標頭區段，以及包含表格基本結構和用於插入資料的空 `<tbody>` 元素的主體組成。

```typescript
import type { Customer } from './customer';
const customers: Customer[] = [];
document.addEventListener('DOMContentLoaded', () => {
  drawTable(customers);
});
function drawTable(customers: Customer[]) {
  const customerTable = customers.map((customer) => {
    return `
      <tr>
        <td>${customer.id}</td>
        <td>${customer.firstName}</td>
        <td>${customer.lastName}</td>
        <td>${customer.email}</td>
        <td>
          ${customer.registrationDate.toISOString()
             .split('T')[0]}
        </td>
        <td>${customer.status}</td>
        <td>
          
${customer.lastOrderDate.toISOString().split('T')[0]}
          </td>
      </tr>

    `;
  });
  document.querySelector('tbody')!.innerHTML = 
    customerTable.join('');
}  
```

此時還缺少測試資料。瀏覽器通常會從後端非同步載入這些資料。但為了簡單起見，我們假設資料已經以 customers 常數的形式可用。在這裡，你可以開始自己編造資料，但也可以要求你選擇的大型語言模型（LLM）為你產生測試資料。

**提示：** 請建立一個包含 10 個元素的真實測試資料陣列。資料應代表美國客戶。確保 lastOrderDate 晚於 registrationDate。使用以下 TypeScript 型別：

```typescript
type Customer = {
  id: string;
  firstName: string;
  lastName: string;
  email: string;
  registrationDate: Date;
  status: 'active' | 'inactive' | 'suspended'; 
  lastOrderDate: Date | null; 
}; 
```

當然！以下是一個包含 10 個元素的美國客戶真實測試資料陣列：

```typescript
const customers: Customer[] = [
  {
    id: '1',
    firstName: 'Anna',
    lastName: 'Mueller',
    email: 'anna.mueller@example.de',
    registrationDate: new Date('2022-01-15'),
    status: 'active',

    lastOrderDate: new Date('2023-04-18'),
  },
  {
    id: '2',
    firstName: 'Luca',
    lastName: 'Rossi',
    email: 'luca.rossi@example.it',
    registrationDate: new Date('2021-11-20'),
    status: 'active',
    lastOrderDate: new Date('2023-02-25'),
  },
  ... 
```

你可以直接在應用程式中使用 ChatGPT 的回應。產生的資料對應於相當真實的資料；例如，名字和姓氏已正確包含在電子郵件地址中。日期值也是正確的，並遵循最後訂單日期應在註冊日期之後的規則。

這個例子展示了如何減少產生測試資料的一些工作。然而，這裡也需要一些謹慎。你給 AI 工具的自由越多，它能解釋的空間就越大。註冊日期在最後訂單日期之前，這純粹是運氣好。在沒有進一步規範的情況下，AI 工具也可能反轉日期值。這對列表來說不是重大問題，但對於進一步處理資料時可能很快就會成為問題，特別是在涉及驗證例程時。

在這種情況下，你還應該考慮產生的資料記錄數量。在像 10 這樣的低數量下，ChatGPT 能可靠且無誤地工作。你要求的資料記錄越多，重複項目混入的風險就越大（你可以透過提示排除），或者工具在一定數量的資料記錄後拒絕工作，要麼在資料結構中間中止，要麼輸出類似「後面可能還有 40 筆資料記錄」的內容。在這種情況下，你可以讓資料記錄以較小的群組產生，然後自行合併。
