# 11.4 使用 Aider

本節提供了使用 Aider 進行各種程式設計任務的實際範例。我們將涵蓋常見的使用案例，並示範如何透過互動式工作階段有效地使用 Aider。

## 入門

### 啟動 Aider

導航至您的 Git 儲存庫並啟動 Aider：

```bash
cd my-project
aider
```

這將啟動一個互動式工作階段，您可以在其中與 AI 進行對話來修改您的程式碼庫。

### 新增檔案

首先，將您想要處理的檔案新增到聊天中：

```
/add src/main.py src/utils.py
```

## 實際範例

### 範例 1：新增功能

**提示：**

```
Add a new function to main.py called calculate_statistics that:
1. Takes a list of numbers as input
2. Returns a dictionary with mean, median, and standard deviation
3. Uses the statistics module
4. Includes proper error handling for empty lists
5. Add type hints and a docstring
```

Aider 將會：
1. 讀取現有的 `main.py` 檔案
2. 理解程式碼風格和約定
3. 在適當的位置新增函式
4. 匯入必要的模組
5. 使用描述性訊息提交變更

### 範例 2：修復錯誤

**提示：**

```
There's a bug in the process_data function in utils.py. When the input
list contains None values, it raises a TypeError. Please fix this by:
1. Adding a check to skip None values
2. Adding a log warning when None values are encountered
3. Add a test case for this scenario
```

### 範例 3：重構

**提示：**

```
Refactor the DatabaseManager class in db.py:
1. Extract the connection logic into a separate method
2. Add context manager support using __enter__ and __exit__
3. Replace the raw SQL queries with parameterized queries
4. Add connection pooling using the connection_pool library
5. Update all existing code that uses this class
```

### 範例 4：新增測試

**提示：**

```
Create comprehensive tests for the UserService class in services/user.py.
Add the tests to tests/test_user_service.py. Include:
1. Unit tests for all public methods
2. Mock the database interactions
3. Test both success and error cases
4. Test edge cases like empty inputs and boundary values
5. Use pytest fixtures for common setup
```

### 範例 5：文件撰寫

**提示：**

```
Add comprehensive docstrings to all functions and classes in the src/
directory. Follow the Google docstring style and include:
1. A description of what each function does
2. Args section with type hints
3. Returns section
4. Raises section for exceptions
5. Example usage where appropriate
```

## 進階使用模式

### 處理大型程式碼庫

在處理大型儲存庫時，策略性地使用 Aider：

**提示：**

```
/map
```

這會顯示程式碼庫地圖，幫助您了解結構。然後：

**提示：**

```
/add src/core/engine.py src/core/models.py src/core/utils.py
```

只新增相關檔案，以將上下文保持在可管理的程度。

### 非互動式使用

Aider 也可用於腳本：

```bash
# 單一命令模式
aider --message "Add type hints to all functions in utils.py" utils.py

# 批次處理多個檔案
for file in src/*.py; do
  aider --message "Add docstrings to all functions" "$file"
done
```

### 建立自訂工作流程

您可以為常見任務建立腳本：

```bash
#!/bin/bash
# aider-refactor.sh - Automated refactoring script

FILES=$@
PROMPT="Refactor the code to follow PEP 8 guidelines, add type hints, 
and improve error handling. Maintain all existing functionality."

aider --message "$PROMPT" $FILES
```

## 有效使用的技巧

### 1. 要具體且明確

良好的提示是清晰且詳細的：

**提示：**

```
In the file api/handlers.py, modify the handle_request function to:
1. Add input validation using the pydantic library
2. Return a 400 status code for invalid input
3. Log all validation errors using the logging module at WARNING level
4. Keep the existing response format for valid requests
```

### 2. 逐步建立

對於複雜的任務，將其分解為更小的步驟：

**提示：**

```
First, let's add the basic data model in models.py.
Then we'll add the CRUD operations in crud.py.
Finally, we'll wire up the API endpoints in routes.py.
```

### 3. 使用 /undo 自由實驗

```
# 如果變更不是您想要的
/undo

# 嘗試不同的方法
Please try again, but this time use a different algorithm for sorting
```

### 4. 提供背景資訊

**提示：**

```
I'm building a REST API using FastAPI. The project follows the 
repository pattern. This function is part of the service layer 
and should not directly access the database.
```

### 5. 與程式碼審查整合

```bash
# 進行 AI 輔助的變更
aider --message "Implement the user authentication feature"

# 在合併前檢閱所有變更
git diff main

# 如果需要，進行選擇性提交
git rebase -i
```

## 常見陷阱

### 1. 檔案過載

新增過多檔案可能會使上下文窗口不堪負荷：
- 只新增必要的相關檔案
- 使用 `/drop` 移除不再需要的檔案
- 讓 Aider 管理上下文

### 2. 模糊的指示

模糊的指示可能會導致意外結果：
- 具體說明要修改什麼以及如何修改
- 提供檔案路徑和函式名稱
- 指定約束條件和偏好

### 3. 不驗證結果

始終驗證 AI 生成的程式碼：
- 執行現有的測試套件
- 手動檢閱變更
- 在合併前在本地測試

### 4. 忽略 Git 歷史

利用 Aider 的自動提交功能：
- 檢閱提交歷史
- 如有必要，使用 `/undo` 復原不良的變更
- 在重大里程碑前建立分支

## 效能考量

### Token 使用量

- **新增更少的檔案**：每個新增的檔案都會消耗 token
- **使用 `.aiderignore`**：從上下文中排除不相關的檔案
- **清除歷史**：使用 `/clear` 重置長工作階段的對話

### 模型選擇

- **Claude 3.5 Sonnet**：整體表現最佳，平衡了速度和品質
- **GPT-4o**：表現優異的替代方案
- **較小的模型**：適用於簡單任務，但可能難以應對複雜的重構

## 總結

Aider 與 Git 的深度整合及其互動式特性使其成為 AI 輔助程式設計的強大工具。透過遵循最佳實踐——提供清晰的指示、驗證結果、適當管理上下文以及利用 Git 整合——開發者可以顯著提升其使用 AI 輔助的生產力。這些實際範例證明，Aider 在廣泛的程式設計任務中表現出色，從簡單的錯誤修復到複雜的重構和測試生成。
