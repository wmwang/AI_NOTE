# 5.2 AI 支援的測試自動化

當你為應用程式開發新功能時，總會到達需要檢查功能是否正常的時刻。手動檢查既耗時又容易出錯，特別是考慮到你還必須至少測試應用程式的其他部分，因為整合時可能在其他地方引入了錯誤。檢查應用程式功能這類任務的自動化潛力是巨大的。這正是測試自動化這門學科存在的原因。你可以根據測試金字塔，在不同的層級上測試你的應用程式，測試金字塔將不同類型的測試分類在不同的層級：

**單元測試**
金字塔的基礎由單元測試組成。它們獨立檢查應用程式中最小的可測試單元。這些單元通常是函式或方法。單元測試執行速度快、易於撰寫，而且數量通常很多。

**整合測試**
這些測試構成測試金字塔的中間層。它們測試單個或多個模組或元件。整合測試比單元測試更複雜、更慢，但它們也提供更高的安全保障，因為它們測試的是應用程式中多個元件的互動。通常，你的應用程式的整合測試數量遠少於單元測試。

**端對端測試**
端對端測試（有時稱為 E2E 測試）構成金字塔的頂端。它們跨越整個應用程式測試功能，從前端到資料管理。這些測試建立起來最複雜，執行起來最慢，但它們提供最高的安全保障。與其他兩個層級相比，端對端測試通常佔比較小。

AI 工具可以在測試金字塔的所有層級上為你提供支援，減輕你在產生測試方面的一些工作。在本章的其餘部分，你將以 TypeScript 為例，了解測試金字塔的不同層級及其特徵。然而，測試在其他程式語言和環境中的運作方式類似，因此你可以轉移你的知識。

## 5.2.1 建立AI支援的單元測試

在 JavaScript 和 TypeScript 中，有多個用於單元測試的測試框架。最受歡迎的之一是 Vitest，一個來自 Vue.js 生態系的測試框架，但它與框架無關，甚至與平台無關。這就是為什麼你可以在客戶端的瀏覽器中和使用 Node.js 的伺服器端使用這個測試框架。Vitest 的一個主要優點是，在簡單的情況下它完全不需要任何設定。它會根據檔案名稱自動找到測試並執行它們。

### AI支援的單元測試

AI 支援產生單元測試的第一步是一個簡單的 TypeScript 函式 add，它將兩個整數相加並返回結果。這個函式的程式碼如下所示：

```typescript
export default function add(a: number, b: number): 
number {
  if (!Number.isSafeInteger(a) || 
!Number.isSafeInteger(b)) {
    throw new Error("Can only add integers");
  }
  return a + b;
}  
```

你可以讓 ChatGPT 為這個函式產生測試：

**提示：** 請為以下函式撰寫一個 Vitest 單元測試：
[函式的程式碼]

僅建立測試；環境已設定好，所有東西都已安裝。

```typescript
import { describe, it, expect } from 'vitest';
import add from './add'; // 調整函式 add 的路徑

describe('add', () => {
  it('should add two integers correctly', () => {
    expect(add(2, 3)).toBe(5);
    expect(add(-1, 1)).toBe(0);
    expect(add(0, 0)).toBe(0);
  });
  it(
    'should throw an error if either argument is not a ' +
    'safe integer', () => {
    expect(() => add(2.5, 3))
      .toThrow('Can only add integers');
    expect(() => add(2, 3.5))
      .toThrow('Can only add integers');
    expect(() => add(Number.MAX_SAFE_INTEGER + 1, 3))
      .toThrow('Can only add integers');
    expect(() => add(2, Number.MAX_SAFE_INTEGER + 1))
      .toThrow('Can only add integers');
  });
  it(
    'should throw an error if any argument is not a number', 
    () => {
    expect(() => add('2', 3))
      .toThrow('Can only add integers');
    expect(() => add(2, '3'))
      .toThrow('Can only add integers');
    expect(() => add(null, 3))
      .toThrow('Can only add integers');
    expect(() => add(2, undefined))
      .toThrow('Can only add integers');
  });
}); 
```

[情境說明]

如果你在提示中指定了完整的原始碼（如本例所示），AI 工具可以涵蓋功能的所有方面。函式的簽章對此有所貢獻，因為你在這裡精確指定了期望的資料型別和返回的型別。函式內的錯誤處理指出了額外的邊界和例外情況，這些也必須被測試涵蓋。像 ChatGPT 這樣的 AI 工具在回答此類任務時喜歡更詳細一些，不僅產生所要求的測試，還提供設定和執行的提示。你可以透過指定你只對測試感興趣，而不需要環境或安裝相關資訊來避免這種情況。

ChatGPT 產生的結果令人印象深刻。產生的程式碼不僅包含一個簡單的測試（例如將值 2 和 3 相加並檢查是否返回正確的結果），還包含一系列的測試。這裡正面的是測試使用 `describe` 進行了分組，並且群組以函式名稱作為標籤。在這個群組中有三個測試：

**成功案例**
測試期望函式能夠將兩個整數相加。第一個測試確保兩個正整數可以相加，第二個測試加入一個負數，第三個測試確保 0 和 0 也可以相加。

**錯誤案例 1**
add 函式的預先實作會檢查兩個數字是否為安全整數，即位於 TypeScript 或 JavaScript 引擎能正確計算的範圍內的整數。第二個測試檢查使用浮點數和過大的數字（`Number.MAX_SAFE_INTEGER + 1`）進行計算是不可能的，並會導致例外。

**錯誤案例 2**
第三個測試更進一步，確保使用無效的資料型別（如字串、null 或 undefined）呼叫 add 函式會導致錯誤。

在成功案例中，ChatGPT 使用 `toBe` 匹配器來檢查結果是否符合期望值。這個檢查函式的選擇很合適，因為它在內部檢查嚴格相等性，不允許型別轉換。這個函式只有在涉及引用型別（如物件）時才會造成困難，但這裡不是這種情況。錯誤案例正確地使用 `toThrow` 匹配器覆蓋測試，你可以用它來檢查函式是否拋出錯誤。

現在讓我們來看看產生的測試的弱點。一個主要問題是 TypeScript 編譯器在處理程式碼時會報告測試中的一些錯誤。第三個測試使用不同的型別呼叫函式，這在 TypeScript 中是不可能的，因為在呼叫時型別與函式的簽章不相容。提示中沒有提到 TypeScript 和型別安全程式碼的任何內容，但在分析函式時這應該是很明顯的。

你可以透過要求 ChatGPT 產生有效的 TypeScript 程式碼來解決這個問題。為此，你可以修改原始提示或按如下所示指定請求：

**提示：** 確保測試以有效的 TypeScript 撰寫。保留任何引數不是數字的測試，但修正它。

當然，以下是以有效的 TypeScript 撰寫的修正測試套件：

```typescript
(existing code ...)
  it(
    'should throw an error if any argument is not a number', 
    () => {
    expect(() => add('2' as any, 3))
      .toThrow('Can only add integers');
    expect(() => add(2, '3' as any))
      .toThrow('Can only add integers');
    expect(() => add(null as any, 3))
      .toThrow('Can only add integers');
    expect(() => add(2, undefined as any))
      .toThrow('Can only add integers');

  });
}); 
```

在這個版本中：每個無效的引數都被明確轉換為 `any`，以確保測試案例按預期執行，同時保持有效引數的 TypeScript 型別檢查。

為了保險起見，你應該在提示中指定第三個測試在任何情況下都應保留。在純 TypeScript 環境中這是不必要的，因為你不能用錯誤的型別執行函式。然而，由於 TypeScript 僅在編譯時使用，而不是在執行時使用，根據環境可能仍會使用錯誤的資料型別。原始的兩個測試保持不變，因為它們已經以有效的 TypeScript 撰寫。為了避免第三個測試中的錯誤，ChatGPT 使用 TypeScript 的型別轉換，將無效的字串資料型別轉換為數字。雖然這種改編不夠優雅，但它滿足了 TypeScript 編譯器。

另一個批評點是測試不只有一個斷言，而是總是有多個。雖然測試在主題上總是相互關聯，但通常有一個要求是每個測試應該只有一個斷言。原因是測試應該只因一個原因而失敗，以便於排除故障。在這種情況下，由你決定是偏好更少、主題分組的測試，還是許多、細粒度的測試。你也可以透過擴展提示並要求 ChatGPT 為每個期望建立單獨的測試來解決這個問題：

**提示：** 請將每個期望放在自己的測試中。並透過將測試放在各自的子 describe 區塊中來保持分組。

當然，以下是修訂後的測試套件，每個期望都在自己的測試中，並使用子 describe 區塊進行分組：

```typescript
import { describe, it, expect } from 'vitest';
import add from './add'; // 調整函式 add 的路徑

describe('add', () => {

  describe('valid inputs', () => {
    it('should add 2 and 3 correctly', () => {
      expect(add(2, 3)).toBe(5);
    });
    ... more tests ...
  });
  describe('invalid safe integer inputs', () => {
    ... more tests ...
  });
  describe('invalid non-number inputs', () => {
    ... more tests...
  });
}); 
```

[程式碼說明]

如果你只提示第一句話，ChatGPT 會將原來的 3 個測試變成總共 11 個測試。提示的第二句話確保分組得以保持，你不會那麼快失去概覽。

總的來說，你還是需要處理測試的主題，雖然像 ChatGPT、Copilot 等工具可以為你做很多工作。你應該熟悉測試的結構和組織，並發展測試模式和測試架構的基本知識。你的提示表述得越精確，結果就越有可能符合你的期望，測試的程式碼也能長期保持可讀性和可維護性。

然而，AI 的可能性不僅止於測試的表述；這些工具也可以在測試驅動開發中為你提供支援。
