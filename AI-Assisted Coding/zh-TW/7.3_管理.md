# 7.3 管理

AI 工具也能協助管理 DBMS。無論是較為簡單的任務，例如建立新資料庫或執行備份，還是需要從頭開始安裝和設定 DBMS，ChatGPT 和其他工具都能提供有用的建議。

根據我們的經驗，在簡單的情況下，這類指示的品質通常不錯。但另一方面，如果您需要真正的專業知識，而即使進行漫長的網路搜尋也只能找到少數幾條可能相互矛盾的建議時，您也應該對 AI 的回答保持高度謹慎。第 7.3.4 節提供了一個相關範例。

## 7.3.1 PostgreSQL：使用者管理

建立新資料庫的程序始終相同：建立資料庫、建立使用者，以及授予使用者相應的權限。在這方面，AI 工具可以幫您省下大量打字的時間。

**提示：** PostgreSQL：我想要建立一個新資料庫並設定三個使用者：一個具有完整管理權限、一個用於一般資料庫操作（基本上就是 SELECT、INSERT、UPDATE、DELETE），以及一個用於自動化備份系統。你能幫我提供所需的 SQL 命令嗎？

當然！以下是在 PostgreSQL 中建立新資料庫並設定三個具有指定權限的使用者所需的 SQL 命令[更多細節]

以超級使用者（通常是 postgres）身分連線到 PostgreSQL。

```bash
psql -U postgres
```

建立資料庫：

```sql
CREATE DATABASE your_database_name;
```

建立使用者：

```sql
CREATE USER admin_user   WITH PASSWORD 'admin_password';
CREATE USER regular_user WITH PASSWORD 'regular_password';
CREATE USER backup_user  WITH PASSWORD 'backup_password';
```

授予權限：

```sql
-- 管理者使用者
GRANT ALL PRIVILEGES ON DATABASE your_database_name
  TO admin_user;
\c your_database_name
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public
  TO admin_user;
...
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT ALL PRIVILEGES ON FUNCTIONS TO admin_user;

-- 一般使用者
GRANT CONNECT ON DATABASE your_database_name
  TO regular_user;
...

-- 備份使用者
GRANT CONNECT ON DATABASE your_database_name
  TO backup_user;
...
```

請將 your_database_name、admin_user、admin_password、regular_user、regular_password、backup_user 和 backup_password 替換為您想要的名稱和密碼。

這個回答幾乎沒有什麼好批評的地方（此處因篇幅原因略作刪節）。這個問題的難度本來就不高。AI 工具在這裡的主要目的是節省時間。如果您在輸入時就指定資料庫和使用者帳號的所需名稱，效率會更高。

**提示：** 完美。請重複 SQL 命令（不加說明），並使用以下名稱：
資料庫：bikerental
管理者使用者：bike_admin
一般使用者：bike_user
備份使用者：bike_backup

## 7.3.2 SQL Server：自動化備份

典型的管理工作是執行備份。根據 DBMS 和資料庫大小的不同，有無數種變體。在這個範例中，我們想要設定一個增量備份流程。

**提示：** SQL Server：我想要為我的會計資料庫設定一個自動化備份系統：每週日晚上進行完整備份，一週中的其他日子進行增量備份。我希望系統在覆寫之前保留最近兩週的備份。備份應儲存在本機目錄中，然後從那裡同步到外部伺服器（此處不相關）。

由於 ChatGPT 的回答在本書中會佔用三頁半的篇幅，因此我們在此僅引用重點。

為您的 SQL Server 資料庫設定符合指定需求的自動化備份系統，您可以使用 SQL Server Agent 來排程備份工作。以下是逐步指南：

建立完整備份的指令碼，儲存為 FullBackup.sql：

```sql
DECLARE @BackupFileName NVARCHAR(500)
DECLARE @BackupPath NVARCHAR(500) = 'C:\SQLBackups\'
DECLARE @DatabaseName NVARCHAR(500) = 
'YourDatabaseName'

-- 產生帶有時間戳記的檔案名稱
SET @BackupFileName = @BackupPath + @DatabaseName + 
  '_Full_' + CONVERT(NVARCHAR(500), GETDATE(), 112) + 
  '.bak'

-- 執行完整備份
BACKUP DATABASE @DatabaseName 
TO DISK = @BackupFileName 
WITH INIT
```

建立差異備份的指令碼，儲存為 DifferentialBackup.sql：

```sql
...
-- 執行差異備份
BACKUP DATABASE @DatabaseName 
TO DISK = @BackupFileName 
WITH DIFFERENTIAL, INIT
```

建立清理舊備份的指令碼，儲存為 CleanupBackups.sql：

```sql
DECLARE @BackupPath NVARCHAR(500) = 'C:\SQLBackups\'
DECLARE @DaysToKeep INT = 14

-- 刪除超過 @DaysToKeep 天的備份檔案
EXECUTE master.dbo.xp_delete_file 0, @BackupPath, 
        'bak', @DaysToKeep, 1
```

建立 SQL Server Agent 工作：開啟 SQL Server Management Studio (SSMS)，連線到您的 SQL Server 執行個體，展開 SQL Server Agent 節點，右鍵點選 Jobs，然後選擇 New Job...

這個解決方案聽起來很合理，您可以實作這個備份系統。您可能會想查看原始文件：

**提示：** 可以提供 BACKUP 和 RESTORE DATABASE 的手冊連結嗎？

當然！以下是 SQL Server 中 BACKUP 和 RESTORE DATABASE 命令的 Microsoft 官方文件連結：

https://docs.microsoft.com/en-us/sql/t-sql/statements/backup-transact-sql?view=sql-server-ver15

https://docs.microsoft.com/en-us/sql/t-sql/statements/restore-statements-transact-sql?view=sql-server-ver15

這些連結指向 SQL Server 15（2019）。在我們測試此範例時，SQL Server 16（2022）是當時的最新版本。不過，備份和還原命令預期不會有重大的語法變更；因此，這些連結是沒有問題的。

設定備份系統時最重要的規則是：也要測試復原！

**提示：** 假設我的主伺服器在週三當機了。我從異地備份來源取回備份。如何復原我的資料庫？

要從備份中復原資料庫（當主伺服器當機時），請按照以下步驟操作[更多細節]。以下是您可能需要的所有命令摘要：

```sql
-- 還原完整備份
RESTORE DATABASE YourDatabaseName
FROM DISK = 
'C:\SQLBackups\Accounting_Full_20240602.bak'
WITH NORECOVERY;

-- 還原 2024 年 6 月 3 日的差異備份
RESTORE DATABASE Accounting
FROM DISK = 
'C:\SQLBackups\Accounting_Diff_20240603.bak'
WITH NORECOVERY;

-- 還原 2024 年 6 月 4 日的差異備份
RESTORE DATABASE Accounting
FROM DISK = 
'C:\SQLBackups\Accounting_Diff_20240604.bak'
WITH NORECOVERY;

-- 還原 2024 年 6 月 5 日的差異備份
RESTORE DATABASE Accounting
FROM DISK = 
'C:\SQLBackups\Accounting_Diff_20240605.bak'
WITH RECOVERY;
```

## 7.3.3 確定資料庫的結構描述

若要讓 ChatGPT 協助您解答有關現有資料庫的問題，您必須先讓它瞭解資料庫的結構描述。最簡單的方法是準備一個包含所有必要 CREATE TABLE 命令的 SQL 檔案。在 ChatGPT 中，您可以使用拖放方式上傳檔案。在 VS Code 或其他編輯器中，您可以載入檔案並在問題中參照該檔案。

但要從哪裡取得這樣的檔案呢？幾乎所有的 DBMS 都有命令可以產生對應的文字格式結構描述檔案。最簡單的方法就是詢問 ChatGPT 關於您正在使用的程式。

**提示：** MySQL：如何將我的資料庫結構描述儲存為 SQL 檔案？

要將 MySQL 資料庫結構描述儲存為 SQL 檔案，您可以使用 mysqldump 命令並加上 --no-data 選項。這將建立一個僅包含資料庫結構（資料表、索引等）而不包含任何資料的 SQL 檔案。

```bash
mysqldump -u [username] -p --no-data [dbname] > schema.sql
```

以下是該命令的詳細說明[更多細節]

**提示：** SQL Server 有類似的命令嗎？

是的，在 SQL Server 中，您可以使用 SQL Server Management Studio (SSMS) 或指令碼來產生僅含結構描述的備份。以下是兩種方法的操作方式：

### 使用 SQL Server Management Studio (SSMS)

1. 開啟 SSMS 並連線到您的 SQL Server 執行個體。
2. 右鍵點選您想要產生指令碼的資料庫。
3. 選擇 Tasks • Generate Scripts。
4. 在精靈中，點擊 Next 跳過簡介畫面。
5. 選擇您想要產生指令碼的特定資料庫物件（或選擇整個資料庫）。點擊 Next。
6. 在 Set Scripting Options 頁面上，點擊 Advanced。

[更多細節]

### 使用 SQLCMD 指令碼

或者，您可以使用 sqlcmd 工具程式搭配自訂查詢來產生僅含結構描述的指令碼。以下是範例指令碼：

```bash
sqlcmd -S [server_name] -U [username] -P [password] \
       -d [database_name] \
       -Q "EXEC sp_generate_inserts @schema_only = 1" \
       -o "[output_file].sql"
```

### 使用 PowerShell 和 SQL Server Management Objects (SMO)

您也可以使用 PowerShell 指令碼來產生結構描述。[更多細節]

在 MySQL 中，mysqldump 命令是最簡單的方法。對於 SQL Server，ChatGPT 提供了三種解決方案供選擇，第一種是關於 SQL Server Management Studio (SSMS) 的。然而，這類指示有一個缺點：通常只有在使用特定版本的使用者介面時，才能以完全相同的方式重現。

**提示：** 您的回答適用於哪個版本的 SSMS？

使用 SQL Server Management Studio (SSMS) 產生僅含結構描述指令碼的步驟適用於 SSMS 2012 及後續版本，包括最新的 SSMS 19。在這些版本中，SSMS 針對這些指令碼工作具有一致的使用者介面，因此這些步驟應該是適用的。

然而，當我們在 2024 年 5 月執行上述提示時，版本 20.1 已經可用，而 ChatGPT 的訓練資料還沒有更新到那麼新。但只要運氣好，這些指示在最新版本中應該也能以相同或類似的方式運作。

## 7.3.4 MariaDB 調校

幾個月前，我們設定了一台 LAMP 伺服器（Linux + Apache + MariaDB + PHP）。MariaDB 對在那裡執行的 Web 應用程式整體效能有重大影響。因此，我們在資料庫伺服器的調校上投入了大量精力。ChatGPT 能幫我們達成目標嗎？

**提示：** 我剛安裝了一台執行 Ubuntu 24.04 的新 LAMP 伺服器。伺服器有 64 GB 的 RAM。它將作為網頁伺服器（Apache）、郵件伺服器（Postfix）和資料庫伺服器（MariaDB 10.11）。我需要協助調校 MariaDB。

MariaDB 最多可使用 50% 的 RAM。有許多資料庫（約 200 個），每個約有 20 個資料表。讀取效能至關重要。（SELECT 與 INSERT/UPDATE/DELETE 的比例約為 100:1。）

請給我五個簡潔的 MariaDB 或 Linux 設定建議以達到良好的效能。

為了優化您的 LAMP 伺服器設定中的 MariaDB，請遵循以下五個建議：

**1. 調整 InnoDB 緩衝池大小**[更多細節]

```ini
[mysqld]
innodb_buffer_pool_size = 32G
```

**2. 優化查詢快取**[更多細節]

```ini
[mysqld]
query_cache_type = 1
query_cache_size = 1G
query_cache_limit = 1M
```

**3. 增加 InnoDB 緩衝池實例數量**[更多細節]

```ini
[mysqld]
innodb_buffer_pool_instances = 16
```

**4. 調整 InnoDB 日誌檔案大小和清除方法**[更多細節]

```ini
[mysqld]
innodb_log_file_size = 2G
innodb_flush_method = O_DIRECT
```

**5. 優化檔案系統和 I/O 排程器：** 使用帶有 noatime 選項的 XFS 或 ext4 檔案系統以減少磁碟 I/O，並將 I/O 排程器設為 deadline 以獲得更好的效能。[更多細節]

```bash
sudo mount -o remount,noatime /
sudo echo deadline > /sys/block/sda/queue/scheduler
```

我們對這些建議並不完全信服。只有第一項措施是真正沒有爭議的。

我們也啟用了查詢快取，這與 ChatGPT 的建議無關。然而，近年來，主流觀點認為查詢快取僅在特殊情況下對整體效能有正面影響。查看快取統計資料後發現，我們的伺服器確實是這些例外之一。無論如何，在目前的 MySQL 版本中，查詢快取已不再可用；在 MariaDB 中，它預設是停用的。ChatGPT 對查詢快取的建議可能源自訓練語言模型時使用的無數舊網站。這是一個相當常見的問題；ChatGPT 傾向於偏好過時的措施，因為這些措施在訓練資料中出現的頻率高於現代指南。

措施 3、4 和 5 不太可能帶來任何可測量的效能差異。如果您真的想使用 deadline 排程器，請注意在目前的核心版本中（包括提示中提到的 Ubuntu 24.04），它叫做 mq-deadline。

然而，在建議清單中，我們缺少了另一項在提示描述的設定中其實很明顯的措施：增加同時開啟檔案數量的上限（在 /etc/security/limits.conf 中加入 `mysql soft nofile 65535` 和 `mysql hard nofile 65535`）。

簡而言之，作為調校輔助工具，ChatGPT 可以提供一些思路，但其回答的品質遠不如編寫 SQL 命令時那麼好。
