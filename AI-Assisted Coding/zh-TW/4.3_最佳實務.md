# 4.3 最佳實務

在重構過程中，您可能會因為不斷發現程式碼中想要改進的新地方，而失去整體視角並偏離原本的計畫。以下是一些幫助您專注於要點的最佳實務。

## 4.3.1 僅可變更內部結構

在重構時，您應該專注於只修改函式的內部結構，而不更動其參數列表或回傳值。如果您要求 AI 工具進行重構，它通常會遵守這項建議。為了保險起見，您也可以在提示中明確指定不應更改方法的簽名。

**提示：** Don't change the signature of the method; only focus on the inner structure.

這可確保重構不會影響應用程式的其餘部分，而僅在個別獨立的地方生效。

## 4.3.2 小步驟方法

即使您只專注於方法或類別的內部結構，根據該結構在應用程式中扮演的角色，重構的規模仍可能變得相當大。為了防止重構失控，您應該嘗試以盡可能小的步驟來進行。

如果您使用 GitHub Copilot 或 ChatGPT 等 AI 工具，應該先分析您的程式碼，看看類別或函式有哪些重構潛力。然後，您可以逐步處理各個要點。這種做法的優點是您可以將每個步驟提交到版本控制系統。因此，如果發生錯誤，您距離正常運作的實作最多只有一個步驟的距離，並且可以使用修改過的提示重複上一步驟。

如果重構步驟太大，一次重構太多方面，發生錯誤和不良行為的風險將顯著增加。

## 4.3.3 測試驅動重構

在開始重構流程之前，您需要確保打算改善的類別或函式已經通過自動化測試的驗證。在大多數情況下，這些是單元測試或整合測試，用於檢查您想要改善之子系統的介面。如果測試在變更後仍然成功執行，這就是重構成功的第一個跡象。

然而，如果測試失敗，您必須撤回變更、進行檢查，或至少分析測試失敗的原因。

自動化測試不能取代對重構結果的手動檢查或手動測試，但它們確實能為您提供保障，並減輕部分手動測試的工作量。

## 4.3.4 定義明確目標

特別是在使用 AI 工具進行重構時，很有可能會失控而重建過多的內容。在開始修改原始碼之前，您應該為自己設定一個具體目標，例如改善原始碼的可讀性或引入特定的結構改善。在每個重構步驟之前，您應該檢查預期的修改是否能讓您更接近目標。如果不是這樣，您就不應該執行。

在本章中，您已經看到 GitHub Copilot 等 AI 工具提供了快速且全面的重構流程。為了真正以小步驟且目標導向的方式工作，您應該使用盡可能精確的提示來限制結果。
