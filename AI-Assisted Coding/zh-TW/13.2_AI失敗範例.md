# 13.2 AI 失敗範例

如果你使用像 GitHub Copilot 這樣的程式設計助手，AI 工具會不斷提出程式碼建議。如果你輸入 for，助手會建議一個遍歷前面行中出現的資料結構的迴圈。輸入 // 或 # 會被 AI 工具完成為完整的註解，諸如此類。顯然，這些未經請求的建議中只有一部分是有用的。

你可以忽略不合適的建議，並儘可能避免分心。在本節中，我們不將不適當的建議視為錯誤；我們不想指控 AI 沒有預知能力。（你經常會有這種印象，但那是具有欺騙性的。）本節討論的是「真正的錯誤」，即 AI 工具在提示明確的情況下仍然提供不正確或誤導性的答案。

## 13.2.2 幻覺

當語言模型回答沒有可靠資訊的問題時，它們有時會編造答案。在程式開發的背景下，AI 工具會建議使用根本不存在的模組、函式庫或函式。

幻覺的發生是因為語言模型使用統計程序來生成答案；也就是說，它們一旦開始，就會盡可能合理地延續文字。例如，可能會出現程式語言 X 中可用的函式出現在實際涉及語言 Y 的答案中。常見的語言模型非常「聰明」，會將函式名稱調整為相應程式語言的慣例（在 PowerShell 中，例如調整為動詞-名詞術語，如 Get-ClusterNameObject）。

幻覺最常發生在訓練材料較少的邊緣 IT 領域。只要你將 AI 工具用於常見語言（Python、C、JavaScript）和常見主題（網頁介面、資料庫、GUI 設計）的程式設計支援，遇到幻覺的可能性就極低。

你可以在網路上找到一些關於如何識別幻覺的建議。然而，在我們看來，這些建議並不是很有效。測試程式碼！如果由於不存在的函式、模組等導致程式碼無法運作，請以傳統方式在網路上搜尋函式名稱。這通常就能解決問題。你可以要求你選擇的 AI 工具進行改進，但如果訓練材料不足，即使在第二次或第三次嘗試中，獲得可用程式碼的機會也不會特別高。

以下範例是使用 ChatGPT 3 建立的。

## 13.2.3 邏輯錯誤

我們在撰寫本書時遇到的最嚴重錯誤之一是程式碼的數學不正確最佳化。三位作者中一位的兒子在 GitHub 上找到了一個計算山地自行車避震器行為的 Python 腳本（四連桿懸吊系統）。這個腳本使用 numpy 和 scipy 模組執行各種計算，父子倆都不太了解，但圖形視覺化看起來是正確的。在沒有真正理解數學原理的情況下，透過 AI 的幫助成功將腳本調整為特定需求。到這裡為止，一切順利。

然而，產生的程式碼相當混亂。AI 工具應該能夠改善它，對吧？

**提示：** Is there a way to further optimize the code?
[270 行 Python 程式碼]

> 是的，有幾種方法可以進一步最佳化程式碼。我們可以專注於提高可讀性、減少冗餘和提高效率。以下是包含這些改進的重構版本：
>
> - **最小化冗餘計算：** 避免多次重新計算相同的值。
> - **為重複程式碼建立函式：** 為重複的程式碼區塊建立函式。
> - **向量化操作：** 盡可能使用 NumPy 的向量化操作來提高效能。
> - **提高可讀性：** 使用更具描述性的變數名稱並新增註解以提高清晰度。以下是最佳化後的程式碼：
>
> [200 行 Python 程式碼]

AI 工具（在這個案例中是 ChatGPT 4o）做出了認真的努力來重構程式碼，並且做了很多正確的事情。特別是，成功地將出現多次的程式碼區塊提取為函式。這部分程式碼變得更短、更容易閱讀。完美！

然而，ChatGPT 並沒有停留在程式的數學函式上，也嘗試簡化其程式碼，例如：

```python
# Jacobian of vector loop function, before optimization
def J_4bar(x, l1, l2, l3, l4, theta1, theta4):
    theta2 = x[0]
    theta3 = x[1]
    J = np.zeros((2, 2))
    J[0, 0] = l1 * np.cos(theta1) - l2 * np.sin(theta2) + 
              l3 * np.cos(theta3) + l4 * np.cos(theta4)
    J[0, 1] = l1 * np.cos(theta1) + l2 * np.cos(theta2) - 
              l3 * np.sin(theta3) + l4 * np.cos(theta4)
    J[1, 0] = l1 * np.sin(theta1) + l2 * np.cos(theta2) + 
              l3 * np.sin(theta3) + l4 * np.sin(theta4)
    J[1, 1] = l1 * np.sin(theta1) + l2 * np.sin(theta2) + 
              l3 * np.cos(theta3) + l4 * np.sin(theta4)
    return J
# Jacobian of vector loop function, after optimization
def J_4bar(x, l1, l2, l3, l4, theta1, theta4):
    theta2, theta3 = x
    return np.array([
        [-l2 * np.sin(theta2), -l3 * np.sin(theta3)],
        [ l2 * np.cos(theta2),  l3 * np.cos(theta3)]
    ])  
```

新程式碼無疑短得多，但一些原始參數（l1、l4、theta1 和 theta4）不再包含在計算中。根據這些參數的值，最佳化後函式的結果可能完全錯誤。ChatGPT 如何得出這個「最佳化」仍然是一個謎。

**這個範例還有什麼問題？**

- （數學方面的）知識不僅對 ChatGPT 來說不夠，對其使用者來說也不夠。如果你的經驗或專業知識不足，就無法確保 AI 結果的合理性，這絕不是一個好主意。
- 要最佳化的腳本已經有大約 10 KB，達到了目前常見語言模型（截至撰寫本文時）還能完整處理的極限。逐段最佳化程式碼的部分會更安全。
- 「最佳化這個程式碼」的指令相當不明確。應該最佳化什麼？效能？功能？程式碼的結構和可讀性？

## 13.2.4 安全問題

也許最大的危險來自於能夠運作但存在固有安全問題的程式碼。這些程式碼不會引起注意，因為它成功通過了常規的功能測試。

根本問題在於網路上充斥著很少考慮安全性的程式碼片段。這些程式碼與「好的」程式碼一樣流入語言模型的訓練材料中。平均來說，AI 工具提供的程式碼品質與網路上或其他訓練來源中可以找到的一樣好或一樣差。

公平地說，必須說使用 AI 工具對於程式是安全還是不安全影響不大。即使在各自專業領域表現出色的許多開發者或程式設計師，也缺乏對安全影響的眼光。最終，你需要讓安全專家審查你的程式碼，無論是手動編寫的還是在 AI 幫助下編寫的。你必須使用模糊測試（fuzzy testing）和其他方法驗證其對錯誤或被竄改輸入的行為。

我們也想用一個範例來說明安全問題。作為我們想在 Python 中建立的 REST API 的起點，我們請 ChatGPT 建立一個最小的 Docker 設定。我們已經縮短了 AI 回覆中的說明。

**提示：** I want to have a minimal setup to test a REST API running in a Docker container.
1. Apache or Nginx as proxy
2. Docker container (Python) with hello world API, GET / should return { "hello": "world"}
As little code as possible. No setup for HTTPS/SSL keys (perhaps later).

> 為了設定一個最小的環境來測試在 Docker 容器中運行的 REST API，我們將使用 Python 搭配 Flask 作為 API，Nginx 作為代理。
>
> **步驟 1：建立 Flask API。** 建立一個專案目錄，例如 hello-world-api。在此目錄中建立一個名為 app.py 的檔案，內容如下：
> ```python
> from flask import Flask, jsonify
> app = Flask(__name__)
> @app.route('/')
> def hello_world():
>     return jsonify({"hello": "world"})
> if __name__ == '__main__':
>     app.run(host='0.0.0.0', port=5000) 
> ```
> **步驟 2：** 在同一目錄中建立 Dockerfile：
> ```dockerfile
> # Use the official Python image from the Docker Hub
> FROM python:3.9-slim
> # Set the working directory
> WORKDIR /app
> # Copy the current directory contents into the container 
> # at /app
> COPY . /app
> # Install Flask
> RUN pip install Flask
> # Make port 5000 available to the world outside this container
> EXPOSE 5000
> # Run app.py when the container launches
> CMD ["python", "app.py"] 
> ```
> **步驟 3：** 在同一目錄中建立 docker-compose.yml 檔案：
> ```yaml
> version: '3'
> services:
>   web:
>     build: .
>     ports:
>       - "5000:5000"
>   nginx:
>     image: nginx:alpine
>     ports:
>       - "80:80"
>     volumes:
>       - ./nginx.conf:/etc/nginx/nginx.conf
>     depends_on:
>       - web 
> ```
> **步驟 4：** 在同一目錄中建立 nginx.conf 檔案，內容如下：
> ```nginx
> events { }
> http {
>   server {
>     listen 80;
>     location / {
>       proxy_pass http://web:5000;
>       proxy_set_header Host $host;
>       proxy_set_header X-Real-IP $remote_addr;
>       proxy_set_header X-Forwarded-For 
>         $proxy_add_x_forwarded_for;
>       proxy_set_header X-Forwarded-Proto $scheme;
>     }
>   }
> } 
> ```
> **步驟 5：** 在終端機中導航到你的專案目錄並執行以下命令來建置和啟動你的容器：
> ```bash
> docker-compose up --build 
> ```
> **步驟 6：** 開啟瀏覽器並導航到 http://localhost。你應該會看到 JSON 回應：
> ```json
> {
>   "hello": "world"
> } 
> ```
> 這個最小設定確保你的 REST API 在 Docker 容器中運行，Nginx 作為反向代理。如果需要，你以後可以擴展此設定以包含 HTTPS/SSL。

基本上，這個答案非常適合快速原型設計！假設有一個正常運作的 Docker 環境，Hello World API 在五分鐘內就能運行，並為進一步的實驗提供一個完美的起點。那麼，問題在哪裡？遺憾的是，問題不止一個，而是整個一系列的安全問題：

- ChatGPT 選擇了完全過時的 Python 3.9（步驟 2）。當時 3.12 是最新版本，下一個版本 3.13 即將完成。出於安全原因，你應該始終使用最新的可能的軟體版本。
- 在 Dockerfile 中額外三行就足以讓 Python 程式以非 root 權限在 Docker 容器中運行 REST API（也是步驟 2）。改進後的 Dockerfile 如下所示（附有變更的註解）：
```dockerfile
FROM python:3.12-slim
# Create custom user and your own group
RUN groupadd -r appgroup && useradd -r -g appgroup appuser
WORKDIR /app
COPY . /app
# Change access rights for app directory
RUN chown -R appuser:appgroup /app
RUN pip install Flask
EXPOSE 5000
# Run Python in the appuser account (instead of root)
USER appuser
CMD ["python", "app.py"]  
```
- 現在通常使用 compose.yaml 而不是 docker-compose.yml（步驟 3）作為檔案名稱。然而，這裡真正的問題是連接埠 5000，它僅用於兩個服務之間的內部通訊。ports: 5000:5000 會將連接埠開放給全世界。這個設定會安全得多：
```yaml
# in docker-compose.yaml use port 5000 internally only
ports:
  - "5000"  
```
- 具有單獨內部網路的設定會是理想的。這需要在 compose.yaml 中增加幾行。
- 在 docker-compose.yaml 中也開放了連接埠 80，以便可以測試 REST API。只要 REST API 僅用於測試目的（這裡顯然是這種情況），將連接埠 80 的使用限制在 localhost 會更安全。這意味著測試系統只能在本地電腦上使用，而不能從外部透過網路使用。
```yaml
# in docker-compose.yaml allow only local access to
# port 80
ports:
  - "localhost:80:80"  
```

這是很多 Docker 特有的細節，本書不是關於這些的。相反，這個範例旨在說明 AI 工具提供的完美運作的解決方案絕不自動等於安全的。這個結論不僅適用於 Docker 設定，也適用於 ChatGPT 等工具輸出的任何類型的程式碼或 IT 指令。只要你在本地嘗試應用程式，安全考量通常只佔次要角色。然而，在生產使用之前，適當的安全審計是必不可少的！

在我們看來，從一開始就注意安全甚至更好。

> **我們自己的錯**
>
> ChatGPT 提供的解決方案在安全方面不足，我們也有部分責任。我們明確要求了盡可能少的程式碼。但我們也可以加上這句話：Please provide a secure setup that can be adapted later for productive usage。這樣 ChatGPT 或其他 AI 聊天機器人雖然仍然不會提供完美的設定，但安全漏洞的數量會明顯減少。

## 13.2.5 不夠理想的解決方案

有許多科學研究使用既定的參考解決方案來評估程式設計回應的品質。我們特別喜歡 Samia Kabir 等人（2024 年 5 月）的「Is Stack Overflow Obsolete?」。它將 ChatGPT 對熱門程式設計問題的回答與 Stack Overflow 上評分最高的解決方案進行了比較。為此，使用了 ChatGPT 3.5。論文的 PDF 和基礎資料（包括所有問題）可以在這裡找到：

https://dl.acm.org/doi/pdf/10.1145/3613904.3642596

https://github.com/SamiaKabir/ChatGPT-Answers-to-SO-questions/tree/main

該論文的結果是矛盾的：

**高錯誤率**
在超過 500 個提示中，ChatGPT 的回應在 52% 的案例中包含錯誤。這並不意味著答案完全錯誤，但它們也不完全正確。我們在本節標題中選擇了「不夠理想」這個詞。

多達 77% 的回應包含冗餘或超fluous的資訊。

**ChatGPT 不理解問題**
錯誤答案的原因通常是 ChatGPT 沒有正確理解問題。

**熱門主題的錯誤較少**
如果涉及既定知識，答案更有可能是正確的。隨著涉及新技術或版本的問題以及不受歡迎、很少被問到的問題，錯誤的可能性會增加。

**ChatGPT 的答案更易於理解**
儘管錯誤率較高，三分之二的開發者仍然更喜歡 ChatGPT 的答案而不是 Stack Overflow 上的解決方案！這是因為答案通常論證更連貫，措辭更禮貌、更積極。

當然，我們想在這份經過仔細研究的出版物中加入我們自己不科學的觀點：

**值得質疑的錯誤率**
錯誤率對我們來說似乎很高。當然，所有語言模型都會犯錯——這正是本章的主題。但我們並沒有經歷每隔一個答案就是錯誤的情況。也許這是因為我們一直在使用下一代 GPT。（到你閱讀本書時，下一代的下一代可能已經問世了！）

**能運作，但不是最佳的程式碼**
在我們的工作中，來自 ChatGPT 的程式碼通常可以運作，所以不是錯的。但往往可以改進程式碼，例如使用現代/較新的函式或應用更優雅的演算法。AI 程式碼沒有錯，但並不總是像人類開發者編寫的高品質程式碼那樣完美。（基準是由經驗豐富的開發者或受過良好訓練、擁有數年培訓和經驗的程式設計師編寫的程式碼。）

**有針對性的回應**
我們認為 ChatGPT 或其他語言模型相對於 Stack Overflow 的最大優勢在於，AI 工具會非常具體地回應我們的個人問題。在實踐中，一個 Stack Overflow 文章回答你確切問題的情況相對罕見。事實上，Stack Overflow 搜尋通常會找到三、四篇回答問題部分方面的文章。然後我們或多或少費力地根據這些組合出自己的程式碼。

該研究的作者為了調查反轉了做法，將既定的 Stack Overflow 問題直接納入提示中。這對我們來說似乎有點不公平。

換句話說，無論 AI 工具犯了什麼錯誤，它仍然確切地回答了被問到的問題（假設提示的措辭足夠清楚）。網路搜尋，無論是在 Stack Overflow 還是其他地方，都會在問題的更廣泛背景下找到無數或多或少相關的資訊。

**少即是多**
我們同意論文作者的批評，即 ChatGPT 的答案幾乎總是太冗長。我們不想把解決方案讀三遍——在簡介中、在程式碼中，然後又在後面的解釋中！

誠然，答案的長度是否適當，在很大程度上取決於你自己的知識水平。先備知識越少，AI 文字的冗餘就越有幫助。就個人而言，我們在這方面更喜歡 Anthropic 的 Claude 工具而不是 ChatGPT：Claude 能抓住要點，而且遠不那麼囉嗦。

**商業語言模型是基準**
與論文作者不同，我們在本書中不僅使用了 ChatGPT，還嘗試了其他語言模型（通常是使用 Ollama 在本地執行的較小 LLM）。雖然我們更希望在這裡寫下相反的內容：商業雲端產品在品質方面更具說服力。
