# 7 資料庫

對於一本探討程式設計的書來說，資料庫似乎是一個不尋常的主題。但 SQL 也是程式碼，而 AI 工具在編寫 SQL 命令方面表現出色。此外，AI 在其他領域也能提供極大的幫助，例如資料庫設計或資料庫管理系統（DBMS）的管理。

在本章中，我們將使用各種範例來說明 AI 工具在資料庫的設計或修改、SQL 命令的撰寫、管理以及最終的用戶端程式碼開發中的應用。

> **資料庫管理系統特定的回答**
> 關聯式資料庫管理系統（RDBMS），例如 Microsoft SQL Server、MySQL、Oracle 和 PostgreSQL 有許多共同點，但在細節上卻有顯著差異。除非您有一個非常基本的問題，否則您應該始終在提示中指定您的問題涉及哪個 DBMS！否則，您可能會收到完全不適用於您的 DBMS 的答案。

如果網路上有大量關於您 DBMS 的訓練資料，您使用 ChatGPT 等工具將會有最佳體驗。在關聯式系統中，這特別適用於 MySQL/MariaDB、PostgreSQL、SQLite 和 SQL Server；在 NoSQL 產品中，非常受歡迎的 MongoDB 和 Redis 則具有優勢。

因此，請要求 ChatGPT 建立程式碼來用 Mermaid 繪製 ER 圖。Mermaid（https://mermaid.js.org）是一個很棒的 JavaScript 函式庫，用於產生圖表和圖形：

**提示：** Can you provide code to draw an ER diagram with Mermaid?

當然！Mermaid 是一個用於建立圖表（包括 ER 圖）的絕佳工具。以下是為您的線上商店資料庫產生 ER 圖的 Mermaid 程式碼：

```
erDiagram
    USERS {
        int UserID PK
        string Username
        string PasswordHash
        string Email
        date DateRegistered
    }
    ...
    USERS      ||--o{  ORDERS : "places"
    CATEGORIES ||--o{  PRODUCTS : "contains"
    PRODUCTS   ||--o{  PRODUCT_PRICES : "has"
    ORDERS     ||--o{  ORDER_DETAILS : "includes"
    PRODUCTS   ||--o{  ORDER_DETAILS : "part of" 
```

要使用此程式碼：
1. 複製上面的 Mermaid 程式碼。
2. 將其貼到 Mermaid 即時編輯器中。
3. 圖表將自動呈現。

此 Mermaid 程式碼定義了實體（表格）、其屬性、主鍵（PK）和外鍵（FK），以及它們之間的關聯。關聯以基數表示：`||--o{` 代表一對多關聯。

### 7.1.4 變更資料庫結構

在現實中，您很少有機會從頭開始開發一個全新的資料庫結構。更常見的是，您只需要針對一些特定細節擴展結構。AI 工具在這方面也能提供幫助。通常，第一步您要傳輸資料庫的當前結構（即所有基礎的 CREATE TABLE 命令），或使其在編輯器分頁中可用。（如果需要，直接問 ChatGPT 如何建立結構檔案。您可以在第 7.3 節中找到相應的範例。）

對於某些問題，只需指定單個表格的定義就足夠了。以下問題是關於文字欄位必須多大，才能在所有特殊情況下正確儲存 IPv6 地址。

**提示：** MySQL: We use the logins table to store IPv4 addresses.
```sql

  ip VARCHAR(15) DEFAULT NULL, 
  ...) 
```
Now we have to deal with IPv6 addresses as well. How do we change the ip column? Could you provide an ALTER TABLE command?

```sql

```

此變更將 `ip` 欄位的長度設為 45 個字元，這足以儲存 IPv4 和 IPv6 地址，包括 IPv6 地址的完整展開形式。

一旦此測試令您滿意，您就可以提出您實際想要的內容。目前 `customers` 表格只能儲存一個人的聯絡資訊。然而，對於某些客戶，您想要儲存多個聯絡人的資料。

**提示：** OK. The customers table currently contains contact information for only one person. For some customers, we need to store data for several contact persons (first name, last name, email, phone number). Can you suggest a schema change? Existing data must be preserved.

為了容納單個客戶的多個聯絡人，您可以建立一個名為 `contactPersons` 的新表格。此表格將有一個連結到 `customers` 表格的外鍵。以下是建立此新表格的 SQL：

```sql
CREATE TABLE contactPersons (
  id         INT(11) NOT NULL AUTO_INCREMENT,
  customerId INT(11) NOT NULL,
  firstName  VARCHAR(100) DEFAULT NULL,
  lastName   VARCHAR(100) DEFAULT NULL,
  email      VARCHAR(100) NOT NULL,
  telNo      VARCHAR(100) DEFAULT NULL,
  PRIMARY KEY (id),
  FOREIGN KEY (customerId) REFERENCES customers(id)
); 
```

在這個新結構中，每個客戶可以有多個聯絡人。`contactPersons` 表格中的 `customerId` 欄位將每個聯絡人連結到 `customers` 表格中的一個客戶。

為了保留現有資料，您可以使用如下 SQL 命令將聯絡人資料從 `customers` 表格遷移到 `contactPersons` 表格：

```sql
INSERT INTO contactPersons (customerId, firstName, 
                            lastName, email, telNo)

SELECT id, firstName, lastName, email, telNo 
FROM customers; 
```

資料遷移後，您可能想要在 `customers` 表格中保留聯絡人欄位一段時間以確保向後相容性。然而，您應該計劃最終移除這些欄位，並更新任何參照它們的程式碼，改為使用新的 `contactPersons` 表格。

我們使用 GitHub Copilot 執行了此範例，該工具在這種情況下真的表現出色：它不僅建議了合理的結構變更（將聯絡資料外包到單獨的表格中），還採用了原始 `customers` 表格中的欄位名稱和資料類型，以之前表格使用的相同 camelCase 命名法為新表格命名，並記住了將新表格連結到現有 `customers` 表格的新外鍵規則。資料遷移的建議也是合理的，並且可以正常運作。

當然，是否按照建議進行資料庫轉換或是否還需要變更，仍然取決於您。如果您有多個聯絡人，您可能想要將其中一個標記為主要聯絡人，在這種情況下，您將需要一個額外的欄位。
