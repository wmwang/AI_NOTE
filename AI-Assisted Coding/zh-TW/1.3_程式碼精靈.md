# 1.3 程式碼精靈

基於聊天的 AI 工具為 AI 輔助程式設計世界提供了簡單且免費的入門方式。然而，在 AI 工具的網頁介面和編輯器或開發環境之間不斷來回切換有些不方便。您需要不斷在編輯器和網頁瀏覽器之間複製程式碼，這肯定可以有更簡單的方式。

事實上，在 OpenAI 向公眾展示其通用 ChatGPT 介面之前，Microsoft 已經啟動了 GitHub Copilot 的技術預覽。GitHub Copilot 自 2022 年 6 月起作為付費服務提供。最便宜的方案目前為每月 10 美元。

> **「Copilot」概念**
> Microsoft 最初僅使用 Copilot 來描述新的程式碼精靈。然而，這個術語現在廣泛用於各種 Microsoft AI 工具、搜尋功能以及 Microsoft Office 工具。自 2024 年中以來，甚至有一個以此命名的筆記型電腦系列（Copilot+ PC）。本書專門關注 GitHub Copilot。

GitHub Copilot 的想法是長期在 Microsoft 世界中已確立的 IntelliSense 概念的邏輯延伸。在輸入程式碼時，Visual Studio Code（VS Code）建議輸入的補全。但 IntelliSense 僅限於方法或屬性的名稱，而 GitHub Copilot 會建議迴圈（包括其內容），甚至建議整個方法或函式。

GitHub Copilot 有時運作得令人難以置信的好。假設您開始定義一個新函式，例如 `findCancelledCustomers`。從檔案其餘部分的上下文中，GitHub Copilot 認出您可能想要從 `customers` 資料表中獲取所有狀態為 Canceled 的客戶列表。

因此，GitHub Copilot 提出了函式的完整程式碼：它產生查詢的 SQL 程式碼，使用已在檔案其他地方定義的資料庫連線變數執行它，並將客戶名稱和 ID 的列表作為函式結果回傳。

當然，GitHub Copilot 的建議並不總是完美的。您通常可以直接使用程式碼，然後需要調整一些細節。如果 AI 工具沒有正確猜測您的意圖，只需繼續手動輸入您的程式碼。在隨後的行中，GitHub Copilot 將根據額外的上下文資訊（例如作為註解或變數名稱）提出更好的新建議。

在幕後，GitHub Copilot 與 ChatGPT 一樣，目前使用來自 OpenAI 的語言模型（最近為 GPT-4）。該語言模型使用來自 GitHub 公開儲存庫的程式碼進行訓練，以及其他材料。這一事實引發了關於版權問題和安全方面的爭議。目前仍不完全清楚使用公開原始碼進行訓練是否真的被視為合理使用，正如 GitHub 和 Microsoft 所主張的那樣。在本書中，我們在第 13 章中討論此問題和其他方面。

## GitHub Copilot 的替代方案

正如 ChatGPT 不是唯一基於聊天的 AI 環境一樣，GitHub Copilot 也有各種替代方案，例如帶有 Duo Chat 的 GitLab Code Suggestions 或當前 Xcode 版本中整合的 Swift Assist 功能。

**Cody** 非常令人興奮。基本上，Cody 也補全程式碼並回答聊天提示。但它有兩個主要優勢：首先，您可以在不同的語言模型之間選擇（本地大型語言模型 [LLM]、Claude Sonnet、Gemini、Mixtral，以及 Pro 帳戶的 GPT-4o 和 Cloud Opus）。其次，Cody 考慮了活動目錄中的整個程式碼庫。（此功能預計運早也會在 GitHub Copilot 中實作。）您可以免費試用 Cody。具有無限聊天功能和更廣泛 LLM 選擇的 Pro 訂閱目前每年 9 美元。您可以在這裡找到更多資訊：https://sourcegraph.com/cody。

除了商業產品外，還有免費的編輯器擴充功能和外掛程式。它們需要透過介面（API）存取 AI 系統。在本書中，我們將以 VS Code 外掛程式 Continue 為例（見第 9 章，第 9.5 節）。它提供與 GitHub Copilot 非常相似的功能，但可以使用本地執行的語言模型。如果您有適當的計算能力，您可以避免向 AI 供應商付費並避免將資料傳輸到雲端。

Continue 也可以連接到本地網路或網際網路上的 AI 實例。例如，您可以將一台配備 GPU 的強大電腦配置為工作群組；所有團隊成員然後共同使用這台電腦進行 AI 程式設計任務。對公司來說的一個主要優勢是它們可以避免任何資料保護疑慮。

### 1.3.1 安裝 GitHub Copilot

GitHub Copilot 最初是作為 VS Code 編輯器的擴充功能開發的。VS Code 與 GitHub Copilot 的組合至今仍然極受歡迎。因此，本書中的大多數程式碼精靈範例都是這個「夢幻團隊」的應用。在使用 GitHub Copilot 之前，您需要安裝同名擴充功能，在 VS Code 中登入您的 GitHub 帳戶，並在 GitHub 網站上註冊 Copilot 訂閱（https://github.com/features/copilot/plans）。

您可以免費試用 GitHub Copilot 30 天。為學生、教師和大型開放原始碼專案的維護者提供免費授權。

> **與 GitHub Copilot 相容的編輯器和開發環境**
> GitHub Copilot 不僅可以與 VS Code 一起使用，還可以與各種其他編輯器或開發環境一起使用，例如 Visual Studio、所有 JetBrains IDE（IntelliJ、Android Studio、PyCharm 等）以及 Vim 和 Neovim 編輯器。還有各種其他編輯器的非官方外掛程式，例如 Emacs 或 Xcode。

### 1.3.2 Visual Studio Code 與 GitHub Copilot 的第一步

安裝後，VS Code 在狀態列中顯示一個 Copilot 圖示，引導您進入 Copilot 狀態選單。選單的第一行必須包含「Status: Ready」條目。

要試用 GitHub Copilot，請用您喜歡的語言建立一個新檔案。我們在以下範例中使用了 Python。現在輸入 `def factorize(x)`，例如。可以假設您想要程式設計一個將整數分解為因數的函式。

GitHub Copilot 以灰色字體建議程式碼的可能延續。如果您想接受整個提案，只需按 **Tab** 鍵確認即可。或者，您可以重複按 **Ctrl+→** 來以小部分（逐字或逐行）接受建議。提案選擇的小型對話框僅在您將游標移到程式碼提案上時才會顯示。

### 1.3.3 切換程式碼建議

VS Code 始終只顯示一個程式碼建議，儘管 GitHub Copilot 內部提供了多個選項。快速鍵 **Alt+]** 或 **Alt+[** 用於在前三個建議之間切換。

| 快速鍵 | 功能 |
|--------|------|
| Tab | 接受建議 |
| Ctrl+→ 或 Cmd+→ | 接受建議的一個字或一行 |
| Alt+] | 下一個建議（美式鍵盤） |
| Alt+[ | 上一個建議（美式鍵盤） |
| Ctrl+Enter 或 Cmd+Enter | 在側邊窗格中顯示 10 個建議 |
| Ctrl+I 或 Cmd+I | 開啟 Copilot 對話框（行內聊天） |
| Ctrl+Alt+I | 開啟聊天側邊欄 |
| Ctrl+Cmd+I | 開啟聊天側邊欄（macOS） |

### 1.3.4 提示與技巧

在新的、幾乎空白的程式碼檔案中進行測試時，GitHub Copilot 的很大一部分潛力仍未被利用。因此，我們建議您在一個您熟悉其程式碼的專案中熟悉 AI 工具。關鍵是您必須能夠毫無疑問地評估 GitHub Copilot 建議的品質。這在熟悉的環境中效果最好。只需在您的專案中新增一個新功能或對現有程式碼進行變更，您就會對這些建議的建設性和有用性感到驚訝！

要充分利用 GitHub Copilot 的全部潛力，請考慮以下提示：

- 為新函式或方法取盡可能有意義的名稱。
- 如有必要，在函式定義之前、迴圈之前或分支之前撰寫一個註解（一到兩行就足夠了），以更精確地描述任務或指出特殊功能或細節。這將幫助 GitHub Copilot 在上下文中提出更合適的建議。
- 您也可以採取相反的方法，先撰寫程式碼（有或沒有 AI 支援），然後讓 GitHub Copilot 幫助您撰寫註解。只需根據語言以 `#`、`//` 或 `/*` 開始註解。GitHub Copilot 解析周圍的程式碼並建議通常有用的文件。
- 在 VS Code 中開啟您專案的幾個相關的其他檔案，包含可比較的程式碼或包含您想在當前工作的檔案中使用的函式或類別的定義。GitHub Copilot 將開啟的分頁的程式碼作為上下文資訊考慮，並認出您迄今為止如何呼叫或評估函式、屬性或方法。這提高了程式碼提案的品質。

目前（2025 年初），GitHub Copilot 僅考慮在編輯器中開啟的專案檔案。未來的版本可能也會分析關閉的檔案。這目前無法運作有兩個原因：首先，當前語言模型的上下文大小非常有限，GitHub Copilot 簡直無法考慮所有檔案。其次，對於大型專案，在數百個檔案中識別相關檔案是困難的。

### 1.3.5 包含額外關鍵字的聊天模式

程式碼精靈和聊天功能絕不互斥。按 **Ctrl+Alt+I** 或 **Ctrl+Cmd+I** 在編輯器中開啟聊天側邊欄。在那裡您可以自由地與 AI 語言模型交流。您可以透過點擊按鈕將回應中包含的程式碼直接傳輸到當前開啟的程式碼檔案中的當前游標位置，從而省去透過剪貼簿的繞道。

與網頁瀏覽器中的 ChatGPT 相比的一個決定性優勢是，您可以透過特殊關鍵字引用開啟的編輯器分頁的上下文，如下所示：

- **@workspace** 表示您引用工作區中的程式碼，不想問一般性問題。（VS Code 工作區由專案目錄中的檔案或所有當前活動的專案組成。）
- **@vscode** 表示您正在詢問有關使用編輯器的問題。GitHub Copilot 可能會在回應中包含您可以直接點擊的 VS Code 命令。
- **@terminal** 將上下文導向終端機，這對與終端機命令相關的問題很有用。根據使用的作業系統，GitHub Copilot 會解釋對 Linux 和 macOS 命令或 PowerShell cmdlet 的答案。
- **/clear** 刪除先前的聊天歷史記錄。
- **/doc** 為程式碼新增註解。
- **/explain** 解釋當前程式碼的任務或功能。這在您編輯第三方程式碼或從網際網路（例如從 Stack Overflow）取得程式碼且不完全理解它時特別有用。
- **/fix** 嘗試修復所選程式碼中的問題或錯誤。
- **/tests** 為當前函式或方法產生單元測試。

> 圖 1.6 GitHub Copilot 在聊天側邊欄中解釋高亮的程式碼行

> **提示**
> `/explain`、`/fix`、`/doc` 和 `/tests` 命令可以直接在程式碼視窗中更方便地執行。為此，在程式碼中選擇相關行，並執行以下上下文選單之一：Copilot • Explain This、Fix This、Generate Docs 或 Generate Tests。

以下提示展示了關鍵字使用的一些範例。在第一個範例中，VS Code 應該解釋剛剛高亮的程式碼。

**提示：** @workspace /explain the selected lines.

您可以使用 GitHub Copilot 在專案中搜尋特定檔案。我們的測試產生了混合的結果。CSS 搜尋成功了，而不安全的明文密碼搜尋失敗了。

**提示：** @workspace Which CSS file defines the invoiceDate class?

> `invoiceDate` 類別在 `css/invoice.css` 檔案中定義。

**提示：** @workspace Which scripts contain unsafe clear text passwords for database connections?

> [沒有有用的結果]

如果您迷失在眾多 VS Code 命令中，GitHub Copilot 有時可以幫助您。同樣，這裡有一個正面和一個負面的範例。第二個提示的答案不完整，因為 GitHub Copilot 沒有提供啟用次要面板並直接將聊天拖到那裡的選項（另見相應的「充分利用空間」框）。

**提示：** @vscode How can I see the differences in the code file since the last commit?

> 您可以使用 Git: Open Changes (git.timeline.openDiff) 命令來顯示活動文件與上次提交相比的差異。

**提示：** @vscode I want to display the chat panel on the right side. All other panels should remain at the left side.

> 您可以使用 View: Move Panel Right 命令將面板移動到右側。然後，使用 View: Show Chat 命令開啟聊天面板。請注意，這將把所有面板（包括聊天面板）移動到右側。VS Code 目前不支援同時在不同側顯示面板。

在執行以下問題時，我們剛剛在 VS Code 中編輯了 Linux 伺服器上的檔案。我們特別喜歡正確的一行答案，沒有任何多餘的廢話。

**提示：** @terminal How can I find files that haven't changed for more than six months?

```bash
find . -type f -mtime +180
```

一旦執行了聊天命令，您就無法直接更改它們。相反，您需要像在終端機中一樣使用游標鍵捲動先前執行的提示，更改它們，然後再次執行。

> **充分利用空間**
> 如果 VS Code 側邊欄中的聊天對您來說太小，您可以透過「Open Chat in Editor」或「Open Chat in Window」在單獨的視窗中開啟它。
>
> 另一種變體是使用「View: Toggle Secondary Side Bar Visibility」在右側顯示第二個側邊欄並將聊天歷史記錄移到那裡。這在您使用非常寬的顯示器工作時特別有用。

### 1.3.7 聊天側邊欄的替代方案

除了在聊天側邊欄中使用 GitHub Copilot 外，您還可以透過 **Ctrl+I** 或 **Cmd+I** 開啟一個小型行內聊天對話框。您可以在那裡表述您的問題，然後直接將答案傳輸到程式碼中。行內聊天對於小變更可以非常高效地使用。此外，聊天歷史記錄保持有條理。

您也可以直接在程式碼中提出小問題。為此，使用 `q:` 開始一個註解，然後表述您的問題。一旦您按 **Enter**，GitHub Copilot 會插入另一行以 `a:` 開頭並包含答案的註解：

```javascript
// q: How do I call bash script from PHP?    <- 您的輸入
// a: Use shell_exec() or exec() function.   <- GitHub Copilot
```

這種類型的問答遊戲特別適合小細節。最大的優勢是您永遠不必離開編輯器。一旦問題解決，您可以刪除包含問題和答案的註解行。

### 1.3.8 其他功能

Microsoft 正努力將 GitHub Copilot 作為一個全方位工具整合到 VS Code 中。您必須自己決定這些功能在您的工作流程中的有用程度：

**Git 提交**

如果您使用 Git 版本管理，您可以讓 GitHub Copilot 撰寫提交文字。AI 工具嘗試總結最近所做的變更。有時這運作得相當好，有時您只需要更改文字。對我們來說，這些訊息往往太冗長了。

**處理錯誤訊息**

如果在程式執行過程中發生錯誤，VS Code 會在終端機區域的錯誤訊息旁顯示一個 Copilot 圖示。如果您點擊它，GitHub Copilot 將嘗試在聊天中解釋錯誤的原因。理想情況下，它還會直接提供解決方案，並在進一步點擊時執行建議。

我們對此功能的體驗好壞參半。對於某些程式語言（包括 Python！），GitHub Copilot 甚至無法識別完全微不足道的錯誤，例如缺少括號。誠然，也有功能按宣傳運作的情況。但這方面當然仍有改進空間。

**終端機中的 GitHub Copilot**

一旦您在終端機中執行了命令，您可以使用 GitHub Copilot 來解釋該命令。

如果在執行上一個命令時發生錯誤，GitHub Copilot 會解釋錯誤的原因並建議更正或改進的命令。

我們已經指出了在聊天中使用 @terminal 提出專門與終端機命令相關的问题的可能性。

### 1.3.9 安全性與資料保護

您應該意識到 GitHub Copilot 可以存取您專案中的所有檔案。這是工具運作所必需的。存取還包括包含密碼、授權金鑰等的檔案。因此，您必須對 GitHub Copilot 不會濫用這些資料有很大的信心。

然而，最終這種考慮不僅適用於 AI 工具，也適用於每個編輯器和每個開發環境：IDE 中的後門或帶有惡意程式碼的外掛程式也可能洩露公司機密。因此，您應該仔細考慮在 VS Code 或您選擇的 IDE 中安裝哪些外掛程式。令人驚訝的是，這方面從未發生過（已知的）安全災難。

> **切勿將密碼儲存在程式碼檔案中！**
> 無論前面的考慮如何，您都不應該將密碼直接儲存在程式碼檔案中。使用單獨的配置檔案，如 `config.json` 或 `.env`，或將密碼儲存在您的程式碼可以存取的環境變數中。

當然，VS Code 也可以存取密碼檔案。然而，在大多數情況下，GitHub Copilot 或其他 AI 工具不需要評估它們。

但是，如果密碼以明文形式存在於程式碼檔案中，當使用 AI 工具時，它將不可避免地上傳到外部伺服器。即使 AI 供應商本身對這些密碼沒有興趣，它們在任何資料庫中不受控制的儲存以改進或訓練未來的語言模型也是一個安全噩夢。

### 1.3.10 遠端模式的問題

Remote SSH 外掛程式允許您使用 VS Code 編輯位於其他電腦上目錄中的程式碼，前提是您對外部電腦有安全殼層（SSH）存取權限。然而，在我們的測試中，GitHub Copilot 在這種情況下拒絕合作。本地電腦上 `settings.json` 配置檔案中的以下行可以幫助：

```json
// 包含在 settings.json 中
{
  "remote.extensionKind": {
    "GitHub.copilot": ["ui"]
  }
}
```

開啟 `settings.json` 檔案最簡單的方式是使用「Open User Settings (JSON)」（**F1**）。您可以在這裡找到更多關於該過程的詳細資訊：https://github.com/orgs/community/discussions/6942#discussioncomment-4962858。
