# 3 除錯

一旦軟體專案超過一定規模，錯誤就會潛入原始碼中。除錯的目的是找出並修正這些錯誤。錯誤通常在開發過程中出現，使用者往往要到後來才會指出它們。然而，我們必須解決問題，如果我們能透過使用 AI 更快地解決它，那就更好了。

在軟體開發中，我們有不同的工具來追蹤問題。程式碼中的臨時列印語句（或 Web 應用程式的 `console.log`）是最簡單且通常最快取得成功的方法。結構化日誌記錄（具有不同的類別和其他元資料）在客戶用於生產環境的應用程式中很有幫助。能夠在執行時分析並停止程式碼的除錯器是複雜應用程式不可或缺的工具。

這些工具幫助您識別和理解錯誤，但它們不是萬靈丹。您仍然需要自己思考問題、排除錯誤來源並調查原因。通常，您會如此專注於自己的程式碼，以至於不會注意到任何不合理的地方。一旦陷入死胡同，就很難獨自找到出路。

這就是為什麼現代軟體開發通常涉及團隊合作：在配對程式設計中，您一起處理程式碼，因為四隻眼睛（和兩個大腦）識別問題的速度簡單來說是兩倍以上。如果沒有同事可用，小黃鴨除錯法通常有幫助：抓一隻橡膠鴨子，向它解釋問題。因為鴨子不是程式設計專業人士，這些解釋必須緩慢且一步步進行。一旦您以這種方式分解了錯誤，解決方案通常就很明顯了。

但如果小黃鴨不只是一個被動的聆聽者，而是能夠貢獻分析、建議的解決方案和自己的想法呢？這將為您提供一個配對程式設計的夥伴，它從不脾氣暴躁或無聊，總是有時間，而且對幾乎每種語言和框架都有良好的基礎知識。

在本章中，我們將嘗試在 AI 的支援下消除自己原始碼中的錯誤。得益於我們在修復錯誤方面的經驗，我們可以估計使用 AI 實現了多少時間節省。

這樣做的缺點是，我們對解決路徑的了解可能會影響問題。這正是提示表述的核心點：如果您已經給了 AI 助手一個方向，並使用關鍵字將其引導到某條路徑上，它很可能會跟隨您的思路並複製它。如果這不能解決問題，開始一個新的会話並以更開放和一般的方式表述請求可能是有意義的。這與我們在書中通常給出的建議相矛盾，即提示應該盡可能具體地表述並帶有大量上下文，以便 AI 工具知道它必須做什麼。

然而，如果您自己不確定問題在哪裡以及正確的解決方案是什麼，最好給 AI 工具更多的餘地。要求它提供關於問題的不同方法和觀點。然後，當然，由您在各種想法之間選擇解決問題的路徑。AI 確實可以幫助您除錯簡單的語法問題，但當情況變得更複雜時，底線更加適用：始終檢查所有答案並牢記在心。僅僅因為您的 AI 助手自豪地說它已經解決了問題，並不意味著一切都如預期般運作。

儘管如此，我們可以證實我們在 AI 支援除錯方面的經驗非常正面。語言模型擅長發現和修正偏離常規的情況。如果您程式中的錯誤可以用這種方式描述，那麼您的 AI 助手找到它的機率很高。

### 3.1.1 CSS 修復

任何曾經嘗試在 HTML 中建立一個據說簡單的佈局的人都會熟悉以下問題：一個元素就是不願意出現在您希望它在頁面上的位置。作為範例，我們使用了一個非常簡單的頁面結尾作為頁尾元素。內容應該分為三個區域：左、中和右。我們的第一個設計有一個缺點，即具有階層式樣式表（CSS）屬性 `float` 的區域不遵循周圍元素的高度，而是可以超出它流動。

> **圖 3.1** HTML 頁面頁尾的不正確佈局

讓我們看看 Claude 3.5 Sonnet 是否能在這裡提供更好的解決方案：

**提示：** How can I make my footer elements all the same height:
```html
<footer style="margin:auto;text-align:center;width:80%;">
  <span style="float:left;">previous page</span>
  <span>(c) 2024</span>
  <span style="float:right;">next page<br />Intro</span>
</footer>
```

> 要讓頁尾元素都具有相同的高度，您可以使用 flexbox。以下是修改後的程式碼版本：
> ```html
> <footer style="margin:auto; width:80%; display:flex;
>     justify-content:space-between; align-items:stretch;">
>   <span style="flex:1;padding:10px;">previous page</span>
>   <span style="flex:1;padding:10px;text-align:center;">
>     (c) 2024</span>
>   <span style="flex:1;padding:10px;text-align:right;">next page
>     <br />Intro</span>
> </footer>
> ```
> 關鍵變更：
> 1. 移除了 float 屬性，改用 flexbox。
> 2. 為 footer 新增了 display:flex。
> 3. 使用 justify-content:space-between 均勻分佈 span。
> 4. 使用 align-items:stretch 使所有項目拉伸到最高項目的高度。[更多細節]

該解決方案使用強大的 CSS Flexible Box Layout（Flexbox）模型正確定位元素。Flexbox 已在主流瀏覽器中實作了將近 10 年，可以做的遠不止將一個區域分為三個等寬的列。然而，這也使得 Flexbox 可能非常複雜，這就是為什麼我們想問 AI 工具是否也可以使用更現代（且通常更簡單）的 grid 佈局。

**提示：** Would CSS display:grid be an option?

> 是的，CSS Grid 絕對是這個佈局的一個好選擇。在許多情況下，Grid 可以提供比 Flexbox 更多的控制和靈活性。以下是使用 CSS Grid 實作相同佈局的方式：
> ```html
> <footer style="margin:auto; width:80%; display:grid;
>     grid-template-columns: 1fr 1fr 1fr;">
>   <span style="padding:10px;">previous page</span>
>   <span style="padding:10px;text-align:center;">(c) 2024</span>
>   <span style="padding:10px;text-align:right;">next page<br>
>     Intro</span>
> </footer>
> ```

### 3.1.4 JavaScript Fetch 使用了錯誤的內容類型

我們的下一個問題來自一個更改，其中我們用現代的 `fetch` 介面替換了舊的 JavaScript 函式庫，用於在瀏覽器中執行非同步 JavaScript（Ajax）Web 請求。大多數呼叫進行得很順利，但在一些地方的轉換有點笨拙。AI 能否也為這個問題提供快速解決方案？

為了實驗，我們將觸發問題的整個函式和瀏覽器主控台中的錯誤訊息複製到提示中：

**提示：** JavaScript: When I call this function:
```javascript
const _newComment = function (topic, meetingId, event) {
  event.target.disabled = true;
  fetch(`api/comments`, {
    body: JSON.stringify({
      id: topic,
      comment: this.newCommentContent.trim(),
      meetingId,
    }),
  })
    .then((json) => {
      this.comments.push({
        id: json.id,
        content: this.newCommentContent.trim(),
        date: moment().format("YYYY-MM-DD"),
        meeting: meetingId,
        user: this.$store("user").fullname,
      });
      this.newCommentContent = "";
    })
    .finally(() => {
      autosize.update(event.target);
      event.target.disabled = false;
      this.$focus.focus(event.target);
    });
};
```
I get this error message in my browser console window:
```
comments.js:8 Uncaught (in promise)
TypeError: Failed to execute 'fetch' on 'Window': Request with
  GET/HEAD method cannot have body.
    at Proxy._newComment (comments.js:8:3)
    ...
```

> 您看到的錯誤是因為您沒有在 fetch 呼叫中指定 HTTP 方法，它預設為 GET。然而，您還試圖發送請求主體，這對於 GET 請求是不允許的。要修復此問題，您需要指定 HTTP 方法（可能是 POST）並包含必要的標頭。以下是修改 fetch 呼叫的方式：[回應程式碼]
>
> 關鍵變更：
> 1. 新增了 method: 'POST' 以指定這是一個 POST 請求。
> 2. 新增了標頭以將內容類型指定為 JSON。
> 3. 將 `then(json =>` 改為 `then(response => response.json()).then(json =>` 以正確解析 JSON 回應。

AI 工具再次給我們留下了深刻印象：使用 `fetch` 發送資料時使用了錯誤的 HTTP 方法（GET 而不是 POST），這個錯誤從瀏覽器主控台中的訊息幾乎是顯而易見的。但我們沒有期望另外兩個錯誤（缺少的內容類型和缺少使用 `response.json()` 將回應從文字轉換為 JavaScript 物件）能立即被解決。

現在我們可能會正當地問這與 AI 有什麼關係。在 Jam 的 Web 介面中，您會在螢幕最右側找到 JamGPT 標籤，這就是 AI 發揮作用的地方。

點擊一個按鈕，您可以將錯誤訊息發送給語言模型，並期望它提供解決問題的方案。此解決方案的便利性確實出色，展示了可能性，因此我們期望獲得卓越的結果。然而，我們有些失望：對於簡單的錯誤，例如後端查詢的 URL 拼寫錯誤，JamGPT 正確地指出了可能的錯誤，但 Web 開發人員不需要 AI 支援來處理這個。錯誤代碼 404 是眾所周知的。

然而，在為 JamGPT 製造的更複雜的錯誤中，AI 工具無法大放異彩。相反，答案甚至指向了錯誤的方向：為了驗證表單資料，我們使用了一個函式庫，它分析元素的 `data-rules` HTML 屬性，然後檢查那裡儲存的規則的有效性。`data-rules` 的值必須是一個可以透過 `JSON.parse` 轉換為 JavaScript 物件的字串，例如：

```html
<input type="text" data-rules='["required"]' name="email" />
```

幸運的是，使用目前的瀏覽器，您可以不需要這種基於 JavaScript 的驗證，但程式碼已經有幾年的歷史了。前面的字串透過 `JSON.parse` 轉換為一個具有一個條目的 JavaScript 陣列，即 "required" 字串。如果您忘記了 required 關鍵字周圍的雙引號，則函式庫的 JavaScript 程式碼中會發生錯誤。JavaScript 主控台提供了轉換失敗的位置資訊：

```
Uncaught SyntaxError: Unexpected token 'r', "[required]" is not valid JSON
    at JSON.parse (<anonymous>)
    at validation.js:17:24
    ...
```

JamGPT 生成的回應建議我們檢查伺服器請求或伺服器回應的有效 JSON 結構：

> 錯誤似乎與程式碼中處理的無效 JSON 有關。
> 1. 檢查回傳或發送的 JSON 結構；它應該正確格式化。
> 2. 尋找 JSON 中的任何語法錯誤，例如錯位的逗號或引號。
> 3. 確保發送到伺服器的資料正確序列化為 JSON。

但我們還沒有走到那一步：錯誤在我們可以向伺服器發送請求之前就發生了。誠然，Web 應用程式中轉換 JSON 時的大多數錯誤發生在與伺服器的通訊期間，這就是為什麼 AI 工具也將此建議為可能的問題。然而，在我們的情況下，問題與伺服器無關，瀏覽器主控台中的上下文會顯示這一點。我們很好奇 Jam 是否會在語言模型的整合方面取得進一步進展，因為潛力無疑是存在的。
