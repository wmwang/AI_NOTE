## 一、概念講授（30 mins）

### 1.1 Legacy Code 的困境

```
┌─────────────────────────────────────────────────────┐
│               Legacy Code 的典型困境                   │
│                                                     │
│  📁 凡萬行 VB6 / .NET / COBOL 程式碼                  │
│  👴 原作者已離職多年                                    │
│  📝 文件過時或不存在                                    │
│  🔗 盤根錯節的依賴關係                                  │
│  🧪 沒有任何測試                                       │
│  😱 「改了 A 就壞 B」的恐懼                             │
│                                                     │
│  結果：沒人敢動，系統越來越腐朽                            │
└─────────────────────────────────────────────────────┘
```

**傳統做法的問題**：

| 做法 | 問題 |
|------|------|
| 直接用 AI 重寫 | AI 不理解隱含的業務邏輯，重寫後遺漏功能 |
| 邊讀邊改 | 改了才發現理解錯誤，返工成本極高 |
| 手動寫文件再重構 | 曠日廢時，文件寫完系統又變了 |

### 1.2 SDD（Spec-Driven Development）：正確的打開方式

**核心策略**：先讓 AI 「只讀不寫」，把舊程式碼的行為規格萃取出來，然後用規格驅動重構。

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Legacy Code  │ ──→ │  YAML 規格    │ ──→ │  新架構實作   │
│  （只讀）     │     │  （Single     │     │  （TDD 驅動） │
│              │     │  Source of    │     │              │
│              │     │  Truth）      │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑                    ↑                     ↑
   Step 1: 讀取舊碼    Step 2: 萃取規格     Step 3: 用規格驅動新實作
   AI 只能觀察         產出行為規格文件      第六堂課的內容
```

**為什麼「只准讀不准寫」這麼重要？**

1. **安全邊界**：AI 不會意外修改正在運行的系統
2. **強制理解**：萃取規格的過程就是深度理解的過程
3. **可驗證**：規格可以讓業務人員確認「這就是系統現在的行為」
4. **可追溯**：每一條規格都能對應到原始程式碼

### 1.3 OpenSpec + Legacy Code：逆向工程工作流

```
/Legacy Code 分析流程：

Step 1: onboard（導入）
  → AI 掃描整個 codebase，建立初步的系統理解

Step 2: explore（探索）
  → AI 深入分析關鍵模組，識別業務邏輯

Step 3: extract（萃取）
  → AI 產出結構化的行為規格（YAML/Markdown）

Step 4: validate（驗證）
  → 人類 Review 規格，確認是否正確反映系統行為
```

#### 規格文件格式（OpenAPI / YAML）

```yaml
# 從 Legacy VB6 萃取出的訂單處理規格
system: MES_Order_Processing
version: "1.0-legacy"
extracted_date: "2026-05-06"
source_module: "OrderProcessing.bas"

endpoints:
  - id: EP-ORD-001
    name: 建立訂單
    method: POST
    path: /api/v1/orders
    description: |
      從 Legacy CreateOrder() 函式萃取。
      原始邏輯在 OrderProcessing.bas 第 45-120 行。
    parameters:
      - name: customer_id
        type: string
        required: true
        validation: "必須為 8 碼數字"
      - name: order_items
        type: array
        required: true
        min_items: 1
        max_items: 500
    business_rules:
      - id: BR-ORD-001
        description: "單筆訂單金額上限 1000 萬"
        source_line: "OrderProcessing.bas:78"
      - id: BR-ORD-002
        description: "VIP 客戶享有 10% 折扣，需驗證 VIP 等級"
        source_line: "OrderProcessing.bas:92"
      - id: BR-ORD-003
        description: "庫存不足時自動建立補貨單"
        source_line: "OrderProcessing.bas:105"
    responses:
      - status: 201
        description: "訂單建立成功，回傳 order_id"
      - status: 400
        description: "參數驗證失敗"
      - status: 409
        description: "訂單號碼重複（冪等性保護）"

  - id: EP-ORD-002
    name: 查詢訂單狀態
    method: GET
    path: /api/v1/orders/{order_id}
    description: |
      從 Legacy GetOrderStatus() 函式萃取。
      原始邏輯在 OrderProcessing.bas 第 150-180 行。
    parameters:
      - name: order_id
        type: string
        required: true
        pattern: "^ORD[0-9]{10}$"
    responses:
      - status: 200
        body:
          order_id: string
          status: "enum: [PENDING, PROCESSING, SHIPPED, DELIVERED, CANCELLED]"
          created_at: "datetime"
          updated_at: "datetime"
```

### 1.4 Prompt 技巧：強制 AI 「只產規格不寫程式碼」

**關鍵**：在 Prompt 中明確限制 AI 的行為邊界。

```markdown
# 分析指令（放入 CLAUDE.md 或直接給 AI）

## 任務：Legacy Code 逆向工程

### 你的角色
你是一個系統分析師。你的工作是「閱讀」舊程式碼並「萃取」出精準的行為規格。

### 嚴格限制
- ✅ 你可以：閱讀程式碼、分析邏輯、產出 YAML/Markdown 規格
- ❌ 你不可以：撰寫任何新的程式碼
- ❌ 你不可以：建議如何重構或改善
- ❌ 你不可以：修改原始程式碼

### 輸出格式
產出 YAML 格式的行為規格，包含：
1. 每個 Function/API 的參數與回傳值
2. 所有業務規則（Business Rules），標注來源行號
3. 錯誤處理邏輯
4. 與其他模組的依賴關係

### 品質要求
- 每條規格必須能追溯到原始程式碼的行號
- 不要推測，只記錄程式碼「實際做了什麼」
- 隱含的業務邏輯必須明確標注為「隱含規則」
```

---

### 1.5 萃取品質的驗證

怎麼確認 AI 萃取的規格是正確的？

| 驗證方式 | 做法 | 適用場景 |
|---------|------|---------|
| **行號追溯** | 每條規格標注 source_line，人工抽查 | 所有場景 |
| **業務人員 Review** | 把規格給懂業務的人看 | 關鍵業務邏輯 |
| **測試案例對照** | 用現有的測試案例驗證規格 | 有測試的模組 |
| **Log 對照** | 比對規格與 production log 的行為 | 可觀測的系統 |
| **雙 AI 交叉驗證** | 用另一個 AI 獨立分析，比對結果 | 高風險模組 |

## 二、Demo 實演（20 mins）

### Demo 1：丟入肥大的 VB6 訂單處理邏輯

```bash
# 準備 Legacy Code 範例
cat > legacy_order.vb << 'VB6'
' OrderProcessing.bas
' 訂單處理模組 - 2008 年版本
' 作者：已離職

Public Function CreateOrder(ByVal custId As String, ByVal items As Collection) As String
    Dim orderId As String
    Dim totalAmount As Double
    Dim rs As ADODB.Recordset
    
    ' 檢查客戶是否存在
    Set rs = db.Execute("SELECT * FROM Customers WHERE CustID='" & custId & "'")
    If rs.EOF Then
        CreateOrder = "ERROR:CustomerNotFound"
        Exit Function
    End If
    
    ' 檢查 VIP 等級
    Dim vipLevel As Integer
    vipLevel = rs.Fields("VIPLevel").Value
    
    ' 計算總金額
    totalAmount = 0
    Dim i As Integer
    For i = 1 To items.Count
        totalAmount = totalAmount + items(i).Price * items(i).Quantity
    Next i
    
    ' VIP 折扣（ hardcoded 在這裡）
    If vipLevel >= 3 Then
        totalAmount = totalAmount * 0.9
    ElseIf vipLevel >= 2 Then
        totalAmount = totalAmount * 0.95
    End If
    
    ' 單筆上限檢查
    If totalAmount > 10000000 Then
        CreateOrder = "ERROR:AmountExceedsLimit"
        Exit Function
    End If
    
    ' 產生訂單號碼
    orderId = "ORD" & Format(Now(), "yyyyMMddHHmmss") & Int(Rnd * 10000)
    
    ' 寫入資料庫（直接 SQL，沒有參數化）
    db.Execute "INSERT INTO Orders (OrderID, CustID, TotalAmount, Status, CreateDate) " & _
               "VALUES ('" & orderId & "', '" & custId & "', " & totalAmount & ", 'PENDING', '" & Now() & "')"
    
    ' 檢查庫存並建立補貨單
    For i = 1 To items.Count
        Dim stock As Integer
        stock = db.Execute("SELECT Stock FROM Products WHERE ProdID='" & items(i).ProdID & "'").Fields("Stock").Value
        If stock < items(i).Quantity Then
            ' 自動建立補貨單
            db.Execute "INSERT INTO RestockOrders (ProdID, Quantity, Reason) " & _
                       "VALUES ('" & items(i).ProdID & "', " & (items(i).Quantity - stock) & ", 'AutoRestock')"
        End If
    Next i
    
    CreateOrder = orderId
End Function
VB6

# 讓 AI 萃取規格
claude
> 請分析這個 VB6 的 CreateOrder 函式（在 legacy_order.vb 中），
> 萃取出完整的行為規格，格式為 YAML。
> 
> 記住：你只能產出規格文件，不可以寫任何程式碼。
> 每條業務規則必須標注來源行號。
```

### Demo 2：觀察 AI 的分析過程

```
AI 的分析過程（預期行為）：

1. 讀取檔案 → 完整閱讀 legacy_order.vb
2. 逐行分析：
   - 識別函式簽名：CreateOrder(custId, items) → String
   - 識別參數驗證：custId 必須在 Customers 表中存在
```

## 三、Quick Win：課堂 10 分鐘動手（10 mins）

### 任務：丟一段 10 行 Function 給 AI，命令它只產出 YAML 規格

```bash
# Step 1：準備一段簡短的 Function（2 min）
cat > sample_function.py << 'EOF'
def calculate_discount(customer_type, purchase_amount, years_of_membership):
    if customer_type == "premium":
        discount = 0.15
        if years_of_membership > 5:
            discount += 0.05
        if purchase_amount > 10000:
            discount += 0.03
    elif customer_type == "regular":
        discount = 0.05
        if years_of_membership > 3:
            discount += 0.02
    else:
        discount = 0
    
    if purchase_amount > 50000:
        discount = min(discount, 0.25)  # cap at 25%
    
    return round(purchase_amount * discount, 2)
EOF

# Step 2：讓 AI 萃取規格（5 min）
claude
> 分析 sample_function.py 中的 calculate_discount 函式。
> 
> 嚴格限制：
> - ✅ 只產出 YAML 規格文件
> - ❌ 不可以寫任何程式碼
> - ❌ 不可以建議改善
> 
> 規格必須包含：
> 1. 函式的輸入參數與型別
> 2. 所有業務規則（含折扣計算邏輯）
> 3. 邊界條件
> 4. 回傳值格式

# Step 3：驗證結果（3 min）

# ✅ regular + 3年以上會員 → 額外 2%
# ✅ 總折扣上限 25%（金額 > 50000 時）
# ✅ 回傳值四捨五入到小數第二位
```

## 四、回家作業

### 作業：反向工程挑戰 — 萃取真實 MES 舊程式碼的規格

**目標**：找一段真實的 MES 舊程式碼（任何語言都可以），讓 AI 萃取出完整的行為規格。

#### 步驟

1. **挑選程式碼**：
   - 建議挑 50-200 行的函式或模組
   - 最好是有複雜業務邏輯的（不是純 CRUD）
   - 可以是 VB6、VB.NET、C#、Java、COBOL 任何語言

2. **準備 CLAUDE.md**：
   ```markdown
   ## 任務：Legacy Code 逆向工程
   
   ### 嚴格限制
   - ✅ 只產出 YAML 規格
   - ❌ 不寫程式碼
   - ❌ 不建議改善
   - 每條規則標注來源行號
   ```

3. **執行萃取**：
   ```
   claude
   > 請分析 [檔案路徑] 中的 [函式名稱]，
   > 萃取出完整的行為規格，格式為 YAML。
   > 記住你的限制：只產出規格，不寫程式碼。
   ```

4. **驗證**：
   - 逐條檢查規格是否正確
   - 標注 AI 遺漏的地方
   - 請同事（最好懂業務的人）Review

#### 繳交方式

1. 原始程式碼（或截圖）
2. AI 萃取的 YAML 規格文件
3. 驗證報告：
   - 正確的規則數 / 總規則數
   - 遺漏的規則（如果有）
   - 錯誤的規則（如果有）

#### 評分標準

| 項目 | 比重 |
|------|:---:|
| 程式碼的複雜度與真實性 | 20% |
| 規格的完整性（有沒有遺漏） | 35% |
| 規格的準確性（有沒有錯誤） | 25% |
| 驗證報告的品質 | 20% |

> ⚡ **重要**：這份 YAML 規格是第六堂課的輸入，請務必保留！

## 五、關鍵 Takeaway

> 💡 **Legacy Code 逆向工程的三大原則**：
> 1. **只讀不寫** — 先理解，再行動。規格萃取階段絕對不碰程式碼。
> 2. **追溯到行號** — 每條規格都必須能追溯到原始碼，確保不是 AI 的幻覺。
> 3. **業務人員驗證** — AI 萃取的規格必須讓懂業務的人看過，技術人員看不出邏輯錯誤。
