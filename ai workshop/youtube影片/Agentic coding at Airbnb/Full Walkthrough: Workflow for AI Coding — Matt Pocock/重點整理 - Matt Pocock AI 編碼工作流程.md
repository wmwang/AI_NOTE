# Matt Pocock 的 AI 編碼工作流程：從想法到交付的完整實戰指南

Matt Pocock 在這場兩小時工作坊中，展示了如何用結構化的工作流程讓 AI 成為你的高效編碼夥伴。核心觀念很反直覺：**AI 時代最管用的，反而不是什麼新發明，而是經典軟體工程基本功。**

## 兩個你必須理解的 LLM 限制

在談工作流程之前，你得先理解 LLM 的兩個硬限制，所有工作流程都是圍繞它們設計的。

| 限制 | 說明 | 實務影響 |
|------|------|----------|
| **聰明區 vs 笨蛋區** | 上下文超過約 100K tokens 後，LLM 的表現明顯下降，不管你用的是 200K 還是 1M 的上下文視窗 | 任務必須控制在聰明區內完成 |
| **像《記憶拼圖》的主角** | LLM 的記憶會隨對話增長而劣化，清除上下文後回到初始狀態 | 每次對話都要重新建立共識 |

說白了，你跟 AI 協作的每一步，都要思考「我現在還在聰明區嗎？」

## 完整工作流程總覽

```
想法 → 拷問對齊 → PRD（目標文件） → 看板（垂直切片） → AFK 實作 → QA/審查 → 團隊分享
 └── 人類在線 ──────────────────────┘                              └── 人類回來 ──┘
                                        └── AI 離線工作 ──┘
```

關鍵分工：**人類負責規劃和品質把關，AI 負責實作細節。**

## 第一步：拷問技能（Grill Me）— 用對話取代文件

面對一個新需求，Matt 的第一步不是進計畫模式，而是啟動「拷問技能」：讓 AI 針對需求的每個面向，一次問一個問題，直到雙方達成共識。

為什麼不直接寫規格文件？因為 Matt 反對「規格到程式碼」的思維——那本質上就是 vibe coding 的另一種形式。**你需要的是跟 AI 達成共同的設計概念（Design Concept），不是產出一份文件。**

一場拷問 session 可能持續 20 到 80 個問題，過程中你會發現很多自己沒想到的問題。例如：「點數應該回溯計算嗎？」——客戶沒想過，你也沒想過，但 AI 問了。

## 第二步：PRD — 不要讀它的目標文件

拷問結束後，用一個簡單指令讓 AI 把對話摘要成 PRD。PRD 包含問題陳述、解決方案、使用者故事、實作和測試決策。

重點來了：**Matt 通常不讀 PRD。** 因為他已經透過拷問 session 跟 AI 達成共識了，讀 PRD 只是在檢查 AI 的摘要能力——而摘要恰好是 LLM 最擅長的事。

## 第三步：看板 + 垂直切片 — 可平行化的任務拆分

比起順序的多階段計畫，Matt 偏好看板（Kanban Board），每張工單標記阻擋關係。這樣可以形成有向無環圖（DAG），支援多代理人平行處理。

### 水平切片 vs 垂直切片

| | 水平切片（AI 的偏好） | 垂直切片（你該要求的） |
|---|---|---|
| 做法 | 第一階段做所有 DB，第二階段做所有 API，第三階段做前端 | 每個切片都從 DB 穿透到前端 |
| 問題 | 到第三階段才能得到回饋 | 每個切片完成後就能端對端驗證 |
| 回饋速度 | 慢 | 快 |

概念來自《程式設計師修煉之道》的「追蹤彈」（Tracer Bullets）——就像防空砲的磷光子彈，讓你隨時知道瞄準方向對不對。

## 第四步：AFK 代理人 — 人類下班，AI 上工

到了實作階段，人類就可以離開鍵盤了。所有工單以 markdown 文件存在本地，代理人（在 Docker 沙箱中）會：

1. 選擇下一個優先任務（bug fix → 基礎設施 → 追蹤彈 → 拋光 → 重構）
2. 用 TDD 完成實作
3. 跑測試和型別檢查作為回饋迴圈
4. 產出 commit 和摘要

## TDD 為什麼對 AI 至關重要

AI 很會在測試上作弊——先寫完所有實作，再補一個「剛好會通過」的測試。TDD 的紅-綠-重構迴圈（先寫失敗測試 → 寫實作讓它通過 → 重構）讓作弊變得困難。

而且，**你的回饋迴圈品質決定了 AI 輸出的天花板。** 如果你的程式碼庫沒有好的測試和型別檢查，AI 基本上是在盲寫。

## 深模組 vs 淺模組 — 程式碼庫架構的關鍵

來自《軟體設計的哲學》的核心概念：

| | 淺模組 | 深模組 |
|---|---|---|
| 特徵 | 很多小檔案，各自匯出少量東西 | 少數大模組，簡單介面，豐富內部功能 |
| AI 導航 | 困難，依賴關係混亂 | 容易，介面清晰 |
| 測試 | 邊界難畫，不知該不該 mock | 一個大邊界就能測到很多 |
| AI 預設行為 | 如果不盯著，AI 會產出這種結構 | 需要刻意在 PRD 中指定 |

**如果你只能帶走一件事：** 在你的 repo 上跑「improve codebase architecture」技能，找出可以加深的模組。Matt 用這個方法改造了他的影片編輯器專案，讓 AI 的表現有天壤之別。

另外，深模組讓你能用一個聰明的策略：**你只管設計模組介面，把內部實作委託給 AI。** 這樣你保持對程式碼庫的掌控感（知道每個模組做什麼），同時大幅減少你需要親自寫和審查的程式碼量。

## Sandcastle：平行化代理人框架

Matt 自建的 TypeScript 工具，用來跑代理人平行迴圈：

1. **Planner** — 看 backlog，選出可平行處理的工單
2. **每個工單** → 獨立沙箱（Docker + git worktree）
3. **Implementer**（用 Sonnet）— 在沙箱中實作
4. **Reviewer**（用 Opus）— 審查程式碼
5. **Merger** — 合併分支，解決衝突

程式碼標準的執行策略：
- **實作階段用 Pull**：讓代理人需要時自行拉取標準
- **審閱階段用 Push**：主動推送標準給審閱者對照

## 程式碼審查的自動化技巧

AI 可以審查自己的程式碼，但要注意一個陷阱：**如果在同一個上下文中實作又審查，審查者會在笨蛋區運作。** 正確做法是清除上下文後再開一個新 session 做審查，讓審查者在聰明區運作。

## 關於文件保留的建議

PRD 和計畫文件要保留在 repo 裡嗎？Matt 傾向不保留，因為**文件腐爛（doc rot）**的風險很高——程式碼改了但文件沒更新，反而會誤導代理人。他 prefer 在 GitHub Issues 上標記為 closed，需要時可以查，但不會主動影響代理人。

## 下一步

1. 在你現有的 repo 上跑「improve codebase architecture」技能，看看你的模組結構有什麼改善空間
2. 下次有新需求時，試試「拷問技能」— 讓 AI 一次問你一個問題，看看你會發現多少自己沒想到的面向
3. 去買幾本經典軟體工程書：《程式設計師修煉之道》、《軟體設計的哲學》、《重構》— 這些 AI 時代之前的智慧，寫進 prompt 裡就是金礦
