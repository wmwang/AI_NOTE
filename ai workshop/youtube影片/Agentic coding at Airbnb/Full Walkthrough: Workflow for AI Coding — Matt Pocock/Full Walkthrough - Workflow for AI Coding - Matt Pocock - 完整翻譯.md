## LLM 的詭異限制

所以我想先談談 LLM 有哪些詭異的限制。這些詭異的限制就是我們很多工作必須圍繞的基礎。現在有一個叫 Dex Horley 的人，他經營一家叫 Human Layer 的公司，他提出了一個概念：當你使用 LLM 時，它們有一個「聰明區」（smart zone）和一個「笨蛋區」（dumb zone）。

當你第一次使用 LLM，就好像你剛開始一個新對話，你從零開始，那個時候 LLM 會做出它最好的工作。因為在那種情況下，注意力關係是最不緊繃的。每次你加一個 token 到 LLM 中，就有點像你加一支球隊到足球聯賽裡。你想想每加一支球隊，比賽場次怎麼增加——它是二次方增長的。這是因為你有注意力關係，從基本上每個 token 到其他 token，包含位置上的和個別 token 語義上的關係。

所以這意味著大約在 40% 左右，或者說，我現在的新指標是大約 100K。因為不管你用的是一百萬的上下文視窗還是 200K，這個臨界點總是大約在這裡。它就開始變笨了。所以當你持續往同一個上下文視窗塞東西，它就越來越笨，直到做出愚蠢的決定。覺得這個經驗很熟悉的請舉手。好。

這意味著我們需要以不超出聰明區的方式來規劃任務大小。我們不希望 AI 好高騖遠、貪多嚼不爛。而這其實呼應了老建議，像是 Martin Fowler 在《重構》裡說的、《程式設計師修煉之道》也談到這個——不要貪多嚼不爛。把任務保持在小規模，這樣你作為一個人類開發者才不會慌張，不會進入笨蛋區。

但你要如何處理大任務呢？你要怎麼把一個大任務——比如，我不知道，克隆一家公司或者做什麼瘋狂的事——拆成小任務，讓每個都塞得進聰明區？

一種方式，當然，你可以——我的意思是，也許是 AI 公司希望你做的，或者是自然而然的做法——就是一直做、一直做、一直做。你進入笨蛋區，每個請求收你一堆 token 費用，然後你壓縮回去。我們待會會好好談壓縮。然後你繼續、繼續、繼續，壓縮回去，繼續、繼續、繼續。我覺得這樣做效果不太好，因為沉積物會越來越多，我們待會會談這個。

所以後來的理論是——這也是我做了一段時間的做法——就是我會使用這種多階段計畫，我會說，好，我們有這個第四號東西，這個很大的任務。讓我們把它拆成小段，這樣我們就能一塊一塊地在聰明區裡做每件小工作。用過多階段計畫的請舉手。對，非常常見的做法，對吧？這基本上就是我們一直在做的。我至少一直到去年十二月都是這麼做的。

而任何有經驗的開發者看到這個都會說：這是一個迴圈嘛，對吧？這是一個迴圈。我們就是第一階段、第二階段、第三階段、第四階段。為什麼不直接有第 N 階段？第 N 階段，基本上我們就說，好，我們有一個在背景運作的計畫，然後我們在上面不斷迴圈，直到完成。

這就是——聽過 Ralph Wiggum 作為軟體實踐的請舉手。好，這個多一些了。好。所以有一個叫 Ralph Wiggum 的概念，基本上是建構在這個想法上的，就是：你只需要指定旅程的終點，你就說，好，我們建立一個 PRD（產品需求文件），說明我們要去哪裡。然後我們就對 AI 說，做一個小改動，做一個小改動，讓我們越來越接近那個目標。Ralph 還行，但我偏好多一點結構。所以這就是我們在思考聰明區方面得到的結論。這也是我希望你們首先開始思考的地方。

## LLM 的第二個詭異限制：像《記憶拼圖》的主角

LLM 的另一個詭異限制是，它們有點像電影《記憶拼圖》（Memento）裡的主角，對吧？它們就是不斷遺忘。它們就是不斷重置回基礎狀態。

讓我拉出這個圖表。我真的應該用投影片，但我就是比較喜歡在無限的 TLDraw 或 Canvas 上隨意滾動。謝謝 Steve。

所以我想讓你們有的另一個概念是：每一個跟 LLM 的對話階段基本上都會經歷相同的階段。首先，你有系統提示詞（system prompt），這個灰色方塊代表永遠在你上下文中的東西。你希望它越小越好，因為如果你塞了一堆東西在這裡——我看過有人塞 250K tokens——那你就連開始都沒開始就直接進入笨蛋區了。所以你希望它很小。

然後你進入一個探索階段（exploratory phase）。這些藍色的部分是程式設計代理人出去探索程式碼庫。然後你進入實作階段（implementation），然後進入測試和確保它能運作的階段，跑你的回饋迴圈之類的。覺得這個經驗很熟悉的請舉手。好，基本上就是任何對話階段的主要里程碑。

當你清除上下文時，你會直接回到系統提示詞。你會直接回到那裡。你刪除了之前所有的內容。

聽過「壓縮」（compacting）的請舉手。好。有一些人沒聽過。讓我快速展示一下那是什麼意思。比如，我一直在跟我的 LLM 聊天。我想確保我們——你們知道，就覆蓋一下基礎知識，讓大家都在同一個頻率上。我一直在跟我的 LLM 聊天，我在聊一個我想建造的東西。字型大小——我應該放大嗎？後面的朋友，放、放、放、放、放。我這個 session 用的是 Claude Code，但你不需要用 Claude Code。事實上，不用 Claude Code 通常也很好。

所以我一直在跟 LLM 聊天，就是在推演我接下來要做什麼。它問了我一堆問題，我可以——我強烈建議你們這樣做。這裡有一個小小的狀態列，告訴我我正在用多少 tokens，確切的 token 數量。我在我的網站 AI Hero 上有一篇文章，如果你想複製這個的話。這是——哇，那東西會震動，是吧？這是每個程式設計 session 中絕對必要的資訊，因為你需要確切知道你用了多少 tokens，這樣你才知道你離笨蛋區有多近。絕對必要。

所以我們來看看。我有兩個選項。我可以清除——嗯，錯了——然後回到空白，或者我可以壓縮。當我壓縮時，它會把所有那個對話——雖然說實話那不是很多——擠壓成一個更小的空間。

用圖表來看，這大概是這樣的。我們把所有 session 的資訊拿出來，你基本上用它建立一個歷史記錄，一份發生了什麼事的書面記錄。

開發者不知道為什麼很喜歡壓縮，但我討厭它。我更喜歡我的 AI 表現得像《記憶拼圖》的主角，因為這個狀態永遠是一樣的，永遠一樣。每次你這樣做，你清除，然後回到起點。所以如果你能這樣做，如果你能最佳化這個模式，那你就處於一個很好的位置。

所以這就是我希望你們思考的關於 LLM 的兩件事，我們要應對的兩個限制。它們有聰明區和笨蛋區，而且它們像《記憶拼圖》的主角一樣。

## 第一個練習：Grill Me Skill（拷問我技能）

所以我們來看第一個練習。在我做這個的同時，我希望的運作方式是我會在上面展示我怎麼走過這個流程，然後我希望你們在下面也跟著操作。

所以剛才是一小段演講。現在讓我們實際開始寫一些程式。遲到的人，還有 Gilgood 房間的任何人，去這個連結——上面這個連結——看練習並 clone repo。你絕對不一定要這樣做。你可以只是看我做，如果你想的話。但讓我自己先去看看有什麼練習在等我們。

基本上我建了一個——這是從我的課程來的。這是一個課程管理平台，基本上是一個給講師、給學生用的 CMS，這就是我們要建功能的平台。所以我會帶你們從功能想法開始，一路到建 PRD，一路到實作功能。希望你們可以從這個過程中獲得靈感，在自己的工作中使用。

我們開始吧。我們要從使用一個我很重視的技能開始。就是拷問我技能（grill me skill）。這個拷問我技能非常小、非常簡潔，它幫助防止一個——我認為——跟 AI 協作時的主要問題，就是不對齊（misalignment）。

我在這裡反對的潛在觀念，我反對的是「規格到程式碼」（specs-to-code）運動。有人聽過規格到程式碼運動嗎？這其實不算是一個運動，只是人們在說規格到程式碼。它的意思是：人們說，好，你要寫一個程式，或者你想建一個 App。建那個 App 最好的方式就是拿一些規格文件，寫某種文件，然後把那個文件變成程式碼。就把它變成程式碼。你怎麼做？你把它傳給 AI。如果產出的程式碼有問題，你不看程式碼。你改規格，然後繼續這樣做。這本質上就是 vibe coding 的另一個名字，你基本上就是在忽略程式碼。你不需要擔心程式碼。你只需要一直改規格，然後就一直繼續。

我試過了。我真的試過了。但它很爛。它不管用。因為你需要持續掌握程式碼。你需要理解裡面是什麼。你需要塑造它，因為**程式碼就是你的戰場**。

所以我們就是要走向這個方向。讓我們來做一些練習。我希望你做的是去這個頁面，拷問我技能的頁面。在 repo 裡面，我們有一封來自我們朋友的 Slack 訊息。在哪裡呢？在 repo 的根目錄。它叫做——在哪裡呢？是來自 Sarah Chen 的 Slack 訊息。不知道為什麼 Claude 總是選 Sarah Chen。上面說在我們的課程平台 Cadence 中，我們的留存數字不太好看。學生註冊了幾堂課就退出了。我很想加一些遊戲化功能到平台中。

當你面對這樣的一個想法時，你需要找到方法把它變成現實。假設 Sarah Chen 是你的客戶。你有時間預算。你需要快速完成。你怎麼做？你會進計畫模式的請舉手。有人是計畫模式的重度使用者嗎？我們快速喊出其他想法，你會怎麼做。你會——你的第一個流程是什麼？請舉手。嗯？對？對，沒錯。讓我們想像 Sarah Chen 已經去度假了。你毫無頭緒，對吧？她就是發了這個東西。你需要在離開前處理它。

我的第一個流程是我會用這個特定的技能。我要清除我的上下文。我要把你去掉。你不需要在那裡。然後我要說我要呼叫一個技能，就是拷問我技能。讓我快速確認一下。不知道這是什麼的請舉手。好。抱歉，抱歉。讓我更具體一點。不知道我在做什麼的——當我在這裡打一個斜線然後打字——不知道這是什麼的人請舉手。有人——大家大概都理解這是什麼。我在呼叫一個技能。我在呼叫拷問我技能。

然後我要做的是說「拷問我」，然後把客戶簡報傳進去。所以現在 LLM 真的只有兩個東西：它只有技能，和我想做什麼的描述。這幾乎就是我如何開始每一項跟 AI 有關的工作的方式。

在它探索程式碼庫的同時，我就來展示拷問我技能是做什麼的。這在 repo 裡面，所以你可以查看。它超級短。「針對這個計畫的每個面向，毫不留情地拷問我，直到我們達成共識。沿著設計樹的每個分支逐一走下去，針對每個問題提供你的建議答案。一次問一個問題。啦、啦、啦。」

這個技能做的是什麼，以及我注意到在跟 AI 協作時——尤其在計畫模式中——它會非常急切地想幫我產出計畫。它會說：「好，我覺得我懂了，我直接規劃吧。」而我發現我真正在找的是——我真的很難找到詞來形容——我在找的是 Frederick P. Brooks 在《設計的設計》（The Design of Design）中談到的「設計概念」（design concept）。當你跟別人一起做新東西時，當你們都在試著一起建造某個東西時，所有參與者之間有一個共享的想法。那就是設計概念。而我意識到我需要跟 Claude 達成的就是這個。我需要達成共同理解。我需要的不是一份文件。我需要的不是一個計畫。我需要跟 AI、跟我的代理人在同一個頻率上。而這是一個非常非常有效的方式。

好，希望——好了，好。它已經完成探索了。首先，它呼叫了一個子代理人（sub agent），這個子代理人在 Opus 上花了 97——93.7K tokens。然後它問了我第一個問題。好。我們可以看到，雖然子代理人消耗了一堆 tokens，我實際上並沒有增加多少 token 使用量。

不知道什麼是子代理人的請舉手。這是個重要的問題。大家都大概清楚什麼是子代理人嗎？好，我簡短定義一下：這個子代理人——這個 explore 子代理人——基本上就是去呼叫了另一個 LLM，那個 LLM 有一個隔離的上下文視窗。然後那個 LLM 回報了一份摘要。所以子代理人有點像一種委派。你把任務委派給子代理人。它很積極地去做了所有事情，探索了一堆東西，然後只把重要的東西一點一滴地回傳給上層的編排代理人（orchestrator agent）、父代理人。

好。希望你們看到了同樣的東西。它做了探索。然後我們有了第一個問題。「點數經濟：什麼行為可以獲得點數，多少點？」好。在這個時候，順帶一提，你可以問它問題來加深你對 repo 的理解。我顯然對這個 repo 很熟，因為是我寫的，但你可能不知道發生了什麼事。

所以比如說我的建議——保持簡單，先用兩個點數來源。這件事多棒的地方是，它不僅給了我一個問題來讓我們對齊，我們還得到了一個建議。而我經常發現 AI 的建議真的很好。所以我就說：「跳過影片觀看事件，它們很吵而且可以被操縱。」「我同意。」「Sarah 要求的——嗯，課程完成是基本盤。」「對。看起來不錯，老兄。」

通常我會用口述的方式跟 AI 對話。通常我實際上是跟 AI 用講的而不是用打的，但這是一台相對新的筆電，而且我沒辦法讓我的聽寫軟體在上面運作，因為 Windows 很爛。

「點數應該回溯計算嗎？現有的課程進度記錄有完成的時間戳。」這是一個很棘手的問題，對吧？我們真的應該回去把所有的課程進度事件都回溯填入嗎？這是一種你在正確實作功能前需要對齊的問題。這不是我想過的問題，Sarah Chen 當然也沒想過。我要讓它回溯計算嗎？嗯。讓我們在這裡做個投票。我們應該回去把所有記錄都回溯填入嗎？認為應該回溯的請舉手。認為不應該回溯的請舉手。現場有很多騎牆派。我要說——你知道，這就是你跟 AI 在進行的那種討論。你越來越對齊了。對，我就直接採用它的建議。我很懶。

現在，你可能會說——我們為什麼不把這個 issues 的生成過程也交給 AI？為什麼我需要參與？因為它給了我們一個不錯的任務組合。為什麼我需要審查這個，然後搞清楚下一步？

我的看法是，這個做起來很便宜——一旦 PRD 做完之後非常快。而且我可以立刻看到一些問題。

這裡有一個非常重要的技巧，當你在搞清楚這段旅程應該長什麼樣子的時候。它來自一個非常經典的概念，來自《程式設計師修煉之道》，叫做**追蹤彈（Tracer Bullets）**，或**垂直切片（Vertical Slices）**。追蹤彈真的徹底改變了我對讓 AI 自己選任務的思考方式。

系統有層次（layers），對吧？你的系統有層次。這些可能是不同的可部署單元（deployable units）。你可能有一個資料庫在某個地方。你可能有一個 API 可能離資料庫比較近但在不同部分。你可能有一個完全在不同地方的前端，像 CDN。或者在這些可部署單元裡面，你可能還有不同的層次。例如，在我們正在工作的程式碼庫中，我們有一堆不同的服務。我們有測驗服務（quiz service）、團隊服務（team service）、使用者服務（user service）、優惠券服務（coupon service）、課程服務（course service）。這些服務彼此有依賴關係。所以它們就像是個別的層次。

我注意到的是，**AI 很喜歡水平編碼**。它很喜歡一層一層地寫。換句話說，在第一階段它會做所有資料庫的東西，所有 schema，所有跟那個單元相關的東西。然後進入第二階段做所有 API 的東西。然後再把前端加在上面。

## TDD：測試驅動開發

我發現 TDD 對於善用代理人來說是**絕對必要的**。知道 TDD 是什麼的請舉手。好。TDD 是測試驅動開發（Test Driven Development）。它本質上做的是一件事叫做紅-綠-重構（red, green, refactor）。如果你看程式碼庫，你可以找到一個技能詳細描述了怎麼做紅-綠-重構，並教 AI 怎麼做。

所以它做的是先寫一個失敗的測試。它說，好，我已經拆解了我正在做的事情的概念，然後我就寫一個會失敗的測試，然後我需要讓實作通過。我發現，首先，這為程式碼庫增加了測試，而且它傾向於增加好的測試。

所以我們有了這個遊戲化服務。看起來它用了一些現有的東西來建立測試資料庫。測試失敗了，因為模組還不存在。好。我們確認了紅燈（red）。然後它去——希望跑起來然後通過了。

我發現——有被 AI 寫過爛測試的請舉手。對。它傾向於在測試上作弊，因為它基本上是用分層的方式在做。它會先做整個實作，然後再做整個測試層。我就說「對，你可以用 npx test」。用這個技巧，它基本上更難作弊了，因為它是在寫程式碼之前先裝設了監測儀器。所以我發現 TDD 在你能實施的地方是超級超級好的。事實上它好到我把整個技術的設計都圍繞在讓 TDD 更好運作上。

## 深模組 vs 淺模組：程式碼庫架構

所以在最後這個部分，我要做兩件事：第一，我要告訴你——你腦中可能有一個問題：假設我有一個正在工作的程式碼庫，而且它是個爛程式碼庫。它是個超級複雜的程式碼庫。AI 就是永遠做不好，也許實際上大多數人類進入那個程式碼庫也做不好。我怎麼改善那個程式碼庫？第二件事是我會展示我的平行化設定。讓我們先談爛程式碼。

在哪裡呢？圖表在哪裡？在這。在《軟體設計的哲學》（A Philosophy of Software Design）這本書中，John Osterhout 談到了理想的模組類型。讓我們想像你有一個看起來像這樣的程式碼庫。這裡每一個區塊都是個別的檔案。這些檔案從裡面匯出東西。它們有你從檔案中拉出來然後在其他地方使用的東西。所以你可能會有這些奇怪的依賴關係，比如這邊的這個檔案可能依賴這個檔案，或者可能依賴那個檔案。

如果這些檔案很小，而且它們不匯出很多東西，那 John Osterhout 會把這些叫做**淺模組（shallow modules）**——基本上它們不怎麼樣。它們大概長這樣。如果我——實際上——我沒辦法畫個好圖。它們基本上就是很多很多小塊。

這對 AI 來說很難導航，因為它不太理解所有東西之間的依賴關係。它沒辦法搞清楚所有東西在哪裡。它必須手動追蹤整個依賴圖，然後說：好，這個依賴這個，這個依賴這個，這個依賴這個。而且這也很難測試，因為你要在哪裡畫測試邊界？你為每個模組單獨測試嗎？比如就真的畫一個測試邊界？不，不要那樣做。繞著這個。然後也許下一個再畫一個測試邊界。然後再下一個。還是你應該做大群組？你說，好，我們把所有這些相關模組放在一起測試，然後就——祈禱它們能運作。

這意味著——如果我認為爛測試大多長那個樣子——AI 基本上試著把每個小函式包裹在它自己的測試邊界中，然後就測試那些個別是否運作。但這樣做的問題是，當——比如說——這邊這個模組呼叫了那兩個，所以它依賴這兩個，那這個模組可能把函式的順序搞錯了，或者在那個可憐的模組裡面可能有一些值得單獨測試的東西。然後如果你用測試邊界把這個包起來，你怎麼辦？你要 mock 另外兩個模組嗎？那怎麼運作？

所以實際上搞清楚如何建一個容易測試的程式碼庫在這裡是必要的，因為如果我們的程式碼庫容易測試，那我們的回饋迴圈就會更好，AI 在我們的程式碼庫中就會做更好的工作。這說得通嗎？

那好的程式碼庫長什麼樣？嗯，不像那樣。它長這樣——你有 John Osterhout 所說的**深模組（deep modules）**——模組上面有一個小介面，暴露出一個簡單的小介面，但裡面有大量的功能。這意味著這些很容易測試，因為你就——比如這一個和這一個之間有依賴——我能用嗎？好了——然後你就在那個模組外面畫一個大測試邊界，繞著上面這個，然後你就能抓到很多好東西，因為你在測試大量功能。而且呼叫者——呼叫模組的人——會有一個簡單的介面可以使用，所以不會太棘手。這說得通。深模組對淺模組。這個是好的。淺的那個版本是壞的。

而我發現的是，如果不受監督，或者如果你不盯著 AI，它就會產出長那樣的程式碼庫。所以你在指導它時需要非常非常小心。這也是為什麼——對——如果我們看 PRD 裡面——PRD 去哪了？它在 issues 裡面。在遊戲化系統裡面。找不到。當然找不到。在這。然後我在裡面有 data model the modules。它特別寫了：「好，這個遊戲化服務是一個新的深模組，我們要圍繞它測試。它會有這個特定的介面。」然後——好，我們也要修改 progress service。我們要修改 lesson route，修改 dashboard route，等等。所以我對我要編輯的模組非常具體，而且我確保在整個規劃過程中以及實作過程中，我都把那個模組地圖放在腦中。這說得通。非常有用。

它還有另一個用處。不只讓你的 App 更容易測試，你還可以做一個小心理技巧。我要去裝水——你們等一下看看是什麼。

讓我問你們一個問題。覺得你在 AI 時代比以前工作更辛苦的請舉手。對。覺得你對你的程式碼庫了解程度不如以前的請舉手。對。

這是一個真實的問題。因為我們跑得快，因為我們委派了更多東西，我們最終失去了對程式碼庫的感覺。而如果我們失去了對程式碼庫的感覺，我們就沒辦法改善它，我們本質上就是把它的形狀委派給 AI 了。我認為那是不好的。但我們又怎麼能快速推進，同時腦中還保留足夠的空間？

我認為這是一個方法。因為你在這裡做的不只是思考在你的程式碼庫中建立大形狀——大服務。我認為你應該做的是**設計這些模組的介面，然後委派實作**。換句話說，這些模組可以變成像灰盒子（gray boxes），你只需要知道它們的形狀，你需要知道它們做什麼、大概怎麼運作。但你可以委派那些模組的實作。我發現這非常棒。我不一定需要程式碼審查模組裡面的每一樣東西。我不一定需要知道它做的每一件事。我只需要知道它在某些條件下有特定的行為，它做它該做的事。

所以就有點像——好，我對我的程式碼庫有了一個大的概覽，我了解裡面的形狀，了解所有介面做什麼，但我可以委派裡面的內容。我發現這是一個非常好的方式來保持對程式碼庫的感覺，同時保持我的理智。這說得通嗎？

所以你可能會問，我怎麼把一個看起來像那樣的程式碼庫變成看起來像這樣的程式碼庫？我怎麼把模組加深？好，我們有——希望它在這裡——我很確定有——我們有一個技能。那個技能叫做「improve codebase architecture」（改善程式碼庫架構）。很直接。讓我們跑跑看。

這個技能要做的事基本上就是掃描我們的程式碼庫，看看有什麼可以用的。如果你在跑練習的話也可以自己跑跑看。它在探索架構，探索基本上是如何在這個程式碼庫中運作。然後它要嘗試找出可以加深模組的地方。很簡單。

它找到的一個很酷的東西是——我的 course video manager App 的一部分是一個影片編輯器，一個建在瀏覽器裡的影片編輯器，這非常硬核。這是相當不錯的工程。我想要一個方法可以把整個前端一直到後端包在一個大的模組裡，這樣我就可以測試「我在前端按了某個東西，然後它一路傳到後端」。然後我找到了一個方法——基本上透過在兩個型別之間使用一種可區分聯集（discriminated union）——我能用這個技能建立一個超級大的模組，從外面測試那個影片編輯器基礎設施。這意味著 AI 能看到整個流程，能對整個流程操作，能測試整個流程。說實話，在 AI 實際做修改的能力上簡直是天壤之別，因為如果你不給它好的測試，AI 在影片編輯器上工作是相當殘酷的。

所以——說實話，如果你今天只帶走一件事，就是試著在你的 repo 上跑這個技能，看看會發生什麼。

讓我們去 Slido。來看看幾個問題，同時這個在跑。

「你有試過 Claude 的 auto mode 嗎？就是 `claude enable auto mode`？這樣你可以避免很多顯而易見的權限檢查。」我們待會會談權限檢查。

「我會保留 markdown 計畫和 issues 供日後參考嗎？」好，這是個好問題。假設你有一個好想法，你把它變成 PRD，然後你實作了那個 PRD，而 PRD 本質上已經完成了。把那份資訊保留在 repo 裡的請舉手。所以你把它變成 markdown 文件，想保留的請舉手。好。想盡快丟掉的請舉手。對，我認為這是一個沒有明確答案的問題。我對任何文件決策真正害怕的是——假設我們有一個遊戲化系統的 PRD，我們把它留在 repo 裡。然後我們繼續、繼續、繼續。假設一個月後，我們想對遊戲化系統做一些修改，然後我們用 Claude 進去，它找到了這個舊的 PRD，然後說：「好，我找到了 PRD 系統的原始文件。」結果實際程式碼跟原始 PRD 已經差到幾乎認不出來了。東西的名稱變了，檔案結構變了，甚至需求可能都變了——我們可能實際上已經跟使用者測試過了。這就是**文件腐爛（doc rot）**——某個東西的文件在 repo 裡慢慢腐爛，而且對 Claude——或代理人——產生不良影響。所以我傾向不保留。我傾向丟掉。對我來說，因為我的設定用的是 GitHub issues，我就把它標記為 closed。它想抓的話還是可以抓，但它有一個視覺指示器顯示它已經完成了。所以我傾向丟掉這些。

「對 Steve 的 beads framework 有什麼看法？」我沒測試過，但看起來像是另一種管理看板和 issues 的方式，看起來很好，但我沒試過。

讓我快速確認一下設定。從現場拿幾個問題。到目前為止有人對我們涵蓋的任何內容有問題嗎？尤其是最後這段？好。

「我希望這回答了你的問題。很抱歉。不不，我認為資料庫 migration 是不同的東西，因為你有一個確切改了什麼的連續記錄，而且它更確定性。而且我認為——對，這是個有趣的類比。我不確定。我們之後再聊。這是說我不知道的好方法。對。對。抱歉大家，我在試著聽這位先生的問題。對，問題是——在早期的規劃階段——我應該試著最佳化計畫嗎？」

這其實是我看到很多人在做的事情，而且是個好想法。所以當你——我們回到那些階段。假設你有所有這些階段，然後你到了跟 LLM 搞清楚一切的那個點。你知道你要去哪裡。你建立了這些旅程目標文件。然後你應不應該試著最佳化、最佳化、最佳化那個 PRD，直到它變成你能想像到的最完美的 PRD？我認為這沒有太大的價值，因為我認為那份旅程真的只是一個你想去哪裡的提示。你需要放功夫的地方是 QA。你可以——我想——AFK 做那個，但根據我的經驗，你不會從中得到太多東西。真正重要的是跟 AI 達成對齊，這就是你一開始在拷問 session 中做的事。再一個問題。然後我還有更多。好。

「這個問題我們之前其實問過了，就是——你怎麼把你的程式碼標準執行在代理人身上？你怎麼讓它按照你要的方式寫程式碼？」

基本上有兩種不同的方式。你有推（push）和拉（pull）。什麼是推和拉？推就是你把指令推給 LLM。所以你說，好，如果你在 claude.md 裡放了「講話像海盜」，那個指令就會一直被送到代理人那裡。對吧？所以那就是推——實際上你在推送 tokens 給它。拉是你給代理人一個拉取更多資訊的機會。那就是——比如——技能。技能是放在 repo 裡面的東西，它有一個小描述標頭說：「好，代理人，你想要的時候可以拉取這個。」

所以我目前對程式碼審查和程式碼標準的思考是這樣的。當你有一個實作者——怎麼——好了，實作者——我等一下把紅色減少——那你希望程式碼標準透過拉的方式提供。如果它有問題，你希望它能回答。但如果你之後有一個自動化審查者，那你就要推。你要把那個資訊推給審查者。你要說「這是我們的程式碼標準。確保這個程式碼遵循它們。」所以如果你有技能的話，你要把那些東西推給審查者，這樣審查者就有了寫出來的程式碼和程式碼標準可以對照比較。

希望這回答了你的問題。我其實也可以展示一個自動化的版本。讓我在記憶猶新的時候做這個。

## Sandcastle：平行化代理人框架

我最近花了大概一週左右建了一個叫 Sandcastle 的東西。我對市場上跑代理人 AFK 的選項不太滿意。它做的事情是——它本質上是一個跑這些迴圈的 TypeScript 函式庫。所以你有一個 `run` 函式，它建立一個 worktree、在 Docker 容器中做沙箱，然後讓你在裡面跑一個 prompt。在那個 worktree 裡面——它就是一個 git branch——你有那個程式碼，然後你之後可以合併它。

如果我打開——有一些非常非常好的觀看方式——它本質上讓你跑這些自動化迴圈，而且讓你非常簡單地跨多個代理人平行化。所以我進入我的 Sandcastle 文件，進入 main.ts，讓我們走過一遍。

這有點像——我之前給你看了一個 Ralph 迴圈的版本。這是我們從順序變成平行的部分。我們這裡有一個規劃器（planner），它接收——它有一個 plan prompt，它看 backlog 然後選擇一定數量的 issues 平行處理。還記得我給你看的那個有所有阻擋關係的看板嗎？它計算出所有的階段。所以這個會說，好——假設我們有——你可以忽略這些膠水程式碼，這基本上就是一組 issues——GitHub issues，有標題和要工作的 branch。然後對每個 issue，我們建立一個沙箱，然後在沙箱裡跑一個實作者（implementer），傳入 issue 編號、issue 標題和 branch。這就像我們剛才跑的那個迴圈。然後如果它建了一些 commits，我們就審查那些 commits。這基本上就是迴圈。我們對那些 commits 做什麼？我們把它們傳進一個合併代理人（merger agent），它接收 merge prompt、建出來的 branches、issues，然後把它合併進去。如果合併有什麼問題——比如型別和測試之類的——它會解決。

這已經是我在大多數專案中工作相當一段時間的流程了。它超級超級好用。我推薦你查看 Sandcastle 如果你想了解更多。然後正確回答你的問題：在審查者（reviewer）中，我會推送程式碼標準。在實作者中，我會讓它拉取。而我實際上是用 Sonnet 做實作，用 Opus 做審查，因為我認為審查——那時候我需要更多智慧。

有任何問題嗎？讓我在——做更多問題之前。讓我們回到這裡。好。我們在哪裡？好。我們在這場演講中到處跳，因為我有點必須平行跑東西。讓我們回到改善程式碼庫架構。它終於跑完了，然後它找到了一堆架構改善候選。它基本上有一組不同模組的群集，它們都有關聯，可能可以作為一個單元來測試。有一號——測驗計分服務（quiz scoring service）。還有一些重排邏輯的提取。它有論述說為什麼它們是耦合的。它還有依賴類別。所以——本地可替換的 SQLite in-memory testdb。測驗計分服務目前有零個測試。這是最大的缺口。所以這就是改善程式碼庫架構回傳的結果。

## 總結

好。我們大概剩下 17 分鐘。我不知道你們怎麼樣，但我累死了。我來幫你們做個總結，因為我覺得我們都快要到體力的極限了。如果你們想來問我問題，我會待到最後。我可能再查一次 Slido，但讓我們總結一下我們到了哪裡。

所以這基本上就是整個流程。在整個過程中，我們一直把程式碼庫的形狀放在心上。這不是一個規格到程式碼的編譯器。這不是一個就只是噴程式碼的 AI。我們對想要的模組類型和程式碼庫形狀非常有意識。我們透過拷問 session 確保我們盡可能對齊，真的把想法錘鍊出來。我們不會過度執著於 PRD。我們不試圖讀它的每一個部分。我們甚至不太深入思考它。然後我們把它變成一組可平行化的 issues，讓代理人可以平行處理。我們實作它，然後我們狠狠地 QA 和程式碼審查，然後不斷回到那個實作。

有一件事我沒怎麼提到的是，在 QA 階段，QA 階段的目的是為看板建立更多 issues。所以甚至在它實作的時候，你就可以在 QA 那些東西，然後回去加更多 issues，而看板讓你可以無限地加入阻擋 issues。一旦全部完成了，一旦你有了你滿意的程式碼，有了你滿意的工作，然後你就可以分享給你的團隊，做完整的審查。到了這裡，這基本上是一個開發者或也許兩個開發者在管理這個，然後就看你要怎麼合併回去了。
