# Airbnb 代理式開發 (Agentic Coding at Airbnb) 演講總結與翻譯

這篇文章整理自 Airbnb 開發者平台團隊（Szczepan 與 Mike）關於「Airbnb 代理式程式開發 (Agentic Coding)」的演講內容，分為兩大部份：重點整理以及完整的原文翻譯。

---

## 第一部分：重點整理 (Key Takeaways)

### 1. 代理式開發的願景與崛起
* **重新定義結對程式設計：** 在傳統開發中，人類工程師分別扮演控制鍵盤的「駕駛員」與負責戰略的「導航員」。在代理式開發時代，開發者成為「永遠的導航員」，負責引導對話，而 AI 代理 (Agent) 則負責實際產出程式碼。
* **驚人的採用率：** Airbnb 高層原本保守預估 2025 年初的採用率可能在 20% 到 40%，但實際採用率已迅速攀升至 64%，這是一條幾乎呈垂直飆升的成長曲線。
* **並行工作流的魔法：** 進階開發者甚至會同時開啟多個（如 5 個）「沙盒工作區 (Sandbox Workspaces)」，讓多個 AI 代理平行執行不同的開發任務。未來的工程師將不再只依賴單一任務的「心流 (Flow)」，而是必須精通「切換上下文 (Context Switching)」與「程式碼審查 (Code Review)」。

### 2. 策略演進與核心技術：Airchat 與 MCP
* **從「IDE 優先」到「擁抱 CLI 代理」：** 初期 Airbnb 開發了自家整合內部知識庫的 IDE 外掛，但發現外部開源的 CLI 代理工具（如 Aider, OpenHands, Gemini 等）發展過於迅速且具備強大的「代理編排 (Agentic Orchestrations)」能力。為融合兩者優勢，他們開發了抽象包裝層 **Airchat**。
* **全面導入 MCP (Model Context Protocol) 標準：** MCP 被視為「LLM 的 REST API」。Airbnb 捨棄了昂貴且難以維護的模型微調 (Fine-tuning)，改以「**有工具呼叫能力就足夠了 (Tool Calling is all you need)**」為核心理念。透過將內部知識庫、CI、分析平台等包裝成 MCP Servers，LLM 代理得以在沒有微調的情況下，自動檢索並套用 Airbnb 的私有上下文與設計模式。

### 3. 文化推廣與六大實戰教訓
* **在開發者所在的地方與他們相遇：** 不強迫開發者改變習慣的工具（不論是 IDE 派還是 CLI 派），而是降低門檻，提供一鍵安裝的代理環境。
* **六大戰場教訓 (War Stories)：**
  1. **需要眾人之力 (It takes a village)：** 打造 AI 開發環境不能只靠單一小團隊，必須建立社群讓各部門專家共同貢獻工具與 MCP 模組。
  2. **不要低估 CLI：** 強大的代理工具讓 80% 的工程師重新擁抱了命令列介面，因為其代理體驗已經超越了傳統的 IDE。
  3. **移除障礙：** 讓啟動代理對話的過程越簡單越好，降低學習與切換成本。
  4. **推動標準化，但別盲目跟風：** AI 市場變化極快，有時等技術稍作收斂後再進行深度投資會更有效率。
  5. **測量所有事物：** 必須持續以質化（情緒回饋）與量化（PR 速度、採用率）的方式追蹤生產力變化。
  6. **不要對 AI 降低標準：** 現代代理工具的能力已經可以與資深工程師匹敵，必須維持並要求同等嚴格的程式碼審查與測試標準。

## 第二部分：完整原文翻譯

我們將向您介紹 Airbnb 的「代理式程式開發 (Agentic Coding)」。所以，我們不是探討 AI 能做的所有事情，而是專門針對代理式寫程式。我是 Szczepan，和 Mike 一起。我們是 Airbnb 開發者平台團隊的工程師，負責為工程師打造工具並整合這些驚人的 AI 技術。我開發工程師工具已經近二十年，也參與過開源專案，或許您還用過其中一些。我對生成式 AI (GenAI) 演進帶來的發展感到驚嘆，至今我從未見過如此巨大的生產力成長，真的很驚人。

在 2025 年初，我們設定了這個願景。這也是我們向高層解釋團隊發展方向的方式。我們喜歡的做法是，尤其是當我們要處理某些大事、做出階躍函數般的巨大改變時，我們會先設定一個大膽的願景，然後思考如何從這個願景反推回來。這很有幫助，因為如果你只考慮漸進式的改善，就很難做出巨大的改變。當時我們覺得這個願景很大膽，現在看來大概已經成為常態了，對吧？我們看到了模型和工具的快速進步。這就是我們今年初的起點。

為了將其具象化，我們從以前工程師的傳統開發方式開始（敲打字母數字產出程式碼），經歷了前幾年從 ChatGPT 複製貼上以及使用 Copilot 的階段。這些都是好工具，我們現在還在用，但這不是魔法發生的地方。這還不是那種「典範轉移」級別的改變——也就是讓一位人類工程師可以同時引導多個代理對話 (agentic sessions) 並實際產出程式碼變更，對吧？這就是我們的方向。

2025 年初向高層解釋時，我們用了「結對程式設計 (Pair Programming)」來比喻。我個人對這有偏好，我很喜歡結對程式設計。我以前常常從週一到週五、朝九晚五整天做結對開發。我熱愛它，它塑造了身為工程師的我，特別是當你與專家結對時收穫會非常多。但同時，傳統的結對程式設計也有缺點，對吧？有時候在做小修改或修復 bug 時會覺得很慢，如果整天結對也會非常耗神。

在傳統的結對程式設計中，你有一個「駕駛員 (driver)」負責控制鍵盤執行戰術任務，還有一個「導航員 (navigator)」負責戰略思考，也就是見樹又見林。套用這個比喻，代理式程式開發就是讓開發者成為「永遠的導航員」，負責引導代理對話，而 AI 代理或工具則負責實際產出程式碼變更。這就是我們的目標。我想我們今年甚至未來都會繼續使用這個比喻。我不確定我們是否會一直堅持這個比喻，特別是當你開始並行處理並擁有多個代理時。但我們在 2025 年初就是這樣理解這件事的。

我們也向高層提供了一些預測。我想我們當時可能太保守了，不夠有野心。我們原本以為到了 2025 年底，透過代理對話來產出程式碼變更的 PR 比例大約會落在 20% 到 40%，或者是每天使用它的工程師比例。我現在展示這些預測給你看，但它們已經不準確了，因為我們現在的比例已經達到了 64%。它的發展速度快得多，這既驚人又有趣。

不過這些預測還是有用的，特別是當你要向高層或員工解釋願景時。但這些不是 KR (關鍵成果)，不是我們衡量成功的方式。為了衡量成功，你需要採用不同的方法。在 Airbnb 有效的做法是，我們試圖全面地檢視開發者生產力，就像 Vic 在他的主題演講中提到的那樣。我們觀察開發者的情緒回饋 (sentiment)，觀察工具的使用情況和採用率，同時我們也看工程系統中的客觀指標。你可以觀察 PR 的速度以及其他各種指標。

讓大家稍微看一下目前的數據。從開發者情緒回饋來看，他們告訴我們 AI 太棒了，這是有史以來最好的東西，他們想要更多。在過去四次的調查中，它都是票選第一的工程生產力提升工具。我們預測未來六次的調查結果也會是一樣的，大家會繼續說這是有史以來最棒的工具。

我們也觀察這些 AI 工具的採用情況。我特別展示了代理式寫程式的採用曲線，這讓我感到驚訝。畫面上顯示兩條曲線：一條是我們建立的「非代理式」工具（輔助工程師的 IDE 外掛，但還不具備代理能力）。大約六到七個月前，我們才開始更深入地發展代理式寫程式。隨著某些模型進步、市場上或開源界出現了特定的工具，這才變得可行，這也是我們看到採用率呈現幾乎垂直飆升的原因。我在打造工程師工具的將近 20 年裡，從沒見過這樣的成長與如此熱切的採用。

我們不會強迫工程師使用它。這是一個輔助工具，你可以用它來增強你的開發能力，但並非強制。我們做了各種事情來鼓勵大家使用：到處推廣、解釋、教學。我們也試著將它整合得盡可能無縫，讓你只需一鍵就能啟動代理對話。但工程社群採用它的程度確實令人驚豔。

此外，我們也看工程系統的產出，雖然這有很多需要注意的地方。這只是你能觀察的指標之一，我們不把它當作衡量個人生產力的標準，而是看整體數據。我們想了解這些 AI 工具是否對工程系統的特定客觀指標產生了影響，也就是 PR 速度。這裡有幾條曲線，一條是關於 AI 輔助工具，還有一條是大約六到七個月前開始的代理式開發，我們可以看出工程系統某些指標（PR 速度）與工程師採用及運用這些工具之間存在相關性。好了，我們一直在談代理式寫程式，現在來定義一下它吧。

我們經常談論內部開發者循環 (inner developer loop)。在開發者循環的某個時刻，工程師會寫好規格或提示詞 (prompt)，然後啟動「代理循環 (agentic loop)」。之所以稱之為循環，是因為它可以多次呼叫 LLM，可以多次呼叫不同的工具或你的 MCPs，它會利用防護機制 (guardrails)，會載入設定。我說的設定指的是內部配置，像是系統提示詞，也包括像是 agents.md、cloud.md 或 cursor rules 等配置檔。

我想讓大家了解的重點是，當一個工程師開啟了代理對話，你不會確切知道代理工具呼叫 LLM 時使用了哪些提示詞，不知道它到底會呼叫 LLM 多少次，也不知道它會使用哪些工具。它可以執行建置、測試，或存取你們的內部知識庫。它擁有某種程度的「自主性 (autonomy)」。

這就是為什麼它被稱為「代理式 (agentic)」，也是我們將其視為一個「循環」的原因。在這個循環結束時，會產生輸出結果，通常就是需要人類審查的程式碼變更。這很重要：在提交 PR 之前，必須由人類工程師進行審查。

我們也喜歡探討在代理式開發中，學習和獲取技能的模型是什麼。這很有用，因為它能幫我們塑造發展方向。它幫助我們了解，也許有些工程師在這個旅程中走得太快或太慢，讓我們能做出適當的調整，並與 Airbnb 的工程社群合作。

快速帶過一下，我們都經歷過起步階段。我想大家都很熟悉「第二級」：使用代理工具。我記得剛開始使用代理工具時，我會批准它的每一個操作，這很正常。我從中學習，在下提示詞和提供上下文方面變得更厲害。到了某個階段，我升級了，開始對把工具設定為「自動批准 (auto-approve)」感到放心。就讓它一直執行，最後把差異 (diff) 給我就好。

現在，我不會說我信任 AI，我會說我信任自己下提示詞及使用工具的技巧，讓我能夠在不批准每一個操作的情況下獲得價值。但這需要時間，這是一趟旅程。最後在「第四級」，就像 Mike 和我一樣，我們在整合這些工具，在工作場所推動轉型。你們之中可能有些人也在開發基礎設施團隊工作，建立 MCP、整合工具、為開源社群做出貢獻，這非常棒。

我想表達的重點是：我們仍然需要人類。在提交 PR 給另一個人類審查之前，人類必須先審查 AI 的產出結果。你必須給工程師適應的時間，每個人掌握這項技能並達到高熟練度的步調不同。太快將工具切換到「自動批准」並不好。這樣一來，工程師可能會建立非常龐大的 PR，而且自己也沒有完全審查過。他們沒有真正去理解並看過每一行程式碼。他們可能不像審查產品程式碼那樣認真地審查測試程式碼。我一直在研討會上強調：確保你的測試程式碼品質跟產品程式碼一樣高，重構你的測試，保持乾淨和可維護性。

另一件對我來說很重要的事是，對於理解代理工具帶來的變更，你必須懂得對工具「施壓/反駁 (push back)」。你不要盲目信任它。如果覺得哪裡怪怪的，「我覺得它不是這樣運作的，為什麼你要這樣做？」你可以問它：「為什麼你在第 120 行要這樣寫？」你的代理工具會非常有自信地向幾解釋原因。但經過幾個回合的反覆詰問，「不，告訴我為什麼...」你會達到一個非常有成就感的時刻，代理工具認輸說：「噢，對，你是對的。抱歉，我弄錯了，我們來修復它吧。」這非常重要。

到了某個階段，我們就能夠並行處理這些工作負載。我記得我的旅程是這樣的：一開始當我有自信將工具設為自動批准時，我其實沒有去開另一個寫程式的任務，我會去審閱設計文件，或者在旁邊寫一份設計文件。但到了某個階段，我變得更有自信，我會開好幾個工作區 (workspaces)。每個工作區都有一個代理對話，在幫我執行不同的編碼任務。

Airbnb 裡有一些進階使用者，他們會同時開啟大概五個平行的工作區，每個都在建構某種技術原型。這聽起來有點混亂 (pandemonium)，就像主題演講中提到的那樣，對吧？但這就是魔法發生的地方，也是我們倍增生產力的方式。所以我們拭目以待，也許未來的工程師不需要擅長「心流 (flow)」，而是必須非常擅長「切換上下文 (context switching)」，這將是我們需要學習的主要技能。

在 Airbnb 對我們來說非常有用的一件事是，我們開發了協助建立沙盒環境、工作區的技術。我們稱之為「AirDev Workspace」。幾年前我們就在這個研討會上展示過，YouTube 上有影片。因為你想想，如果你要並行執行那些代理對話，你當然可以在你的筆電、一台機器上做，可以使用 Git worktrees 或一些開源技術。但我們不建議這麼做。我認為當你能擁有沙盒環境（像是雲端上獨立的機器），效果會好得多、更一致也更可靠。我們很高興在 Airbnb 能夠運用這個，因為我們有為此開發的技術。

我想讓大家帶走的一個觀念是：光有代理工具是不夠的，你周圍還需要有很多技術配套。你需要有遠端 IDE 的技術、沙盒環境的技術，還需要非常好的程式碼審查工具和相關的文化。因為誰知道呢，也許未來的軟體工程就只剩下「程式碼審查」了，對吧？這就是我們唯一要做的事。我對此完全可以接受。這是我最後一張投影片。現在交給 Mike，他會更深入探討這些，我保證。交給你了，Mike。

**Mike:** 大家好，哇，多棒的開場。在進入我的投影片之前，我只想說我簡直不敢相信這一切能行得通。我不敢相信這一切讓事情變得更簡單。但 AI 讓像我這樣的人——我是個大學輟學生，做了 10 年的 Android 開發，最近才轉到開發平台團隊——能夠跟上像 Szczepan 這樣的人的步伐。就像你說的，他做這個已經 20 年了，以前絕對是把我遠遠甩在後頭的。所以，遊戲開始了，我現在能跟上了。

首先，我想呼應我們 VP 說過的話：我們必須「在開發者所在的地方與他們相遇 (meet developers where they are)」。要學習代理式寫程式和 AI 已經夠難了，如果這時還有人告訴你「嘿，你必須換用這個新工具，用這種方式來做」，那簡直是在扯你後腿。

我認為在 2025 年第一季，就是這個巨大轉變發生的時刻。我們都在學習一種寫程式的新方式。寫程式代理 (Coding agents) 開始出現在 CLI 和 IDE 中，而且這是整個業界的趨勢。無論你是基礎設施工程師、前端、後端，甚至你根本不是工程師，這都深深影響了我們整個產業，並改變了我們每天工作的方式。

現在我想談談 Airbnb 典型的工程師是什麼樣子。劇情反轉：在 Airbnb，我們並沒有所謂「典型的」工程師。我們是一家成立 17 年的公司，技術堆疊非常廣泛。就連我們的開發環境也不一致：有 60% 的工程師喜歡在 Szczepan 提到的那些遠端工作區工作；另外 40% 的人（像我）則喜歡在本地端工作，熱愛本地的開發體驗。

在 IDE 還是 CLI 的選擇上也類似。坦白說，我不知道怎麼在命令列使用 Git，所以我絕對是 IDE 派的。我愛 IntelliJ，我一直都很愛它，它能滿足我的需求。但我們也有其他熱愛 Vim、喜歡在命令列操作的工程師，我們同樣支持他們，讓他們用自己覺得舒適的方式寫程式。

我們使用的語言也包羅萬象，包括 TypeScript、Java、Kotlin、Python、Go、Swift。我在 Airbnb 是在 AI 的幫助下寫出我的第一支 Go 和 TypeScript 類別。正是在這樣的時刻，讓我能夠跟上腳步，並開始轉換到以前可能不那麼有把握的平台。

同樣地，當 AI 開始在 Airbnb 內部普及時，它也是透過多種方式進行的，我會逐一深入說明。

首先，我們有 IDE 外掛。你可以把它想像成我們為 VS Code、IntelliJ 和 Xcode 開發的類似 Cursor 的外掛。但後來，市場上開始湧現各種開源工具。我團隊面臨的一個問題是，當人們回報 bug 時，我們需要非常多的上下文。我們總是得問：你在哪裡跑這個工具的？你用了什麼斜線指令 (slash commands)？你用了什麼設定？我們做事的方式分歧太大了。大家用的都不同，所以我們必須退一步分析，釐清我們未來的策略會是什麼。

我們分析的第一個策略是「IDE 優先 (IDE first)」。就像我說的，我們有這些 IDE 外掛，它們有很棒的 UX，與當前的上下文整合得很緊密。它們能看到你開了哪些分頁、編輯過哪些程式碼、以及你本地的變更。我們還有一個優勢，就是與 Airbnb 知識庫的緊密整合。當時我們領先市場的一點是，我們擁有 RAG (檢索增強生成) 流程。我們將它與我們的 IDE 外掛緊密結合，它掌握 Airbnb 的知識。所以它能搜尋索引、RAG 流程，針對某個詞彙給出來源，並提供 Airbnb 特有的資訊。起初這是一個很大的優勢。

但後來我們看到了另一面。當代理工具開始出現時，像是 Aider、OpenHands、Gemini、CloudCode、Codex 等各種 CLI 工具，它們都具備了可以一次執行多個步驟的「代理編排 (agentic orchestrations)」能力。這不再只是我問一個問題然後得到回覆，不只是聊天。現在它實際上會呼叫工具、呼叫多個工具、做決策，並能自行互動，而不需要我像我們剛才在投影片看到的「第一級」那樣不斷地複製貼上。

好，那我們試試新策略。這次，我們試試「CLI 優先 (CLI first)」。它很強大，能給你最好的答案，擁有最多的工具。它實際上能在任何地方執行，CLI 是一個終端機應用程式，在這些工作區，甚至在沒有 UI 的環境中也比較容易安裝。但很不幸地，它「缺乏對 Airbnb 的認知」。

我們很多工程師很喜歡這些工具，在個人專案或家裡都在用，然後一直跑來問：「嘿，我們為什麼不用 X 工具？為什麼不用 Y 工具？這在家裡用超棒的。」我們分析之後發現，跟我們的內部工具相比，它的表現並沒有預期的好。它某些部分很棒，但它不了解我們的程式碼庫 (repos)，不懂我們的設計模式。它用在個人專案很棒，但對於巨大的單體程式庫 (mono repos) 或是我們多數工程師寫程式的地方來說，其實沒那麼好用。

所以，我們想要那些高品質的答案，同時也想要包含 Airbnb 的內部知識。我們退一步思考，意識到我們必須在這個系統上投入更多心力。既然趨勢已經形成、動力已經產生，那就讓我們實際成立一個團隊吧。

在談論成立團隊之前，我只想說，在過去一年裡，我們已經將代理式開發的覆蓋率擴展到了 60% 的 PR 作者。再說一次，我不敢相信這真的成功了，不敢相信大家接受得這麼快，也不敢相信如果我們把這些工具收走，會有多少像我這樣的人可能會想辭職。

於是，我們成立了 DevAI 團隊。我們建立了一個核心團隊和一個跨部門工作小組，以實現所有服務的對等性。我們希望無論你是在 IDE 還是 CLI，都能獲得同樣出色的體驗。我們也開始建立一個充滿活力的社群。這包含了從舉辦黑客松、演講，到成立一個有 30 人的工作小組；他們回到各自團隊後可以擔任推廣大使。這就像是我們當初開始做行動開發 (mobile) 時一切重新來過一樣，所有的東西都需要從頭建立。

我們思考了一下。我們是個小團隊，有四名工程師以及一些產品支援。我們該專注在什麼地方？什麼才是最具影響力的？我們把賭注押在三件事上。

第一，我們想「將代理 (agents) 引入 Airbnb」。老實說，是我們的主管希望我們這麼做。這源自於一封週五晚上的 Email：「嘿，我在外面用 Cursor，太棒了。為什麼我們內部的東西沒這麼好？我們什麼時候能有代理功能？」所以我們成立了團隊，找到了一個方法，並規劃出了一條路徑，將大家喜歡用的外部代理工具引進公司內部。

第二方面是，我們必須「將 Airbnb 引入代理中」。我們不僅要把代理帶進來，為了讓它們具備對 Airbnb 的認知，我們必須設法將這兩者結合，並且要找到一個不需要自行訓練模型的方法，因為我們團隊很小，你也看到了我們只有七個人。

最後，出現了一個名為 MCP 的新標準。對於不熟悉它的人來說，你可以把它想像成「LLM 的 REST API」。它是一種通訊協定，一種讓你包裝程式碼並開放存取權限的方式，讓 LLM 能夠呼叫那些工具。

從第一點「將代理引入 Airbnb」說起。外面有各種 CLI 工具，像 Aider, OpenHands, CloudCode, Codex, Gemini。這領域發展得非常快。這裡的答案品質是市場頂尖的。它們能在遠端環境中運作，為我和許多同事找到解決方案的速度比我們內部工具還要快。我們需要一個安全、簡單的方法來採用新的 CLI 工具。我們透過建立一個包裝層來實現，這是一個我們稱為「Airchat」的抽象層。Airchat 讓我們能擁有多個引擎，讓我們可以快速地將這些代理部署並派發到工程師的機器上。只要工程師安裝了 Airchat (單行指令安裝)，當有新的模型推出時，我們就能自動幫他們更新。這給了我們一個可以持續投入的單一進入點。

現在我們有了這個管理與部署代理的機制，接著我們轉向了 MCP。我們非常早就「全力投入 (all in)」MCP。意思是我加入這個團隊的第三週，我就說：「我們來採用這個昨天才出的瘋狂標準 MCP 吧。」當時我看到很多困惑的臉，但同時，當我們開始檢視這個標準時，發現它與我們在 Airbnb 已經在做的事情非常契合。我們已經有工具呼叫機制，已經在使用 LLM 的 API 和 SDK。我們意識到這個標準與我們目前的做法非常接近。因為它是一個標準，我們就能發布我們的工具，並與其他生態系結合。

所以我們沿著這條路走：首先，我們喜歡 IDE 工具。我們將最愛的 IDE 工具和功能透過一個 MCP server 暴露出來。這樣一來，即使我們在 CLI 裡工作，也能使用那些相同的工具。另一方面，我們建立了 MCP 用戶端，一種在我們的 IDE 外掛內部與工具介接的方式。最後，我們在 Airchat CLI 內部建立了一個無縫的配置與同步機制，這樣我們現在就能派發這些 MCP servers、配置它們，並確保它們正確運作。

它不斷地成長，像滾雪球一樣。各個團隊開始從四面八方貢獻。有些團隊引進開源的 MCP servers；另一些團隊則開始為分析平台、CI 以及各種內部工具建立 MCP servers。現在我們內部有超過十幾個 MCP servers，可以綁定到我們所有的客戶端。

回到核心團隊。我們現在持續投資那些能加速這個 MCP 生態系的領域。其中之一是驗證 (Auth)。沒人喜歡處理驗證問題，所以我們找到了一種自動化的方式，讓你建立託管的 MCP servers，然後我們會將你的使用者驗證注入進去。我們也確保通過了資安審查。現在只要你用這種方式做，就不需要經過資安團隊漫長的審查。我們還製作了專案範本，讓各團隊能夠以極低的成本進行實驗、建立 MCP servers 進行快速迭代。我們讓安裝變得非常簡單，就像 Homebrew 一樣，一行指令就搞定。

最後，我們正在建置用來建立客戶端的 SDK。人們想為所有東西建立代理應用程式，整合進自己的網頁或桌面應用程式。所以我們現在正在標準化一個 SDK，將過去九個月學到的經驗轉化成框架，讓人們能夠建立支援 MCP 的客戶端。

給大家稍微看一下我們的 Airchat。這是我們的 IDE 外掛，當時它還很陽春，就只是一個聊天應用程式。我們想把代理功能帶進 IDE 裡。大家說：「我想要在 IntelliJ 裡有這種好體驗，你能讓它的回答跟在 CLI 裡一樣好嗎？」當然可以。我們做了研究，發現 Anthropic 的「Effective Agents (高效代理)」文章中提到了很多設計模式。這給了我們一個明確的輪廓：你有一個編排器，允許你註冊許多代理。代理就是一個提示詞、一個工具清單，以及呼叫 LLM 的方式。你可以串聯多個代理。

但這是一個新產品。我們能力不足，而且 AI 發展得太快了。市場已經超前我們五步了。這行不通。我們團隊太小了。所以，既然打不過，就加入他們。我們改變了方向，改為委派給我們的 Airchat CLI，委派給這個可以呼叫其他代理的工具，把它當作一個具有單向資料流且單一職責的系統。我們現在的代理是一個 CLI 代理，但我們在它上面加了一層很薄的轉接層，讓我們能夠從其他應用程式或服務與其介接。

我們的哲學是：不想重複發明輪子。我是開源的支持者。我們也不想讓使用者感到困惑。弄清楚你的需求是什麼。盡量堅持使用某種工具，除非有新東西帶來了巨大的進步。我們希望有能力讓對話在不同工具間接續。所以將客戶端與引擎解耦是很棒的，讓使用者可以在 CLI 中開始對話，然後切換到 IDE 繼續。良好的開發實踐仍然適用。AI 不會替你建立規範。我們想做的是在各處提供相同的核心功能，然後加上附加改進，比如 IntelliJ 內建的 diff 檢視器、互動選單。畢竟不是每個人都喜歡看著 CLI 畫面一直捲動，我們有些人還是喜歡圖形介面的。

現在我想談談我們如何「將 Airbnb 知識帶給代理」。我們有了 CLI 和代理，但那個代理對 Airbnb 一無所知。我之前提到過，我們有一個 RAG 流程和 REST API。但這是單次檢索的、線性的，在代理模式下運作得並不好。我們試著重新想像知識庫應該如何運作。我的想法是：「擁有工具呼叫能力就足夠了 (tool calling is all you need)」。世界已經改變了。如果去年，我可能會說我們將進行微調 (fine-tuning)，訓練我們自己的模型。但在其他公司這非常昂貴且效果不佳。

我的觀點是：有工具呼叫就夠了。這完全取決於你賦予你的代理存取哪些工具。因此，我們重新架構了流程，把它包裝成一個 MCP server 和 MCP 工具。現在，LLM 能夠成為「導航員」了。它能夠自主呼叫知識庫拿取資訊，如果不滿意還能重新搜尋。我們等於「免費」獲得了代理式搜尋和深度研究的能力。因此，即使是像「我該如何設定 S3？」這樣的問題，它會告訴你「在 Airbnb 使用我們特定的工具，你該怎麼做」。這對我們來說意義重大。

最後我想談談我們學到的教訓，這就像是戰場上的故事。我將其分為六大類：
第一，需要眾人之力 (It takes a village)。這件事太龐大了。這發展得太快，需要每個人的參與。要賦權給你身邊的專家。現在這感覺更像是一個我和其他人共同貢獻的生態系，而不僅僅是我們團隊專屬的。
第二，不要把 CLI 看作劣勢工具 (Don't call CLI an underdog)。我連怎麼用命令列操作 Git 都不會，但我用代理工具時就會了。現在的 CLI 代理工具已經變得非常棒了。今天，我敢說我已經好幾個月沒打開 IDE 了。80% 的工程師跟我有一樣的感覺。這帶來了 CLI 代理工具曲棍球棒式的採用成長。
第三，移除障礙 (Remove barriers)。讓點擊一下就能輕鬆開始代理對話。把它放在你的開發者所在的地方。不要強迫 IntelliJ 用戶轉到 VS Code，這行不通。把工具放到他們習慣的地方，讓它越簡單越好。
第四，你必須標準化 (Standardize)。市場變化很快，但不要盲目跟風。在市場稍微穩定下來之前，深思成為早期採用者所投入的心力是否值得。
第五，測量所有的東西 (Measure all the things)。這很困難，有量化和質化的指標。許多過去衡量生產力的方法仍然適用，但我們需要弄清楚這一切帶來的影響是什麼。
最後，不要對 AI 降低標準 (Don't drop your standards for AI)。我曾認為 AI 寫的都是垃圾。但現在我個人已經被代理式開發超越了。如果你做不到，那麼你現在擁有了一個工具，能讓你達到過去覺得遙不可及的高手水準。

最後我想做個總結並向大家道謝。我是 Mike，這位是 Szczepan。我們很高興來到這裡，也很享受在 Airbnb 打造代理式開發的時光。
