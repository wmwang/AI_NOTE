### 1.2 為什麼「裸奔 AI」是企業的噩夢

「裸奔 AI」= 開發者直接打開 AI 工具，用自然語言下指令，零約束、零規範。

**典型災難場景：**

```
開發者：「幫我加一個登入功能」

裸奔 AI 的回應：
→ 自動選了 Session-based 認證（公司標準是 JWT）
→ 把密碼明文存在 localStorage
→ 用了 lodash（公司用 Ramda）
→ 建了新的 users 表（已經有 auth schema）
→ 沒寫測試
→ commit message 寫 "add login"
```

**問題本質**：AI 不知道你們公司的規範，它用「統計上最可能」的方式回答——而這通常不是你們要的。

#### Layer 1：CLAUDE.md — 專案憲法（Memory 系統）

CLAUDE.md 是 AI 的「最高指導原則」，定義了 AI 在這個專案中**能做什麼、不能做什麼、必須怎麼做**。

**關鍵最佳實務**（來自 Boris Cherny 和社群）：

> 📏 **CLAUDE.md 應控制在 200 行以內**。超過後 AI 開始忽略後面的規則。
> 
> 🗂️ **用 `.claude/rules/` 分拆大型規則**，搭配 YAML frontmatter 的 `paths:` 實現按路徑自動載入。
> 
> 🔒 **用 `settings.json` 處理強制行為**（如 attribution、permissions），不要寫在 CLAUDE.md 裡——settings 是確定性的，CLAUDE.md 是建議性的。

```markdown
# CLAUDE.md 範例

## 專案概述
這是一個 Spring Boot 3.2 的 MES 系統後端，使用 Gradle 建置。

## 架構邊界
- Controller 層：只做參數驗證與轉發，不含業務邏輯
- Service 層：所有業務邏輯必須在這一層
- Repository 層：只使用 Spring Data JPA，禁止手寫 SQL

## 技術約束
- 認證：一律使用 JWT（Keycloak）
- 資料庫：PostgreSQL 15，禁止使用 MySQL 特有語法
- 快取：Redis，TTL 不超過 30 分鐘
- 版本：Java 17，禁止使用 var 關鍵字

## 禁止事項
- ❌ 禁止在 Controller 直接注入 Repository
- ❌ 禁止使用 System.out.println，一律用 SLF4J
- ❌ 禁止建立新的資料庫連線池，使用既有的 HikariCP
- ❌ 禁止修改 schema.sql，所有變更走 Flyway migration
```

**Memory 系統的層級體系**：

```
~/.claude/CLAUDE.md               ← 全域：對所有專案生效
~/.claude/rules/*.md              ← 全域規則（自動載入）
  └── project/CLAUDE.md           ← 專案：覆蓋全域設定
      ├── .claude/rules/*.md      ← 專案規則（可搭配 paths: 條件載入）
      └── project/src/CLAUDE.md   ← 子目錄：最細粒度的約束
```

**條件式規則載入**（.claude/rules/ 的進階用法）：

```markdown
---
# .claude/rules/java-rules.md
paths:
  - "src/main/java/**/*.java"
  - "src/test/java/**/*.java"
---

## Java 開發規範
（只有當 AI 操作 Java 檔案時，這些規則才會被載入）
- 使用 SLF4J 而非 Log4j
- 所有 public 方法必須有 Javadoc
```

#### Layer 2：Commands — Slash Command 工作流

Commands 是**可重複的 Slash Command**，把你每天都在做的事變成一鍵觸發：

```
.claude/commands/
├── build-and-test.md      ← /build-and-test 一鍵建置測試
├── code-review.md         ← /code-review AI Code Review
├── deploy-staging.md      ← /deploy-staging 部署到 Staging
└── check-health.md        ← /check-health 系統健康檢查
```

> 💡 **Boris 的建議**：*"Use commands for your workflows instead of sub-agents"*。如果你一天做超過一次的事，就把它變成 Command。

**Command 範例**：

```markdown
# .claude/commands/code-review.md
請對目前的 git diff 進行 Code Review，檢查：
1. 是否符合 CLAUDE.md 定義的架構規範
2. 是否有安全漏洞（SQL Injection、XSS 等）
3. 是否有遺漏的錯誤處理
4. 測試覆蓋率是否足夠
5. 命名是否符合團隊規範

輸出格式：
- ✅ 通過項目
- ⚠️ 建議改善
- ❌ 必須修正（附上原因和建議修改方式）
```

#### Layer 3：Skills — 漸進式揭露的專業能力

Skills 是 Claude Code 最強大的擴展機制。它不只是文字指令，而是**一個完整的知識目錄**：

```
.claude/skills/
└── order-api/
    ├── SKILL.md           ← 技能定義（名稱、觸發條件、指令）
    ├── references/        ← 參考文件（API 規格、架構圖）
    ├── scripts/           ← 可執行腳本
    └── examples/          ← 範例程式碼
```

**Skill 的關鍵最佳實務**（來自 Thariq，Anthropic 工程師）：

> 🎯 **Skill description 是觸發器，不是摘要**——寫給 Model 看（"when should I fire?"）。
> 
> 📦 **Skills 是資料夾，不是檔案**——用 references/、scripts/、examples/ 實現漸進式揭露。
> 
> 🚫 **不要在 Skill 裡說廢話**——專注於把 AI 推出它的預設行為。
> 
> 🚫 **不要用逐步指令綁死 AI**——給目標和約束，讓它自己找路。
> 
> ⚠️ **建立 Gotchas 段落**——記錄 AI 常犯的錯，這是最高價值的內容。

#### Layer 4：Agents（Subagents）— 子代理平行處理

Subagent 讓主 Agent 可以派發任務給專門的子 Agent：

```
主 Agent：「重構這個模組」
  ├→ Sub-agent 1：寫單元測試
  ├→ Sub-agent 2：重構 Service 層
  └→ Sub-agent 3：更新文件
```

> 💡 **Boris 的建議**：*"Have feature-specific sub-agents with skills instead of general qa, backend engineer"*
>
> 💡 **Test Time Compute**：*"Separate context windows make results better; one agent can cause bugs and another (same model) can find them."*

#### Layer 5：Workflows — Command → Agent → Skill 編排模式

這是 Claude Code 的核心設計模式（**The Orchestration Pattern**）：

```
┌─────────────────────────────────────────────────────────────┐
│             Orchestration Pattern                            │
│                                                             │
│  /command（觸發）                                            │
│    │                                                        │
│    ├→ 讀取 SKILL.md（載入專業知識）                           │
│    │                                                        │
│    ├→ 派發 Sub-agent（執行任務）                              │
│    │    └── 在獨立的 Context Window 中工作                    │
│    │    └── 只有最終結論返回主 Context                        │
│    │                                                        │
│    └→ 彙整結果，產出最終交付物                                │
│                                                             │
│  好處：                                                     │
│  1. 主 Context 保持乾淨                                     │
│  2. 子任務的失敗和嘗試不污染主對話                            │
│  3. 可以平行處理                                             │
└─────────────────────────────────────────────────────────────┘
```

**業界主流開發工作流都遵循同一個模式**：Research → Plan → Execute → Review → Ship

#### Layer 6：Hooks — 行為攔截與強化

Hooks 可以在 AI 執行工具的前後插入自訂邏輯：

```
Hooks 觸發時機：
├── PreToolUse   → AI 執行工具前（可阻擋危險操作）
├── PostToolUse  → AI 執行工具後（可自動格式化）
├── Notification → 通知事件
└── Stop         → AI 結束回合時（可要求它繼續或驗證）
```

> 💡 **Boris 的建議**：*"Use a PostToolUse hook to auto-format code — Claude generates well-formatted code, the hook handles the last 10% to avoid CI failures."*

### 1.4 Claude Code 的 Context 管理（關鍵技能）

這是區分「新手」和「高手」的核心能力：

```
Context 生命週期管理：

🟢 0-30%   → 最佳狀態，AI 智商最高
🟡 30-40%  → 開始衰退，注意品質
🟠 40-60%  → "Dumb Zone"，結果明顯變差
🔴 60%+    → 應該 /compact 或 /clear

管理工具：
├── /compact   → 壓縮 Context（有損但保持動能）
├── /clear     → 清空 Context（重新開始，更精確）
├── /rewind    → 回到之前的檢查點（比修正更好）
└── Subagent   → 把探索工作外包，只收回結論
```

> 💡 **Thariq 的建議**：
> - *"Context rot kicks in around 40% — aggressively keep it below 30%."*
> - *"/compact with a hint beats letting autocompact fire — the model is at its least intelligent point when auto-compacting."*
> - *"Rewind > correct — go back to before the failed attempt and re-prompt, instead of leaving failed attempts polluting context."*
> - *"Use subagents for context management — 20 file reads + 12 greps + 3 dead ends stay in the child's context, only the final report returns."*

### 1.5 MCP（Model Context Protocol）— 系統連接層

MCP 讓 AI 能夠「看見」並「操作」外部系統資源：

| 能力 | 說明 | 範例 |
|------|------|------|
| **資源讀取** | AI 可查詢外部系統狀態 | 「查一下 production 的 CPU 使用率」 |
| **工具呼叫** | AI 可執行預定義的操作 | 「幫我重啟 staging 的 Pod」 |
| **Prompt 模板** | 標準化常用指令 | 「用 RCA 模板分析這次故障」 |

### Demo 2：「掛載完整架構規範後的 AI」

```bash
# 建立另一個測試目錄
mkdir demo-structured && cd demo-structured
git init

# 建立 CLAUDE.md
cat > CLAUDE.md << 'EOF'
# 專案規範

## 技術棧
- 語言：Java 17
- 框架：Spring Boot 3.2
- 建置：Gradle（Kotlin DSL）
- 認證：JWT with Keycloak
- 資料庫：PostgreSQL + Spring Data JPA
- 測試：JUnit 5 + Mockito，測試覆蓋率 ≥ 80%

## 架構規則
- 嚴格分層：Controller → Service → Repository
- Controller 禁止直接存取 Repository
- 所有 API 必須有對應的 Integration Test
- 所有錯誤回傳統一使用 GlobalExceptionHandler
- commit message 格式：`feat(scope): description`

## 禁止事項
- ❌ 禁止使用 Lombok
- ❌ 禁止在 Service 層返回 Entity，必須用 DTO
- ❌ 禁止硬編碼設定值，一律使用 @Value 或 Configuration Properties
EOF

# 建立條件式規則
mkdir -p .claude/rules
cat > .claude/rules/test-rules.md << 'EOF'
---
paths:
  - "src/test/**/*.java"
---
## 測試規範
- 測試命名：should_ExpectedBehavior_when_Condition
- 每個業務規則至少一個測試
- 使用 @DisplayName 加上中文說明
- Mock 只 Mock 外部依賴，內部物件用真實實例
EOF

claude
> 請幫我建立一個使用者登入 API

# 觀察：AI 會嚴格遵守 CLAUDE.md
# ✅ 用 Spring Boot 3.2 + Gradle
# ✅ Controller → Service → Repository 三層分離
# ✅ JWT 認證整合
# ✅ DTO 轉換，不返回 Entity
# ✅ Integration Test + GlobalExceptionHandler
# ✅ commit message: "feat(auth): add user login API"
# ✅ 測試檔案會自動套用 test-rules.md 的規範
```

## 三、Quick Win：課堂 10 分鐘動手（10 mins）

### 任務：讓 AI 嚴格遵守一條特定規則

```bash
# Step 1：建立測試 Repo（1 min）
mkdir quickwin-test && cd quickwin-test
git init

# Step 2：建立 CLAUDE.md（2 min）
cat > CLAUDE.md << 'EOF'
# 測試規則

## 絕對禁止事項
- ❌ 你「絕對不可以」在回應中提到 Python 這個詞
- ❌ 你不可以建議使用 Python 相關的任何工具或框架
- ✅ 所有建議必須基於 Java 生態系

## 回應格式
- 每次回應必須以 "[Java Guardian]" 開頭
- 回應結尾必須加上 "— Java Guardian v1.0"
EOF

# Step 3：測試 AI 是否遵守（5 min）
claude
> 我想做一個網頁爬蟲，你推薦什麼工具？

# 預期結果：
# ✅ AI 不會提到 Python / BeautifulSoup / Scrapy
# ✅ AI 會建議 Jsoup、HttpClient 等 Java 工具
# ✅ 回應以 "[Java Guardian]" 開頭
# ✅ 回應以 "— Java Guardian v1.0" 結尾

> 請比較三種程式語言的優缺點

# 預期結果：
# ✅ AI 會刻意避開 Python，只比較 Java/Kotlin/C# 等
# （如果 AI 失敗了，觀察它是在哪裡破功的）
```

## 五、Claude Code 精髓速查表

### 必學指令

| 指令 | 用途 | 場景 |
|------|------|------|
| `/compact [hint]` | 壓縮 Context | Context > 30% 時 |
| `/clear` | 清空 Context | 換任務時 |
| `/rewind` | 回到之前狀態 | AI 走偏時（比修正更好） |
| `/context` | 查看 Context 用量 | 隨時監控 |
| `/model` | 切換模型 | Plan 用 Opus，Code 用 Sonnet |
| `/doctor` | 診斷問題 | 安裝/認證出問題時 |
| `/config` | 設定 | 開啟 thinking mode、output style |

### 核心心法（來自 Boris Cherny & Thariq）

```
1. 🚫👶 Don't Babysit
   → 給 AI 目標和約束，不要逐步指導

2. 📏 CLAUDE.md < 200 行
   → 超過就拆分到 .claude/rules/

3. 🔄 新任務 = 新 Session
   → 不要在同一個 Session 裡跳任務

4. ⏪ Rewind > Correct
   → 走偏時回溯，不要在錯誤上疊加修正

5. 🤖 Subagent 是 Context 管理利器
   → 探索工作外包，只收回結論

6. 🔧 一天做兩次的事 → 寫成 Command
   → 一週做兩次的事 → 寫成 Skill

7. 📋 Plan Mode 先行
   → 永遠先用 Plan Mode，想清楚再動手
```
