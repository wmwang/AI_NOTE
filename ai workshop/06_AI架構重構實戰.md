## 一、概念講授（30 mins）

### 1.1 為什麼「規格即法律」

```
傳統開發流程：
  需求（模糊） → 直接寫程式碼 → 測試（隨便寫） → 上線
  問題：AI 自由發揮，結果偏離需求

Agentic TDD 流程：
  規格（精確 YAML） → 先寫測試（紅燈） → 實作（綠燈） → 驗證 → 上線
  好處：AI 的產出被規格和測試雙重約束
```

**TDD 的三個階段（紅綠重構）**：

```
🔴 紅燈：寫測試 → 測試失敗（因為還沒有實作）
🟢 綠燈：寫實作 → 測試通過（滿足規格要求）
♻️  重構：改善程式碼品質 → 測試仍然通過（行為不變）
```

### 1.2 Agentic TDD：讓 AI 執行 TDD 迴圈

### 1.3 CLAUDE.md 中的 TDD 約束

在 Spring Boot 專案中強制 TDD 的 CLAUDE.md 設定：

```markdown
# CLAUDE.md

## 開發紀律（不可違反）

### TDD 強制規範
1. **先寫測試，再寫實作**。沒有測試的程式碼不接受。
2. **測試必須先失敗**。如果測試一開始就通過，代表測試寫錯了。
3. **每次只寫一個測試**。通過後再寫下一個。
4. **紅燈 → 綠燈 → 重構**。嚴格遵守這個順序。

### Clean Architecture 規範
- **domain/**：純業務邏輯，不依賴任何框架
- **application/**：Use Case，定義業務流程
- **infrastructure/**：外部整合（DB、MQ、外部 API）
- **interfaces/**：Controller、DTO、Request/Response

### 測試規範
- 單元測試：JUnit 5 + Mockito
- 整合測試：@SpringBootTest + TestContainers
- 測試命名：`should_ExpectedBehavior_when_Condition`
- 每個業務規則至少一個測試案例
- 使用 `mvn test` 執行，不接受跳過測試

### 禁止事項
- ❌ 不可以在沒有測試的情況下寫實作
- ❌ 不可以跳過重構階段
- ❌ 不可以在 domain 層引用 Spring Framework
- ❌ 不可以提交有 @Disabled 的測試
```

### 1.4 從 YAML 規格到 JUnit 測試

第五堂產出的 YAML 規格，每條 business_rule 直接對應一個測試：

```yaml
# 第五堂的規格（簡化版）
business_rules:
  - id: BR-ORD-001
    description: "VIP Level ≥ 3 → 10% 折扣"
  - id: BR-ORD-002
    description: "VIP Level = 2 → 5% 折扣"
  - id: BR-ORD-003
    description: "單筆金額上限 10,000,000"
  - id: BR-ORD-004
    description: "庫存不足時自動建立補貨單"
```

對應的 JUnit 測試：

```java
// AI 產出的測試（紅燈階段）
class OrderServiceTest {

    @Test
    void should_apply10PercentDiscount_when_vipLevelIs3OrAbove() {
        // BR-ORD-001: VIP Level ≥ 3 → 10% 折扣
        Customer customer = new Customer("C001", 3); // VIP Level 3
        List<OrderItem> items = List.of(new OrderItem("P001", 1000, 1));
        
        OrderService service = new OrderService(); // 還沒實作！
        OrderResult result = service.createOrder(customer, items);
        
        assertThat(result.getTotalAmount()).isEqualTo(900.0);
    }

    @Test
    void should_apply5PercentDiscount_when_vipLevelIs2() {
        // BR-ORD-002: VIP Level = 2 → 5% 折扣
        Customer customer = new Customer("C002", 2);
        List<OrderItem> items = List.of(new OrderItem("P001", 1000, 1));
        
        OrderService service = new OrderService(); // 還沒實作！
        OrderResult result = service.createOrder(customer, items);
        
        assertThat(result.getTotalAmount()).isEqualTo(950.0);
    }

    @Test
    void should_rejectOrder_when_amountExceeds10Million() {
        // BR-ORD-003: 單筆金額上限 10,000,000
        Customer customer = new Customer("C003", 1);
        List<OrderItem> items = List.of(new OrderItem("P001", 10000001, 1));
        
        OrderService service = new OrderService(); // 還沒實作！
        
        assertThrows(AmountExceedsLimitException.class, 
            () -> service.createOrder(customer, items));
    }

    @Test
    void should_createRestockOrder_when_insufficientStock() {
        // BR-ORD-004: 庫存不足時自動建立補貨單
        Customer customer = new Customer("C004", 1);
        List<OrderItem> items = List.of(new OrderItem("P001", 100, 10));
        // 庫存只有 5 個
        
        OrderService service = new OrderService(); // 還沒實作！
        OrderResult result = service.createOrder(customer, items);
        
        verify(restockService).createRestockOrder("P001", 5);
    }
}
```

## 二、Demo 實演（20 mins）

### Demo 1：餵入 YAML 規格，觸發 Agentic TDD

```bash
# 準備 Spring Boot 專案
spring init --dependencies=web,jpa,validation \
  --java-version=17 \
  --build=maven \
  order-service

cd order-service

# 確保 CLAUDE.md 包含 TDD 約束
cat > CLAUDE.md << 'EOF'
# 開發紀律

## TDD 強制規範
1. 先寫測試，再寫實作
2. 測試必須先失敗
3. 紅燈 → 綠燈 → 重構

## Clean Architecture
- domain/ → 純業務邏輯
- application/ → Use Case
- infrastructure/ → 外部整合
- interfaces/ → Controller + DTO

## 測試
- JUnit 5 + Mockito
- mvn test 執行
EOF

# 將第五堂的 YAML 規格放入專案
cp ~/workshop/session5/order-spec.yaml specs/

# 開始 Agentic TDD
claude
> 請根據 specs/order-spec.yaml 的規格，使用 TDD 方式開發。
> 
> 步驟：
> 1. 先分析 YAML 規格，列出所有 business rules
> 2. 為每個 business rule 寫一個 JUnit 測試（應該會失敗）
> 3. 執行 mvn test 確認全部紅燈
> 4. 逐個實作，直到測試通過
> 5. 執行重構，確保測試仍然通過
```

### Demo 2：觀察 AI 的 TDD 過程

```
AI 的執行過程：

[Step 1] 分析規格
  → 識別 4 個 business rules + 2 個 endpoints

[Step 2] 🔴 寫測試
  → 建立 OrderServiceTest.java（4 個測試方法）
  → 執行 mvn test → 編譯失敗（OrderService 不存在）

[Step 3] 建立 Domain Model
  → 建立 OrderService.java（空殼）
  → 建立 Customer.java、OrderItem.java、OrderResult.java
  → 執行 mvn test → 測試失敗（方法沒有實作）

[Step 4] 🟢 逐個實作
  → 實作 BR-ORD-001（VIP 折扣）
  → mvn test → 1/4 通過 ✅
  
  → 實作 BR-ORD-002（VIP Level 2 折扣）
  → mvn test → 2/4 通過 ✅
  
  → 實作 BR-ORD-003（金額上限）
  → mvn test → 3/4 通過 ✅
  
  → 實作 BR-ORD-004（自動補貨）
  → mvn test → 4/4 通過 ✅ ✅ ✅ ✅ 🎉

[Step 5] ♻️ 重構
  → 抽取 DiscountStrategy 介面
  → 抽取 InventoryChecker 服務
  → mvn test → 仍然 4/4 通過 ✅
```

## 三、Quick Win：課堂 10 分鐘動手（10 mins）

### 任務：用 Prompt 強制 AI 先寫出會 fail 的單元測試

```bash
# Step 1：建立簡單的測試目標（2 min）
mkdir -p tdd-demo/src/test/java/demo && cd tdd-demo

cat > spec.yaml << 'EOF'
function: isEligibleForDiscount
description: 判斷客戶是否符合折扣資格
rules:
  - 會員年資 > 1 年 → 符合資格
  - 消費金額 > 5000 → 符合資格  
  - 其他 → 不符合資格
EOF

# Step 2：強制 AI 先寫測試（5 min）
claude
> 根據 spec.yaml 的規格，用 JUnit 5 寫單元測試。
> 
> 嚴格限制：
> 1. 你現在只能寫測試，不能寫任何實作程式碼
> 2. 測試類別中不能引用任何還不存在的類別（可以用 Object 暫代）
> 3. 測試必須能夠編譯通過
> 4. 測試執行時必須失敗（因為沒有實作）

# Step 3：觀察結果（3 min）
# 確認：
# ✅ AI 產出了 3 個測試方法（對應 3 條規則）
# ✅ 測試可以編譯
# ✅ 測試執行時失敗（紅燈）
# ✅ AI 沒有產出任何實作程式碼
```

### 檢查點

```bash
# 編譯測試
javac -cp junit-jupiter-api test.java

# 執行測試 → 應該全部失敗
# 紅燈 = 測試正常運作
```

**目標**：拿第五堂課產出的 YAML 規格，用 Agentic TDD 完整實作 Spring Boot 服務，截圖綠燈。

   ```bash

   ```

2. **設定 CLAUDE.md**：加入 TDD 約束和 Clean Architecture 規範

3. **放入第五堂的 YAML 規格**

4. **啟動 Agentic TDD**：
   ```
   claude

   > 嚴格遵守：紅燈 → 綠燈 → 重構。
   > 每個階段執行 mvn test 並回報結果。
   ```

5. **截圖綠燈**：
   - `mvn test` 全部通過的截圖

3. `mvn test` 綠燈截圖

## 五、關鍵 Takeaway

> 💡 **Agentic TDD 的核心紀律**：
> 1. **規格先行** — 沒有 YAML 規格就不開始寫測試
> 2. **紅燈先行** — 沒有失敗的測試就不開始寫實作
> 3. **測試即文件** — 每個測試都是業務規格的可執行版本
> 4. **重構不變** — 重構前後測試都必須通過
